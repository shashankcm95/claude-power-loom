## Summary

**Concept**: JMH (Java Microbenchmark Harness) — the canonical, JIT-aware way to measure JVM code, defeating the optimizations (dead-code elimination, constant folding, warm-up) that make naive timing meaningless.
**Key APIs**: `@Benchmark`, `@BenchmarkMode` (`Throughput`/`AverageTime`/`All`), `@OutputTimeUnit`, `@Warmup`, `@Fork` (isolated JVMs + `jvmArgs`), `@State`/`@Param`/`@Setup`, `@Group`, `Blackhole.consume(...)`; benchmark uber-jar via `maven-shade-plugin` + `jmh-generator-annprocess`.
**Gotcha**: skip `Blackhole`/warm-up/forks and JMH lies — DCE deletes "unused" results and constant folding inlines literals; read `@State`-held input (not a local). `@sun.misc.Contended` is inert without `-XX:-RestrictContended`.
**2026-currency**: JMH is the most durable module in the domain (still the canonical tool); virtual threads (Java 21 JEP 444) and AOT cache change WHAT you benchmark, not how.
**Sources**: Baeldung `jmh`/`performance-tests`; JEP 444 / Java 25 performance coverage.

## Quick Reference

**Canonical benchmark**:

```java
@Benchmark
@BenchmarkMode(Mode.Throughput)
@OutputTimeUnit(TimeUnit.MILLISECONDS)
@Warmup(iterations = 5)
@Fork(value = 1, warmups = 5)
@State(Scope.Benchmark)
public void measure(Blackhole bh, @Param("...") String input) {
    bh.consume(work(input));   // else DCE makes the number meaningless
}
```

- Consume results with `Blackhole.consume(...)` (or return them), else dead-code elimination deletes the work.
- Read `@State`-held input (not a local constant) to defeat constant folding.
- `@Fork` runs isolated JVMs; `@Warmup` lets the JIT compile before measuring.

**Build the runnable benchmark jar**: `maven-shade-plugin` with `ManifestResourceTransformer` → `org.openjdk.jmh.Main`, excluding `META-INF/*.SF/.DSA/.RSA` (shading signed jars breaks without this). Benchmarks generated by `jmh-generator-annprocess` (`provided`). Run `java -jar target/benchmarks.jar`.

**Taught lessons** (the most reusable, least-stale content in the domain):
- Avoiding DCE and constant folding.
- False sharing + `@Contended` (inert without `-XX:-RestrictContended`; internal/unsupported API).
- `boolean[]` vs `BitSet` (with JOL footprint).
- Exception cost decomposition (the stack-trace fill is the expensive part).
- JIT warm-up (Java is both compiled and interpreted).
- Benchmarking mapping frameworks (MapStruct/JMapper/Orika/Dozer/ModelMapper).

**Top gotchas**:
- JMH lies if you skip `Blackhole`/warm-up/forks; hand-rolled timing loops (the `Fibonacci.main` demo) are illustrative only.
- `@sun.misc.Contended` is inert without `-XX:-RestrictContended`.
- Shading signed jars breaks without excluding `META-INF/*.SF/.DSA/.RSA`.

**Current (mid-2026)**: JMH is current and canonical. What changed is WHAT to benchmark — virtual threads (Loom) and AOT/CDS startup are the new performance surfaces.

## Full content

JMH is the domain's most durable cluster — two modules (`jmh`, `performance-tests`) with many distinct lessons, and it remains the canonical JVM microbenchmarking tool. The whole point is that the JVM is a moving target: the JIT eliminates dead code, folds constants, and only reaches steady state after warm-up, so naive `System.nanoTime()` loops produce numbers that say nothing.

### The harness mechanism

A benchmark is a `@Benchmark` method tuned by annotations: `@BenchmarkMode` (throughput vs average time), `@OutputTimeUnit`, `@Warmup`, `@Fork` (isolated JVMs, optionally with `jvmArgs`), and `@State`/`@Param`/`@Setup` for inputs. The two load-bearing defenses are `Blackhole.consume(...)` (so the result isn't dead code) and reading `@State`-held input rather than a local (so it isn't constant-folded). The runnable benchmark is an uber-jar built with shade and the `jmh-generator-annprocess` processor.

### The teaching lessons

The modules teach a catalogue of JVM-performance truths: DCE and constant folding, false sharing and `@Contended`, the memory footprint of `boolean[]` vs `BitSet` (measured with JOL), the real cost of exceptions (the stack-trace fill, not the throw), JIT warm-up, and a mapping-framework shootout that names MapStruct the fastest. These lessons are model-level and don't expire.

### 2026 currency

- **JMH remains current and canonical** — the most durable module in the domain; the harness API and the DCE/constant-folding/warm-up lessons carry forward unchanged. [JEP 444 coverage — Java 25 and the new age of performance](https://javapro.io/2026/03/05/java-25-and-the-new-age-of-performance-virtual-threads-and-beyond/)
- **What to benchmark moved**: **virtual threads (Project Loom)** were finalized in **Java 21 via JEP 444** — lightweight JVM-scheduled threads make thread-per-request scalable again (pitfall: `synchronized` pinning in JDK 21, improved later). This is a new performance surface worth measuring.
