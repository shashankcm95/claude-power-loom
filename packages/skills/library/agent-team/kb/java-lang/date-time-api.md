## Summary

**Concept**: The immutable `java.time` (JSR-310) API — `LocalDate`/`LocalDateTime`/`ZonedDateTime`/`Instant`, `Period` vs `Duration`, `TemporalAdjuster`, formatting, and the legacy↔modern bridge.
**Key APIs**: `LocalDate.of`/`parse`/`now`, `plusDays`/`minus`/`with(...)`, `Period`/`Duration`, `ChronoUnit.DAYS.between`, `TemporalAdjusters.firstDayOfMonth`/`next(...)`, `DateTimeFormatter.ofPattern`, `Date.toInstant`/`Date.from`.

## Quick Reference

**Core types**:

| Type | Holds | Zone |
|---|---|---|
| `LocalDate` / `LocalTime` / `LocalDateTime` | date / time / both | none |
| `ZonedDateTime` | date-time | full `ZoneId` (DST rules) |
| `OffsetDateTime` | date-time | fixed `ZoneOffset` |
| `Instant` | machine timestamp | UTC |

**Creation**: `of(...)`, `parse(...)` (ISO-8601), `now()`.

**Immutable manipulation**: `plusDays`/`minus`/`with(...)` all return **new** instances.

**`Period` vs `Duration`**:
- `Period` — date-based (`P2Y3M5D`); `between` is a normalized y/m/d breakdown, NOT total days; no `getWeeks()` — for total days use `ChronoUnit.DAYS.between`
- `Duration` — time-based (`PT10M`)

**Adjusters**: `TemporalAdjusters.firstDayOfMonth()` / `next(DayOfWeek.SUNDAY)`; custom adjuster via `adjustInto`.

**Comparison**: `isBefore`/`isAfter`/`isEqual`; `truncatedTo(ChronoUnit.DAYS)` for same-day equality.

**Formatting**: `DateTimeFormatter.ISO_DATE` / `ofPattern(...)` / `ofLocalizedDateTime(...)`.

**Legacy ↔ modern bridge**: `Date.toInstant`/`Date.from`, `Calendar.toInstant`, `GregorianCalendar.toZonedDateTime`/`from`, `TimeZone.toZoneId`.

**Date ranges**: `LocalDate.datesUntil()` (Java 9).

**Top gotchas**:
- `Period.between` ≠ total days — use `ChronoUnit.DAYS.between`.
- Legacy `SimpleDateFormat` is NOT thread-safe (`DateTimeFormatter` is).
- All `java.time` types are immutable — discarding the return of `plus`/`with` is a no-op.

**Current (mid-2026)**: `java.time` (JSR-310) is stable and current; no successor. Legacy `Date`/`Calendar` kept only for interop.

## Full content

`java.time` (JSR-310, Java 8) replaced the error-prone `java.util.Date`/`Calendar`/`SimpleDateFormat` trio with an immutable, fluent, and clearly-modeled API. The core types separate concerns by whether they carry a zone: `LocalDate`/`LocalTime`/`LocalDateTime` hold a date and/or time with **no** zone; `ZonedDateTime` adds a full `ZoneId` (and thus DST rules); `OffsetDateTime` adds a fixed `ZoneOffset`; and `Instant` is a UTC machine timestamp. Every type is created with `of(...)`, `parse(...)` (ISO-8601 by default), or `now()`, and is **immutable** — `plusDays`, `minus`, and `with(...)` all return new instances, so discarding the result is a silent no-op.

Two amount types model spans: `Period` is date-based (`P2Y3M5D`) and `Duration` is time-based (`PT10M`). The recurring trap is that `Period.between` produces a *normalized* years/months/days breakdown, **not** a total day count, and there is no `getWeeks()` — a "how many days between" question must use `ChronoUnit.DAYS.between`. The `TemporalAdjuster` abstraction encapsulates date arithmetic: built-ins like `TemporalAdjusters.firstDayOfMonth()` and `next(DayOfWeek.SUNDAY)`, and custom adjusters implementing `adjustInto`. Comparison uses `isBefore`/`isAfter`/`isEqual`, with `truncatedTo(ChronoUnit)` to compare at a coarser granularity (same day, same hour). Formatting uses the thread-safe `DateTimeFormatter` (`ISO_DATE`, `ofPattern(...)`, `ofLocalizedDateTime(...)`) — in contrast to the legacy `SimpleDateFormat`, which is **not** thread-safe.

Because legacy code persists, the API provides a **conversion bridge**: `Date.toInstant`/`Date.from`, `Calendar.toInstant`, `GregorianCalendar.toZonedDateTime`/`from`, and `TimeZone.toZoneId`. Random temporal values are generated by round-tripping through the numeric domain (epoch-day, second-of-day, epoch-second + `ThreadLocalRandom`, end-exclusive), and `LocalDate.datesUntil()` (Java 9) produces a stream of dates over a range.

### 2026 currency

`java.time` (JSR-310) is **stable, evergreen, and current** across all modern JDKs (17/21/25) with no successor on the horizon ([Java version history](https://en.wikipedia.org/wiki/Java_version_history)) — it is the post-`Date`/`Calendar` API and the base's teaching carries forward verbatim. The legacy `java.util.Date`/`Calendar`/`SimpleDateFormat`/`DateFormat` classes remain in the JDK solely for interop with old code; new code should use `java.time` exclusively, and the conversion bridge above is the migration path. The immutability and `Period`-vs-`Duration` distinctions, the `ChronoUnit.DAYS.between` rule, and the thread-safety contrast between `DateTimeFormatter` and `SimpleDateFormat` are unchanged.
