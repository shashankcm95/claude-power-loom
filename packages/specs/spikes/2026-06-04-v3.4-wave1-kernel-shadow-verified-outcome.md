# v3.4 Wave 1 — kernel-shadow + "verified outcome" resolution (A6/E4 grounding)

**Date:** 2026-06-04 · **Gate:** opens v3.4 Wave 1 (A6 snapshot mechanism + E4 reputation). **Method:** firsthand recon → architect design pass (opus, every claim re-probed). **Verdict: COHERENT-WITH-RESHAPINGS — A6/E4 IS buildable in v3.4 with the kernel in shadow, via Resolution C.**

## The question
Re-shaped v3.4 leads with A6 → E4. A6 snapshots a derived view (E4 reputation) that v6 §0a.3.1 requires to ground on **kernel-emitted records of VERIFIED OUTCOME**. So: does the kernel ever leave shadow, and what counts as a "verified outcome" for A6 to snapshot?

## Firsthand facts
- The kernel resolver runs in **SHADOW**; the **enforce-path EXISTS but is flag-gated OFF** (`stage-promote.js` behind `LOOM_RESOLVER_ENFORCE==='1'`; three arms at `spawn-close-resolver.js:719/732/749` — enforce / `LOOM_STAGE_CANDIDATES` / shadow). Leaving shadow is an *activation* decision, but high-risk (never run in anger; worktree ≠ sandbox — absolute-path writes escape).
- Shadow `commit_outcome=COMMITTED` = **"the agent finished," NOT a verified kernel commit** (`spawn-close-resolver.js:228-230,:539`). The real A9 two-phase commit (intent→COMMITTED after K14 validation+fsync) is enforce-path only.
- Reviewer-verdicts are a **manual CLI** (`verdict-recording.js:90` `data.verdicts[v] += 1`), **persona-keyed aggregate counts**, self-reported. **Decisive probe: grep for `spawn_id|transaction_id|evidence_ref|writer_spawn` across `identity/` → ZERO matches.** No spawn-record linkage anywhere in the verdict store.
- v6 §0a.3.1 (`:175,:187`) forbids trust-by-frequency; `:504` splits a finding's **emission** (Attestation, grounded) from its **content** (Stochastic Sample). A3b (`:383`): advisory LLM-verdict verification is *permitted, provided it cannot block*.

## The resolution (Resolution C — confirmed + 2 refinements)

**The kernel STAYS in shadow for v3.4.** Flipping to enforce (Resolution B) is a separate major high-risk phase — and the `LOOM_STAGE_CANDIDATES` middle tier does **NOT** shortcut it (it emits *structural-genesis provenance* with `commit_outcome:COMMITTED` hardcoded = "structural gate passed", no verdict/verified field; even the enforce tier no-ops K14 and defers to human review — `stage-promote.js:28-29,:294`). **Reject B as a v3.4 sub-wave.** Reject Resolution A (E4 over the existing persona-keyed counters) as **§0a.3.1-ILLEGAL** (trust-by-frequency + self-reported, two independent failures).

### v3.4's v6-legal definition of "verified outcome"
> **The EMISSION ATTESTATION of an advisory verdict that is evidence-linked (via `evidence_refs`) to a kernel-emitted spawn-record (K2 `transaction_id` / `writer_spawn_id`) of the judged work.**
>
> NOT the kernel's `COMMITTED` ("agent finished"). NOT the verdict *content* (stochastic, advisory — A3b). NOT a frequency count over self-reports (§0a.3.1-illegal).

**The legality hinges (PT#1 REFINE):** counting emission attestations *alone* IS trust-by-frequency (forbidden). C is legal ONLY because each verdict-emission is **itself evidence-linked to a kernel spawn-record** → E4 becomes "P produced N kernel-attested work-products, each independently judged" (passes the `:187` trace test), NOT "P was judged N times." The evidence-link is load-bearing for legality.

**Refinement 1 (the synthesis under-stated):** the Wave-1 build is a **NET-NEW evidence-linked verdict-emission record**, NOT a relabel of the existing counters (which have zero spawn-link). Real Wave-1, not a ~1-day rename.

**Refinement 2 (strategic decoupling — the most useful insight):** E4's grounding is **ORTHOGONAL to whether the kernel ever enforces.** Even a fully-enforcing kernel feeds E4 *advisory* verdict-emissions (the verdict is an LLM judgment regardless — A3b). The "does Power Loom ever enforce / Execute-pillar" positioning is an explicit **wager** (`v6:39`), not settled — and it does NOT block A6/E4. Resolve A6/E4 with C now; let B/positioning resolve on its own timeline.

## GATING PROBE — RESOLVED ✓ (empirical end-to-end, 2026-06-04)

**The verdict↔spawn-record correlation IS expressible under Pattern A.** Confirmed firsthand, not inferred:
- **Mechanism (code):** the resolver sets the kernel record's `writer_spawn_id`/`spawn_id` = `tool_response.agentId` (`spawn-close-resolver.js:232,469`). The orchestrator receives that same `tool_response.agentId` in the Agent tool result (firsthand — every spawn carries it).
- **End-to-end (empirical):** spawned a delta-bearing probe agent → orchestrator observed `agentId: a104143b476ed011f` → a kernel record materialized at `~/.claude/spawn-state/cde7d2eb.../resolver-journal-a104143b476ed011f.jsonl` with `spawn_id: a104143b476ed011f`. **Byte-identical on both sides.**
- **The evidence-link is the `agentId`** — the harness correlation id present at the natural boundary (Agent-tool-result for the orchestrator; `tool_response.agentId` for the kernel hook). The orchestrator forms the link (it holds the agentId + knows which spawn a verdict judges, since it ran the review). **No structural exception layer needed.**
- **Scope boundary (honest, code `:250-251`):** ONLY delta-bearing spawns get a kernel record (the resolver no-ops on no-`worktreePath`). So E4 grounds reputation on **work-producing spawns** — the correct boundary (reputation = "does this persona's WORK pass review?", NOT "are its read-only advisory findings good?" — the latter is stochastic content E4 can't ground on regardless).
- **K13 concurrency — RESOLVED (empirical, 2026-06-04):** 3 spawns launched concurrently (one message) each received a **distinct** agentId (`a9c8a180…`, `a881733c…`, `a7457de6…`); the kernel wrote **3 distinct journals + 3 distinct `transaction_id` content-addressed records**, all `record_appended:true deduped:false` — no collision, no corruption under concurrent close. **Within-run agentId uniqueness under concurrency holds** (it must — the harness routes results by agentId among concurrently-live spawns; that is the harness's own correctness requirement, not just ours).
- **runId is NOT orchestrator-derivable — the load-bearing correction (empirical, 2026-06-04):** `runId = sha256(session_id).slice(0,16)` (`spawn-close-resolver.js:153-176`; envelope `run_id_source:"session_id_sha256"`). The 3 concurrent probes landed under runId `d37d788b…` (session `2879b7c3…`) — **NOT** my session's `cde7d2eb…` (`9539c742…`) where the solo probe landed 13 min earlier. **The session_id ROTATED across compaction**, and the orchestrator was **not** aware of its new id. ∴ the orchestrator **cannot self-compute the runId** at evidence-link time. The pre-compaction note that "the full address is `(runId, agentId)`, trivially known" was **wrong-on-mechanism** — runId is a moving, kernel-side value.

**→ Wave 1 is UNBLOCKED, with a corrected evidence-ref design:**
- **The durable, orchestrator-reliable key is `agentId` alone** — always present in the Agent tool result; 17-hex (~2⁶⁸) ⇒ globally collision-resistant in practice; byte-equal to the kernel record's `spawn_id`/`writer_spawn_id`.
- **runId is a kernel-internal namespacing detail, NOT part of the orchestrator-formed ref.** The Lab resolves `agentId` → kernel record by **glob** `~/.claude/spawn-state/*/resolver-journal-<agentId>.jsonl` (resolves to exactly one file; a 2-hit is a *detectable* collision, disambiguated by `transaction_id`/timestamp — a fail-loud edge, not silent corruption).
- **Two-part evidence_ref (recommended shape):** `agentId` (orchestrator-formed join key, written at verdict time) + `kernel_record_ref:{runId, transaction_id}` (Lab-enricher-resolved via the glob; `null` until enriched). The enricher is a **pull** pass (E1-style: the Lab reads kernel on-disk output as DATA — no runtime/kernel→lab import; K12-clean). It upgrades the practical-unique agentId into the precise content-addressed link the §0a.3.1 "evidence-linked, not frequency-only" clause demands.
- **New on-disk data contract to FREEZE:** the Lab now depends on the kernel's `~/.claude/spawn-state/<runId>/resolver-journal-<agentId>.jsonl` + `records/record-<transaction_id>.json` layout (analogous to E1's decompose-outbox contract). Document + version it so a kernel layout change cannot silently break the link.
- The fallback ladder (degrade-to-run-level / defer) is moot — the primary path works.
