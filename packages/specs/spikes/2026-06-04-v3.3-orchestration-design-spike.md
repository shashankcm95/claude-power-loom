## The question

How/when does a persona decompose its task and invoke `decompose-run` **live**, and how do the resulting
`failure_signature`s reach E1? = "wire the v3.2 tier to a live consumer" — the un-darkening (the tier ships
LIVE-but-INERT: 0 refs in `packages/kernel/hooks.json`). This is **item 1** of the USER's "v3.3 real work."

## The load-bearing harness fact (empirically probed — `p-depthone-findings.md`)

A plugin-spawned persona has **NO Agent / Task / TaskCreate tool** ("no sub-agent spawn primitive") but **DOES
have Bash + Read/Write/Edit**. Therefore:
- A persona **cannot sub-spawn** → **Pattern-B (per-leaf LLM sub-spawns) is impossible from within a spawn.**
- The ONLY way a persona invokes decompose-run is via **Bash** (`node decompose-run.js …`) = the Pattern-A serial
  module composition (leaves verified as code: R12 subprocess test-runs + R9 structural checks; no LLM sub-spawns).
- Pattern-B is **root-orchestrator only** (root has the Agent tool) → E12/v3.5, deferred.

## The design

**TRIGGER = persona-instinct.** An `agents/<name>.md` instruction block: *"when your task decomposes into ≥3
independent, individually-verifiable leaves, write them to `swarm/run-state/<run-id>/leaves.json` and run
`node packages/runtime/orchestration/decompose-run.js --leaves <file> --run-id <fresh-id> --persona <your-type>
--task <id> --cwd <repo>`; if any leaf is rejected, run the E1 ingest; report the admitted/rejected split."*
- **Why instinct, not a hook:** there is NO harness event that fires *inside* a running persona at a useful
  decision point. `PreToolUse:Agent`/`PostToolUse:Agent` fire in the *root* (before/after the spawn), not within
  it; `PreToolUse updatedInput` is inert on Agent (ADR-0012). A hook is right for CAPTURE (root-side), not TRIGGER.
- **Why instinct, not a skill:** the Skill tool is **absent from the depth-1 probe inventory** — unproven inside a
  spawn. Bash is the only reliably-owned invocation channel. (A SKILL.md can still be *documentation the instinct
  references*, just not the load-bearing trigger.)
- **First persona = `code-reviewer`** (its `tools:` includes Bash; review tasks naturally fan into leaves).

**CAPTURE = outbox pattern** (the K12-clean, axiom-honest path):
1. `decompose-run` (runtime) ALSO writes its result `{admitted, rejected, …}` to a neutral runtime-owned file
   `swarm/run-state/<run-id>/decompose-result.json` (~15 LoC; reuse the trampoline's `checkWithinRoot` guard).
2. E1 (lab) gets a CLI `record-from-decompose --run-id X` that **Reads** that file, pulls `rejected[].failure_signature`,
   and writes each into E1's own store.
- **Zero `require()` edge** runtime↔lab in either direction → `layer-boundary-lint.js` (K12) finds nothing (it only
  models import edges, not data-file reads). **Lab PULLS data inward; nothing pushes into Lab** → the advisory axiom
  holds. No fragile Bash-stdout parsing; no unprobed harness-payload-completeness assumption.
- **Rejected (a)** a `PostToolUse:Bash` hook parsing stdout — fragile (stdout completeness on the payload is UNPROBED)
  AND a kernel-hook-writing-into-lab inverts the Lab axiom (dodges the import-lint via subprocess, but violates intent).
- **(b)** persona shells to the E1 CLI directly — viable FALLBACK; **composes** with the outbox (the file makes the
  E1 step recoverable if the persona skips it — root/session-end sweep can ingest later).

## E1 contract corrections (forced by real provenance — fold BEFORE E1 freezes)

1. **`attestation_id` keys on `rejected[].id`**, not an assumed top-level `leaf_ref`. The `decompose-run` rejected
   entry is `{id, failure_signature}` (`decompose-run.js:83-86`); the signature-internal `leaf_ref` is **often null**
   (`failure-signature.js:52`). So the null-fallback→append-always path (E1 §3c) is **essential, not edge-case.**
2. **`subagent_type` grain = bare agentType** (`code-reviewer`), NOT numbered-roster (`04-architect`). A depth-1
   spawn has **no identity env-injection** (`v3-entry-probes.md:74`) — it knows only its bare agentType. The
   numbered-roster grain needs root-side prompt injection → deferred to the E4 era / `build-team`-driven spawns.

ingest (`negative-attestation/record-from-decompose.js` + `cli.js`) + the E1 store. All on the runtime↔lab DATA

**Live dogfood VERIFIED end-to-end (firsthand on disk, NOT self-report):** a real `code-reviewer` spawn authored
leaves (Bash heredoc), ran `decompose-run` (run-id `wave0-dogfood`) → the outbox carried

un-darkening works:** spawn → decompose → verify → attest, end-to-end.

**DEFERRED — a design-spike element corrected against reality: baking the instinct into `agents/code-reviewer.md`.**
On build, the **path problem** surfaced: `agents/*.md` are SHIPPED general personas used in *arbitrary user
projects*, where the claude-toolkit-internal `packages/runtime/orchestration/decompose-run.js` path does not exist —
baking a substrate-internal invocation into a general persona would be dead/broken instruction everywhere else. So
the persona-binding is **deferred to the open "how does production decomposition get triggered" question**
(v3.3+/v3.4 — needs `$CLAUDE_PLUGIN_ROOT`-relative paths + a decision on whether decompose-run is meaningful outside
claude-toolkit). Wave 0 ships the **durable pipeline** (outbox + ingest) + proves it via the dogfood with the
instinct given **inline** (the snippet below), operating *in* claude-toolkit. Honest claim: *"a persona CAN and DID
drive the tier from a real spawn,"* NOT *"a shipped persona reflexively decomposes."* (Same match-build-to-reality
discipline that deferred E2/E3/E4.)

**The reusable instinct snippet** (for dogfooding claude-toolkit's OWN substrate; given inline to a spawn — NOT
shipped in a persona def):
> When your task decomposes into ≥3 independent, individually-verifiable leaves: (1) author them as decompose-run
> leaves to `/tmp/<run>-leaves.json` via a Bash heredoc (Bash is the one reliably-present spawn tool); (2) run
> `node packages/runtime/orchestration/decompose-run.js --leaves <file> --run-id <FRESH> --persona <self> --task <tag> --cwd <repo>`
> (FRESH run-id each time — R7 replaces the checkpoint); (3) if any leaf is rejected,
> `node packages/lab/negative-attestation/cli.js record-from-decompose --run-id <FRESH>`; (4) report the split.
