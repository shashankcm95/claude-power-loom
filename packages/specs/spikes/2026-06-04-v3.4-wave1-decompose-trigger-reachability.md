# v3.4 Wave 1 spike — can a production persona invoke `decompose-run`?

**Date:** 2026-06-04 · **Gates:** v3.4 Design B Wave 1 (production decompose trigger + E11 breaker). **Verdict: NOT buildable as conceived — blocked on a probed packaging gap + an unsettled semantic-scope question. Two decisions required before any Wave-1 build.**

## The question
The cumulative-coherence gate put Wave 1 = "make the decompose tier live so E1's producer stops being starved," flagged "blocked on the depth-1 / general-persona question — may need a spike." This spike answers: **how (and whether) does a SHIPPED persona, spawned in an arbitrary user repo, actually come to run `decompose-run` in production?**

## Probed facts (firsthand, 2026-06-04)

**Mechanism — settled by the prior orchestration spike (`2026-06-04-v3.3-orchestration-design-spike.md:26-35`):** the trigger is a **persona-instinct invoked via Bash** (an `agents/<name>.md` block: "when your task decomposes into ≥3 verifiable leaves → author leaves.json + Bash-run decompose-run"). NOT a hook (no harness event fires *inside* a running persona; `PreToolUse updatedInput` is inert on Agent — ADR-0012). NOT a skill (the Skill tool is absent from the depth-1 probe inventory). **Bash is the only reliably-owned invocation channel in a depth-1 spawn.** The prior spike empirically confirmed a spawned `code-reviewer` given the instinct DID author leaves (via Bash heredoc — Write also absent at depth-1) and run decompose-run — **but only operating IN claude-toolkit, with the instinct given INLINE.**

**The blocker — the PATH/reachability gap (NEW; this spike falsifies the prior spike's proposed fix):**

| Probe | Result | Source |
|---|---|---|
| `CLAUDE_PLUGIN_ROOT` in the main-session Bash env | **UNSET** | `echo` in main session |
| `CLAUDE_PLUGIN_ROOT` in a **depth-1 spawned sub-agent's** Bash env | **UNSET** | a real Agent-spawn probe |
| plugin `bin/` on the sub-agent's PATH | **YES** (`…/power-loom/3.3.0/bin`, inherited from main) | the spawn probe |
| what the plugin `bin/` actually ships | **only `migrate-to-plugin.sh`** — `decompose-run NOT on PATH` | `ls` + `command -v` |
| where `decompose-run.js` is reachable | ONLY a **version-pinned absolute cache path** `~/.claude/plugins/cache/power-loom-marketplace/power-loom/3.3.0/packages/runtime/orchestration/decompose-run.js` | `find` in the plugin cache |

**Conclusion:** the prior spike (`:122`) proposed "`$CLAUDE_PLUGIN_ROOT`-relative paths" as the v3.4 fix. **That is falsified — `CLAUDE_PLUGIN_ROOT` is substituted by the harness ONLY in hooks.json `command` strings, never exported into a Bash tool's environment** (main OR sub-agent). So a persona's instinct cannot say `node $CLAUDE_PLUGIN_ROOT/…/decompose-run.js`. The only reachable path is the version-pinned cache absolute path — brittle (breaks every release), and a persona has no robust way to discover the current version. **A depth-1 persona in an arbitrary repo cannot cleanly invoke `decompose-run` today.**

## The two decisions Wave 1 needs BEFORE building

**D1 — Packaging (the reachability fix).** How is `decompose-run` exposed to a spawned persona's Bash?
- **(a) Ship it as a plugin `bin/` executable** (a `decompose-run` wrapper in the plugin's `bin/`, which IS on PATH). Cleanest: instinct becomes a bare `decompose-run --leaves …` command, version-agnostic, portable. Cost: a real CLI surface + the leaf-schema becomes a public contract.
- **(b) Export a stable env var** (the plugin sets e.g. `POWER_LOOM_ROOT` in the spawn env). Requires a harness mechanism to inject env into spawns — **probe first** (ADR-0012; `CLAUDE_PLUGIN_ROOT` itself proves the harness does NOT export plugin paths to Bash, so this likely needs the bin/ route anyway).
- **(c) Accept toolkit-internal-only** — decompose-run is reachable only when the persona operates ON claude-toolkit (the path `packages/runtime/…` resolves). Production = dogfood-only.

**D2 — Semantic scope (is it even meaningful outside the toolkit?).** `decompose-run` runs the R9 leaf-criteria (cost-justified / interface-clean / validation-supported / resource-bounded / discipline-gate) over a persona's proposed task-decomposition. Those criteria are **HETS-orchestration concepts** (does a leaf DECLARE `output_schema`, `estimated_tokens`, a verification runner?). For a persona doing arbitrary PRODUCT work (review a user's PR), decomposing into R9-conformant leaves is *possible* but its **value is unproven** (the prior persona-instinct A/B found "legibility, not coverage"). If decompose-run is effectively substrate-orchestration-only, then **the decompose→E1 signal is inherently LOW-VOLUME regardless of packaging** — which directly reinforces the coherence gate's "don't build amplifiers over a thin producer."

## Implication for Design B (refines the gate)
The gate ordered Wave 1 (decompose trigger) FIRST to un-starve E1. This spike shows Wave 1 is **gated on D1+D2**, and — more importantly — that **un-starving E1 via "production decomposition" may have low payoff**: the signal is hard to trigger (D1) AND inherently thin (D2). That strengthens the gate's own thesis. **Candidate refinement (USER scope call):** if D2 resolves "substrate-orchestration-only," DEMOTE the decompose-trigger and lead v3.4 with **A6 (Wave 2) + E4 reputation (Wave 3)** — the already-live, quality-bearing signal — accepting decompose→E1 as a low-volume dogfood-only path. The decompose trigger then becomes a smaller "expose decompose-run as a bin/ command for substrate work + E11 breaker" item, not the v3.4 keystone.

## Honest boundary
D1 is a probed fact (the path gap is real + measured). D2 was framed as a design judgment — but reading `leaf-criteria.js` made it largely evidence-backed (below).

## D2 RESOLVED (firsthand, `leaf-criteria.js`) — decompose-run is ORCHESTRATION-native, NOT a general-product-work tool
The R9 leaf-criteria are explicitly **spawn-declaration gates**:
- Header (`:3`): the checks decide whether a leaf is "well-formed enough to be **SPAWNED**/verified."
- `#1 cost-justified` (`:55`): a 500-token floor, because below it a leaf is "too trivial **to justify a spawn**." → would actively **mis-reject** a valid small product sub-task (a 200-token "check the auth flow" review-area is fine for product work, rejected here).
- `#3 interface-clean` (`:115`): requires a non-empty `output_schema` = the SPAWN's structured return.
- `#5 resource-bounded` (`:57,:169`): spawn-sizing + `allows_subspawn` forbid.
- R9 is a VERIFY-time gate consumed by **R11 the spawn-verify dispatcher** (`:7-11`); R6's decompose step never calls it.

**A depth-1 persona cannot spawn its leaves** → these spawn-declaration criteria validate declarations for spawns that never happen = semantically empty for non-orchestration work, and the 500-token spawn-floor actively mis-judges valid small product sub-tasks. **decompose-run's real home is the ROOT-orchestration boundary** (the main session / `/build-team` deciding how to fan out persona-spawns — where it is already reachable + meaningful), NOT inside a shipped depth-1 persona. **The "production decompose trigger for shipped personas" was a mis-framing** — depth-1 personas don't decompose-into-spawns, so they don't need it.

**Counter considered + rejected:** as a pure planning-rigor discipline (decompose + declare per sub-task) it adds "legibility, not coverage" (the prior persona-instinct A/B) AND the spawn-economics criteria actively fight a product-work decomposition. Marginal value, mis-calibrated tool.

**→ SETTLES the v3.4 order.** The decompose→E1 signal is inherently LOW-VOLUME (only root-orchestration ≥3-leaf fan-outs produce it) + dogfood-scoped. **Lead v3.4 with A6 (Wave 1) → E4 reputation (Wave 2)** — the live, quality-bearing, generalizable signal. **DEMOTE the decompose trigger** to a small later "expose decompose-run as a plugin `bin/` command for root/substrate work + E11 breaker" item (the D1 bin/ fix, when/if it earns its keep). decompose→E1 stays dogfood-only. *(USER ratification pending.)*
