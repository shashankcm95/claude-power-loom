## 1. Architecture Overview — Transactional Agent Runtime

The substrate is a **deterministic envelope around untrusted non-deterministic processes**. Agents are treated as non-cooperative non-deterministic execution units; the substrate guarantees behavioral correctness at the interface boundary, not in the agent's internals.

(Round-1 architect noted "byzantine" imports specific BFT semantics that don't quite fit; revised terminology above.)

Same architectural pattern as: database transaction managers, CI gates, BFT consensus wrappers, iOS sandboxes. The substrate's job is to make **state transitions** deterministic given non-deterministic LLM trajectories.

```
        ORCHESTRATOR (depth-0; deterministic kernel)
                       │
       ┌───────────────┴───────────────┐
       │                               │
   user-space spawn               user-space spawn
   (non-deterministic, LLM)       (non-deterministic, LLM)
       │                               │
       │ writes deltas to              │ writes deltas to
       │ isolated worktree             │ isolated worktree
       ▼                               ▼
   ┌─────────────────────────────────────────┐
   │       INTERFACE BOUNDARY (kernel)         │
   │  - Deterministic gates (tests, lint,      │
   │    schema, contract-verifier)             │
   │  - Capability validators (attestation)    │
   │  - Path-rewriting / cherrypick            │
   │  - Spawn-record envelope capture          │
   │  - Negative-attestation extraction        │
   └─────────────────────────────────────────┘
                       │
                       ▼
           HOST CODEBASE (validated state)
```

## 2. The Six Axioms (was 5; A6 added in v2)

### Axiom 1 — Transactional Determinism

> Validated filesystem deltas (or contract-conformant text outputs) are the substrate's unit of truth. LLM trajectories are non-deterministic by construction and recoverable-by-resampling.

**Forbids**: storing LLM reasoning traces as ground truth; treating agent self-reports as authoritative; implicit trust in agent claims of completion.

**Implies**: every spawn's output must be verifiable by deterministic gates; failures invalidate the spawn's deltas atomically; replay-from-validated-state is supported, replay-from-decisions is not.

### Axiom 2 — Kernel / User-Space / Interface Boundary

> Kernel = hooks + GC + validators + recall-CLI + contract-verifier (pure functions, versioned, testable, deterministic). User-space = agent spawns (non-deterministic, ephemeral). Interface = filesystem deltas + contract-conformant text outputs (the verification surface).

**Forbids**: LLMs writing to kernel paths (`hooks/`, `scripts/kernel/`, `.claude-plugin/`); kernel code calling LLMs in the verification path; agents bypassing the interface to directly mutate host state.

**Implies**: kernel additions require TDD + versioning; user-space additions can iterate freely; the boundary IS the verification surface.

### Axiom 3a — Gating Verification Must Be Pure Functions AND Semantically Adequate

> Verification gates that BLOCK promotion/merge/cherrypick are pure functions whose semantics are *adequate* for the property being verified. Surface-keyword checks (e.g., `containsKeywords ["Principle Audit"]`) for substantive semantic properties are FORBIDDEN in the blocking path — they are pure-function-deterministic but semantically shallow and trivially gamed.

**Forbids**: LLM-as-judge as a blocking gate; surface-token-only gates as proxies for substantive compliance; gates whose pass-condition is satisfiable by spawn-side token mimicry.

**Required**: where the property requires semantic judgment, push to advisory gates (Axiom 3b); where the property is deterministically encodable (type-check, schema, test, contract-verifier on functional/antiPattern keys), use pure-function gating.

### Axiom 3b — Advisory Verification May Be LLM-Mediated (Audit-Logged, Non-Blocking)

> Advisory verification (architect review, honesty-auditor findings, code-reviewer pair-runs, cross-persona test review) IS permitted as LLM-judgment, provided it (a) cannot block promotion/merge/cherrypick, (b) emits a structured audit record into the spawn-record's `advisory_findings` field, and (c) optionally informs reputation scoring per Axiom 6.

**Implies**: HETS pair-review patterns from v2.9.x continue to work; existing personas (architect, code-reviewer, honesty-auditor) operate exclusively in the advisory path; their findings shape downstream routing via reputation but never directly gate a merge.

### Axiom 4 — Algorithmic Discipline is Kernel Responsibility (Positive Form of A2)

> Where A2 says "LLMs SHALL NOT write to kernel paths," A4 says "kernel scope SHALL include algorithmic logic." When an operation requires deterministic logic (sorting, subset-check, traversal, budget arithmetic), it lives in kernel code with unit tests — not as prose discipline in persona prompts or as embedded pseudocode for the LLM to mentally execute.

**Forbids**: "iterate through children in topological order" in a persona contract (LLM may choose slightly-wrong order under cognitive load); embedded pseudocode that the LLM is expected to trace; algorithmic correctness depending on agent compliance.

**Implies**: kernel grows a `scripts/kernel/algorithms/` library (~400-500 LoC; budgeted within Phase 2 Wave 2). Persona contracts REFERENCE algorithms by name; kernel runs them; LLM provides judgment inputs.

This is the **template method pattern** at substrate scale.

### Axiom 5 — Substrate Evolution is Designed-First-Class (Phase 3 Deliverable)

> The substrate IS DESIGNED TO measure its own components' quality (personas, tests, decompositions, recalls) and IS DESIGNED TO use those measurements to evolve. Components below quality thresholds WILL be demoted/culled; components above thresholds WILL become defaults. **All evolution infrastructure is Phase 3 work; this axiom describes design intent, not current capability.**

**Forbids**: static persona contracts that never update (anti-pattern); reputation tracking that doesn't feed back into spawn-routing; KB that never grows from substrate experience.

**Implies (when built)**: reputation per-persona × per-task-type × per-quality-axis; attribution graph for tracing bugs to test-gaps; evolve/forge triggers from reputation decay; convergence metrics as Phase 3 validation criterion.

**Round-1 caveat**: cross-persona test review (anti-gaming mechanism) is itself an LLM-judged signal and therefore susceptible to recursive Goodhart drift. Phase 3 design must include either (a) out-of-band ground-truth signal (e.g., human-rated tasks at random sample rate, RLHF-style preference oracle) or (b) acknowledge ConvergenceRate is a proxy and Phase 3 success requires separate qualitative human gate.

### Axiom 6 — Reputation as Snapshotted Axiom (NEW in v2, resolves A1 ↔ A5 tension)

> When evolution-layer signals (reputation, trait health, KB freshness) influence spawn-routing, they are captured into the spawn's `axioms` block at spawn-init as `axioms.evolution_snapshot`. Re-spawn-from-axioms produces deterministic routing; evolution affects the *next* spawn, not replay of the current one.

**Resolves**: A1's replay-determinism guarantee vs A5's living-reputation flow. Without A6, the same axiomatic inputs at different times produce different routings (A1 violation). With A6, the reputation context is itself axiomatic-at-spawn-init.

**Implies**: Phase 2 spawn-record envelope gains `axioms.evolution_snapshot` field (schema bump to v2; see §3.5). Re-spawn replays from the snapshot, not from live reputation. Forward evolution flows into the NEXT spawn's snapshot.

### 3.2 Capability Primitives (4) — INJECTION vs ENFORCEMENT distinction

**Critical clarification (Round-1 absorbed)**: capabilities in Phase 2 are **attested** (declared + post-hoc audited), not **enforced** at runtime, except for:
- **Write capabilities**: enforced by worktree filesystem scope (Anthropic native)
- **All other capabilities**: declared in contract, audited post-hoc at promote-deltas. A runaway agent's runtime network/subprocess/recall reads cannot be blocked mid-spawn — only detected at spawn-close.

| # | Capability axis | Declared in contract | Phase 2 enforcement |
|---|---|---|---|
| 4 | Read | Broad allowlist | None (audited post-hoc) |
| 5 | Write | Sandbox-only | **Worktree filesystem scope** (Anthropic-enforced) |
| 6 | Network | Explicit allowlist | None (advisory; Phase 3+ candidate) |
| 7 | Subprocess | Scoped command whitelist | Settings.json `permissions.allow` *per-Claude-Code-invocation* (not per-spawn) — advisory at persona level |

**Capability traits as mixins** (composition over inheritance):

```yaml
traits:
  worktree_writable: { isolation: worktree, write: ["sandbox://**"] }
  network_anthropic: { network: ["api.anthropic.com"] }
  bash_test_runner: { subprocess: ["npm test", "vitest", "pytest", "tsc --noEmit"] }
  recall_global: { read_recall: ["@library/*", "@thoughts/*"] }

06-node-backend:
  traits: [worktree_writable, network_anthropic, bash_test_runner, recall_global]
```

**Trait composition conflict-resolution** (Round-1 absorbed; required normative rule):
- **Narrowing-direction** capabilities (subprocess whitelists, write paths): **intersection** (tightest scope wins)
- **Broadening-direction** capabilities (read paths, recall scope): **union** (widest scope wins)
- **Same-direction conflicts** at trait composition: **contract-load-time error** in `contract-verifier.js`

Without these explicit rules, trait composition for security-relevant capabilities is exactly where ambiguity becomes exploitable. ~50 LoC addition to `contract-verifier.js`.

Honesty discipline: do not claim Liskov enforcement for natural-language fields.

### 3.4 Decomposition Primitives (3)

**The merge-sort-for-tasks principle**: recursive atomization until each leaf satisfies the leaf criteria.

**Leaf criteria (revised after Round-1 — criterion #5 reworded for falsifiability)**:

| Criterion | Operational check |
|---|---|
| Cost-justified | Expected work ≥ ~500 tokens / ~30s wallclock (estimate, deterministic check) |
| Semantically cohesive | One clear purpose (advisory check; tag-overlap proxy) |
| Interface-clean | Focused input; structured output (deterministic schema check) |
| Validation-supported | Deterministic gate exists for this leaf type (kernel registry lookup) |
| Resource-bounded (revised) | `estimated_tokens ≤ N AND capability declaration forbids sub-spawning` (deterministic check; replaces v1's unfalsifiable "high confidence completes without internal sub-spawning") |

**Leaves at domain-artifact granularity, not operation granularity**: "add a migration" is a leaf; "add a semicolon" is inline noise.

**Atomicity cutoff is empirically tuned**, not hardcoded. Initial value at Wave -1 calibration probe: TBD (see §8 open question 1).

**Decomposition disciplines** (persona-declared, kernel-routed at spawn-verify):

| Discipline | When to use | Verification gate |
|---|---|---|
| `tdd` | Code-producing leaves (add migration, implement endpoint, refactor) | Test suite passes |
| `spec-driven` | Configuration / structural work | Schema validate |
| `exploratory` | Research / subjective writing / design docs | Structural check (required sections present) |

**Pattern A — Persona-internal trampoline**:
TodoWrite tree + folder hierarchy + save states within ONE Anthropic spawn. Same cognitive thread handles all sub-tasks sequentially.

**Round-1 caveat absorbed**: Pattern A produces **no per-leaf filesystem delta** — only the spawn-close delta covers the whole subtree. This means A1's filesystem-delta-as-truth applies at *spawn-close granularity*, NOT at *leaf granularity* within Pattern A. Leaf criterion #4 (validation-supported) is checked via the persona's TodoWrite trail + structured artifact emission per leaf (within-spawn save states), NOT via per-leaf delta capture. This is a deliberate limitation: Pattern A trades per-leaf delta isolation for cognitive-thread continuity and lower coordination overhead.

**Pattern B — Multi-spawn trampoline** (Phase 3):
Orchestrator queue + continuations + parent_state_id chain. Different personas, different capability surfaces, parallel execution. Each "logical node" is a separate top-level spawn with its own filesystem delta. ~300-400 LoC kernel work, deferred.

**Negative attestation: split into witness (Class-4) and derived policy (Class-1)** (Round-1 absorbed):
