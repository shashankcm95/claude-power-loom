## 1. Architecture Overview — Transactional Agent Runtime

The substrate is a **deterministic envelope around untrusted non-deterministic processes**. Agents are non-cooperative non-deterministic execution units; the substrate guarantees behavioral correctness at the interface boundary, not in the agent's internals.

```
                    ORCHESTRATOR (deterministic kernel)
                              │
              ┌───────────────┴───────────────┐
              │                               │
       agent spawn (non-det)            agent spawn (non-det)
              │                               │
              │ writes deltas to              │ writes deltas to
              │ isolated worktree             │ isolated worktree
              ▼                               ▼
       ┌─────────────────────────────────────────┐
       │       INTERFACE BOUNDARY (kernel)         │
       │  Deterministic gates only:                │
       │  - schema / capability subset / path /    │
       │    test-suite / lint / contract-verifier  │
       │  - Spawn-record envelope capture          │
       │  - Negative-attestation extraction        │
       └─────────────────────────────────────────┘
                              │
                              ▼
                  HOST CODEBASE (validated state)
```

## 2. Three-Layer Architecture (NEW in v3)

Per GPT's recommendation: formalize the substrate as three distinct layers with separate shipping units, verification surfaces, and stability commitments.

### Layer 1 — Loom Kernel (microkernel)

**Property**: minimal, deterministic, portable, auditable. Strongest backward-compat guarantees.

**Verification surface**: pure-function gates only. No LLM in the trust path.

**Portability story**: could detach from Claude-specific assumptions via adapter layer (Claude / OpenAI / Local). Kernel logic is runtime-agnostic.

**Components** (assigned in §3.1):
- Hook framework integration
- Deterministic validators (schema, secrets, path, capability subset)
- Spawn-record envelope (capture, write, retrieve)
- Lineage primitives (parent_state_id chain)
- Recall-CLI deterministic ranker
- Kernel algorithm library (sort, allocate, traverse, etc.)
- Worktree integration (declarative use of Anthropic-native primitives)
- Promote-deltas (cherrypick + path-rewrite)
- LOOM_DISABLE_WORKTREE escape hatch

**Stability commitment**: kernel API is MAJOR-bump-protected. Changes require ADR.

### Layer 2 — Loom Runtime (operational)

**Property**: HETS orchestration + persona contracts + decomposition discipline. Stable across MAJOR versions but can iterate within.

**Verification surface**: kernel gates (deterministic, blocking) + advisory verification (LLM-mediated, non-blocking, audit-logged).

**Portability story**: HETS pattern is portable; Claude-adapted specifics are encapsulated.

**Components** (assigned in §3.2):
- 16-persona system + two-tier contracts (interface + defaults)
- HETS hierarchical supervision pattern
- Capability traits + composition (intersection/union rules)
- Decomposition disciplines (tdd / spec-driven / exploratory)
- Leaf criteria (5 deterministic conditions)
- Pattern A trampoline (intra-persona)
- Budget envelope propagation
- TodoWrite-as-checkpoint primitive
- Spawn-verify dispatcher (routes by discipline)
- Test-runner adapters

**Stability commitment**: runtime APIs are MAJOR-bump-protected; persona contracts are MINOR-bump additive.

### Layer 3 — Loom Evolution Lab (experimental)

**Property**: adaptive cognition; explicitly isolated from kernel determinism. Iterates freely without affecting kernel/runtime guarantees.

**Verification surface**: advisory only. Outputs feed reputation asynchronously; never blocks kernel-gated operations directly.

**Portability story**: highly Claude-adapted (uses Claude's cognitive properties); less portable; deliberately experimental.

**Components** (assigned in §3.3):
- Negative attestation (Class-4 witness + Class-1 derived-policy)
- Policy-axiom store
- Reputation tracking + extension (per-persona × per-task-type × per-quality-axis)
- Attribution graph + queries
- Convergence metrics + CLI verb (dashboard deferred to Phase 4)
- Evolve/forge triggers
- Cross-persona test review (+ out-of-band ground-truth signal)
- TDD-craft KB (seed + self-grown)
- Reference test suites (cold-start calibration)
- Pattern B trampoline (multi-spawn; deferred)
- Dream-Lite cycles (per RFC v3.2 §Dreaming)
- Circuit-breaker on denials

**Stability commitment**: experimental. Components may be removed/replaced between PATCH versions. NEVER promoted to kernel without explicit ADR + full pair-review.

### Why this separation matters

Without explicit three-layer separation:
- Experimental cognition (Lab) accidentally creeps into kernel determinism (drift risk)
- Verification surface explodes (Lab metadata pollutes kernel audit)
- Portability story collapses (Claude-specific evolution wiring locks the kernel to Claude)
- Stability commitments leak (kernel must support Lab's iteration speed)

With it:
- Kernel stays small and auditable (microkernel discipline)
- Runtime stays focused on HETS operational concerns
- Lab can iterate freely without contaminating shipping commitments
- Each layer can be reasoned about independently

This is the load-bearing architectural commitment in v3.

## 3. The Six Axioms (mapped to layers)

### Axiom 1 — Transactional Determinism (Kernel)

> Validated filesystem deltas (or contract-conformant text outputs) are the substrate's unit of truth. LLM trajectories are non-deterministic by construction and recoverable-by-resampling.

**Forbids**: storing LLM reasoning traces as ground truth; agent self-reports as authoritative; implicit trust in agent claims of completion.

**Implies**: every spawn's output must be verifiable by deterministic gates; failures invalidate deltas atomically; replay-from-validated-state is supported.

### Axiom 2 — Kernel / User-Space / Interface Boundary (Kernel)

> Kernel = hooks + validators + recall-CLI + contract-verifier (pure functions, versioned, deterministic). User-space = agent spawns (non-deterministic, ephemeral). Interface = filesystem deltas + contract-conformant text outputs.

**Forbids**: LLMs writing to kernel paths; kernel code calling LLMs in verification; agents bypassing the interface to mutate host state.

### Axiom 3a — Gating Verification Must Be Pure AND Semantically Adequate (Kernel)

> Verification gates that BLOCK promotion/merge/cherrypick are pure functions whose semantics are *adequate* for the property being verified. Surface-keyword checks (e.g., `containsKeywords ["Principle Audit"]`) for substantive semantic properties are FORBIDDEN in the blocking path — pure-function-deterministic but semantically shallow and trivially gamed.

### Axiom 3b — Advisory Verification May Be LLM-Mediated (Runtime)

> Advisory verification (architect review, code-reviewer pair-runs, cross-persona test review) IS permitted as LLM-judgment, provided (a) it cannot block promotion, (b) it emits structured audit records into spawn-record's `advisory_findings`, (c) optionally informs reputation per Axiom 6.

**Implies**: HETS pair-review patterns continue to work; existing personas operate exclusively in advisory path.

### Axiom 4 — Algorithmic Discipline is Kernel Responsibility (Kernel; positive form of A2)

> Where A2 says "LLMs SHALL NOT write to kernel paths," A4 says "kernel scope SHALL include algorithmic logic." Deterministic operations (sorting, subset-check, traversal, budget arithmetic) live in kernel code with unit tests — not as prose discipline in persona prompts or as embedded pseudocode for the LLM to execute.

### Axiom 5 — Substrate Evolution is Designed-First-Class (Evolution Lab; Phase 3+ deliverable)

> The substrate IS DESIGNED TO measure its own components' quality and IS DESIGNED TO use those measurements to evolve. **All evolution infrastructure is v3.3 and v3.4 work; this axiom describes design intent, not current capability.**

**Round-2 caveat absorbed**: cross-persona test review is itself an LLM-judged signal subject to recursive Goodhart drift. Phase 3 design must include out-of-band ground-truth signal (RLHF-style preference oracle) OR honest framing that ConvergenceRate is a proxy.

### Axiom 6 — Reputation as Snapshotted Axiom (cross-cutting; bridges Lab → Kernel)

> When evolution-layer signals (reputation, trait health, KB freshness) influence spawn-routing, they are captured into the spawn's `axioms` block at spawn-init as `axioms.evolution_snapshot`. Re-spawn-from-axioms produces deterministic routing. Evolution affects the *next* spawn, not replay of the current one.

**Round-2 sibling-wave clarification**: within a single orchestrator dispatch wave, all sibling spawns share the same evolution_snapshot; updates apply only to spawns initiated after the wave closes. This locks determinism at wave granularity.

**Round-2 transitive-path clarification**: advisory_findings (from A3b) feed reputation asynchronously; reputation is snapshotted at spawn-init per A6; **kernel never reads live advisory_findings in a gating decision**. Without this explicit statement, a future implementer might wire `if findings.severity == HIGH then block` and unknowingly violate A3a.

**WOULD RESOLVE A1↔A5 tension** when implemented (Phase 3 schema extension dependency).

## 5. Capability Model (split for clarity in v3)

**Critical clarification (Round-1 absorbed; reaffirmed in v3)**: capabilities in v3.1 are **attested** (declared + post-hoc audited), not **enforced** at runtime, except for:

- **Write capabilities**: enforced by worktree filesystem scope (Anthropic native via K1)
- **All other capabilities**: declared in contract, audited post-hoc at promote-deltas

| Capability axis | Declared in contract | Phase 2 enforcement |
|---|---|---|
| Read | Broad allowlist | None (audited post-hoc) |
| Write | Sandbox-only | **Worktree filesystem scope** (kernel) |
| Network | Explicit allowlist | None (advisory; Phase 4 candidate) |
| Subprocess | Scoped command whitelist | Settings.json `permissions.allow` *per-Claude-Code-invocation* (not per-spawn) — advisory at persona level |

**Trait composition rules** (normative; per R4):
- **Narrowing-direction** capabilities (subprocess whitelists, write paths): **intersection** (tightest wins)
- **Broadening-direction** capabilities (read paths, recall scope): **union** (widest wins)
- **Same-direction conflicts** at trait composition: **contract-load-time error**

## 10. Phase 2 Capacity Envelope (Round-2 HIGH-7 absorbed)

**Phase 2 (v3.0-alpha through v3.2) is SERIAL-ONLY by policy.** Concurrent-spawn capacity envelope is deferred to Phase 4 (or v3.5+ when explicit need surfaces).

**Why**: at 10× parallel spawn rate, shared resources break:
- `~/.claude/spawn-state/<run_id>/` directory (single run_id per session; many parallel spawns contend)
- `~/.claude/self-improve-counters.json` counter file (single-writer)
- `~/.claude/library/_meta/gc.lock` (10s acquire timeout; second session writes `gc_sweep_deferred` — at scale, GC effectively stops)

**Mitigation in v3**: serial-only contract documented in K3 (parent_state_id chain). If concurrency lands later, `parent_state_id` grows from ~15 LoC to ~80-120 LoC (file locking + atomic-CAS + cycle detection). Flag in K3's acceptance criteria.
