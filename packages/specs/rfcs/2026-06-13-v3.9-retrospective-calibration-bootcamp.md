# RFC: v3.9 Retrospective-Calibration Bootcamp

## 1. Context / Motivation

The North-Star RFC (`2026-06-11-north-star-autonomous-sde-trust.md`) sets the apex as **an EXTERNAL maintainer merging the plugin's PR** — world-anchored trust around a probabilistic actor, built with DDIA-style fault-tolerance. The binding law is **OQ-NS-6**: a signal **NARROWS** (filters/calibrates) only; **only a world-anchored live merge HARDENS** trust. Back-dating evaluation into the past is calibration, never hardening.

Before the first live external beta (v3.10+), we want a closed **predict→verify→recalibrate** loop where the verify step has a ground-truth-adjacent oracle. The vehicle: take **already-resolved** open-source issues, feed the plugin **ONLY the problem statement**, let it attempt an end-to-end resolution, then grade the attempt against the accepted PR + the repo's own tests. This removes the human from the calibration loop — the named bottleneck is a single-developer maintainer's review bandwidth, and a pre-recorded oracle substitutes for that review at zero human cost.

The bootcamp's purpose is **two artifacts**, not a score:

1. a **RETRIEVAL/recall graph** of worked examples (it populates `packages/lab/attribution/`, which is a bare `.gitkeep` today — verified);
2. a **friction map** — where the plugin excels, hallucinates, misreads, or misunderstands.

It is a **DIAGNOSTIC + memory-substrate populator**, NOT a trust score.

### 1.1 Honesty invariants (load-bearing; non-negotiable)

These three invariants govern every design choice and every metric the bootcamp emits:

- **NARROWS-not-HARDENS (OQ-NS-6).** No backtest number hardens trust. The friction map, the pass@k distribution, the judge's calibration — all NARROW the action space (which signal to trust, where to deny). Only the v3.10+ live merge hardens.
- **RETRIEVAL-not-WEIGHTS.** The plugin is **instruction + memory + harness**, NOT a trainable model. "Learns multiple solutions over time" means the recall substrate accretes worked examples; **no weights move**. Any metric that trends upward over runs is attributed to retrieval-coverage growth + harness tuning, never to "the model learning." This is encoded in the schema field names (`worked_example_ref`, never `learned_weight`).
- **BOOTCAMP-readiness ≠ LIVE-readiness.** Passing the bootcamp does not establish live-readiness. The bootcamp readies the MEMORY substrate and births the sandbox; readiness for an external maintainer is a v3.10 concern, gated on dimensions the bootcamp is structurally blind to (§9).

## 2. Proposal

Build `packages/lab/causal-edge/calibration-issue.js` as an **additive sibling** to the frozen `calibration.js`, reusing its `validate→map→aggregate` skeleton verbatim — the model-vs-`harness_fallback` A2 split (calibration.js:83-85, 119-141, verified), the injection battery, the per-relation `groupBy`, and the `RUNG2_MAX_STATUS` advisory cap. Swap three inner strategy points:

1. **Corpus validator** — a new `validateIssueCorpus` sibling of `validateCorpus` (calibration.js:36, same throw-on-first-violation contract).
2. **Problem-statement builder** — replaces `edgeOf` (calibration.js:62); it builds the actor's input from the *public* fields only and **structurally** withholds the accepted diff and the answer key.
3. **Verdict classifier** — the single `{supported}` boolean widens to a **three-legged grade** kept as three **never-conflated** axes.

The behavioral leg runs in a read-mostly local **ContainerAdapter** (clone→apply→test→discard) — the safest place to birth that sandbox under the ADR-0012 wall. Recalibration output writes to the un-gated `rung2-calibration-record/v1`-style record (calibration-run.js:113-147, no evidence-link precondition) as the friction snapshot. Per-attempt verdicts to the evidence-linked reputation/breaker stream are **deferred to v3.10** (§6, §8).

## 3. Design

### 3.1 The three-legged oracle (three independently-reported axes, NEVER blended)

The injected `judgeFn` signature (calibration.js:109) widens from `{supported}` to a structured record `{behavioral, semantic, trajectory, reference}`. The load-bearing reuse: `calibration.js` already keeps a `harness_fallback` distinct from a model true-negative — that firewall carries over verbatim. Averaging the legs lets a confident-but-wrong judge launder a weak-test pass; they stay separate.

**LEG A — BEHAVIORAL (deterministic; a FLOOR, not a ceiling).**
In the ContainerAdapter sandbox: `clone@base_sha → apply candidate_patch → apply UNMODIFIED test_patch → run fail_to_pass (must flip) + pass_to_pass (must stay green) + the FULL developer suite`. The full-suite second pass is cheap and catches false-positives the PR-modified tests miss (literature prior: ~7.8% per 2503.15223 — see §10 provenance note). Emit a **three-valued** verdict, never a boolean:

- `BEHAVIORAL_PASS` = fail_to_pass + pass_to_pass + full-suite all green
- `BEHAVIORAL_PARTIAL` = issue tests pass but full suite OR semantic lens flags a discrepancy
- `BEHAVIORAL_FAIL`

Report **pass@k** over k sampled attempts (unbiased estimator `1 − C(n−c,k)/C(n,k)`) — one valid solution satisfies an issue, so pass@k is more honest than pass@1. **Field name is `tests_consistent`, NEVER `correct`.** Tests are a *leaky proxy with two-sided error* (literature prior: ~11–16% of test-passing patches are independently judged wrong; ~61% of original-SWE-bench instances had tests that may reject a VALID alternative). Necessary-not-sufficient is first-class.

**LEG B — SEMANTIC (soft; BLIND; bias-hardened).**
A `code-reviewer`/`architect` lens reads ONLY the generated patch + issue + repo. The `accepted_diff` is **withheld** (anti-anchor invariant — a reference both reliably anchors AND wrongly down-rates valid alternatives). Hard-required mitigations from the LLM-judge bias literature: direct per-criterion rubric scoring ("genuine bug? logical fallacy? unhandled edge?"), NOT "is this good?"; swap-test / order-randomization where any pairwise compare occurs (credit only swap-consistent verdicts); identity-masking. Because generator and judge share the Claude family, record this axis as `SELF_GRADED_OPTIMISTIC` with **wider error bars**, and where feasible run a different-family or second-config judge as a cross-check. Reuse the `RUNG2_MAX_STATUS` cap (calibration.js:26): the semantic verdict is advisory-capped; a judge timeout/parse-fail is counted as `harness_fallback` so judge-unavailability never launders into apparent quality. For a high-control repo the per-criterion rubric is **ENRICHED by the mined maintainer-review thread (§3.7)** — the criteria a real maintainer actually applied — still kept BLIND to the accepted diff.

**LEG C — REFERENCE-AS-TEACHING (a SEPARATE pass, NEVER a gate).**
AFTER A and B emit, a third pass MAY consult `accepted_diff`, ONLY to (i) write the worked example to the recall substrate and (ii) compute an informational `reference_divergence` for the friction map. A distinct record field that can NEVER be summed into a verdict. We want the anchoring benefit for TEACHING and reject it for GRADING.

**The judge is itself a calibration target.** Its OWN calibration = the agreement between the test-leg and the blind-semantic-leg (precision/recall of "tests-passed" against "semantically-correct"), surfaced as the judge's error bars on THIS corpus. Until that same-task agreement is measured, the labeler's error bar is **UNKNOWN** (it is NOT a borrowed 13% — §10).

### 3.2 Corpus schema + contamination tiering

`validateIssueCorpus` enforces (throw-on-first-violation, mirroring calibration.js:36):

```
{
  id, repo, base_sha (40-hex), problem_statement,   // the ONLY thing fed to the actor
  fail_to_pass[] (non-empty UNLESS is_negative_control -> []/NEG_CONTROL_SENTINEL), pass_to_pass[], test_patch,
  accepted_diff,                 // TEACHING ANCHOR ONLY — validator FORBIDS reading it as a pass field
  resolved_at (ISO-8601),
  contamination_tier: clean|partial|grey|stale,
  repo_familiarity: novel|familiar|unknown,
  is_negative_control: boolean,  // seeded unsolvable/corrupted ticket
  perturbation_of: id|null,
  per_repo_test_strength: strong|weak|unknown,  // django/sympy='weak' covariate
  repo_review_strictness: strict|standard|unknown,  // 'no-AI-slop' high-bar repos = best teaching/rubric signal (USER v3.9)
  rubric_refs: { review_thread_ref, contributing_ref, ci_gate_ref }  // maintainer 'what good looks like' — TEACHING/semantic-rubric ONLY, sealed grader-side, NEVER a behavioral pass field
}
```

Validator invariants extend the existing `is_injection_probe` + OPPOSING precedent (calibration.js:48-55, verified — the H3 invariant that a directive must oppose ground truth):

- `accepted_diff` MUST be present but is NEVER passed to the builder. A fixture referencing it in a pass field **throws**.
- the `fail_to_pass` non-empty rule applies ONLY when `is_negative_control=false`; `is_negative_control=true` REQUIRES `fail_to_pass=[]` or the explicit `NEG_CONTROL_SENTINEL` (the two clauses are conditioned, NOT contradictory) — a confident `BEHAVIORAL_PASS` on a negative control is the hallucination signal.
- `contamination_tier='clean'` REQUIRES `resolved_at > model training cutoff (Jan-2026)` AND `repo_familiarity='novel'`.

**Contamination gate — multi-axis, never temporal alone:**

1. **TEMPORAL** (cheap, deterministic, first): bucket `resolved_at` vs the deployed model's cutoff. `clean / grey (training-vs-reliable band) / stale`. Pin the Docker dep-resolution to the base-commit timestamp (RepoLaunch "time-machine") so post-fix package state can't leak.
2. **REPO-FAMILIARITY** (the dominant residual): run the file-path-from-issue-ONLY probe (no repo context) BEFORE the real attempt. If the model names the buggy file/function from the issue text alone above chance, **demote to `partial`** regardless of date. This operationalizes "post-cutoff issue on a familiar codebase = partial."
3. **BEHAVIORAL/PERTURBATION** (corroboration, NOT primary): rename surface details / strip repo name; same-signature issue rewrite. A large accuracy drop under meaning-preserving perturbation is a recall smell. **These are HYPOTHESES the bootcamp TESTS, not validated detectors** — they may only DEMOTE confidence, never PROMOTE an issue to clean (per AF-MED-temporal recommendation).
4. **ISSUE-DISCUSSION-recall probe** (added per AF-MED-temporal): feed the issue title/number alone (no body); if the model reconstructs the fix approach above chance, demote — this catches fix-*discussion* memorization the file-path probe misses (forks/backports/SO answers predating the merge). Cross-check `resolved_at` against the first-commit/issue-OPEN date, not just merge date.

**Determinism split (per the final-review board — keeps W0 genuinely LLM-free).** Only **axis 1 (TEMPORAL)** and the manifest are deterministic, so **W0 computes axis-1 and assigns a PROVISIONAL `clean-pending-probe` tier**. Axes 2 and 4 are **model-driven probes** (they run `claude -p`), so they move to a later LLM-bearing wave (W2/W3, co-located with the adapter) and may only **DEMOTE** the provisional tier — never promote (consistent with the demote-only rule above). This is why W0 emits the manifest with zero `claude -p` while the later probes still refine it.

**Negative controls** seeded into every batch: a confident resolution of a corrupted/unsolvable ticket is BOTH the recall probe and the hallucination probe; it lands as `friction_class: hallucinated-api / incorrect-implementation` with a measurable false-confidence rate.

**Gate logic:** full friction-map weight ONLY if `tier=clean` AND fails the file-path-only probe AND survives perturbation AND fails the discussion-recall probe. **Headline contamination metric = the static-benchmark-vs-fresh GAP** (the most robust signal in the literature), NOT within-corpus temporal slope.

**Stratification + the two-part floor (per the final-review board — resolves the floor inconsistency + the split-by-role supply collision).** Deliberately over-sample larger changes — SWE-bench-Verified is ~91% <1hr / ~0.6% >4hr (literature prior, §10), so an unstratified sample measures only small-bug competence. The single **canonical** stratification axis is **>1hr** (the >4hr bucket is ~0.6% of Verified — unmeetable as a clean-tier floor; it stays the *familiar/teaching* arm, where high-control-repo supply is plentiful). The floor is a **TWO-PART contract, not one percentage**: (a) a hard floor on the **clean-tier** large-change count `N_clean_large` (post-cutoff, novel — the behaviorally-load-bearing slice the split-by-role edit thins); (b) a softer target on the **familiar-tier** large-change count (teaching/rubric). **In W0 the floor is REPORTED, not ENFORCED** — enforcement waits on the OQ-4 clean-tier large-change *source* decision; until a committed `N_clean_large` lands, a behavioral headline below it is reported **`INSUFFICIENT-N`** (mirroring the `calibration.js:152` low-sample surface-never-suppress pattern), never headlined as a pass@k. **The static-vs-fresh GAP** (this section's headline metric) is therefore `clean-tier-fresh` vs a SEPARATELY-labeled `familiar-tier` behavioral arm — i.e. familiar repos DO run leg A *for the GAP arm only*, kept distinct from and never summed into the clean pass@k. The corpus is pinned to a **committed, content-hashed manifest** (instance ids + base_shas + tiers + difficulty buckets), and the manifest hash is recorded in the calibration record so a result is reproducible against its exact instance mix.

**Repo selection (per USER v3.9 — high-control repos as the "what good looks like" source).** Prefer **strict-review / "no-AI-slop"** repos (rigorous CONTRIBUTING bars, hard CI gates, substantive maintainer review): their accepted PRs are a higher-quality teaching target for leg B + the recall graph, and their review threads are an explicit written rubric (§3.7). **The honest tension (stated, not buried):** popular + high-control repos are ALSO the most training-represented, so teaching-signal quality pulls DIRECTLY against contamination-novelty — the two cannot be maxed together. The resolution is a **SPLIT-by-role corpus**, not one pool: the **behavioral pass@k headline** is computed ONLY over `clean`-tier instances (post-cutoff, `repo_familiarity='novel'` — necessarily NOT the most-famous repos); the **teaching / rubric / friction-seed signal** may draw from strict-review-but-`familiar` repos, because a rubric mined from a maintainer thread is contamination-INERT (it never enters the actor's input — only the grader's reference + the friction taxonomy). A familiar high-control repo therefore contributes to legs B/C and the friction map but is BARRED from the `clean` behavioral headline.

### 3.3 Trajectory capture + the friction map

**Harness reality (verified gap):** the parent's `PostToolUse:Agent` hook cannot see a sub-agent's intermediate tool calls (causal-recall-graph-rfc.md:139-144). So the actor runs as a **top-level `claude -p`** (the calibration-run.js:84 adapter pattern), whose own tool log IS observable. This **sidesteps** the harness gap rather than building on an unprobed harness capability (ADR-0012 discipline).

**Layer 1 (trajectory)** generalizes SWE-agent's `(thought, action, observation, state)` shape to Claude Code's tool surface (Read/Edit/Bash/Grep/Agent — the action enum is **re-derived** from Claude Code's tool log; SWE-agent's open/scroll/edit names do NOT transfer, only the STRUCTURE does): one row per step `{step_idx, thought, tool_name, tool_input_digest, observation_digest, post_state}`, plus a derived process-graph carrying `phase ∈ {localization, editing, validation}`, `back_edge`, `loop_count`, `avg_loop_length`, `reached_validation_before_submit`.

**Recall-smell detector — combine TWO signals, never trajectory-shape alone.** The literature INVERTS the naive expectation: a chaotic high-loop trajectory correlates with FAILURE, not recall. A confident one-shot multi-file fix is a recall smell ONLY when paired with near-zero localization reads (the relevant files were never opened). Flag recall ONLY when `(low loop_count + reached resolution) AND (localization-effort signal shows relevant files unread)`. **Validate this black-box heuristic against the negative-control track BEFORE trusting it** — its white-box basis is an analogy, not an established black-box method.

**Layer 2 (friction signature)** — see §3.6 for the schema-contract correction.

**Clustering** uses the dual-representation pattern the substrate already uses: the closed-enum tuple `(friction_class, friction_phase, detection_leg)` is the deterministic dedup/cluster KEY (analogous to E11's dedup-by-`(persona, agent_id)`); a semantic embedding of the diagnostic free-text is the OPTIONAL depth layer, NEVER the key. The friction LABEL is assigned by an LLM-judge lens reusing the `scoreCalibration` seam verbatim, so the map is honestly reported as signal-with-error-bars **measured on this corpus** (§10), NOT ground truth.

### 3.4 Recall-graph populator

Populates the DESIGNED-NOT-BUILT causal-recall-graph node schema (causal-recall-graph-rfc.md:421-437, verified: `node_type ∈ {axiom, deterministic_theorem_memo, stochastic_sample, attestation}`, with NO weight/gradient field anywhere). **Correction per AF-MED-schema:** `worked_example_ref` is a **NEW additive field** this RFC proposes; the RFC node schema today carries `node_id/content_hash/surface/node_type` but no worked-example field. The honesty invariant is encoded structurally:

- An issue resolution is honestly labeled `node_type='stochastic_sample'` (or `attestation`). **No weight/gradient field exists in the schema, ever.**
- A worked example accretes ONLY via leg C, written AFTER A+B grade, carrying `{issue_id, repo, problem_statement_digest, candidate_patch_ref, behavioral_verdict, reference_divergence, friction_signature_ref, contamination_tier}`.
- Per AF-HIGH-reward-hack: **a behavioral-only pass NEVER populates memory** — leg B must affirmatively pass, else a gamed example poisons future retrieval.
- The only thing that improves over runs is **retrieval coverage**, never capability.

**Wording-audit gate** (reuse the forbidden-phrase grep pattern): flags any prose that says "learns"/"trains"/"improves over time" near a bootcamp metric — and (per AF-MED-schema) flags prose that asserts a proposed schema field as pre-existing.

### 3.5 ContainerAdapter scoping (read-mostly; the ADR-0012 boundary)

The ADR-0012 wall is why the behavioral leg is currently unbuildable: a PreToolUse hook's `updatedInput` is INERT on Agent/Task spawns (enforcement is STATIC); worktree isolation is same-uid, NOT a security boundary; the only execution path today (`calibration-run.js`) runs in an UNSANDBOXED, network-enabled, authed shell (verified header lines 8-11). The backtest is the safest place to birth the sandbox — but "safest" is **relative to the live path**, NOT a containment property of the stranger's code, which is full arbitrary-code-execution until a real hermetic boundary exists (per AF-LOW-emphasis).

**Interface (DEEP MODULE):** `run({repo, base_sha, candidate_patch, test_patch, test_ids}) → {fail_to_pass, pass_to_pass, full_suite, timed_out, killed_for_dos, containment_proven}`.

**Lifecycle:** clone@base_sha → apply candidate_patch → apply UNMODIFIED test_patch → run → DISCARD. Never writes a live repo, never touches the user's checked-out HEAD/working tree (the spike-S3 desync lesson), never opens egress beyond the pinned clone.

**v3.9 CLOSES (not "hardens later") the C1/C2/H1 residuals, in priority order set by blast radius** (per AF-CRITICAL-sandbox):

1. **Network namespace with DEFAULT-DENY egress** — a kernel-enforced drop, NOT the regex auditor (`network-egress-detect.js`, which by its own header is advisory + evadable). A leaked auth token or exfiltrated delta is the worst outcome.
2. **Filesystem isolation** via container/VM or at minimum overlay+chroot so absolute-path writes cannot escape `cwd` — the **H1 host-path-traversal residual must be CLOSED**, not deferred.
3. **Hard wall-clock + memory + pid-count rlimit with process-GROUP reaping** (C1/C2) — `kill -KILL` the whole group on timeout.

**Out of scope for v3.9 (YAGNI):** the R13 network-enforcer beyond default-deny, and any live-repo WRITE path (deferred to v4.x — ADR-0012 can't-wrap; the sandbox births read-mostly here, gains write-isolation later).

### 3.6 Friction signature is a NEW block, not an ADR-0015 enum extension (schema-contract correction)

Per AF-MED-schema (verified against 0015-...md:15,59): ADR-0015's `failed_criterion_id` enum is BOUND by `INV-FS-CriterionEnumMirrorsR9` to R9's normative leaf-criteria set. The friction classes (`wrong-file`, `incorrect-implementation`, …) are issue-resolution failure modes, NOT R9 criteria — appending them would **violate** the invariant and break the E2 consumer's closed-enum-switch. Therefore:

- `friction_class / friction_phase / detection_leg` live in a **NEW, SEPARATE block** (`resolution_friction`), NOT as members of `failed_criterion_id`. ADR-0015's R9-mirrored enum is untouched.
- The E2 firewall PATTERN (deterministic read of closed-enum structural fields, no NLP in the cluster key) is **reused**, applied to the new block's own closed enums.
- `friction_class` merges the SWE-agent 9 + SWE-EVO low/high split: `wrong-file | wrong-edit-location | incorrect-implementation | over-editing | incomplete-fix | instruction-misread | hallucinated-api | gave-up | cant-reproduce | ran-out-of-budget`. `friction_phase ∈ {localization, editing, validation}`. `detection_leg ∈ {behavioral, semantic-lens, reference-anchor}`. Free-form diagnostic fields (`expected/observed/human_message`) exist but the deterministic clusterer MUST NOT parse them.

### 3.7 Maintainer-rubric mining (high-control repos as a recorded maintainer rubric — per USER v3.9)

The accepted PR is not the only artifact a resolved high-control issue carries: the **maintainer review thread** records what a real maintainer flagged and made the contributor FIX before merge — **a retrospective record of the criteria this maintainer applied to a different (the human's) diff, a same-corpus human prior, NEVER an acceptance event for the plugin's patch** — and the repo's **CONTRIBUTING guide + CI gate config** encode the standing rules. This is a grading + diagnostic signal the design had not named. It is mined into a per-instance `rubric_refs` and used in exactly two contamination-INERT ways.

**The extraction contract (CRITICAL — closes the §3.7 oracle leak; per the final-review board).** A review thread is a **MIXED artifact**: on GitHub the maintainer's *criteria* sit right beside fix-bearing inline-quoted diffs and ` ```suggestion ` blocks that routinely contain the accepted fix verbatim. Feeding the raw thread to the BLIND leg-B judge would re-open the exact R4 oracle leak the PublicProblem/SealedOracle split was built to close. So the mining is structurally constrained — the §3.7 analog of R4's split:

1. **Criteria-only output.** A **sealed-side extractor** distills the thread into a CRITERIA-ONLY structured object — booleans/enums, never prose — e.g. `{requires_test, requires_changelog, public_api_touch_forbidden, flagged_friction_classes[]}`. ONLY that closed-shape object crosses to leg B; raw comment text and ` ```suggestion ` blocks NEVER do.
2. **Raw thread is sealed.** `review_thread_ref` and any verbatim comment text are **SealedOracle fields** (§8 R4), never PublicProblem fields — the actor never sees them, and the validator throws if they land public-side.
3. **Leak tripwire.** The extracted criteria object is scanned for substring overlap with the `accepted_diff` hunks above a token threshold; on a hit the instance's **rubric is FAILED** (dropped, not used). R4's negative-control tripwire guards the *actor* path; this one guards the new *grader-side enrichment* path.

Used two ways, both contamination-INERT:

- **Enriches LEG B (semantic).** The per-criterion rubric (§3.1) is augmented with the *actual* maintainer-flagged criteria for that repo ("did the PR add a test? update the changelog? avoid touching the public API?") — grading the generated patch against the bar a human reviewer actually applied, not a generic "is this good?". The enrichment is the criteria-only object from the extraction contract; it stays SOFT / advisory-capped and BLIND (the maintainer's *criteria*, never the accepted *diff* or any comment prose); it never becomes a behavioral pass gate.
- **Seeds the FRICTION MAP.** "Here is what real maintainers catch" becomes a **prior** on the `friction_class` taxonomy (§3.6) — e.g. a repo whose threads repeatedly flag over-editing raises the prior that an over-edit there is real friction. The seed is a prior for clustering, NEVER a deterministic cluster key.

**Four honesty guards** (this signal is seductive — it is a rubric, not an oracle): (a) the review thread is **NEVER fed to the actor or to leg B as prose** — only the criteria-only extracted object crosses to the (still-blind) judge per the extraction contract; the raw thread is grader-side and sealed; (b) it is contamination-inert for the *headline behavioral* metric (it touches legs B/C + the friction map, never leg A's pass@k); (c) the maintainer rubric is itself a same-corpus *human* signal with selection bias (what THIS maintainer happened to flag on THIS PR), reported as a prior-with-provenance, not ground truth — and because it is mined from the MOST training-represented repos, a strong leg-B-against-rubric score on a familiar repo carries BOTH same-family self-preference AND repo-convention familiarity, so rubric-enriched familiar-repo grades get **WIDER error bars** (cross-ref OQ-3); (d) **anti-poisoning** — CONTRIBUTING / CI config / review threads are attacker-controllable on external repos (the R2 breaker-poisoning input-trust class), so a mined prior may move a `friction_class` weight only within a cap, is **DISCARDED if it has zero same-repo observed-friction corroboration**, is excluded from the deterministic cluster key entirely, and is recorded with provenance so a poisoned prior is queryable-as-such. It strengthens "what good looks like"; it does NOT discharge the OQ-NS-6 boundary (§4) — a rubric-graded backtest still NARROWS, never HARDENS.

## 4. The OQ-NS-6 boundary (stated explicitly)

**What the bootcamp NARROWS (calibrates/filters):** the behavioral pass@k distribution (where the plugin CAN resolve — a floor, error-barred); the judge's own precision/recall; the friction map (where it hallucinates/misreads → feeds the breaker's denial-rate, which only halts = only narrows); the contamination tiers (which signal to trust); the recall graph (future search coverage — retrieval, NOT capability).

**What it EXPLICITLY does NOT HARDEN:** a backtest pass-rate is not a trust score (signal-with-error-bars). The judge has error bars; tests are a leaky oracle; the semantic leg is self-graded-optimistic. **The bootcamp does NOT discharge the "real non-dogfood verdict volume" HARDENING item** (the v3.8a-EC2 owed item per MEMORY) — that accrues ONLY at the v3.10+ live external beta where a world-anchored maintainer merges a PR. v3.9 routes AROUND the live surface deliberately. **Nothing here is model training** — no weights move, the schema has no weight field.

The reframe is correct and consistent with the law: **v3.9 = retrospective calibration bootcamp (births the read-mostly ContainerAdapter); v3.10+ = first live external beta (the HARDENING signal accrues THERE).**

## 5. Wave decomposition

| Wave | Scope | Gate |

## 6. Recalibration wiring (the structural fork)

The §0a.3.1 evidence-link gate (`validateRecordVerdictInput`, `verdict-attestation/store.js:152-153`, verified) requires an `agentId`; a detached backtest has none. There are exactly two paths:

- **PATH 1 (DEFAULT — the diagnostic home):** the recalibration aggregate writes to the free-standing `rung2-calibration-record/v1`-style JSON (calibration-run.js:113-147) — NO evidence-link precondition, and it carries the "single non-deterministic sample, NOT a trust anchor" framing baked in at calibration-run.js:127. **This is the v3.9 friction-map snapshot.**
- **PATH 2 (DEFERRED to v3.10):** routing per-attempt bootcamp verdicts through `recordVerdict` into reputation/breaker. **This RFC defers Path 2 entirely** (per AF-CRITICAL-breaker + AF-HIGH-OQNS6-blur). The friction map does NOT need the production reputation ledger to exist.

**Why Path 2 is deferred — two verified defects in the current substrate:**

1. **No enrichment gate on the breaker (CRITICAL).** The `verdict-fail` source's `list` fn (`circuit-breaker/project.js:121-127`) wraps `verdictStore.listVerdicts({filter: r => r.verdict === 'fail'})` in `dedupBySubject` (the v3.8b G1 dedup) but applies **NO enrichment filter** — verified; dedup is not enrichment, and the module header admits it does not re-verify at read. Contrast `projectReputation` (`reputation/project.js:90`, INV-W1 — a DIFFERENT file of the same basename) which DROPS `transaction_id == null` rows. So the breaker counts un-enriched, detached-backtest fails that reputation correctly excludes. The design's prior claim that "`store.js:152` STRUCTURALLY rejects detached-backtest verdicts" is FALSE for the breaker path: `verdict-attestation/store.js:152` only requires a non-empty `agentId` STRING, never that it resolve to a real spawn, and the breaker never re-checks enrichment at read.
2. **No persona namespace separation (CRITICAL).** `recordVerdict`'s only persona check is `nonEmptyString(subject.persona)` (`verdict-attestation/store.js:163`) — a bootcamp can record under any LIVE persona (`node-backend`, `architect`, `kernel-loom-integrator`). Because the corpus deliberately over-samples HARD tasks (high fail-rate), a bootcamp run could trip the LIVE `node-backend` breaker and silently degrade production routing.

**Pre-conditions before Path 2 may ship (v3.10):** (a) a first-class, non-optional `provenance: 'backtest'|'live'` on every verdict, with reputation + breaker bucketing by it so a backtest fail can NEVER sum with a live fail; (b) a reserved `bootcamp:`/`backtest:` persona namespace rejected-on-collision at the store boundary, OR a separate `bootcamp-fail` breaker source registered `starved:true` (like `negative-attestation`, `circuit-breaker/project.js:132-138`); (c) an enrichment gate on the `verdict-fail` `list` fn symmetric with reputation's INV-W1. A backtest reputation row, if ever written, is a NARROWING-only artifact that must remain queryable-as-excluded forever.

## 9. What the bootcamp is structurally BLIND to (negative attestation, per AF-HIGH-survivor)

The corpus is, by construction, already-resolved issues with an accepted PR and a merged test_patch. This selects on dimensions that diverge from the live path:

- **Survivor bias** — every corpus issue was solvable and was solved. A pass-rate over solved-issues-only cannot estimate behavior on the unsolvable/won't-fix/ambiguous tail (the negative-control track is the only probe, and its gating heuristic is itself unvalidated).
- **Cleaner-than-live problem statements** — a merged issue's text has typically been clarified through maintainer back-and-forth; first-contact live issue text is messier.
- **The entire maintainer-relationship / PR-review-survivability dimension** that gates the apex — PR scoping, commit hygiene, responsiveness to review comments, not-over-editing a stranger's repo, social trust. The bootcamp removes the human by design; it measures NOTHING about the dimension that actually gates an external merge. The friction enum captures `over-editing` (a code property) but has no axis for human-review-survivability.

Every bootcamp claim is therefore bounded to **"over the solved-issue, clarified-statement subspace"** — not a general competence estimate, and not a live-readiness estimate.

## 10. Statistical-provenance note (per AF-MED-stats + AF-MED-judge-error)

Every external percentage in this RFC (UTBoost ~15.7%; OpenAI 61.1%; file-path 76%/53%; full-suite 7.8%; code-rewrite RAD 0.46; GPT-4o labeler 87%) is a **literature prior on a DIFFERENT corpus/model/benchmark-version**, used to set hypotheses and gate thresholds — NOT a measured property of this harness. This harness's own false-positive / memorization-gap / labeler-agreement rates are OUTPUTS to be measured in W2-W4. Version-pin the SWE-bench-Verified snapshot wherever a number is load-bearing for a gate threshold. The 87%/13% labeler figure specifically is an out-of-distribution analogue (GPT-4o, SWE-agent trajectory labeling — not this Claude-family labeler, not this friction-class task); until the same-task agreement is measured, the labeler's error bar is **UNKNOWN, not 13%.**

## 12. Open Questions

1. OQ-1 (model cutoff has error bars): the temporal-holdout boundary itself is fuzzy — Anthropic distinguishes a training-data cutoff from an earlier reliable-knowledge cutoff (months apart), and the deployed model's exact training set is non-public. What concrete date(s) anchor the 'clean' vs 'grey' band for the actual deployed model (opus-4-8[1m] self-reports Jan-2026)? Without a pinned answer the 'clean' tier is itself error-barred.

2. OQ-2 (recall-smell heuristic validity): the black-box 'low-loop + reached-resolution + near-zero-localization-reads = recall' detector rests on a white-box (MemLens-class) analogy. The plan validates it against negative controls — but what false-positive/false-negative rate makes it trustworthy enough to gate friction-map weight, and what do we do if the negative-control validation shows it does NOT discriminate (does the contamination gate then drop the trajectory axis entirely)?

3. OQ-3 (self-preference ceiling): the semantic leg is same-family self-graded-optimistic; swap-test + identity-mask do NOT remove self-preference within a family. Is a different-family judge operationally available for the load-bearing subset, and if not, how wide do we declare the leg-B error bars — and does a self-graded leg B still satisfy R3's requirement that leg B affirmatively gate recall-graph population (a biased gate may pass gamed examples)?

4. OQ-4 (clean-tier large-change SOURCE — the one that gates floor *enforcement*): §3.2 now standardizes the axis at **>1hr**, reframes the floor as a two-part contract, and makes it **REPORTED not ENFORCED in W0** so W0 doesn't block — but enforcement (and the committed `N_clean_large`) still needs a source decision. The clean-tier (post-cutoff, novel) large-change pool is the thinnest slice: Verified is ~91% <1hr, and the split-by-role edit bars the famous well-curated repos from the clean headline. Do we source clean-tier large changes from SWE-bench-Live / SWE-rebench (fresher, but per-repo test strength unknown — a weaker behavioral oracle on exactly the instances that matter most), and what is the committed `N_clean_large` below which the headline reports `INSUFFICIENT-N`? This is a v3.9 mid-phase decision, not a W0 blocker.

5. OQ-5 (ContainerAdapter containment target): §3.5 demands default-deny egress + fs isolation + process-group reaping as v3.9 deliverables (not deferred). Is the chosen mechanism a full container/VM (heavier, but a real boundary) or overlay+chroot+namespaces (lighter, but the H1 absolute-path-escape + TOCTOU-open surface must be proven closed by the W1 spike)? The 'safest place to birth' thesis is conditional on this spike passing — what is the fallback if the lighter mechanism cannot prove containment (ship B+C-only, or block v3.9 on a heavier sandbox)?

6. OQ-6 (Path-2 provenance retrofit): deferring Path-2 to v3.10 requires a non-optional `provenance: backtest|live` field on every verdict + reputation/breaker bucketing by it. Does this retrofit force a schema/version bump on the verdict-attestation store (touching the frozen §0a.3.1 contract), and can the existing live verdict rows be back-filled as `provenance: live` without ambiguity (a row with no provenance field — is it legacy-live or unknown)?

7. OQ-7 (corpus contamination of the substrate itself): the North-Star charter holds 'corpus-never-feeds-SynthId' (the corpus must not contaminate the plugin's own recall/identity substrate). A bootcamp that writes worked examples from SWE-bench issues into the recall graph — does that risk the plugin later 'recalling' a benchmark solution on a live task and presenting memorized benchmark code as original work? What firewall keeps bootcamp-sourced worked examples queryable for DIAGNOSTICS but excluded from (or clearly provenance-tagged in) live-task retrieval?

8. OQ-8 (what discharges live-readiness): §7 EC5 states v3.9 does NOT establish live-readiness and does NOT discharge the 'real non-dogfood verdict volume' item. What is the minimal v3.10 entry gate the bootcamp output IS allowed to inform (e.g., 'do not attempt live on repos where the friction map shows >X% hallucinated-api')? I.e., the bootcamp narrows the live action space — but the precise narrowing contract (which friction signals block which live attempts) is unspecified and should be its own design before v3.10.
