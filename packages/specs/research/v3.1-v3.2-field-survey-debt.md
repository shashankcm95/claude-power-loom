# Field-Survey Debt — Category B (post-v5.4)

**Created**: 2026-05-26 (v5.4 BLUEPRINT-LOCK day)
**Source**: v5.4 Category-A absorption (deep-read of 8 academic papers + 7 industry artifacts). Category-A landed in `swarm/thoughts/shared/design/v3.3-substrate-synthesis.md` as positioning fixes. **Category-B is the rest** — real design gaps surfaced by the field survey that don't block Phase 0 but need work in v3.1 / v3.2 release windows.
**Status**: backlog-tracking only. Each item has a target release and effort estimate. No work begins on these until v3.0-alpha implementation is shipping.

## Why this lives separately from the synthesis doc

The v5.2 GPT review's strongest single recommendation was: *"You are ready to stop architecture expansion and begin substrate hardening."* Folding all 10 items below into v5.4 would violate that explicitly. The synthesis doc captures the architecture we will build; this doc captures the field-validated improvements queued for the natural release windows once that architecture is shipping.

## Items

### B1 — Σ session-level sensitivity state (vs Policies-on-Paths arxiv 2603.16586)

**Source**: Kaptein/Khan/Podstavnychy 4-tuple formalization `(A, P_i, s*, Σ)` — `Σ` is per-organization accumulating state (max sensitivity touched, active info-barriers, delegation records).

**Gap**: Power Loom captures policy at spawn-time (K2 envelope) and at write-time (K14). Neither captures *session-accumulating* sensitivity. **Multi-step exfiltration where each individual write is innocuous but the trajectory leaks is invisible to K14.** Example: agent reads `/secrets/key.pem` (allowed read), then writes `~/notes.md` containing the key (allowed write to allowed path) — both K14 checks pass; the leak is in the trajectory not in any single step.

**Proposed mechanism**: extend K2 envelope with a `session_state.sensitivity_tags: string[]` field. Tags get propagated forward when an agent reads a flagged path. K9 promotion check fails if `session_state.sensitivity_tags` is non-empty AND the delta touches a non-flagged path.

**Target release**: v3.1 (alongside K6/K8 capability work).
**Effort**: M (~5-7h; schema-additive to K2 + ~30 LoC trigger logic in K14 + ~20 LoC gate logic in K9).
**Pillar served**: Pillar 2 (Byzantine Treatment of LLM — outputs AND inputs).

### B2 — K2 envelope signing (vs SAGA C3 self-replication attack, arxiv 2504.21034)

**Source**: SAGA's threat model C3 — "unregistered child agents via self-replication sharing keys with parent."

**Gap**: K2 spawn-records are plain JSON. Any tool with FS write access can construct a fake envelope and submit it as a spawn request. K1 isolation contains the blast but doesn't prevent unauthorized capability use *within* the worktree.

**Proposed mechanism**: Kernel-signed envelopes. K2 emits envelope + HMAC-SHA256 over `(spawn_id, capability_set, policy_version, parent_id)` keyed by a Kernel secret. K6/K8 verify the signature before honoring the declared capability subset.

**Target release**: v3.1 (alongside K6/K8 — natural pairing).
**Effort**: S (~3-4h; HMAC is one-liner via Node crypto; key management is the harder bit — recommend reading from `~/.claude/loom-kernel.key` with 0600 perms, fail-open with warning if missing for dev ergonomics).
**Pillar served**: Pillar 2.

### B3 — A6 reputation observation-only ADR (vs Trust-Vulnerability Paradox arxiv 2510.18563)

**Source**: Xu et al. empirically demonstrate that increasing inter-agent trust monotonically increases information leakage (OER + AD metrics) *regardless of implementation*. The paradox is structural, not a function of how trust is measured.

**Gap**: v5.4 doesn't explicitly forbid A6 reputation from feeding back into K8 capability scope. If "this persona has 95% success rate → grant broader write access" is ever implemented, we structurally reproduce the OER/AD failure mode the paper demonstrates.

**Proposed mechanism**: ADR-0009 explicit constraint: A6 reputation is **observational and ranking-only**. It MAY reorder candidate persona lists. It MAY trigger `paired-with` recommendations. It MAY NOT widen K6/K8 capability scope. Tested by an INV-A6-NonAuthorizing invariant in §6.9 (NEW): for any (persona, task_type, capability_set), capability_set is independent of A6 reputation score.

**Target release**: v3.3 (with the rest of A6 work) but write the ADR NOW so v3.0-alpha doesn't accidentally lay groundwork for the wrong shape.
**Effort**: XS (ADR-only, ~1h).
**Pillar served**: Pillar 2.

### B4 — Goal-conditioned trajectory drift detection (vs MI9 arxiv 2508.03858)

**Source**: MI9's drift detection runs Jensen-Shannon divergence on event-sequence distributions per goal-context vs rolling baseline (+ Mann-Whitney U on continuous metrics like latency, token spend, tool-call cardinality).

**Gap**: A6 tracks per-persona outcomes (success/fail counters). We don't track *how* a persona accomplishes things. If `omar.ml-engineer` typically reads 5-10 files and runs `pytest` once, and in this session reads 47 files and runs `rm -rf` adjacent to test paths, that's a real drift signal independent of whether tests pass.

**Proposed mechanism**: K2.c per-tool-call observability already records tool-call sequences. Add a histogram aggregator per `(persona, task_class)` with rolling N=50 window. Emit advisory finding (NOT gating per A3a/A3b split) when JSD against baseline crosses threshold. Cheap variant: tool-call cardinality + tool-call type distribution; no trajectory ordering.

**Target release**: v3.3 (Evolution Lab; this is fundamentally a reputation/learning extension).
**Effort**: M (~6-8h; statistics library + per-persona JSON aggregator + advisory hook).
**Pillar served**: Pillar 3 (deterministic/auditable execution — drift signal is *advisory*, not gating, preserving the split).

### B5 — Multi-agent correlation accounting (vs TRiSM Category 3 arxiv 2506.04133)

**Source**: TRiSM Cat 3 — agent collusion / mode collapse. Two `code-reviewer` instances reviewing the same diff with same persona+model+similar prompt context are NOT statistically independent draws; they're highly correlated.

**Gap**: Power Loom's self-improve loop counts approvals as if they were independent (e.g., "5 approvals → auto-graduate to low-risk rule"). HETS pair-review is treated as redundant when it's correlated. Our claim that paired-review provides "convergence" is empirically questionable.

**Proposed mechanism**: Two parts.
(1) **Honesty fix**: rename approval counts to "correlated-approval counts" in self-improve UI; document the correlation explicitly.
(2) **Independence-promoting fix**: HETS pair-review template requires different personas (architect + code-reviewer; not 2× code-reviewer) and SHOULD use different model temperature or model family if available. Add an `independence_dimensions: ["persona", "temperature", "model"]` field to HETS spawn config; track how many dimensions differ.

**Target release**: v3.2 (HETS work) and v3.4 (self-improve auto-graduation review).
**Effort**: S (~3-4h docs + ~2-3h HETS schema additive).
**Pillar served**: Pillar 4 (TDD / role-separation — correlation undermines the role-separation claim).

### B6 — AGENTS.md interop layer (vs Linux Foundation stewardship + 60K+ project adoption)

**Source**: AGENTS.md is now Linux-Foundation-stewarded, 60K+ projects including Apache Airflow, Temporal SDK, Cursor, Copilot, Devin, Cognition. Power Loom currently bets on CLAUDE.md which is Anthropic-specific.

**Gap**: Real interop debt. Users coming from other agent systems will expect their AGENTS.md to work.

**Proposed mechanism**: K8 capability injection reads both AGENTS.md (cross-vendor) and CLAUDE.md (Anthropic-specific) with precedence: explicit prompt > AGENTS.md nearest > CLAUDE.md nearest > defaults. Proximity-based directory traversal as AGENTS.md spec defines. Document the precedence in v3.1 release notes.

**Target release**: v3.1 (alongside K8 injection work).
**Effort**: S (~3-4h; both formats are markdown; precedence logic is straightforward).
**Pillar served**: Pillar 2 (input handling) + interop / non-pillar (community standard).

### B7 — K9 cherry-pick latency benchmarks (vs DeltaBox arxiv 2605.22781)

**Source**: DeltaBox publishes 14ms checkpoint / 5ms restore with detailed methodology on SWE-bench Verified. We have not measured K9 cherry-pick latency.

**Gap**: "How fast is K9 promote-deltas?" is the obvious first question; we cannot answer. If K9 cherry-pick is 100ms-1s as estimated, we should know this and be honest about the perf gap vs DeltaBox-style OS-overlayfs implementations.

**Proposed mechanism**: benchmark suite under `bench/k9-latency/` measuring cherry-pick wall-clock across (small/medium/large repo) × (clean/dirty working tree) × (number of files in delta). Match DeltaBox's methodology so numbers are comparable. Publish in v3.0-alpha release notes.

**Target release**: v3.0-alpha (alongside K9 implementation; benchmarks should land WITH K9, not after).
**Effort**: M (~5-7h; benchmark harness + 3 representative repos + result tables).
**Pillar served**: Pillar 1 (empirical grounding of the substrate's transactional claim).

### B8 — Praetorian 5-role HETS template (vs 39-agent Praetorian platform)

**Source**: Praetorian's "Deterministic AI Orchestration" ships a battle-tested 5-role assembly line per domain: Lead → Developer → Reviewer → Test Lead → Tester.

**Gap**: HETS currently has the architect + code-reviewer pair as the documented template. The 5-role pattern is more crisp and has battle-tested provenance (their 39-agent system).

**Proposed mechanism**: codify (Lead/Developer/Reviewer/Test-Lead/Tester) as a documented HETS template alongside the existing pair pattern. Add `skills/agent-team/templates/5-role-per-domain.md`.

**Target release**: v3.2 (decomposition + HETS work).
**Effort**: S (~2-3h documentation; no code changes).
**Pillar served**: Pillar 4.

### B9 — Concurrency-strategy first-class naming (vs LangGraph 4-mode model)

**Source**: LangGraph names four concurrency strategies (`enqueue`/`reject`/`interrupt`/`rollback`). Power Loom has K13 "serial-only" but does not name the rejected alternatives.

**Gap**: Users coming from LangGraph won't know what K13 implies operationally. Documentation has no first-class vocabulary for the strategies we DIDN'T pick.

**Proposed mechanism**: §8 (Phase 2 Capacity Envelope) addendum naming the four strategies we considered: `serial-only` (K13 default; what we ship), `parallel-isolated-per-worktree` (theoretically sound but unimplemented; v3.5+ candidate), `parallel-unsafe` (rejected — A1+A7 violation), `interrupt` (rejected — Anthropic primitive limitation per Wave -1).

**Target release**: v3.0-alpha (documentation only; ships with the existing K13 spec).
**Effort**: XS (~1h documentation).
**Pillar served**: Pillar 3 (auditable choices).

### B10 — Policy DSL pluggability decision (vs Microsoft AGT PolicyProviderInterface)

**Source**: Microsoft Agent Governance Toolkit supports YAML / Rego (via Janssen) / Cedar / Python policies natively via `PolicyProviderInterface` with 992 conformance tests across 20+ framework adapters.

**Gap**: Power Loom policies are hook scripts (bash/node). Users with existing OPA bundles or Cedar policies have no story.

**Decision needed BEFORE v3.1**: either commit to a `PolicyProviderInterface`-shape with Rego/Cedar adapter slots, or explicitly scope it out as "policy stays in hook scripts; bring-your-own-DSL via Bash invocation." Both are defensible. AGT has set the user expectation bar; we either match it or position around it.

**Proposed decision**: scope out for v3.1; revisit at v3.5+ when ContainerAdapter work happens (natural integration point). Hook scripts remain the canonical policy substrate. Document the decision in ADR-0010.

**Target release**: v3.1 (ADR only); v3.5+ for actual implementation if revisited.
**Effort**: S (~2h ADR + decision call).
**Pillar served**: Non-pillar (developer-experience / interop).

### B11 — v3.3 Evolution Lab MUST consume the learnings log + retire the prior generation

**Source**: empirical audit on 2026-05-26 of `~/.claude/checkpoints/self-improve-pending.json` — the current self-improve loop (Stop hook + UserPromptSubmit hook + PreCompact hook auto-store) produced **0 actionable promotions across 47 candidates in 21 days** (91.5% dismissal rate; 100% of skill-candidates dismissed). Full diagnosis at `swarm/thoughts/shared/learnings/2026-05-26-self-improve-loop-empirically-broken.md`.

**Gap**: the prior-generation self-improve loop asked "which files / commands were observed most often?" — a frequency counter, not a learner. The v5.4 Evolution Lab spec (E1/E2/E3/E4) asks the right question instead ("which spawn outcomes failed; which personas succeed at which task classes") but is not yet built. The empirical 0-signal corpus is direct evidence that frequency-based auto-graduation is the wrong category of signal. v3.3 design must NOT silently inherit the prior generation's hook substrate.

**Proposed mechanism**:
1. v3.3 design phase reads `swarm/thoughts/shared/learnings/` as empirical input — every entry classified into E1 (negative-attestation candidate) / E2 (policy-extraction candidate) / E4 (reputation-shaping candidate) / N/A. If a learning doesn't fit cleanly into any E-primitive, that's a schema gap to address.
2. v3.3 spec must EXPLICITLY name the deprecation of the prior-gen hooks (Stop / UserPromptSubmit / PreCompact self-improve scan) — and provide a migration path or formal retirement.
3. E1 schema MUST encode (a) what was attempted, (b) why it failed, (c) typed failure category, (d) suggested constraint. NOT "file X observed N times."
4. Auto-promotion thresholds (5+/10+ pattern from prior gen) are suspect by default — v3.3 must require contract-verified or human-in-loop promotion paths.

**Target release**: v3.3 (Evolution Lab design phase; the retirement is part of the spec, not a separate phase).
**Effort**: incorporated into v3.3 design budget (~14-20h per §6.4); no separate effort needed.
**Pillar served**: P3 (Deterministic/Auditable Execution) — the prior-gen loop violated this by producing noise that *looked* auditable but carried no signal.

**Why this is in the backlog and not a v5.4 amendment**: the diagnosis is empirical observation about the *prior generation* of self-improvement. The v5.4 architecture (E1-E4) is conceptually correct; what we're flagging is that the prior-gen hooks should not be silently inherited or treated as "already shipped." This is a v3.3 design-time consideration, not an architecture change.

## Items deliberately NOT included

These were considered during the Category-A absorption and explicitly excluded:

- **Output-scoring layer (Cleanlab-style)** — Power Loom is in Work-Bench Execute+Constrain, not the reliability stack. Out of scope.
- **Durable execution (Temporal-style)** — different layer (workflow continuity through process death, not effect containment). v5.4 §0 explicitly positions against this.
- **Container/microVM isolation (E2B-style)** — already deferred to v3.5+ ContainerAdapter per §10c. Field-survey doesn't change the existing deferral.
- **OS-overlayfs K9 backend (DeltaBox-style)** — §10a.1 acknowledges DeltaBox; backend choice deferred to "future v3.x if cherry-pick perf becomes a constraint." Not on the critical path.
- **APG (Action Provenance Graph) cryptographic provenance (AGENTSAFE-style)** — covered by B2 envelope signing + existing OQ-3 hash-chained-tamper-evidence open question. No new line item needed.
- **Anything in the Trust-Vulnerability Paradox paper beyond B3** — B3 captures the load-bearing fix (ADR forbidding reputation→capability widening). The paper's k-of-n threshold-reconstruction and Guardian-Agent mitigations are tactics; we don't need them if we don't reproduce the paradox in the first place.
