## Design

### `kernel/_lib/recency-decay.js` (the extraction — activates the dark function)
- The leaf exports: **`computeRecencyDecayAt(history, nowMs)`** (the core — uses the injected `nowMs`, NOT `Date.now()`) + **`computeRecencyDecay(history)`** = `computeRecencyDecayAt(history, Date.now())` (the back-compat wrapper, observable behavior byte-identical to the original) + `RECENCY_HALF_LIFE_DAYS`. Pure, no deps. `trust-scoring.js` does `const { computeRecencyDecay, RECENCY_HALF_LIFE_DAYS } = require('../../../kernel/_lib/recency-decay')` and re-exports both — its public surface is unchanged (existing `_h70-test` + the `registry.js`/`agent-identity.js` re-export chain see the same fn). A test asserts `trustScoring.computeRecencyDecay === leaf.computeRecencyDecay` (no drift). **(verify-plan HIGH-2)** E4 calls the **injectable** `computeRecencyDecayAt(history, nowMs)` so the projection is genuinely deterministic under a pinned `now`; the runtime keeps the `Date.now()`-wrapping `computeRecencyDecay`. **(verify-plan MEDIUM-1)** `RECENCY_HALF_LIFE_DAYS` (decay-weight) and the store's `DEFAULT_EXPIRES_AFTER_DAYS` (ledger size-bound) are **coincidentally-equal (both 30), NOT the-same-knob** — a one-line comment at each site prevents a future "DRY" merge.

### `packages/lab/reputation/project.js` (E4 — the pure projection)
- `projectReputation({ now?, halfLifeDays? })` → reads `verdictStore.listVerdicts({now})`, **filters to enriched only** (`(r.evidence_refs || {}).transaction_id != null` — INV-W1, **guarded** per verify-plan MEDIUM-3: a record with a missing `evidence_refs` must NOT throw the group-by — it's fail-soft-counted as excluded), groups by `subject.persona`:
  ```
  { generated_at, source:'verdict-attestation', label:'advisory-verdict distribution over kernel-attested spawns — NOT a quality score',
    excluded_unenriched: <count>,                         // GLOBAL INV-W1 transparency (links not yet resolved)
    personas: [ { persona, total, distinct_spawns,        // distinct evidence_refs.agent_id = delta-bearing (work-producing) spawns
                  pending_enrichment,                      // PER-PERSONA unenriched count (verify-plan MEDIUM-2 — no silent omission)
                  by_verdict:{pass,partial,fail},          // partial stays a DISTINCT bucket BY DESIGN (collapsing = the scalar-score move §0a.3.1 forbids — verify-plan LOW-2)
                  by_verifier_kind:{ <kind>:{pass,partial,fail}, … },   // R1 stratification
                  recency_decay_factor,                   // computeRecencyDecayAt(records.map(r=>({ts:r.recorded_at})), nowMs) — the ADAPTER (verify-plan HIGH-1)
                  last_seen } … ] }
  ```
- **(verify-plan HIGH-1) the adapter is load-bearing**: the leaf filters on `entry.ts`; the store emits `recorded_at`. E4 MUST map `records.map(r => ({ ts: r.recorded_at }))` before calling the leaf — else recency is silently `null` for every persona (the "activation activates nothing" trap). Asserted non-vacuously in the test (recent > old).
- **Pure**: one ledger read, no writes, no other I/O; deterministic given (ledger, now) — recency via the **injectable** `computeRecencyDecayAt(…, nowMs)` (HIGH-2). Imports `../verdict-attestation/store` (lab→lab) + `kernel/_lib/recency-decay` (lab→kernel). Imports **no runtime identity STATE** (containment test enforces — like Wave 1).
- **Honest semantics**: a *distribution of advisory judgments grounded on attested work*, not a scalar "score". `distinct_spawns` is the honest denominator (one spawn judged by 3 reviewers = 3 attestations but 1 spawn). Per-persona `pending_enrichment` makes real-but-unlinked work visible (not silently dropped).

### Performance boundary (answers the standing question)
E4 is a **pure projection** (v6:509 deterministic theorem) — bounded by the Wave-1 ledger caps (`MAX_LEDGER_RECORDS` 10k + `MAX_FIELD_LEN` 512); a group-by over ≤10k records is single-digit ms. In **W2 it is off the spawn path entirely** (no event-loop interaction — the Lab is synchronous CLI/module code). The spawn-path meeting point is **W3 (A6)**: the spawn-init capture hook (`spawn-record.js`, <50ms p99 budget) reads a **pre-computed** snapshot (O(1)), never a live projection; per v6:408 sibling-wave semantics, all sibling spawns in a dispatch wave share ONE snapshot (captured at wave-open) — the burst is batched **structurally**, not via async. v6:410 confirms: reputation feeds routing "asynchronously" = out-of-band/decoupled, never a live gating read. The W3 A6 seam will be **probed firsthand before build** (no code seam exists yet — ADR-0012 discipline).
