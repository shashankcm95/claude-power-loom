## H-1 — the design decision: Option (b), reject-on-mislabel (id basis UNCHANGED)

`attestation_id = sha256(canonicalJsonSerialize([agentId, verifier.identity, verifier.kind, verdict]))`
(`store.js:174`) omits `subject.persona`. Two `recordVerdict` calls with the same agentId + verifier-tuple
but different persona collapse to one id → the 2nd silently dedups away.

**Decision: (b) reject-on-mislabel. Reject (a).** Rationale:
- `agentId` uniquely pins ONE subject spawn = ONE persona. "Same agentId, different persona" is a
  **false state** (a mislabel), never legitimate.
- **(a) add `subject.persona` to the id basis is WORSE** — it makes the two-persona records *accumulate*
  (distinct ids), **encoding + amplifying** the false state into E4 (one spawn double-counted under two
  personas). It hides the mislabel instead of catching it.
- **(b) reject** catches the mislabel at the boundary (fail-loud), enforces agentId→one-persona, and
  keeps the id basis UNCHANGED (so the W1 dedup/MEDIUM-3 tests stay green — less churn, no E4 impact).
- `subject.persona` is functionally dependent on `agentId` (1:1), so it does NOT belong in the dedup
  key; the reject-check *enforces* that functional dependency.

**Mechanism:** under the lock (after `readLedger` + prune → `live`), before the dedup check: if a live
record has `evidence_refs.agent_id === o.agentId` AND `subject.persona !== o.subject.persona` → **throw**
(fail-loud; `withLabLock`'s `finally` releases the lock on throw — safe). Message names both personas +
the agentId. Checked against LIVE (non-expired) records only (an aged-out persona is forgotten — fine).

## M-1 — store-boundary input screening (defense-in-depth)

`recordVerdict` validates only `nonEmptyString` + `MAX_FIELD_LEN`. Add, in the pre-flight block (before
the lock):
- **`isSafePathSegment(o.agentId)`** (import from `kernel/_lib/path-canonicalize`) → reject a path-hostile
  agentId (`../../etc`, separators, NUL). The store's OWN write path never path-joins agentId (fixed
  `LEDGER_PATH`), so this is not a store-write CWE-22 fix — it makes the store the FIRST screening layer
  for the **enrich READ path** (`enrich-from-spawn-state` path-joins the stored `agent_id`) + any future
  consumer that joins it. Today the enricher's pre-join `isSafePathSegment` is the SOLE layer for that
  read path; this adds the store as a defense-in-depth first layer (architect FLAG — scoped to read path).
- **`hasControlChars` char-code screen** (≤ 0x1f OR === 0x7f) on agentId + verifier.identity +
  verifier.kind + subject.persona → reject control chars (NUL/newline/CR/tab/DEL) that would corrupt the
  JSONL ledger / E4 grouping keys. **Char-code loop, NOT a control-char regex** (a `/[\x00-\x1f]/`-class
  regex trips eslint `no-control-regex` — learned in W6 when the same screen was reverted; this is its
  proper home).
  - agentId needs BOTH (`isSafePathSegment` misses newline/CR/tab — verified `path-canonicalize.js:197-205`
    rejects only sep/`..`/NUL; a newline in agentId → JSONL-line-injection).
