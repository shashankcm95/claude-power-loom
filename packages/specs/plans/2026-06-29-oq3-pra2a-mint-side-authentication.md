## Context

W3 persisted a broker-signed `(approval_hash, lesson_commitment)` binding onto the chain but **no code crypto-verifies it** — the stores hold no verify key by design (integrity-at-rest, provenance-at-PR-A2). `world-anchor-mint.js` today reads only `record.approval_hash` + `record.merge_commit_sha` and joins the lesson by `lesson_signature` **alone** (never the body). PR-A2a makes the world-anchor mint the **first + only crypto-verifier** of the persisted broker_sig, and strengthens the lesson join from signature-only to **body-binding** (re-derive `computeLessonCommitment` === the recorded commitment). This NARROWS #273 on the inbound side (a same-uid forger lacking the broker key can no longer bind an attacker lesson to a genuine `approval_hash`); it does **not** close it (the same-uid co-forge of a self-consistent lab body + the deployment of a real cross-uid broker remain — OQ-NS-6).

## Goal

At `mintFromMergeOutcome`, add a fail-closed authentication layer (RFC §5.5 steps 1+2), engaged when a custody verify-key is provided:

1. **Verify** (step 1): reconstruct `basis = approvalSigBasis({ hash: record.approval_hash, approvedAt: record.approvedAt, nonce: record.nonce, key_id: record.key_id, lesson_commitment: record.lesson_commitment })` and `verifyRecordSig(basis, record.broker_sig, { publicKeyPem: verifyKeyPem, allowEnvFallback: false })` — fail-closed. Runs **before** lesson resolution (needs only the record bundle).
2. **Re-derive** (step 2): `computeLessonCommitment({ lesson_signature: lesson.lesson_signature, lesson_body: lesson.lesson_body })` over the **resolved** floor lesson, assert `=== record.lesson_commitment`. Runs **after** lesson resolution (needs the body). The mint's own join is by `lesson_signature` alone (`world-anchor-mint.js:217-226`); the commitment binds the **body**, so step 2 is strictly stronger and catches a `lesson_body` reword under a kept signature.

Both fail-closed + **observable** (new `mint_reason` tokens + emit, per `security.md`). The mint stays **TOTAL** (never throws): `approvalSigBasis` and `computeLessonCommitment` THROW on bad input, so every call is wrapped → a throw becomes a fail-closed reason, never a crash (the existing `lesson-build-failed` try/catch is the template).

## Engagement model (the load-bearing design decision — VERIFY-board stress point #1)

Authentication **engages on verify-key presence**, mirroring the established subsystem precedent (`edgeSigner` undefined → UNSIGNED edge; `authenticatedWorldAnchorIds` no-key → empty set, fail-closed at the consumer):

- **Key provided** (`opts.verifyKeyPem` present — tests + the future armed/deployed wire): authentication is **MANDATORY + fail-closed**. A bad sig → refuse `broker-sig-invalid`; a commitment mismatch → refuse `lesson-commitment-mismatch`; a throw in either primitive → refuse `auth-verify-error`. No node/edge written (the verify runs before the node mint).
- **No key provided** (today's un-armed production — production passes none): the mint proceeds **un-authenticated SHADOW** (UNSIGNED node/edge, exactly as today) BUT emits an observable `mint-unauthenticated` alert — the skip is **never silent** (`security.md`: a fail-closed decision must be observable; here the non-engaged path is observable too, so no silent bypass).

**Why engagement-on-presence is SHADOW-safe (not fail-open):** the mint OUTPUT is UNSIGNED + SHADOW and gates nothing — `LIVE_SOURCES` is empty and the eventual consumer (PR-B) fail-closes via `authenticatedWorldAnchorIds`, which requires a verify key, which requires the mint to have been authenticated **and** signed (PR-A2b). So an un-authenticated mint produces an edge no live consumer admits. The fail-closed boundary correctly lives at the CONSUMER; PR-A2a is the PRODUCER-side enrichment. **PR-B residual (named):** when a consumer goes live, the verify-key MUST become mandatory (refuse-on-absent), because at that point the un-authenticated path WOULD be a fail-open. PR-A2a does not make it mandatory because no consumer is live.

## OQ3-5 — the no-commitment (static grandfather) floor (RFC §6 named residual)

A record from the static seed (Branch A, the spec-kitty grandfather) carries `lesson_commitment === ''` (it predates the capture lane). When engaged AND `record.lesson_commitment === ''`:

- The **broker_sig verify still runs** over the `''`-commitment basis (proving a legitimate approval bound this merge — the `''` basis is what the broker signed for a no-lesson emission).
- The **commitment re-derive is SKIPPED** (there is no committed lesson to bind to; `computeLessonCommitment` would THROW on the body-vs-`''` mismatch anyway), and emits an observable `mint-uncommitted-lesson`.

This ADMITS the static grandfather on its broker-sig provenance without a body commitment. The final policy (commitment-REQUIRED vs admit-on-human-vetted-provenance) is **OQ3-5, deferred to the PR-B HARDEN gate** — PR-A2a's choice (admit + observe) preserves the current grandfather mint and is named, not silent.
