## 4. Boundary decisions (surfaced for architect VERIFY + USER ratification)

### B1 — Module-level vs spawn-level producer (THE load-bearing one)

v3.3 builds E1–E4 to consume `failure_signature` records. **Where do they come from at runtime?**

- **Today (module-level):** `decompose-run`'s `rejected[]` + R11/`spawn-verify` emit real `failure_signature`s
  when *called* (CLI / module / fixture). This is a real, unit-testable, live producer (P3).
- **Deferred (spawn-level):** a real persona spawn that decomposes its task and fires the verify tier
  end-to-end. The tier is hook-inert (P8); wiring it is the **orchestration design** (how/when a persona
  decomposes) — that is E12 Pattern-B / a separate activation concern, **NOT** in the E1–E4 charter.

**Proposed resolution:** v3.3 builds + tests E1–E4 against the **module-level** producer (fixtures + the
`decompose-run` rejected[] path). The spawn-level end-to-end stays **DEFERRED**. This is **honest and
RFC-consistent**: the RFC acceptance criterion is literally *"5 deliberately-failing **test fixtures**
produce structured negative-attestation"* — fixture-level by design. Same "built + validated, activation
gated separately" posture as v3.2's tier. The plan must NOT claim spawn-level liveness.

**Not ADR-0012-class** (architect VERIFY F1): ADR-0012's trap was building enforcement on a harness
mechanism *that does not exist*. Here the mechanism EXISTS and is tested (`verifySpawn` → `buildFailureSignature`
is real code with real fixtures); only the *trigger* (a persona spawn that calls `decompose-run`) is deferred.
A wiring gap, not a capability void — the consumed `failure_signature` contract is identical today and when
the trigger lands.

**The double-inert honesty boundary (carry into every wave + the phase-close):** v3.3 stacks a **second
inert layer on the first**. The v3.2 tier is LIVE-but-INERT (hook-unwired, P8); v3.3's Lab layer consumes it
but is itself inert-on-inert until the v3.3+ spawn-level wiring lands. So the honest claim is: **v3.3 proves
the Lab tier *composes and runs as code* over fixture-produced signatures — it does NOT prove the Lab tier
observes any real persona failure, because nothing in production emits a `failure_signature` yet.** This is the
exact analog of the v3.2-close "MODULE composition, NOT Agent-spawn-driven" boundary. The REAL un-darkening
(spawn-level wiring) is v3.3+. Never let "failure becomes capital" be read as a live production capability.

### B2 — K2.c (per-tool-call observability) in v3.3, or deferred?

RFC §6.4 charters **only E1–E4**. K2.c is a v3.1 carry-list item ("→ v3.3 with A6, its consumer").
**Proposed: DEFER K2.c.** **Corrected rationale (architect VERIFY F2):** the right reason is NOT that
`failed_criterion_id` *substitutes* for per-tool-call data — it's that **K2.c's signal is orthogonal.**
E4's reputation signal is *"persona P fails criterion X at rate R,"* which comes entirely from the
attestation records (E1), not from tool-call telemetry. K2.c would add *a different* reputation signal
(per-tool-call behavior) that v3.3 simply does not need. Deferring it keeps v3.3 a pure E1–E4 release.
(Revisit only if a concrete need surfaces.)

## 4.5 Cumulative substrate-coherence pass (USER-requested; the reshape driver)

A cumulative-coherence architect pass (does kernel→runtime→Lab compose, given the *built* reality — not v6's
blueprint?) returned **COHERENT-WITH-RESHAPINGS**. The substrate composes (every "live" claim verified
firsthand against `hooks.json` + code), but two findings reshaped v3.3:

- **B4 — defer the volume-amplifiers (the headline; USER-ratified).** E2/E3/E4 turn *many* observations into
  trends/policies/rankings. The producer (one persona decomposing on rare ≥3-leaf tasks) yields single-digit
  attestations; the kernel's *high-volume* signal (every spawn close) is shadow-only AND the Lab doesn't
  consume it. Building amplifiers over a near-empty producer is YAGNI/speculative-generality. → **v3.3 = Wave 0
  + E1; E2/E3/E4 → v3.4** with a volume trigger, designed against real attestation distributions.
- **R1 — E1 must surface `verifier_kind`, not flatten it (must-fix, load-bearing).** R9 is a
  **declaration-conformance** gate, not a measured-property gate (`leaf-criteria.js:13-23`): only `tdd` leaves
  get a real R12 test-run; `spec-driven`/`exploratory` leaves are checked purely structurally. So a
  `failure_signature` for a non-tdd leaf means *"the persona mis-DECLARED its leaf"* (forgot `output_schema`),
  **not** *"the persona did bad work."* Conflating them would violate Axiom 1 ("agent self-reports aren't
  authoritative"). **E1's record must carry `verifier_kind` as a first-class stratification field** (it's
  already in the 8-field signature — E1 just stores + exposes it) so the future E4 weighs `test-run` (measured)
  separately from `structural`/`predicate` (declared). Folds into E1's frozen contract NOW (Wave 1).
- **Miss-2 (honesty):** `semantically-cohesive` is advisory-only → never produces a `failure_signature`, so the
  Lab's entire input vocabulary is the 4 *structural* (mechanical declaration) enums. "Failure becomes capital"
  can only ever mean *"which mechanical leaf-declaration gate a persona trips"* — state this at phase-close.

**Earlier-layer notes (v3.4 framing — NOT v3.3 blockers):**
- **Kernel shadow-permanence:** the resolver has *never promoted in anger* (`spawn-close-resolver.js:719`,
  enforce-path flag-gated OFF). Before a Lab *learns from* the substrate, decide whether the kernel will ever
  leave shadow — else it learns about a hypothetical.
- **Dormant decay:** `computeRecencyDecay` is built-but-OBSERVABLE-ONLY in the *existing* reputation system
  (`trust-scoring.js:145-159`). v3.4's E4 must either activate it (sign-off-gated) or explicitly supersede it —
  don't run two half-live decay systems.
- **Reconsider the Lab's primary input for v3.4:** the kernel's per-spawn resolver-journal (high-volume,
  already-live) may be a richer Lab input than decompose-run's per-leaf rejects (low-volume, rare). Put on the
  v3.4 design table. (ADR-0012 + depth-1 themselves need NO revisit — correctly applied.)

## 6. Per-wave workflow (the standing cadence — applied to EVERY wave)

1. **PLAN** — a per-wave build-plan in `packages/specs/plans/` with a `## Runtime Probes` section verifying
   that wave's current-state claims against the actual repo.
2. **VERIFY (pre-build)** — spawn an architect (+ lenses by cognitive need) to pressure-test the wave plan;
   fold corrections BEFORE building.
3. **BUILD** — TDD (test → red → green). Honor the Lab boundary invariants (advisory-only; K12-clean).
4. **VALIDATE (post-build)** — multi-lens on the diff. Attestation semantics / Lab-boundary integrity is the
   high-stakes class → **3-lens parallel tier**: code-reviewer (correctness) + honesty-auditor (claim-vs-
   evidence: is it really advisory-only? really deterministic?) + architect (layer-boundary soundness). A
   `hacker` lens if the wave touches a store/recall surface (path traversal, forged-record).
5. **GATE + PR** — `bash install.sh --hooks --test` (all green) + full kernel + runtime suites green; PR at
   the USER merge gate. Never auto-merge; never cache-hotfix (fix source, ship on `claude plugin update`).

Read-only verify/review passes use **read-only personas** (architect / code-reviewer / honesty-auditor /
hacker), never Write-capable.
