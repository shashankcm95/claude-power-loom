## THE DESIGN FORK (for the architect/hacker VERIFY to settle)

**What the breaker counts = committed destructive MINTS in a window** (the blast-radius bound — "no more than N TOMBSTONE/SUPERSEDE in the window"). Halting a mint NARROWS (refuses a destructive op; grants nothing) → §0a.3.1-safe by construction, same argument as the existing breaker. (NOT promote *denials* — a refusal already prevented harm; the safety signal is committed destruction rate.)

**Source — two options:**
- **(A) record-store-scan source** *(my lean — KISS/YAGNI + strongest trust)*: a `manage-promote` source that scans the kernel record-store for `TOMBSTONE`/`SUPERSEDE` records with `recorded_at` in the window. NO new store, NO new mutation surface; counts AUTHENTIC content-verified mints (#273). Cost: a cross-run scan (destructive mints are rare; bounded). Needs P4's scan helper (or add one).
- **(B) dedicated append-only promote-event ledger**: `promote.js` appends `{persona, op, target_count, recorded_at}` on each committed mint; a `manage-promote` source reads it. Cleaner read + EC3 observability, BUT adds a new writer-unauthenticated Lab store (p-writescope) — acceptable for a HALT-only breaker (a forged line only OVER-halts → narrows → §0a.3.1-safe, per project.js:14-18), but it is a new mutation surface to review.

**Halt placement + shadow default:** evaluate the breaker AFTER per-target eligibility, BEFORE the mint (step 5.5). ## VERIFY RESOLUTION (architect + hacker, 2026-06-10 — both NEEDS-REVISION; folded)

The two lenses **disagreed on the source** (architect → B/ledger; hacker → A/scan). Orchestrator synthesis: **Option A (record-store cross-run scan)**, because for a SAFETY CAP the hacker's **H2 asymmetry is decisive** — to under-count in A the attacker must DELETE the very TOMBSTONE/SUPERSEDE records that ARE the destruction (suppression is self-defeating; the W1 reader then reports the target live again), whereas B's writer-unauthenticated ledger is free to truncate → a pure fail-open (the §0a.3.1 "forge only over-halts" argument covers forgery, NOT suppression). A also adds no new mutation surface. The architect's B-preference was driven mainly by F1 (which BOTH options must fix) + the cross-run-scan cost (mitigated: the scan reuses the hardened `record-locate.js:40-54` enumeration, runs only on rare human-gated promotes, and needs NO content-verify for a halt-only count).

**Folded design (the build contract):**
- **C1/F1 (CRITICAL) — window on FS `mtime`, NOT `intent_recorded_at`.** The kernel record's only time field is caller-chosen + content-hashed (`manage-op-record.js:96,103`), so a back-dated `nowIso` would read 0-in-window. The `manage-promote` source maps each record's `fs.statSync(file).mtimeMs` → `recorded_at` for the projection. Residual: same-uid `utimes()` (the documented OQ-E/sandbox boundary, same class as `updateDisposition`).
- **H1 (HIGH) — CROSS-run.** The source enumerates ALL run dirs (reuse `record-locate.js`'s `readdirSync` + `isSafePathSegment` + `realpathSync` + `checkWithinRoot` gates verbatim) and counts `operation_class ∈ {TOMBSTONE, SUPERSEDE}` (F4 — both, not just TOMBSTONE) with `mtime` in-window. A per-run source fails-to-trip by construction (spread mints across runs).
- **F2 (HIGH) — NO second flag.** The breaker-halt rides the existing `LOOM_MANAGE_ENFORCE` (the breaker refuses under the SAME flag that authorized the mint — a blast-radius cap must be live exactly when destruction is live; a second opt-in = fail-to-trip-by-config). Kill-switch = the existing `LOOM_DISABLE_CIRCUIT_BREAKER` opt-OUT (safe-by-default). The breaker ALWAYS evaluates + the decision rides in the result (observability/EC3), even in shadow.
- **H3/F3 (HIGH) — off-by-one.** Evaluate at step 5.5 (after IDOR eligibility `promote.js:191-194`, BEFORE the mint `appendRecord` line 204). The scan counts PRIOR committed mints (the in-flight one isn't appended yet) → cap the (N+1)th; keep `>= maxD`. Boundary tests: prior=maxD-1 → pass; prior=maxD → halt.
- **M3 (MEDIUM) — fail-CLOSED on scan ERROR.** Distinguish "scan threw" (corrupt dir / perm) from "scan returned [] (clean empty)". Under enforcement, a scan THROW → `refuse('breaker-source-unavailable')` (NOT a silent clear). A clean empty store → 0-and-clear → proceed.
- **F5/L1 (LOW) — GLOBAL cap.** All mints carry `writer_persona_id: 'lab:manage-promote'` → per-persona is degenerate; the consumer calls `evaluate({ now })` WITHOUT a persona to read the GLOBAL count + GLOBAL threshold.
- **M2 (MEDIUM) — mint-cadence window.** Don't blind-copy the denial-tuned defaults silently; document that the global window/threshold (`LOOM_BREAKER_WINDOW_MS` / `LOOM_BREAKER_GLOBAL_MAX_DENIALS`, default 10min/10) is the burst blast-radius bound; note the slow-drip tuning knob in the source comment.
- **Finding-8 — confirmed:** a tripped breaker GRANTS nothing (it only adds a `refuse`; never skips eligibility/post-condition). §0a.3.1 holds by construction; all risk is on the fail-to-trip side.

## §0a.3.1 / safety invariants (load-bearing — for the hacker VERIFY)

- The breaker HALT only NARROWS (refuses a destructive mint) → grants nothing → §0a.3.1-safe regardless of source read-trust (the existing project.js:10-18 argument extends: a forged/over-counted source can only OVER-halt, never grant a destruction).
- "fails-to-trip" OUTRANKS "crashes" — a breaker that silently never trips is the dangerous failure (the W4 `WINDOW_MS_FLOOR` lesson). The count MUST exclude future-dated (CR-1) + the breaker MUST NOT be silenceable by a bad env (fail-safe clamps).
- The breaker must count the in-flight mint as NOT-yet-counted (it caps PRIOR mints; the current is the (N+1)th that gets halted) — an off-by-one that counts itself would let the first mint always pass even at threshold, or block the legitimate first.
