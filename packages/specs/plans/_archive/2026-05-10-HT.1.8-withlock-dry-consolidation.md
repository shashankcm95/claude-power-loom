# HT.1.8 — `withLock` DRY consolidation

## Context

Per HT.0.2 finding D.4 (substrate-scripts-audit) + HT.0.8 Trajectory.2 confirmation: the `_lib/lock.js` shared helper exists since H.3.2 (extracted from agent-identity.js + pattern-recorder.js inline copies; applied to kb-resolver + budget-tracker + tree-tracker per CS-1 code-reviewer.nova X-3). 8 substrate scripts currently consume the helper directly. Three sites diverge:

1. **`scripts/prompt-pattern-store.js:41-114`** — owns its own 50 LoC inline lock primitive (predates H.3.2; never migrated). Constants: `LOCK_TIMEOUT_MS=5000` + `LOCK_STALE_MS=30000` (time-based stale detection vs `_lib/lock.js`'s PID-based). Logger usage: `log('lock_timeout', ...)` on timeout. Single `withLock(fn)` call site at line 188.
2. **`scripts/self-improve-store.js:46-65`** — 3-tier require fallback: tries `~/.claude/scripts/agent-team/_lib/lock` first, then `__dirname/agent-team/_lib/lock`, then no-op fallback with `_warnLockFallback()` stderr warning (H.5.3 hacker.kai H-1 + code-reviewer.blair H-2). The 3-tier shape pre-dates the modern `findToolkitRoot()` convention.
3. **`scripts/agent-team/spawn-recorder.js:37-41`** — 2-tier try-fallback: tries `./_lib/lock`, then no-op fallback (silent — no `_warnLockFallback()` parity with self-improve-store.js).

The HT.1 backlog (lines 175-185) ranks this score-4 (impact-2 / effort-2): substrate hygiene; `_lib/lock.js` adoption surface convergence; closes the 5000ms-vs-30000ms timeout divergence in prompt-pattern-store.js (a real bug surface — concurrent invocations could race past `_lib/lock.js`'s 3000ms default if prompt-pattern-store.js's primitive holds for up to 30s before stale reclamation).

**Why now (HT.1.8 timing)**: `_lib/lock.js` is the substrate's canonical lock primitive at 8 callers; the 3 inline / try-fallback sites are the remaining DRY divergence post-H.3.2 extraction arc. Mechanical fix-in-code; well-bounded scope; matches HT.1.2 parseFrontmatter DRY precedent (4 inline copies → `_lib/frontmatter.js` adoption with no version bump).

**Why sub-plan-only methodology**: mechanical refactor against established helper; no fresh design surface (PID-based vs time-based stale detection — `_lib/lock.js`'s PID-based is strictly better for single-machine substrate scripts; timeout values preserved per-site via `{maxWaitMs}` opt). Per-phase pre-approval gate would not catch HIGH severity bugs that empirical smoke wouldn't catch. Matches HT.1.4 (mechanical bash extraction; sub-plan-only) + HT.1.6 (mechanical persona MD authoring; sub-plan-only) + HT.1.2 (parseFrontmatter DRY; sub-plan-only) precedents. EXPLICIT decision rationale per HT.1.6 methodology — avoids the failure mode of skipping pre-approval by default without principled assessment.

### Migration targets — per-site shape

**Site 1: `scripts/prompt-pattern-store.js` (substantive migration)**

- Currently: 50 LoC inline impl spans lines 41-114 (`LOCK_TIMEOUT_MS=5000` + `LOCK_STALE_MS=30000` constants + `sleepMs` helper at ~30-59 + `acquireLock` at 65-95 + `releaseLock` at 97-99 + `withLock` at 103-114). 1 call site at line 188 (`withLock(() => { ... })`).
- After:
  - Add `const { withLock } = require('./agent-team/_lib/lock');` near top imports
  - Drop `LOCK_TIMEOUT_MS`, `LOCK_STALE_MS` constants (lines 41-42)
  - Drop `sleepMs` helper (NB: verify this helper is only used by lock primitive; if used elsewhere, preserve)
  - Drop `acquireLock`, `releaseLock`, `withLock` functions (lines 61-114)
  - Update call site at line 188: `withLock(() => { ... })` → `withLock(LOCK_PATH, () => { ... }, { maxWaitMs: 5000 })` (preserves original 5000ms timeout via opts; lockPath becomes 1st positional arg per `_lib/lock.js` API)
  - Preserve `LOCK_PATH` constant (used as 1st arg)
  - Trade-off: lose `log('lock_timeout', ...)` logger call on timeout (line 105); `_lib/lock.js` uses `console.error` + `process.exit(2)`. lock_timeout is rare; console.error is sufficient observability per ADR-0001 invariant 3 ("every fail-open path goes through `logger('error', ...)`" applies to HOOK fail-open paths, not substrate-script fail-hard paths).
- LoC delta: -52 (drop 52 LoC inline; add 1 LoC require + adjust 1 call site)

**Site 2: `scripts/self-improve-store.js` (collapse 3-tier → 1-tier)**

- Currently: 3-tier try-fallback (lines 57-65):
  ```js
  try { withLock = require(path.join(HOME, '.claude', 'scripts', 'agent-team', '_lib', 'lock')).withLock; }
  catch {
    try { withLock = require(path.join(__dirname, 'agent-team', '_lib', 'lock')).withLock; }
    catch { withLock = (_lockPath, fn) => { _warnLockFallback(); return fn(); }; }
  }
  ```
- After: 1-tier with `__dirname`-relative require (matches spawn-recorder.js shape):
  ```js
  try { withLock = require('./agent-team/_lib/lock').withLock; }
  catch { withLock = (_lockPath, fn) => { _warnLockFallback(); return fn(); }; }
  ```
- Why this is safe: `__dirname` resolves to wherever the script lives. If installed at `~/.claude/scripts/`, `__dirname` is `~/.claude/scripts/` → require resolves `./agent-team/_lib/lock` to `~/.claude/scripts/agent-team/_lib/lock.js`. If at `~/Documents/claude-toolkit/scripts/`, same relative resolution works. The explicit `~/.claude/...` path attempt was redundant — `__dirname`-relative covers both scenarios.
- Preserve: `_warnLockFallback()` H.5.3 stderr warning (load-bearing observability)
- Preserve: `path` require (used elsewhere in file)
- LoC delta: -7 (collapse 9 LoC try-fallback to 2 LoC; preserve warning function ~10 LoC unchanged)

**Site 3: `scripts/agent-team/spawn-recorder.js` (add observability parity)**

- Currently: 2-tier try-fallback (lines 37-41):
  ```js
  let withLock;
  try { withLock = require('./_lib/lock').withLock; }
  catch { withLock = (_lockPath, fn) => fn(); }
  ```
- After: same 2-tier shape with parity `_warnLockFallback()` stderr warning:
  ```js
  let withLock;
  let _lockFallbackWarned = false;
  function _warnLockFallback() {
    if (_lockFallbackWarned) return;
    _lockFallbackWarned = true;
    process.stderr.write(
      '[spawn-recorder] WARNING: lock primitive (_lib/lock.js) unreachable; ' +
      'using no-op fallback. Concurrent record operations may corrupt SPAWN_HISTORY. ' +
      'Install or symlink the agent-team _lib helpers to enable real locking.\n'
    );
  }
  try { withLock = require('./_lib/lock').withLock; }
  catch { withLock = (_lockPath, fn) => { _warnLockFallback(); return fn(); }; }
  ```
- Why add observability: parity with self-improve-store.js's H.5.3 fix; closes the silent-degradation observability gap per ADR-0001 invariant 3 spirit (fail-open paths must be observable). Empirically the spawn-recorder fallback path is unlikely to fire (substrate ships `_lib/lock.js` always) but the warning is institutional discipline insurance.
- LoC delta: +9 (add warning function); shape change is 2-line require to 8-line block

## Drift-note 67 (NEW; deferred to HT.2 sweep)

**Title**: `session-end-nudge.js` inline lock primitive — hook fail-soft contract divergence from `_lib/lock.js` `withLock()` exit-on-timeout

**Source**: HT.0.1 D.3 finding (`hooks-audit.md:612`) — "inline lock primitive in `session-end-nudge.js:38-95` (`acquireLock`, `releaseLock`, `sleepMs`) duplicates the `_lib/lock.js` primitive that `error-critic.js` uses via dynamic require"

**Why deferred**: `session-end-nudge.js` is a Stop hook with explicit fail-soft contract (line 138-145: `if (!haveLock) { log('lock_timeout', ...); process.stdout.write(input); return; }` — passes input through silently on lock_timeout per ADR-0001 + ADR-0003 hook fail-soft invariant). `_lib/lock.js`'s `withLock()` wrapper calls `process.exit(2)` on timeout — incompatible with hook contract. Migration would require:
- Use `acquireLock()` + `releaseLock()` direct imports (not `withLock` wrapper)
- Different shape than other 8 consumers (which use `withLock` wrapper)
- Preserve `log('lock_timeout', ...)` observability call
- Preserve fail-soft pass-through semantics

This is structurally distinct from HT.1.8's mechanical migration and warrants its own analysis (drift-note candidate for HT.2 sweep OR standalone phase if HT.2 deems it priority). HT.1.8 SCOPE: 3 substrate scripts only.

**Sibling drift-note**: `hooks/scripts/error-critic.js:57-72` — 2-tier try-fallback for `_lib/lock.js`. Out of HT.1.8 scope per backlog (HT.0.2 D.4 explicitly enumerates only the 3 substrate scripts; error-critic.js wasn't in scope at HT.0.2 audit). HT.2 sweep candidate alongside session-end-nudge.js — both are HOOK consumers with different fail-soft contract requirements than substrate-script consumers.
