## In-scope target shape

### Component A — Extend `validate-kb-doc.js` with additional frontmatter checks (HARD block)

Add HARD-block validation for objective, auto-fixable, currently-100%-compliant fields:

1. `kb_id` matches file path (auto-derivable; mismatch is always a bug; quality bar L42)
2. `version: 1` present (trivial; quality bar L43)
3. `tags` ≥3 entries (quality bar L44)
4. `sources_consulted` array present + ≥2 entries (quality bar L45+L56)

All 15 current docs comply (baseline 0 violations). NEW docs MUST comply (zero-regression). Auto-fixable in seconds; non-controversial.

**Failure mode shape**: Block decision with `reason` field like `[KB-DOC-INVALID] frontmatter field 'version' missing — quality bar requires version: 1 per _PRINCIPLES.md L43`.

**Implementation note (architect HIGH-1 + code-reviewer HIGH-CR3 absorption)**: `kb_id`-matches-path is computed against the POST-edit `kb_id` value vs the file's current relative path. Per the existing validator's Edit-mode behavior (validate-kb-doc.js L210-225, content-source-approximation), a stale `kb_id` after a `git mv` would HARD-block edits to the relocated doc until `kb_id` is updated. Mitigation: `SKIP_KB_DOC_CHECK=1` env bypass remains the operator escape; relocation-fixup is a 2-edit pattern (one frontmatter edit + one substantive edit; acceptable friction aligns with discipline encoding goal — relocations SHOULD update `kb_id`). For Edit events that CHANGE `kb_id` to an incorrect value: validator reads pre-edit content (H.8.8 content-source-approximation); block fires on subsequent Write rather than the current Edit. This is a known limitation of the pre-edit approximation pattern.
