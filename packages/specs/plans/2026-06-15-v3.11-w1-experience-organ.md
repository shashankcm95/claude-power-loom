## Content-hash basis (the architect C1 resolution — decided WITH the freeze, a one-way door)

`node_id` + `content_hash` derive from `worked_example_ref` ONLY (`recall-graph.js:146`; `verifyNode`
re-derives both on read, `recall-graph-store.js:54-63`). The lesson fields are placed as follows:

- **`node_id` / `content_hash` stay patch-stable** (`worked_example_ref` only, UNCHANGED). `lesson_body` is seedless
  `claude -p` output; putting it in the node-id basis would mint a different id per re-derivation -> dedup breaks.
- **Lesson scalar fields are TOP-LEVEL siblings** (outside `node_id`, like `built_by`/`recorded_at`): the block
  `trigger_class`/`gotcha_class`/`corrective_class`, plus `lesson_signature`, `lesson_body`, `accepted_diff_ref`,
  **`candidate_patch_sha`** (the full-sha pointer into the sidecar; it backs the `lesson_content_hash` at the next
  bullet, so it is a named node field, not just a hash input), `fail_to_pass`, `friction_signature_ref`. **These are
  TOP-LEVEL node fields, NOT added to the frozen `WORKED_EXAMPLE_FIELDS`** (which stays frozen + its immutability
  test untouched); `fail_to_pass` is threaded out-of-band to the mint as a top-level field, not via
  `WORKED_EXAMPLE_FIELDS`.
- **The candidate patch is a SEPARATE content-addressed sidecar** keyed by **full** `sha256(candidate_patch)` (NOT
  the 16-char slice), verified-on-read by **RE-HASHING the patch content** and rejecting a filename<->content
  mismatch (the #273 content-not-just-key discipline on a new artifact -- never a filename-only check).
- **`verifyNode` is EXTENDED** to (a) re-derive `lesson_signature` from the stored lesson block and reject a
  mismatch (the closed-enum key cannot be forged independent of its body) and (b) verify a new
  `lesson_content_hash = sha256(canonical({lesson_signature, lesson_body, accepted_diff_ref, candidate_patch_sha}))`
  so the leak-bearing `lesson_body` is tamper-evident even though it is outside the patch-identity hash.
  **`friction_signature_ref` is EXCLUDED from `lesson_content_hash`** (re-derivable from the friction block;
  `verifyNode` may re-derive it as a cross-check). **The verify is PRESENCE-CONDITIONAL** (re-verify board): a
  worked-example-only node (no lesson layer) must still PASS; only a node that HAS a lesson layer with an
  absent/wrong `lesson_content_hash` or a forged `lesson_signature` is rejected -- do not brick the legitimate
  lesson-less nodes. The lesson layer dedups/merges by `lesson_signature` (a separate index), NOT by `node_id`.

that test; update it deliberately and note the contract change.

   1 weighted; the diagnostic emits per-signature counts.
7. **The capture re-run spike** (`_spike/`) — `runIssueCalibration` over a small seed slice with the real actor,
   minting derivation-ready nodes; OUT of CI. **Cost (pinned):** `N` seed issues x `M` attempts ~= `3*N*M`

## Exit (internal-N, honesty-corrected)

- `derive -> lessonClusterKey -> lessonLeaks -> store` runs end-to-end on the capture re-run's nodes (a
  machinery-runs claim; W1 is the machinery wave).
- `lessonLeaks` shows **no >=12-char run shared with a sealed `accepted_diff` OBSERVED across the N re-run nodes**
  (N below the discrimination floor of 20 -- RFC Sec 8; "observed", not "proven").
- Two same-signature lessons merge into one weighted node, **collision observed on fixtures OR the re-run set**;
  a real-data collision is NOT guaranteed at N~11 (deferred to W3 per RFC:204-206).
- The lesson key provably carries the `lesson:` prefix and never collides with a `friction:` key.
- Persona-agnostic: key + recurrence from D1 axes only; `built_by`/`graded_by` never an input.
