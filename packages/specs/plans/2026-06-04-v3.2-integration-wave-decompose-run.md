## Goal

Close the integration/dogfood gap the phase-close flagged: build the **minimal consumer that composes the two v3.2 tiers end-to-end**, prove it with a real cross-tier integration test, and **dogfood it live** in this session.

- **Decompose tier:** R6 `runTrampoline` (R7 checkpoint + R8 disciplines + R10 budget).
- **Verify tier:** R11 `verifySpawn` (R9 `validateLeaf` + R12 `node-runner` subprocess).
- These two tiers have **zero live caller** today and are **not wired to each other** (probed below). This wave builds the one composition that makes them a working whole.

**Non-goal (YAGNI):** a production orchestrator, a hook wiring (that un-darkens the deployment — a separate USER-gated step), or Pattern-B multi-spawn (v3.5+). This is a *thin* consumer + its test + a live CLI dogfood.

## The composition (design)

The intended flow (honoring the R9-header contract that R6 is decompose-time and never verifies; R11 is the verify-time gate — DAG R11→R9, NOT R6→R9):

```
runDecomposition({ runId, personaId, taskId, leaves, maxDepth, ctx, stateDir }):
  # Phase 1 — VERIFY each leaf (R11 → R9 + R12). "A bad leaf never runs."
  verdicts = leaves.map(leaf => ({ leaf, verdict: verifySpawn(leaf, ctx) }))
  admitted = verdicts.filter(v => v.verdict.accepted).map(v => v.leaf)
  rejected = verdicts.filter(v => !v.verdict.accepted)
                     .map(v => ({ id: v.leaf.id, failure_signature: v.verdict.failure_signature }))
  # Phase 2 — TRAMPOLINE only the admitted leaves (R6 → R7 + R10)
  trampoline = admitted.length > 0
    ? runTrampoline({ runId, personaId, taskId, leaves: admitted, maxDepth, stateDir })
    : null
  return { admitted: admitted.map(l => l.id), rejected, trampoline, allRejected: admitted.length === 0 }
```

**Why verify-all-then-trampoline-admitted** (vs interleaved per-leaf): it respects the tier separation (the consumer orchestrates both tiers without coupling R6→R11 — Dependency Inversion); it makes "a bad leaf never runs" structural (the gate precedes all processing); and it keeps the consumer a pure composition with no new policy. *(Flag for architect VERIFY: is interleaved — verify leaf N, trampoline it, then N+1 — ever required for Pattern-A cognitive-thread semantics? My read: no, because R6 is already serial and the budget is depth-based, not interleave-dependent.)*

**Leaf-shape bridge (no transformation needed):** the consumer takes **extended** leaves (the R9/R11 fields: `estimated_tokens`, `output_schema`, `verification`, …). R6 reads only the `{id, content, discipline}` subset and ignores the extra fields (probed: `trampoline.js:151-158` reads only `leaf.discipline` + `leaf.id`). So a full leaf is a valid R6 leaf — pass the admitted full leaves straight through.

## Deliverables

1. **`packages/runtime/orchestration/decompose-run.js`** — `runDecomposition(opts)` (import-friendly, pure of process.exit on the API path) + a `require.main === module` **CLI** (`node decompose-run.js --leaves <file.json> --run-id X --persona Y --task Z --cwd <dir> --max-depth N`) that prints the JSON result. The CLI is the dogfood vehicle (a real process exercising real R12 subprocesses).
2. **`tests/unit/runtime/contracts/decompose-run.test.js`** — the **end-to-end integration test** (real modules, NO mocks; auto-CI-gated by the recursive `find tests/unit/runtime` glob — probed). Asserts the composed behavior the per-tier unit tests cannot: a bad leaf rejected with the right `failure_signature` AND never trampolined; admitted leaves trampolined to COMPLETED; a budget-starved run emits the R6 ABORTED record; a tdd leaf with a real failing test rejected by R11 via a real R12 subprocess.
3. **A dogfood fixture** — a realistic 4-leaf scenario JSON (valid spec-driven; valid tdd w/ a passing test; tdd w/ a failing test → R11 reject; bad spec-driven w/ no `output_schema` → R9 reject), reusing the existing `tests/unit/runtime/verify/fixtures/{passing,failing}.fixture.js`.
4. **The live dogfood** — invoke the CLI via Bash in this session on the fixture, observe the real end-to-end run (real R12 subprocesses, real R7 checkpoint, real R10 budget + ABORTED record), and record the observed output in the plan + the PR.

## Realism boundary (honesty MEDIUM-2 — what the dogfood does NOT prove)

The dogfood runs **hand-authored fixture leaves** through the **module composition** with real subprocesses + real persisted records. It is **NOT Agent-spawn-driven**: no `isolation:worktree`, no `PostToolUse:Agent` close, no LLM doing the decomposition — and it does **NOT un-darken the deployment** (the plugin stays at the 3.1.0 cache; nothing is hook-wired). Wiring the composition to a live Agent spawn is the separate, USER-gated un-darkening step. This wave proves the tiers **compose and run end-to-end as code**, not that the spawn lifecycle is integration-tested.
