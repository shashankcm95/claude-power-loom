## Design

### R8 — freeze the discipline vocabulary (do FIRST; freeze-point) — ✅ RATIFIED: OPTION A

> **USER decision (2026-06-03): Option A — freeze `Object.freeze(['spec-driven', 'tdd'])`.** Scope-faithful
> ("drive off that field"); YAGNI; widening to add `exploratory` later is additive/backward-compatible, so
> deferring it is cheap. **No contract migration** (read-only personas stay as-is; revisit when R11 routing
> or a contract-accuracy fix gives `exploratory` a real consumer). The build implements A; B/C below are
> retained as the rejected alternatives for the record.

**The central decision (USER ratifies — a freeze-point R6 consumes now, R9/R11 consume in Wave 2).** The
field is uniformly `{spec-driven, tdd}` across all 18 contract files (16 personas + the `challenger` and
`engineering-task` task-type contracts). What vocabulary do we freeze?

**Honest framing (corrected after VERIFY — honesty-auditor HIGH):** the parent scope explicitly directs
R8 to *"drive off that field, **not** invent a parallel `tdd/spec-driven/exploratory` enum"* (scope §57-59).
Read literally, that points at **Option A**. Adding `exploratory` is a deliberate **divergence** from the
scope's directive — defensible on forward-routing + honesty grounds, but it must be ratified *as* a
divergence, not dressed up as the scope's own intent (my first draft did the latter — evidence-laundering).

- **Option A — freeze exactly `{spec-driven, tdd}`** (the two observed values). Matches the scope's literal
  directive; purest YAGNI (no enum value without a present consumer). Adding a value to an allowed-set
  later is *additive / backward-compatible*, so deferring `exploratory` is cheap, not a hard reversal.
  Honest-to-today. Cost: R11's Wave-2 routing sees 2 near-identical buckets.
- **Option B — freeze `{spec-driven, tdd, exploratory}`; DEFER contract migration.** Forward-flex: three
  buckets R11 *would* route on in Wave 2 (tdd→tests/R12, spec-driven→`output_schema` coverage,
  exploratory→findings-shape for read-only personas). Honest cost: `exploratory` is used by **zero**
  contracts today and R11/R12 are **unbuilt** — a *speculative* slot (mild speculative generality)
  justified only if Wave-2 routing materializes. *That is the trade, stated plainly.*
- **Option C — freeze `{spec-driven, tdd, exploratory}` AND migrate the read-only contracts now**
  (`codebase-locator` / `codebase-analyzer` / `codebase-pattern-finder` → `primary: exploratory`).
  Resolves both the scope-divergence *and* the "unused enum" critique: a `codebase-locator` produces no
  spec to satisfy and no tests to run, so today's uniform `{spec-driven, tdd}` is a **demonstrable
  mis-description** — C fixes that truth now. Cost: edits 3 contracts inside Wave 1 (small, but real scope).

**No pre-baked recommendation.** The architect favors B on forward-routing merits; the honesty-auditor
flags B's unused enum as genuine speculative generality and that A matches the scope. The USER picks; the
build sets the frozen array (and, for C, migrates the read-only contracts) accordingly.

### R7 — TodoWrite-as-checkpoint primitive (parallel with R8)

NEW `packages/runtime/orchestration/todo-checkpoint.js`, modeled on `budget-tracker.js` (import-friendly,
atomic write, no `process.exit`, lock-guarded). It is the **durable ledger** the trampoline writes — NOT
an observer of the TodoWrite tool (probe #6).

- Record shape: `{ runId, leaves: [{ id, content, status, discipline? }], createdAt, updatedAt }` where
  `status ∈ {pending, in_progress, completed}` (mirrors TodoWrite).
- Functions: `writeCheckpoint(runId, leaves)`, `updateLeafStatus(runId, leafId, status)`,
  `readCheckpoint(runId)`.
- **Liskov invariant (mirrors TodoWrite):** at most one leaf `in_progress` at a time; `updateLeafStatus`
  enforces it (reject or auto-demote — architect call).
- State file `todo-checkpoint.json` under `runStateDir(runId)` (never the user's tree). **Its own lock**
  (`_lib/lock.js withLock` keyed on the checkpoint file) — `withBudgetLock` does NOT cover a separate file
  (Q4). Build the lock now so Wave-2 parallel writers inherit it.

### R6 — Pattern-A trampoline (LAST; integrates R7 + R8 + R10)

NEW `packages/runtime/orchestration/trampoline.js`. Serial, persona-internal. **Consumes** (no edits to):
R7 `todo-checkpoint.js`, R8 `_lib/decomposition-disciplines.js`, R10 `budget-tracker.js`, and the kernel
`_lib` record path (`transaction-record.js` `validateTransactionRecord`/`computeTransactionId` +
`record-store.js` `appendRecord`). Flow per decomposition level:

1. `enterDepth(runId, MAX_DEPTH)` (R10). `depthExhausted` is `currentDepth > maxDepth` (**STRICT** — a
   3-level descent needs `MAX_DEPTH ≥ 3`). If exhausted → emit the ABORTED record (below), **stop**.
2. Otherwise: validate the leaf-id (write-scope guard, Q3) → `mkdir` the leaf folder under
   `runStateDir(runId)/decomposition/` → write/advance the R7 checkpoint (`pending → in_progress →
   completed`, at most one `in_progress`).
3. `exitDepth(runId)` on unwind.
4. On clean completion: **no transaction record** (build decision — see below). The folders + checkpoint
   are gitignored run-scratch, not canonical state → no canonical delta → no record; completion is
   evidenced by the checkpoint (all leaves `completed`) + the recursion unwound to 0. Only the
   budget-exhaust path emits a record (the ABORTED provenance). *(This refines the original "a COMMITTED or
   NOT_APPLICABLE record" line — emitting a NOT_APPLICABLE/CREATE record for a no-delta run would be
   semantically odd and add post_state_hash complexity the exit demo doesn't need. Flagged by the
   honesty-auditor as a plan↔code divergence; reconciled here.)*

**The ABORTED record — built DIRECTLY, NOT via `buildSpawnRecord` (Q5 CRITICAL):** `buildSpawnRecord`
hardcodes `operation_class:'CREATE'` + `commit_outcome:'COMMITTED'` (`quarantine-promote.js:297,300`) — it
cannot emit ABORTED. R6 constructs a plain object → `validateTransactionRecord` → `appendRecord`:

- `commit_outcome: 'ABORTED'`; `abort_reason: 'budget-exhausted'` (the **canonical** value, `schema:117`;
  the recursion budget IS a budget dimension — NOT the invented `recursion-depth-exhausted`). Depth
  specifics go in the structured `abort_detail` block (`schema:135-160`).
- `writer_spawn_id`: R6 is persona-internal (NOT a spawn) → synthesize a deterministic id matching the
  documented format (`schema:49`), e.g. `sp-<iso>-<persona>-trampoline-<level>` (`minLength:1` required).
- `operation_class` (required): every state-changing enum value triggers A10's non-empty-`evidence_refs`
  enforcement (`transaction-record.js:303-310`); a recursion-abort has no natural canonical-delta edge →
  use the bootstrap-sentinel `evidence_refs` pattern (`ROOT_TASK_RECORD:<id>`, the escape
  `genesisRecordFields:298` already uses). Pick the most honest value and PROVE it round-trips
  `appendRecord` in a test (this field is the likeliest runtime `validateTransactionRecord` failure).
- `transaction_id` via `computeTransactionId`; plus `prev_state_hash`, `writer_persona_id`,
  `intent_recorded_at`, `schema_version` (all `required[]`). **NO `idempotency_key`** — keyless is
  admissible (`idempotency_key` not in `required[]`; `appendRecord` gates dedup on presence,
  `record-store.js:223,246`), so no INV-22 `computeContentHash` machinery.

Each leaf declares a `discipline` validated against the R8 frozen set. Pattern B (sibling multi-spawn) is
explicitly out of scope (v3.5+).

- **SRP:** each primitive owns exactly one reason to change — R8 = the vocabulary, R7 = the checkpoint
  ledger, R6 = the decomposition orchestration. The folder-hierarchy I/O is R6's, not R7's.

  (R10) and the `transaction-record.js` ABORTED outcome rather than inventing parallel machinery; the R8
  constant is the single source of truth R9/R11 import in Wave 2.

  deferred v3.5+; folder hierarchy is the minimum the exit demo needs.
- **Liskov:** R7's checkpoint honors TodoWrite's at-most-one-`in_progress` contract, so any consumer
  written against TodoWrite semantics behaves correctly against the ledger.

- **Scope-inversion / evidence-laundering (honesty-auditor HIGH — load-bearing lesson):** my first draft
  cited the scope's *"do **not** invent a `tdd/spec-driven/exploratory` enum"* as if it *endorsed* that
  triple. It is the opposite — a prohibition. The strongest-sounding support for my recommendation was the
  exact sentence arguing against it. Lesson: when a plan cites a source *for* a position, re-read the
  source in context — a prohibition read as an endorsement is the most dangerous citation error because it
  *looks* evidence-backed. This is why the honesty-auditor lens runs pre-build, not just post-build.
- **A4 kernel/runtime boundary — RULE (architect, codify candidate):** *"A4/K11 binds branching /
  derivation logic with a non-trivial spec (route scoring, subset math, path canonicalization). A
  static-set membership lookup or a single-field presence check is a runtime constant, NOT a kernel
  algorithm — do not register it in `algorithms/manifest.json`."* Stops future trivial checks from
  re-litigating the boundary or cargo-culting the gate.
- **The `path.join` pre-normalization trap (hacker VULNERABLE, build-caught + fixed):** a
  `checkWithinRoot(path.join(BASE, untrusted), BASE)` guard is **blinded** — `path.join` runs
  `path.normalize`, collapsing `..` *before* `checkWithinRoot`'s `hasTraversalMarkers` screen can see it,
  so an in-base traversal (`a/../a`, `x/..`) passes. The canonical defense (record-store `isSafeRunId`) is
  a RAW-token check BEFORE the join. The first Wave-1 build applied this to leaf-ids but NOT the runId,
  leaving a cross-run-clobber hole (`realrun/../realrun` overwrote `realrun`); fixed with a shared
  `_lib/safe-segment.js isSafePathSegment` on runId + leaf-ids in R6 AND R7. **Class lesson:** every
  `checkWithinRoot(path.join(...))` site needs a raw-token pre-check; `checkWithinRoot` alone is a
  second layer, never the primary gate. A follow-up audits the other runtime call sites.
