## Load-bearing risks / catches (a builder must not miss)

1. **The schema is additive, but a store without OQ-27 is dark.** The whole point of Wave 1's gate: do not build W2.1 until Spike B proves the walker is buildable. The RFC's "an edge is just a canonical record" is true *and* is the trap.
2. **OQ-E mechanisms (a)/(b) are blocked/weak by ADR-0012 — a substrate-bricking class** (the repo shipped `pre-spawn-tool-mask` INERT on exactly this). Mechanism (c) makes "is this a kernel writer?" an *execution-path* fact, not a string check. **Probe before any R1/R2 line. Multi-reviewer blessing is NOT runtime verification** (the RFC passed *two* security reviews and the primitive is still unbuilt).
3. **`writer_persona_id` is the silent dependency under R1 too.** Fail-closed default (un-attested ⇒ `assertion_class=stochastic_sample`; stochastic-source CREATE ⇒ `faithfulness_status=unvalidated`) is load-bearing, not polish — ship it in W3 regardless of OQ-E outcome.
4. **No live SUPERSEDE/TOMBSTONE emitter exists today** (only CREATE via spawn-record + the shadow resolver). Record-emitting manage-ops must ride a *new* write path → keep them human-gated candidates in v3.5 (the v3.2/v3.4 inert-code lesson).
5. **Stays shadow.** The graph walker is an advisory read surface (never a kernel gate, never a `hooks.json` deep ref). `grep -c "lab/\|causal" packages/kernel/hooks.json` stays 0. Live destructive writes = a separate leave-shadow phase.
6. **`conflicted` stays retrieval-eligible** — a `flag-conflict` (dream-cycle, not R2-gated) must NOT bury two legitimate records.
