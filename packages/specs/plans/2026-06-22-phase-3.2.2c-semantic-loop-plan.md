## Decision trail (why semantic-only, why now)

The original ③.2.2b plan proposed a Docker-executed **no-regression** behavioral leg. The pre-build VERIFY
board (architect lens; the parallel run died on a transient 529, this lens re-run solo + FIRSTHAND-verified)
falsified its load-bearing premise and surfaced an intrinsic tension:

- **CRITICAL (verified):** the no-regression leg cannot REUSE `runTests` — the pytest wrapper emits results
  ONLY for explicitly-named `test_ids` (`out = {t: res.get(t,'missing') for t in ids}`, [pytest-runner.js:75])
  and `parseTestStatus` iterates the caller's id list ([container-adapter.js:143]); empty `test_ids` →
  empty baseline `G` → `G ⊆ pass(G′)` is VACUOUSLY true for every candidate. It's a substantial NET-NEW
  pytest-harness build, not reuse.
- **Intrinsic tension (verified):** the grade sandbox is `--network none`, HARD-enforced + ATTESTED
  ([docker-backend.js:48,122]; the attestation proves egress denied), and the image is `python3 + pytest`
  ONLY with no deps-install step ([Dockerfile]). Running an arbitrary repo's tests needs its deps; installing
  them needs network AND runs the repo's untrusted build code (`setup.py`). So a no-regression leg either
  (β) limits the corpus to offline-runnable repos, or (α) adds a separate contained-online-install phase —
  a security sub-project that runs untrusted build code with network.
- **The architecture clarified (USER):** Docker is NOT used for delta capture (that's host-side in
  `real-solve.js`); Docker exists ONLY to EXECUTE the repo's tests. The actor runs with no Bash
  (`ACTOR_TOOLS`, [real-solve.js:65]) so it never executes the repo regardless. **The network/deps roadblock
  exists ONLY because the no-regression leg executes tests.** A semantic-only grade executes nothing → no
  Docker, no roadblock, no repo limitation.

**USER decision:** ship the non-limiting semantic-only loop now (works on every repo), run a confirmation
round on real repos, and take up the no-regression signal LATER as its own wave — built on the
contained-online-install (α) network path so it ALSO doesn't limit repos (never the offline-only β). The
behavioral signal is a SHADOW one that narrows-not-hardens anyway (OQ-NS-6) — additive, not foundational.

## Scope (③.2.2c — the full semantic-only loop)

**IN scope:**
- A live-issue semantic grade `gradeLiveIssueSemantic({ record, candidate, semanticFn, frictionFn })` →
  a SHADOW verdict, composing ONLY the oracle-free legs (verified usable on a public-only record):
  - **Leg B (REUSE)** — the blind `claude -p` semantic judge: does the candidate plausibly address the
    `problem_statement`? Public-safe (`buildLegBInput` no-ops the absent `criteria_only_rubric`/`accepted_diff`
    — verified); fail-closed to a non-PASS on refuse/throw.
  - **Leg D (REUSE)** — friction diagnostic, report-only, public-safe input.
  - Verdict: `{ behavioral: 'UNAVAILABLE', semantic_supported: bool|null, friction: {...}|null,
    oracle: 'none', shadow: true }`. Structural markers, NOT a pass/fail proof-of-fix; nothing reads it as a
    gate (`recall_eligible` is structurally false for a live issue — [calibration-issue.js:208-213]).
- An actor-solve-for-live path (no sandbox backend needed — nothing executes): clone @ `base_sha` on the
  host → `claude -p` actor (Read/Grep/Glob/Edit/Write, no Bash) → capture the candidate diff via the
  ③.2.0-hardened `_clone-lifecycle` (`prepareClone` snapshot + `captureActorDiff` pristine-config restore +
  diff-scope). NOTE: `real-solve.js:makeRealSolve` GATES on an attested backend
  ([real-solve.js:134]) — for semantic-only we either (a) thread a "semantic backend stub" whose
  `containmentAttested` is true but whose grader is the semantic composer (minimal real-solve reuse), or
  (b) a lean new live-solve that reuses `_clone-lifecycle` directly without the backend gate. The VERIFY
  board picks; (a) maximizes reuse, (b) is cleaner — lean toward (a) to avoid duplicating the hardened clone.
- The DRAFT artifact via the egress chokepoint (REUSE — verified): call `emitPR({repo, issueRef, diff},
  {dry-run custody opts})` → it scrubs the diff + builds `{title, touched_paths, diff}` + returns
  `{ok:true, emitted:false, draft}` WITHOUT emitting (`armedEmit` only fires on `disposition.mode==='live'`
  — [emit-pr.js:332-337]). `issueRef` is parsed from the record `id` (`owner__repo-issue-N`; the public
  record carries no separate number field — validatePublicRecord forbids extra keys). Write the draft to a
  local artifacts dir; emit NOTHING. (This makes ③.2.3 a disposition flip, nothing more.)
- The orchestration driver: loop over live-puller records → solve → grade → DRAFT → write artifact +
  a JSON run report. Emits nothing.
- Unit tests (mock semanticFn/frictionFn + mock actor + a dry-run `emitPR`) + a live dogfood
  (`_spike`) that runs the REAL `claude -p` actor on ONE real good-first-issue repo end-to-end (Rule
  2a-corollary — a green mock suite is a hypothesis; the real `claude -p` path is where bugs hide).

**OUT of scope (deferred):**
- The no-regression behavioral leg → a LATER dedicated wave (③.2.2d), built on the contained-online-install
  (α) network path (its own security plan + VERIFY; runs untrusted build code with network).
- Any emission / token / live PR → ③.2.3 (just flips the `emitPR` disposition to live).
- Establishing a "does it FIX the bug" oracle for open issues — impossible without a human (OQ-NS-6).

## Security & correctness model (this wave executes NOTHING)

The risk surface SHRINKS vs the Docker-execution plan — no stranger code runs anywhere. What remains:
- **The actor reads/edits UNCURATED stranger content** (issue text + repo files). The threat is
  prompt-injection of the actor (the issue/README/code telling it to write somewhere malicious or produce a
  hostile diff). Mitigations (reuse): the actor has no Bash/network tools (can only Edit/Write within the
  throwaway clone); the ③.2.0 capture hardening (no `.git`-poisoning, pristine-config restore, diff-scope);
  `problem_statement` is fence-as-DATA into the prompt (already handled) + bounded (③.2.2a 64 KiB cap).
  Blast radius: a prompt-injected actor produces a bad DIFF in a throwaway clone → caught downstream
  (scrubbed by `emitPR`, graded SHADOW, emits nothing).
- **The semantic judge sees the issue + diff** (both attacker-influenceable) — prompt-injection of the
  judge. Reuse the blind-judge fence-as-DATA discipline; the judge's verdict is SHADOW + gates nothing, so
  a poisoned judge verdict has no action consequence this wave.
- **The DRAFT diff is scrubbed** by `emitPR` (reuse `scrubEmitDiff`) before it lands in an artifact —
  defense-in-depth vs a secret in the candidate. `emitPR` rejects a `.github`/`.git*` diff.
- **No emission, structurally:** `armedEmit` throws; the disposition is dry-run; no token in any env.
- No secrets in source (ADR-0006; ASCII-only); immutable record handling.

**Explicitly NOT claimed:** a `semantic_supported:true` verdict does NOT mean the candidate fixes the
issue — only that a blind reviewer found it plausible. `behavioral:'UNAVAILABLE'` is honest (no oracle).
Trust stays ZERO; only a maintainer merge hardens (OQ-NS-6).
