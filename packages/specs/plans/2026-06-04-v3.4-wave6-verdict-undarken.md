# v3.4 Wave 6 — verdict→E4 dogfood un-darkening

## Goal

Un-darken the **verdict-emission → E4 reputation → A6 snapshot** chain by establishing the missing
piece: a **routine producer**. All consumer machinery is already built + functional-when-invoked
(W1 store, the enricher, E4, A6, E11). The gap is that nothing routinely *calls* `recordVerdict`.

This wave **dogfoods the loop on its own construction**: delegate a real build sub-task to a builder
persona (a delta-bearing **subject** spawn), 3-lens review it, then record the **first real
verdict-attestation** linking that spawn → enrich → watch E4 + A6 reflect it. The substrate measures
its own development (Axiom 5), and the Pattern-2 + recording convention is established *by use*.

### `record-review` design spec (resolves verify-plan code-reviewer FAIL + FLAGs)

**The batch-arg FAIL fix (Finding 1):** do NOT reuse the shared last-wins `parseArgs` for the batch (it
collapses repeated flags). The triple is expressed as **one self-contained repeatable flag**, avoiding
zip-misalignment across three separate flags entirely:

```
cli.js record-review --subject-persona <P> --agent-id <A> \
       --review "<verifier-identity>|<verifier-kind>|<verdict>"   # REPEATABLE, pipe-delimited, exactly 3 parts
       [--review "..."] ... [--expires-after-days N] [--no-enrich]
```

- A **dedicated arg walk** for `record-review` collects ALL `--review` occurrences into an array;
  `--subject-persona`/`--agent-id`/`--expires-after-days` parse single-value; `--no-enrich` is a bool.
  **It MUST NOT modify `parseArgs` or any existing subcommand's path** (the existing single-value
  callers stay byte-identical). Pipe `|` is the delimiter (no legal identity/kind/verdict contains it;
  e.g. `03-code-reviewer.nova`, `structural`, `pass`). A `--review` value that does not split into
  exactly 3 non-empty parts → clean exit 1 with a precise message (the zip-misalignment case).
- **Atomicity FLAG fix (Finding 3): all-or-nothing.** PRE-VALIDATE every triple + the subject/agentId
  BEFORE writing any record, using the store's EXPORTED constants (`store.VALID_VERDICTS`,
  `store.MAX_FIELD_LEN`) — not a re-implementation, a pre-flight using the published contract. On any
  failure → exit 1, **write nothing**. (Residual: a store-internal mid-batch throw after pre-validation
  is near-impossible; if it occurs, already-written records are dedup-safe on re-run — advisory ledger,
  idempotent.) At least one `--review` is required (empty batch → exit 1).
- **Auto-enrich exit-code FLAG fix (Finding 2):** records persisted = success. After recording, unless
  `--no-enrich`, run `enrichLedger()`. If enrich **throws**, exit **0** with a stderr WARNING ("records
  persisted; enrich failed: <e>; run `enrich` to resolve links") — the recording (the load-bearing part)
  succeeded and enrich is idempotent/re-runnable. A usage/validation/record failure still exits 1.
- **Size FLAG fix (Finding 6):** extract the batch logic to a named helper
  `recordReviewBatch({ subjectPersona, agentId, reviews, expiresAfterDays, enrich })` so the
  `if (cmd === 'record-review')` dispatch block stays a thin router (<20 lines); the helper returns a
  summary `{ recorded, deduped, enriched }` the dispatch prints. (SRP — dispatch routes, helper implements.)
- Output: a JSON summary `{ recorded, deduped, enriched, unresolved }`.
