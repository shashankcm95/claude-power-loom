## Context

**What R11 is** (scope:72, :48-52): a **NEW runtime dispatcher** (the largest component, ~600–750 LoC) that
decides whether a decomposition leaf is admissible — routing by the leaf's R8 **discipline**, running the
R9 + R12 gates, and emitting a structured ADR-0015 `failure_signature` on rejection. It is explicitly **NOT
a 4th arm of `spawn-close-resolver.js`** (that resolver dispatches spawn-CLOSE git-delta disposition by
env-flag; R11 routes leaf-VERIFY by discipline — a NEW routing surface, scope:48-52).

**Its three-phase lifecycle** (mirrors ADR-0015's `detection_phase` enum). **NB (post-VERIFY honesty — see Pre-Approval Verification §2): the declared-vs-measured reconciliation SHIPS OUTCOME-ONLY** — only the test pass/fail half is honestly measurable now; the token + output-schema halves DEFER (R10 records spend per-persona not per-leaf; a node test's stdout is not schema-shaped). The feasible half is NOT bolted on; it IS the `post-spawn-verify` phase below (`budget-abort` is reserved-not-produced):

1. **`pre-spawn-leaf-check`** — structural admission: call R9 `validateLeaf(leaf)`. A failed error-criterion
   → reject before anything runs (a bad leaf never spawns).
2. **`post-spawn-verify`** — runtime verification (the declared-vs-measured reconciliation, part 1): for a
   `tdd` leaf, run its test via R12; reconcile the actual outcome/output against what the leaf declared.
3. **`budget-abort`** — runtime reconciliation (part 2): compare the leaf's declared `estimated_tokens`
   against the **actual R10/`budget-tracker` recorded spend** for its spawn; a material overrun rejects.

**What R11 produces + LOCKS**: the `failure_signature` (ADR-0015, 8 fields). R11 is the **producer**, so it
**locks the `verifier_kind` member-set** (currently provisional: `schema|test-run|structural|registry-lookup|predicate`)
and **promotes the R9 6-criteria set into the v6 RFC** (both deferred-to-the-producer per the lock-at-producer discipline).

**Folded in (USER-approved 2026-06-03):** (a) a thin `./CLAUDE.md` **signpost** (router to where memory/rules/specs
live — NOT memory content); (b) the **declared-vs-measured reconciliation** as a tested R11 acceptance criterion.

## Design (PROPOSED — architect VERIFY settles the open questions)

**Module(s)**: `packages/runtime/verify/spawn-verify.js` (the dispatcher) + `packages/runtime/verify/failure-signature.js` (the builder/validator for the 8-field object — enforces the enums + the structural/diagnostic firewall). Tests under `tests/unit/runtime/verify/` (CI-covered by the widened runtime job).

**Entry point** (the R11 contract): `verifySpawn(leaf, ctx) → { accepted: boolean, failure_signature: object | null, advisories: [...] }` where `ctx = { runId?, cwd?, runTests?: boolean, tunables? }`.
- `accepted === false` ⇔ a non-null `failure_signature`.
- `advisories` surfaces R9's `semantically-cohesive` advisory (never a rejection).

**The dispatch (route by discipline + the 3-phase lifecycle):**

```
verifySpawn(leaf, ctx):
  // Phase 1 — pre-spawn-leaf-check (structural; R9)
  r9 = validateLeaf(leaf)
  if !r9.ok:
     failed = first error-criterion key            // R9's keyed result → failed_criterion_id directly
     return reject(failure_signature(failed, leaf.discipline, verifierKindFor(failed),
                                     'pre-spawn-leaf-check', r9.criteria[failed]))
  // Phase 2 — post-spawn-verify (runtime; only tdd runs a test; spec-driven's gate was R9)
  if leaf.discipline === 'tdd' AND ctx.runTests:
     result = nodeRunner.run({ testFile: leaf.verification.testFile, cwd: ctx.cwd, ... })
     if !result.passed:
        return reject(failure_signature('validation-supported', 'tdd', 'test-run',
                                        'post-spawn-verify', expected/observed from result))
     // declared-vs-measured (output half): actual output vs declared output_schema
     if !outputConforms(result, leaf.output_schema):
        return reject(failure_signature('interface-clean', leaf.discipline, 'schema',
                                        'post-spawn-verify', ...))
  // Phase 3 — budget-abort (runtime; declared-vs-measured token half, needs R10)
  if ctx.runId AND R10-spend(ctx.runId, leaf) materially exceeds leaf.estimated_tokens:
     return reject(failure_signature('resource-bounded', leaf.discipline, 'predicate',
                                     'budget-abort', expected/observed from spend))
  return { accepted: true, failure_signature: null, advisories: r9.advisories }
```

**`verifierKindFor(criterion)`** — the producer's criterion→verifier_kind map (this is what LOCKS the 5-member set; the lock = confirming exactly these 5 are used + sufficient):

| R9 criterion (failed_criterion_id) | verifier_kind | why |
|---|---|---|
| `discipline-gate` | `predicate` | a boolean `isValidDiscipline` check |
| `cost-justified` | `predicate` | a threshold check |
| `interface-clean` | `schema` | an output-shape check |
| `validation-supported` (R9 definition-time) | `registry-lookup` | the `getAdapter` availability lookup |
| `validation-supported` (R11 runtime test) | `test-run` | the actual R12 run |
| `resource-bounded` | `predicate` | a boolean/threshold field read |
| `semantically-cohesive` | (advisory — not a hard reject; surfaced, never emits a rejecting signature) | — |
| `structural` | reserved (no current producer) — see Q2 | — |

**`failure-signature.js`** — `buildFailureSignature({failed_criterion_id, discipline, verifier_kind, detection_phase, leaf_ref?, expected?, observed?, human_message})` validates: required fields present; each of the 4 structural fields ∈ its frozen enum; `human_message` non-empty; returns a frozen object. The **structural/diagnostic firewall** (INV-FS-StructuralDiagnosticFirewall) is honored by construction (E2 reads only the 4 enums).
