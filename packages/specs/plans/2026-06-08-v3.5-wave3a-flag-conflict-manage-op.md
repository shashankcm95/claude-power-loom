## Context — why this wave, why now

v3.5's named thrust is the Memory **Manage**-Layer. What shipped so far: Wave 0 = the *read/project* half (mark-stale / retention-archive projections, which emit NO record); Wave 2 = the causal-*graph* half (the edge store + walker). **Nothing yet builds the *write* half of manage** — the dream-cycle ops (RFC §3) that emit records. This wave starts it.

The scoping pass found that Wave 2 **absorbed scope-W3.2 (R3/R4 enforcement)** — the store self-owns R4 (closed enums + NFC) and the walker owns the R3 read-tier gate + R1 fail-closed default. Combined with **OQ-E NO-GO** (R1 *kernel-derivation* + the writer-identity primitive defer to v3.6), the planned "W3 security spine" collapses to its **Manage-write-layer**. Of the RFC §3 ops, only **`flag-conflict` is a non-destructive CREATE** (`CREATE causal-edge relation=contradicts`); the rest (`content-dedup`→`SUPERSEDE`, `cull`→`TOMBSTONE`, `merge`→candidate summary, `quarantine`→marker) are either R2-destructive (live enforcement is the v3.6 leave-shadow event) or need a new candidate-record shape. So this wave (**3a**) builds `flag-conflict` + its `conflicted` consumer; **Wave 3b** (the destructive-proposal / candidate-record ops) is named-deferred below.

`flag-conflict` is the cleanest first op precisely because it **reuses Wave 2 verbatim**: a `contradicts` edge is already a first-class `RELATIONS` member, `conflict_type` is already required-iff-contradicts + validated, and the walker's EC4 test already proves a contradicts edge keeps its endpoints reachable. So 3a closes a **second producer->consumer loop** (manage-op -> edge store -> walker / projection) rather than building anything parallel.

## D1 (3a) — `flag-conflict` is a thin validated CREATE over the Wave 2 store; the edge is born a CANDIDATE by construction

`flagConflict({ blockX, blockY, conflictType, origin, now? })` emits `store.createEdge({ relation:'contradicts', sourceBlock:blockX, targetBlock:blockY, conflictType, sourceOrigin:origin })`. By the Wave 2 store contract the edge is born `faithfulness_status='unvalidated'` (R1 fail-closed) -> **AUDIT-ONLY** (NOT walker-eligible). **That unvalidated/AUDIT-ONLY birth IS the candidate safety-tag** (see D4 for the full firewall reconciliation). `flag-conflict` is CREATE-only — **no destructive write**.

**Thin wrapper, but it OWNS two things the store does not (VERIFY architect FLAG-5 / code-reviewer):** (1) it **pins `relation='contradicts'`** (the caller never supplies `relation`), and (2) it **validates `conflictType` PRESENCE before the store call** so a missing one yields a clean error naming *flagConflict's* contract, not the store's internals (the store would also reject it, but with a store-internal message). All other validation (closed-enum `conflictType`, the block free-strings, the lock, the content-address, R1/R4) **delegates to `store.createEdge`** — no second admission point (DRY; the single-gate discipline the walker header warns to preserve). It returns the candidate edge record, OR the existing row (dedup), OR a clean error / soft-skip — never a stack dump (the advisory contract).

**`conflict_evidence` is NAMED-DEFERRED, NOT folded (VERIFY architect FAIL-1 — load-bearing honesty).** The RFC §3 table gives `flag-conflict` an `evidence_refs` triple `[x, y, conflict_evidence]` — separating what the op ACTS ON (the two blocks) from what JUSTIFIES it (the conflict evidence), the same affected-vs-justifies discipline the Wave-0 projection established. **The Wave 2 edge record has NO `evidence_refs`/`affected_records` field** (firsthand: `store.js:224-235` — the frozen record is `{edge_id, relation, source_block, target_block, conflict_type, faithfulness_status, source_origin, recorded_at}` and nothing else). So a 3a edge **cannot carry `conflict_evidence` without a Wave-2 store schema change** — which would break the thin-wrapper thesis. Therefore 3a carries the flag's **`origin` only** (the authoring provenance — *who/what flagged it*, NOT *why x⊥y*); **`conflict_evidence` (the justifying-record list) is deferred to 3b/v3.6 with its human-disposition consumer** (where a candidate-record shape with `evidence_refs` is built). Do NOT conflate evidence into `source_origin`.

## D2 (3a) — `conflicted` is a LAB projection over the edge store, NOT a kernel lifecycle state (D1 reshaped the Wave-0 anticipation)

`provenance-projections.js:11` (Wave 0) says "`conflicted`/`quarantined` arrive with W2/W3" — it **anticipated `conflicted` as a kernel `derivedLifecycleState`**. But D1 (Wave 2) moved the contradicts edge into the **Lab** causal-edge store, and **the kernel cannot read Lab (inner->outer = K12-illegal)** — so the kernel projection can never compute `conflicted` from a Lab edge. Resolution: **`conflicted` is a pure LAB projection** (`conflictedBlocks(edges)`), mirroring `provenance-projections`' pure-over-passed-in-set style but in the Lab layer over the edge set. A block is:
- **`conflicted` (confirmed)** iff it is an endpoint of an R3-**eligible** (`advisory_llm_checked`/`human_confirmed`) `contradicts` edge;
- **`conflict-candidate`** iff it is an endpoint of an `unvalidated`/`surface_overlap_only` `contradicts` edge (a flagged-but-unjudged conflict).

Both tiers are **advisory ANNOTATION, retrieval-eligible — never suppression** (RFC §3.1 / the security F-finding: `conflicted` annotates, it does not hide). The Wave-0 `provenance-projections.js:11` comment is now stale and gets a one-line reconciliation (conflicted is Lab-derived post-D1). **VERIFY must confirm this kernel-vs-Lab placement** (the load-bearing design call of this wave).

## D3 (3a) — the loop is closed function-level: `flag-conflict` -> candidate edge -> rung-2 judge -> confirmed -> `conflicted`

This wave wires the producer (`flag-conflict`) to the consumer (`conflictedBlocks`) THROUGH the Wave 2 rung-2 gate: a flagged conflict is a *candidate* until a rung-2 judge (the injectable `rung2AdvisoryCheck`, Wave 2) supports it and the caller promotes it via `updateEdgeStatus`; only then does `conflictedBlocks` count it `confirmed`. This is what gives the **W2.3-owed real judge a home** — and it stays narrowing-safe (an un-judged flag never confirms a conflict). The loop is **function-level, NOT production-wired into live recall** (no production edge-producer beyond `flag-conflict` itself; wiring into K4 recall is a v3.6 leave-shadow event - architect). State this so VALIDATE honesty does not catch an overclaim.

## D4 (3a) — the §0a.3.1 shadow firewall: clauses (a)/(b) are N/A by LAYER; clause (c) = the walker-exclusion

The scope's §0a.3.1 candidate-firewall (scope:55) has 3 conjuncts, authored for KERNEL candidate-records: (a) `commit_outcome:PENDING`/lifecycle `candidate`; (b) `assertion_class=stochastic_sample`; (c) never feeds a forbidden sink (L_spawn->L_persona promotion / persona-contract update / capability assignment) without a human-disposed kernel-attested transaction. A flag-conflict edge is a **Lab advisory edge, NOT a kernel record**, so the reconciliation is by LAYER, not by field (VERIFY architect FAIL-2):

- **Clauses (a) + (b) are N/A by LAYER.** `commit_outcome` and `assertion_class` are kernel-envelope tags; the Lab edge record has neither (firsthand: `assertion_class` absent, `grep -c`=0; no `commit_outcome`). A Lab edge is firewalled by a **STRONGER boundary than either tag** — it is structurally ABSENT from the kernel substrate (0 `hooks.json` refs; the kernel CANNOT read Lab, K12 inner->outer / ADR-0012). There is no kernel candidate-record to tag, so clauses (a)/(b) do not apply rather than being "satisfied by construction."
- **Clause (c) = the walker-exclusion** — the analog for the ONE sink a Lab edge can reach (advisory walker reads). Traced firsthand: a flag-conflict edge is born `unvalidated` -> `isEligible` (`walker.js:63-72`, admits only `advisory_llm_checked`/`human_confirmed`) excludes it -> FILTER-THEN-INDEX means it never enters the adjacency index -> it surfaces in NO walker mode (`walker.test.js:72-85`). The only promotion path is `updateEdgeStatus`, gated by the fail-closed `rung2AdvisoryCheck` (`faithfulness.js:84-103`, `supported!==true` -> no-promote, ceiling `advisory_llm_checked`). So an un-judged flag cannot feed any forbidden sink.

**Net:** the firewall holds by LAYER-ABSENCE + walker-exclusion, NOT by a missing `assertion_class` field (resolves the architect's #1 mid-build trap: a builder reading the scope/RFC literally would reach for `assertion_class`). This is the framing VALIDATE honesty will check.

### W3a.2 — `conflicted` projection (`packages/lab/causal-edge/projections.js`; test at `tests/unit/lab/causal-edge/projections.test.js`)
- PURE over a passed-in edge array (the `provenance-projections` precedent; the STORE does the bounded read + feeds edges in). **Return shape PINNED (VERIFY code-reviewer F1 — an ambiguous shape can't have a deterministic red test): `conflictedBlocks(edges) -> Map<string, {tier:'confirmed'|'candidate', edges: object[]}>`** (keyed access, matching the provenance-projections precedent; tests assert `result.get('A').tier === 'candidate'`).
- **The confirmed tier MUST filter `relation==='contradicts'` FIRST, THEN apply eligibility (VERIFY code-reviewer F3 — a real correctness trap):** `isEligible` (`walker.js:63-72`) admits ALL 9 relations, so `edges.filter(isEligible)` alone would let a `caused_by`/`advisory_llm_checked` edge falsely mark its endpoints conflicted. The contradicts gate is `e.relation === 'contradicts' && isEligible(e)`. Reuse `isEligible` (DRY) but pre-filter to contradicts. **Test:** a non-contradicts eligible edge in the input -> its endpoints are NOT annotated.
- **Precedence PINNED (VERIFY code-reviewer F2): a block with AT LEAST ONE R3-eligible contradicts edge is `confirmed`, regardless of co-existing unjudged contradicts edges; `candidate` applies ONLY when ALL touching contradicts edges are unjudged.** Test the MIXED case (a block with both an eligible and an unjudged contradicts edge -> `confirmed`).
- **Annotation, not suppression — re-scoped to what 3a can actually test (VERIFY architect/code-reviewer F4):** 3a has NO retrieval layer, so the projection cannot prove a downstream reader honors the annotation. The TESTABLE form is STRUCTURAL: (i) after `flagConflict(A,B)`, `conflictedBlocks(store.listEdges())` returns BOTH `A` and `B` as keys (surfaced, not filtered); (ii) `store.listEdges().length` is UNCHANGED after the call (additive metadata — neither filters nor mutates). The real annotation-not-suppression guarantee (a retrieval reader returns conflicted blocks) is OWED to the v3.6 K4-recall wiring (NAMED, not tested in 3a).
- Single pass over edges (endpoint index); NOT a walker (no graph traversal). **Scoped to `contradicts` edges ONLY in 3a** (VERIFY architect FLAG-4); 3b's quarantine/dedup states are a SEPARATE projection (a 3b builder must not assume `conflictedBlocks` covers them).
