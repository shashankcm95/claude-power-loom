## Design

### D1 â€” the pure module `trajectory-friction.js` (additive sibling; import allow-list = `corpus.js` only; Linux-CI-safe)

NO `child_process`, NO `claude`, NO sandbox backend, NO `*-run` module â€” the deterministic core, TDD'd with synthetic NDJSON fixtures shaped like the RP-1 real capture. Exports:

- **`parseTrajectory(streamEvents)` â†’ `{rows, dropped_noise, unpaired}`** â€” filter to `assistant.tool_use` + `user.tool_result`; pair by `tool_use.id`; one row per tool_use: `{step_idx, tool_name, tool_input_digest, target_path|null, thought_digest, observation_digest, is_error}`. `thought_digest` = digest of the `thinking`+`text` blocks preceding the tool_use in the same assistant message. `target_path` = a grader-side extraction of `input.file_path|path|pattern` for file-targeting tools (Read/Edit/Write/Grep/Glob) â€” used ONLY by the grader-side recall-smell, NEVER emitted to a blind leg. `tool_input_digest`/`observation_digest` are bounded sha256[:16] (privacy + bound). Robust to RP-1a (string|array content, absent is_error, noise events).
- **`computeProcessGraph(rows)` â†’ Layer-1 metrics** â€” map each tool to a `phase` via a frozen `TOOL_PHASE` table (Read/Grep/Glob â†’ `localization`; Edit/Write â†’ `editing`; Bash â†’ `validation` if the command looks test-like else `editing` â€” a bounded heuristic, documented as such; Task/other â†’ `null`/unclassified). Then derive `{phases:[...], loop_count, avg_loop_length, back_edge, reached_validation_before_submit, localization_reads:[paths], n_localization, n_editing, n_validation}`. `back_edge` = a transition to an earlier phase (editingâ†’localization); `loop_count` = count of phase oscillations; `reached_validation_before_submit` = a `validation` step occurred at all.
- **`detectRecallSmell({processGraph, relevantFiles, reachedResolution, lowLoopMax})` â†’ `{recall_smell, signals}`** â€” the TWO-signal heuristic. `low_loop = loop_count <= lowLoopMax`; `relevant_files_unread = relevantFiles.length > 0 && relevantFiles.every(f => !localization_reads.includes(f))`; `recall_smell = low_loop && reachedResolution && relevant_files_unread`. Fail-CLOSED: with no `relevantFiles` (can't tell) â†’ `recall_smell=false`, `signals.relevant_files_unread='UNKNOWN'`. NEVER fire on trajectory-shape alone.
- **`FRICTION_CLASS` / `FRICTION_PHASE` / `DETECTION_LEG`** (frozen enums, Â§3.6): class = `wrong-file | wrong-edit-location | incorrect-implementation | over-editing | incomplete-fix | instruction-misread | hallucinated-api | gave-up | cant-reproduce | ran-out-of-budget`; phase = `localization | editing | validation`; leg = `behavioral | semantic-lens | reference-anchor`.
- **`buildResolutionFriction({friction_class, friction_phase, detection_leg, expected, observed, human_message})` â†’ block** â€” validate each enum against the frozen set; **throw on an unknown value** (fail-closed, the closed-enum contract). `expected/observed/human_message` are free-form diagnostics stored on the block but **the deterministic clusterer MUST NOT parse them** (carried under a `_diagnostic` sub-object the cluster key ignores).
- **`frictionClusterKey(block)` â†’ `"class|phase|leg"`** + **`clusterFriction(blocks)` â†’ `{clusters, n}`** â€” the dual-representation clusterer. The tuple string is the deterministic dedup/cluster KEY (analogous to E11's dedup-by-`(persona, agent_id)`). An OPTIONAL `block.embedding` is ACCEPTED for the depth layer but is NEVER read into the key (a RED test asserts two blocks with the same tuple but different embeddings cluster together).
- **`validateRecallSmellAgainstControls(labeled)` â†’ discrimination report** â€” the RFC Â§3.3 mandate. Given an array of `{is_negative_control, recall_smell}`, compute the heuristic's **false-positive rate on negative controls** + true-positive shape on the clean track, reported as `{neg_control_fp_rate, n_neg, n_clean, discriminates: <boolean-with-threshold>, error_bar: 'UNKNOWN-until-measured'}`. Honest: the bar is MEASURED on this corpus (Â§10), never the borrowed 87%/13% (that is an out-of-distribution analogue â€” different model, different task).
