# v3.6 Wave 2a — the first destructive MINT

## Context

W1 (#269) shipped the *reader* (`manageLifecycleStatus`) that surfaces a record's manage-lifecycle — but its
kernel-half is dark until something MINTS a COMMITTED `SUPERSEDE`/`TOMBSTONE`. W2a ships that mint, in its
narrowest real form: **a human promotes ONE explicit `approved` `cull` proposal with ONE target -> a COMMITTED
`TOMBSTONE` -> the W1 reader reports `tombstoned`.** This is the **leave-shadow event** (the first time the
substrate mutates the kernel WAL on a human verdict). It is **opt-in + shadow-default** (a flag gates it; the
human is the trust anchor) under the cooperative/trusted-local-fs threat model. Multi-target, the `SUPERSEDE`
ops, and the E11 breaker are W2b.

## Design decisions (the load-bearing calls — for VERIFY to pressure-test)

1. **The manage-op record shape (genesis-rooted, non-tree-advancing):** `operation_class:'TOMBSTONE'`;
   `prev_state_hash=computeGenesisHash(schemaVersion,'per-project')` (rooted, not forked); `post_state_hash:null`
   (no git tree — a logical op; `findAffectedByOp` scans by op+`affected_records`, not by `post_state_hash`);
   `affected_records:[<target txid>]`; `evidence_refs:['USER_INTENT_AXIOM:<sha256(canonical(approved-proposal))>']`
   (the human approval IS the A10 bootstrap evidence — sentinel-valid, non-empty, distinct from
   `affected_records`); `commit_outcome:'COMMITTED'`; `writer_persona_id:'kernel:manage-promote'`;
   `writer_spawn_id:'manage-promote:<proposal_id>'` (stable per proposal -> re-promoting an approved proposal
   is an INV-22 idempotent no-op); `content_hash`/`idempotency_key`/`transaction_id` derived per `buildSpawnRecord`.
2. **op_type -> operation_class mapping:** `cull` -> `TOMBSTONE` (W2a). `content-dedup`/`merge` -> `SUPERSEDE`
   (W2b). `quarantine` -> a **recall-layer suppression** (v3.8a K4-recall, per `enums.js:9-11`), NOT a kernel
   op — W2a's `promote` REFUSES a `quarantine` proposal with a clear "deferred to recall" message.
3. **The run-scoping seam — RESOLVED to "write the op into the TARGET's run" (Option A):** a new content-
   addressed locator `findRecordRun(txid,{stateDir})` globs `<stateDir>/*/records/record-<txid>.json` -> the
   target's `runId`; the TOMBSTONE is `appendRecord`-ed into that SAME run, so `listByRun(R_T)` returns both
   the target and the op -> `findAffectedByOp` links them -> the W1 reader (given `listByRun(R_T)`) reports
   `tombstoned`. **Rejected Option B** (write to a manage run + a global reader scan): it globalizes reads and
   breaks the run-scoping invariant. A target that exists in NO run -> `findRecordRun` null -> the promotion
   REFUSES (fail-closed; cannot tombstone a phantom).
4. **TOCTOU re-verify (the architect VERIFY-HIGH from the arc):** `promote` takes an EXPLICIT `--proposal-id`
   (the human names what they reviewed). It re-loads via `listProposals` (which re-derives `proposal_id` —
   `isAuthenticProposal`), asserts `disposition==='approved'`, and re-derives `proposal_id` from the body.
   A target-set swapped after approval changes the id -> not the named id -> REFUSED; a forged proposal has a
   different id -> not named. The content-address IS the defense.
5. **Flag-gate (shadow stays default):** `promoteProposal` is a no-op-REFUSE unless `LOOM_MANAGE_ENFORCE==='1'`
   (opt-in; mirrors `LOOM_RESOLVER_ENFORCE`). Default = shadow (the mint never fires).
6. **OQ-E residual (documented, sandbox-deferred):** `writer_persona_id` is un-attested — a same-uid actor can
   forge a COMMITTED `TOMBSTONE` directly (the OQ-E gap). Accepted under the cooperative threat model; the
   human approval (`USER_INTENT_AXIOM` evidence) is the trust anchor; closes only at the ContainerAdapter sandbox.
7. **Layer (K12):** `lab/manage-proposal/promote.js` reads the Lab proposal + calls the kernel/_lib builder +
   `appendRecord` (lab->kernel/_lib LEGAL). The builder + locator are kernel/_lib (they mint/locate kernel records).

## Pre-Build VERIFY (architect + hacker) — findings folded

The first WAL mutation drew a 2-lens pre-build VERIFY. **architect = SOUND-WITH-FIXES; hacker =
RESIDUAL-UNDERSTATED (1 CRITICAL + 3 HIGH, TRIGGERED with live `node` probes, not traced).** The core design
holds (the genesis-rooted `null`-post TOMBSTONE is coherent — validated firsthand; a synthetic
`post_state_hash` would be a *fabricated* chain edge; run-seam Option A holds; quarantine-out + the
decomposition are right). But the mint needs real hardening. All folded:

- **CRITICAL (hacker) — INV-22 poison-key fail-OPEN.** `content_hash` binds only `{post_state_hash,
  writer_spawn_id, head_anchor}` (`transaction-record.js:167-174`); `affected_records` is in NEITHER hash
  input, and `writer_spawn_id='manage-promote:<proposal_id>'` is deterministic+public. An attacker pre-plants a
  decoy COMMITTED TOMBSTONE with the SAME `idempotency_key` but `affected_records:[decoy]`; the human's promote
  -> INV-22 dedup (`record-store.js:249-253`) -> writes nothing -> returns the decoy's id -> the real target is
  NEVER tombstoned while the operator sees success. The shipping `deriveIdempotencyKey` defense does NOT catch
  it (the poison's key genuinely derives from its own body). **FOLD — a POST-CONDITION check in
  `promoteProposal`:** after `appendRecord`, re-read the result record (INCLUDING the `deduped:true` path) and
  HARD-FAIL unless `operation_class==='TOMBSTONE'` AND `affected_records` contains the expected target. (Binding
  `affected_records` into `content_hash` is REJECTED — it requires changing the kernel `deriveIdempotencyKey`/
  `computeContentHash`, breaking INV-22 for all records; the lab-level post-condition is the correct-scoped fix.)
- **HIGH (hacker) — IDOR: unconstrained kill-list.** A cull can tombstone ANY record — a kernel provenance
  record, another persona's/run's record (`findAffectedByOp` matches only `operation_class`+`affected_records`;
  no ownership check). **FOLD — a TARGET-ELIGIBILITY gate in `promoteProposal`:** resolve the target
  (`findRecordRun`+`readById`) and REFUSE if `writer_persona_id` is `kernel:`/`kernel-`-namespaced (VALIDATE NEW-2)
  OR `operation_class` is a manage op (SUPERSEDE/TOMBSTONE/DERIVED-VIEW-INVALIDATE) OR the target is absent. A
  cull may tombstone only an eligible memory record (a non-kernel CREATE/APPEND).
- **HIGH (hacker) — run-seam glob escape.** `isSafePathSegment(runName)` passes a SYMLINKED run dir (clean
  basename; symlink invisible), and the glob returns first-match over UNVALIDATED filenames (a decoy
  `record-<T>.json` in an earlier-sorted run wins). **FOLD — `findRecordRun` gets the readers' S1/S1b
  discipline:** hex-gate the txid; after the glob, `loadRecordFile`+validate the candidate (reject
  garbage/non-records); `checkWithinRoot(matchedDir, stateDir)` on the REALPATH-resolved (symlink-collapsed)
  path before returning the runId.
- **MEDIUM (architect) — multi-run first-match ambiguity.** A txid content-addressed into >1 run -> readdir
  roulette. **FOLD — REFUSE on >1 match** (fail-closed; "target in multiple runs -> W2b"), with a test.
- **MEDIUM (architect) — idempotency binding indirect.** **FOLD —** `buildManageOpRecord` derives
  `writer_spawn_id` from `proposalId` internally (no free-form param) + a test: different target sets ->
  different keys; re-promote -> same key (dedup).
- **MEDIUM (architect) — no breaker.** **FOLD (doc, not breaker-pull-forward):** the per-invocation explicit
  `--proposal-id` + INV-22 dedup IS the W2a rate-bound; the E11 breaker generalizes it in W2b. Probe: promote
  acts on exactly one proposal/call + no-ops on re-promote.
- **MEDIUM (hacker) — `kernel:`-namespace mint by the Lab layer.** A `lab/` orchestrator stamping
  `writer_persona_id:'kernel:manage-promote'` lies about authorship. **FOLD — stamp `lab:manage-promote`** (the
  mint is Lab-originated + un-attested until the sandbox; never claim kernel authorship).
- **MEDIUM (hacker) — TOCTOU honesty.** The content-address closes target-swap + forged-id, but NOT a same-uid
  `updateDisposition('approved')` on an attacker's GENUINE proposal. **FOLD (reframe §Design 4):** under the
  cooperative model this collapses to OQ-E — `promote` trusts `disposition==='approved'` as a same-uid
  assertion; the human-in-loop is the ONLY gate distinguishing a real approval. Stated, not overclaimed.
- **LOW (both) — A10 axiom must bind the real proposal.** **FOLD —** `evidence_refs[0]` =
  `USER_INTENT_AXIOM:<sha256(canonicalJsonSerialize(<the loaded approved proposal>))>` + assert it in the builder test.

**Net — W2a is now the HARDENED mint:** builder (proposal-derived `writer_spawn_id` + bound axiom + `lab:`
namespace) + a hardened locator (validate + realpath + multi-run-refuse) + `promoteProposal` (TOCTOU re-verify +
**target-eligibility gate** + **POST-CONDITION verify** + flag-gate) + CLI. The CRITICAL fail-open + both HIGHs
close in the Lab orchestrator + the locator; the residuals (OQ-E forge, same-uid self-approval) are documented +
cooperative-model-accepted. The 3-lens VALIDATE will re-run the hacker's 9 probes against the built code.

  the check (closes the `KERNEL:` + leading-whitespace MED-1 evasions). + test `IDOR NEW-2`. Test fixed to seed the real persona.
- **code-reviewer HIGH — CREATE plan-vs-code divergence.** The §VERIFY IDOR fold wrongly parenthesized
