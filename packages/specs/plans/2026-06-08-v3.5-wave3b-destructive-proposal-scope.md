## Context — what Wave 3b is, why now

v3.5's Manage-Layer has a **read/project** half (Wave 0), a **causal-graph** half (Wave 2), and the first **write** op (Wave 3a `flag-conflict`). Wave 3b is the rest of the manage **write** half — the **destructive-proposal / candidate-record ops** (RFC §3): `quarantine`->marker, `content-dedup`->SUPERSEDE-proposal, `cull`->TOMBSTONE-proposal, `merge`->candidate-summary. The 3a plan named-deferred these as needing "a NEW candidate-record shape (not a causal edge) + a human-disposition surface; the destructive class stays a PROPOSAL in v3.5 (live enforcement = the v3.6 leave-shadow event)."

The pre-3a scope's **W3.3** (`2026-06-07-v3.5-memory-manage-scope.md:55`) assumed a candidate is a KERNEL record (`commit_outcome:PENDING` + `assertion_class=stochastic_sample`). **3a's D1/D2 reshape falsifies that.** This scope resolves the central question via the probed current state.

## D1 (3b) — destructive proposals live in a dedicated ADVISORY Lab store, NOT a kernel PENDING record

**Decision: a new Lab proposal store** (a `packages/lab/causal-edge/` sibling — recommend a new `packages/lab/manage-proposal/` directory, see Open Q2), the §10b/OQ-24 advisory-cache class the causal-edge store occupies. The eventual v3.6 promotion (proposal -> kernel-attested COMMITTED `SUPERSEDE`/`TOMBSTONE`) is a SEPARATE v3.6 event. A clean **two-layer model**, NOT a §4.2 dual-write trap.

The pre-3a kernel-PENDING framing is **falsified on three independent probed axes:**

1. **The inert-control trap fires harder than in 3a.** RFC §3 maps the destructive ops onto `operation_class`, the **v6-LOCKED** kernel enum (`transaction-record.schema.json:60-70`; RFC explicitly "§3 mints no new `operation_class`"). The kernel schema is **documentary — draft-07, no ajv runtime** (3a D1). A proposal discriminator on a kernel record is an INERT control (ADR-0012) — nothing reads the schema at write time to enforce it. Identical to the reasoning that moved the contradicts edge to Lab in Wave 2.
2. **`assertion_class` does NOT exist — pure RFC fiction, never built.** `grep assertion_class` over the kernel schema = **0**; the schema `required` list has `operation_class`/`commit_outcome` but no `assertion_class`; `transaction-record.js` validates those two only. It appears solely in RFC prose ("DERIVED from `writer_persona_id` ... not stored"). 3a D4 already caught this (the "#1 mid-build trap"). A kernel PENDING record cannot carry the very tag the firewall names.
3. **A kernel PENDING destructive record is BOTH inadmissible AND invisible.** Inadmissible: `SUPERSEDE`/`TOMBSTONE` are `isStateChanging`, so A10 requires **non-empty `evidence_refs` pointing at records existing in the chain at `prev_state_hash`** (`transaction-record.js:434-437`); a dream-cycle "these are dupes" proposal has no kernel-chain evidence -> fails validation today, or needs the NO-GO OQ-E writer. Invisible: `findAffectedByOp` counts **`commit_outcome:'COMMITTED'` only** (`provenance-projections.js:59`) -> a PENDING record is read by nobody (Risk #4: "no live SUPERSEDE/TOMBSTONE emitter exists today").

**Not a dual-write trap (§4.2):** a Lab proposal store is not canonical anything — it is a regeneratable advisory cache (E1/E4/causal-edge class). It holds a *proposed future* kernel state that has no canonical existence until v3.6 promotion mints a real kernel transaction. The kernel `affected_records`/`evidence_refs` chain stays the single source of truth for what *happened*; the Lab store holds what a dream-cycle *suggests*. Exactly the 3a D1 argument (the edge is §10b-advisory, not a §4.2 amendment).

## D2 (3b) — the candidate-record shape (a new Lab record, distinct from a causal edge)

A proposal asserts *a recommended destructive op on N existing kernel records* (identity = op + targets, mutable disposition) — NOT a semantic relation between blocks. Different identity, different mutable axis, different consumer -> a **separate store + separate projection** (3a W3a.2 already warned: "3b's quarantine/dedup states are a SEPARATE projection").

Record (mirroring `causal-edge/store.js` hardening verbatim — DRY against the proven pattern):

| Field | Source / semantics |
|---|---|
| `node_type` | `'manage-proposal'` (a Lab discriminator, like `'causal-edge'` — NEVER a kernel `operation_class`) |
| `proposal_id` | content-address = `sha256(canonicalJsonSerialize([op_type, ...sorted target_records]))`; re-derived + checked on read (INV-22 `isAuthenticEdge` discipline) |
| `op_type` | closed enum `{quarantine, content-dedup, cull, merge}` — R4-validated (`validateEnum` NFC/homoglyph reuse) |
| `target_records` | the kernel `transaction_id`s the op acts on — the proposal's `affected_records` analog. **HEX64-validated (these ARE 64-hex kernel txids, NOT free strings — Trap 8); reuse `HEX64` from `provenance-walk`** |
| `justification` | free-string "why" (the slot 3a's `flag-conflict` lacked; where `conflict_evidence`-class content can finally live — Open Q3). Length-capped + control-char-rejected (`validateFreeString`) |
| `proposer_origin` | provenance of the FLAG (the dream-cycle run) — free string. Authoring provenance, NOT authority (writer-unauthenticated, OQ-E NO-GO) |
| `disposition` | mutable `{pending, approved, rejected}` — the supersede-the-status axis (the `updateEdgeStatus` analog `updateDisposition`); born `pending` (R1 fail-closed analog) |
| `recorded_at` | ISO; the same `MAX_DATE_MS` guard |

Dedup-on-`proposal_id` (one live row per identity), idempotent create, status-supersede-not-second-row. Do NOT accumulate (the E1/W1 contrast).

## D3 (3b) — firewall mapping: 3a's layer-absence reasoning extends, with a sharpened sink

The §0a.3.1 forbidden sinks (`v6-substrate-synthesis.md:175`): a derived view MUST NOT widen capability, enter a state-transition that writes a kernel-canonical record, propagate as instruction-text into peer LLM contexts, or grant trust not evidence-linked to a verified-outcome kernel record.

- **Conjuncts (a) `commit_outcome:PENDING` + (b) `assertion_class` -> N/A BY LAYER** (kernel-envelope tags; a Lab record has neither — the stronger layer-absence boundary, 3a D4).
- **Conjunct (c) "never feeds a forbidden sink" = disposition-exclusion + execution-absence.** **Naming a kernel txid in `target_records` is INERT data** — provable: `findAffectedByOp` reads `affected_records` from kernel records ONLY (never Lab); there is no Lab->kernel write path (K12; the store imports only `kernel/_lib` primitives, never `record-store`/`transaction-record` STATE). So a proposal naming `txid X` cannot supersede X, cannot enter a K9 promote-delta, cannot update a persona contract. It is a string in an advisory cache.
- **The one reachable sink = the advisory disposition/listing read.** The walker-exclusion analog: an **un-approved proposal must not be presented as actionable** (the consumer gates on `disposition==='approved'`, exactly as `conflictedBlocks` gates `confirmed` on `isEligible`). And **even an `approved` proposal does nothing destructive in v3.5** — approval is recorded, not executed. Narrowing-safety holds: a forged `approved` (writer-unauthenticated, the accepted OQ-E NO-GO boundary) reaches advisory reads only.

**Net (state it explicitly for VALIDATE honesty):** the firewall holds by LAYER-ABSENCE + DISPOSITION-EXCLUSION + EXECUTION-ABSENCE — NOT by `assertion_class`/`commit_outcome`.

## D4 (3b) — the human-disposition surface

A CLI mirroring the proven Wave-2 shape (`causal-edge/cli.js` has `create`/`list`/`update-status`):

- `propose <op-type> --targets <txid,txid> --justification "..." --origin <run>` -> a `pending` proposal (the producer; the `flag-conflict` analog).
- `list-proposals [--disposition pending]` -> the human review surface (the `list` analog).
- `dispose --proposal-id <id> --decision approved|rejected` -> the human acts (the `update-status` analog; the CLI operator IS the disposing human, the 3a model).

Disposition lives in the record's mutable `disposition` (updated via `updateDisposition`, RMW under the advisory lock, replace-in-place-with-a-new-frozen-record — no separate disposition store). **v3.5 `approved` = recorded-not-executed (NON-destructive — confirmed; the blast radius is one string flips).** v3.6 promotion (named) reads `approved` proposals and mints the kernel-attested COMMITTED op. The seam = the `approved` flag (Producer-Consumer Phasing: the v3.6 consumer is named now, never silent debt).

A clean producer->store->`quarantinedBlocks`-projection loop, zero destructive surface. **Honesty caveat (load-bearing):** RFC §3.1 says `quarantined` is retrieval-SUPPRESSED, but **v3.5 has no retrieval layer to suppress** (3a F4) -> in 3b.1 quarantine is a **recorded marker + a surfacing projection**; the actual suppression is OWED to v3.6 K4-recall (named, not built). Same shape as 3a (a marker + a projection, no live effect). This is the clean NO-GO fallback unit.
- **3b.2 — the destructive-PROPOSAL ops** (`content-dedup`->SUPERSEDE-proposal, `cull`->TOMBSTONE-proposal, `merge`->candidate-summary). Thin validated wrappers over the 3b.1 store (the `flag-conflict`/`manage-ops.js` pattern: each owns its op-type pin + target-presence guards, delegates validation/lock/content-address). **CREATE-only into the Lab store — zero kernel writes, zero `operation_class`.** The "destructive" is notional (a proposed future op) until v3.6. Sequence: `content-dedup` + `cull` are symmetric (targets + justification -> pending proposal); `merge` carries a proposed-summary payload (richer) -> last.
- **3b.3 — the human-disposition surface + the closed loop.** The `propose`/`list-proposals`/`dispose` CLI subcommands + the function-level loop test (propose -> pending -> dispose approved -> `disposition:approved`, NON-destructive) mirroring 3a's W3a.3 loop. Firewall asserts: (a) an un-approved proposal is not actionable; (b) no destructive write in any manage-op (source-scan: 0 `SUPERSEDE`/`TOMBSTONE` in the Lab files); (c) SHADOW (`grep -c "lab/\|causal\|manage-proposal\|proposal" packages/kernel/hooks.json` = 0).

Order rationale: 3b.1 proves the store + the safest op in isolation (clean fallback); 3b.2 adds the proposal ops once the store is proven; 3b.3 closes the human loop last. The consumer-first/safest-first discipline 3a + Wave 0 followed.

## Load-bearing traps (a builder must not miss)

1. **The `assertion_class` field-trap (#1, recurs from 3a).** The scope/RFC firewall names it; it does NOT exist (`grep`=0). The firewall holds by LAYER-ABSENCE, not a field.
2. **The kernel-PENDING-record trap.** RFC §3's "emits: SUPERSEDE/TOMBSTONE" is CONCEPTUAL; a literal kernel producer fails A10 validation OR needs the NO-GO OQ-E writer AND is invisible to `findAffectedByOp`. The realization is a Lab advisory write (the 3a `flag-conflict` precedent).
3. **"No live destructive write" is the hard invariant.** No live SUPERSEDE/TOMBSTONE emitter exists; 3b must NOT introduce one. Every manage-op is CREATE-into-Lab-only. The source-scan firewall assert is mandatory (3a W3a.3(c)).
4. **3b does NOT do detection.** The contradiction/near-dup DETECTOR is weak (55.2%, deferred v3.6+). 3b records/disposes proposals; it does not generate them. State this so VALIDATE doesn't catch an implied "3b detects dupes" overclaim.
5. **Inert-control (ADR-0012).** Any attempt to make the kernel schema enforce a proposal discriminator is inert theater. Enforcement lives in the Lab store's own validation boundary (R4 self-ownership).
6. **Writer-unauthenticated is the accepted boundary.** `propose`/`updateDisposition` are writer-unauthenticated (OQ-E NO-GO); a forged `approved` is bounded by narrowing-safety. Hacker M-findings here are DEFERRED-with-reason (the v3.6 OQ-E item), like 3a's M1/M2 — pre-state it in the plan.
7. **Quarantine suppression vs conflicted annotation asymmetry.** `quarantined` is retrieval-SUPPRESSED in the RFC; `conflicted` annotates. With no v3.5 retrieval layer, 3b.1 quarantine is a recorded marker + surfacing projection (real suppression owed v3.6). Don't claim 3b.1 "suppresses" anything.
8. **`target_records` are HEX64 kernel txids, NOT free-string blocks.** Validate with `HEX64`, not `validateFreeString`. A builder copying the edge store verbatim will get this wrong.
