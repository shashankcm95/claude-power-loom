## Context — what 3b.2 is

3b.2 is the rest of the manage **write** half: the three remaining `op_type` wrappers (`content-dedup` -> SUPERSEDE-proposal, `cull` -> TOMBSTONE-proposal, `merge` -> candidate-summary). The op-type enum **already lists all four** (`enums.js:28` `OP_TYPES = ['quarantine', 'content-dedup', 'cull', 'merge']`); 3b.1 built only the `quarantine` wrapper. 3b.2 adds the other three as **thin validated wrappers over the EXISTING store** — the `quarantineRecord` pattern — plus their CLI subcommands. **CREATE-into-Lab-only: zero kernel writes, zero `operation_class`, zero live destructive effect** (the "destructive" is notional until the v3.6 promotion pass). This wave **completes the planned write-op set**; whether it CLOSES the manage-write-layer is the `/phase-close v3.5` gate's call (EC1-EC5), not this plan's (see "Sub-wave boundary" below).

## The central finding (drives the whole plan): the store needs NO schema/identity change (ONE defensive read-path freeze)

`createProposal({opType, targetRecords, justification, origin, now})` already takes a **multi-target array** and hardens it fully: `validateTargets` enforces non-empty + HEX64-per-element + dedup+sort + `MAX_TARGETS=256`; `computeProposalId` content-addresses over the canonical target set; `freezeProposalRecord` deep-freezes `target_records` on three of four return paths (create/dedup/update). So `content-dedup`/`cull`/`merge` are **pure additions to `manage-ops.js` + `cli.js`** — no schema/field/identity/enum change.

**The ONE store change (VERIFY code-reviewer HIGH F1):** `listProposals` (`store.js:307-311`) returns raw `readLedger()` output — JSON-parsed, **un-frozen** — so the 4th return path leaks a mutable record (the 3b.1 shallow-freeze lesson, applied incompletely). Add `.map(freezeProposalRecord)` at the `listProposals` return boundary so the store's immutability guarantee is uniform across all read/return paths. Internal RMW callers (`createProposal`/`updateDisposition`) keep raw `readLedger` (they slice+spread, never mutate in place — frozen would be safe too, but the boundary freeze is the minimal, targeted fix). This makes the planned read-back-freeze test GREEN-able. (causal-edge `listEdges` has the same structural gap but no nested array field to leak -> a named promote-candidate, NOT folded here.)

## Decisions — the 3 open questions, resolved against the probed store

- **OQ1 (identity basis — does `merge[A,B]` dedup with `merge[A,B,C]`?): RESOLVED by the existing store — DISTINCT identities.** `computeProposalId = sha256(canonicalJsonSerialize([opType, ...canonicalizeTargets(targets)]))`. Different target SETS -> different canonical array -> different `proposal_id`. A different blast radius is a different identity (correct). **No code; a test pins it** (`merge[A,B]` != `merge[A,B,C]`; and `content-dedup[A,B]` == `content-dedup[B,A]` — order-independent within one set).
- **OQ3 (merge summary / `conflict_evidence` slot): RESOLVED — rides the existing `justification` free-string; NO new store field.** The record shape is fixed at 9 fields with no `summary`/`evidence_refs` slot; the scope's D2 explicitly designates `justification` as "where `conflict_evidence`-class content can finally live." So a `merge` proposal's proposed-summary text IS its `justification`; a `flag-conflict`'s evidence can land there too. The **structured** evidence slot (a distinct field / `evidence_refs`) stays **v3.6** (named-deferred — the structured close is not built here). KISS/YAGNI: do not touch the hardened 9-field store.
- **OQ-arity (NEW — should `content-dedup`/`merge` require >=2 targets?): NO special-casing (KISS).** The store's `validateTargets` already rejects an empty set; a 1-target dedup/merge proposal is harmless advisory data a human rejects. Semantic arity (a dedup needs >=2 to be meaningful) is a **v3.6-execution** concern (when the op actually runs); enforcing it now is YAGNI for an advisory, recorded-not-executed layer. All wrappers uniformly require a non-empty target set (delegated to the store). **FLAG for architect VERIFY** — if the lens rules a >=2 boundary guard earns its keep for legibility, it is a 2-line add per wrapper; lean = no.

## Build

### `manage-ops.js` — add the three plural-target wrappers (DRY helper + 3 thin shims)

`quarantineRecord` (singular `target`) stays **UNTOUCHED** (don't touch tested code; the singular/plural boundary is real). The three new ops share the genuine repetition via one internal helper:

```
function proposeMultiTargetOp(opName, opType, input) {
  const { targets, justification, origin, now } = input || {};
  // PRESENCE guards (the quarantineRecord pattern): a MISSING field names THIS wrapper's contract;
  // FORMAT validity (is-array / non-empty / HEX64-per-element / length / lock / content-address)
  // delegates to store.createProposal — one admission gate (DRY).
  if (justification == null) throw new Error(`${opName}: justification is required (the why)`);
  if (origin == null)        throw new Error(`${opName}: origin is required (the provenance of the flag)`);
  if (targets == null)       throw new Error(`${opName}: targets is required (an array of 64-hex kernel transaction_ids)`);
  return createProposal({ opType, targetRecords: targets, justification, origin, now });
}
const contentDedupRecord = (input) => proposeMultiTargetOp('contentDedupRecord', CONTENT_DEDUP, input);
const cullRecord         = (input) => proposeMultiTargetOp('cullRecord', CULL, input);
const mergeRecord        = (input) => proposeMultiTargetOp('mergeRecord', MERGE, input);
```

- Local op-type consts `CONTENT_DEDUP='content-dedup'`, `CULL='cull'`, `MERGE='merge'` (mirror the existing `QUARANTINE` const; a typo'd const is caught by the store's `validateEnum`). None contain the `SUPERSEDE`/`TOMBSTONE` tokens the firewall source-scan greps for.
- **Presence -> wrapper-named; FORMAT -> store-named** (the proven contract): a present-but-non-array / empty / bad-hex `targets` flows to `validateTargets` and surfaces the store-level `target_records` message — exactly as `quarantineRecord`'s delegation test #5 expects for a bad format. The wrappers own only the null/undefined presence guard.
- CREATE-only — never call `updateDisposition` (the firewall source-scan #9 asserts this over the whole file).
- `merge`'s "summary" is its `justification` (OQ3). No special handling — `mergeRecord` is structurally identical to `cullRecord`/`contentDedupRecord`; the `op_type` is the only semantic carrier (the destructive distinction is realized by the v3.6 promotion reading `op_type`). State this honestly so VALIDATE catches no "merge synthesizes a summary" overclaim (trap #4 — 3b does not detect or execute).

### `cli.js` — add `content-dedup` / `cull` / `merge` subcommands

One combined block (the three are identical but for the wrapper fn):

```
if (cmd === 'content-dedup' || cmd === 'cull' || cmd === 'merge') {
  if (typeof args.targets !== 'string') { fail(`${cmd} requires --targets <txid,txid,...>`); return; }
  const targets = args.targets.split(',').map((s) => s.trim()).filter(Boolean);
  const fn = { 'content-dedup': contentDedupRecord, cull: cullRecord, merge: mergeRecord }[cmd];
  try { emit(fn({ targets, justification: args.justification, origin: args.origin || 'cli' })); }
  catch (e) { fail(e.message); return; }
  process.exit(0);
}
```

- `typeof args.targets !== 'string'` catches a bare `--targets` (parses to boolean `true`) AND a missing flag -> clean usage error (the `list --disposition` bare-flag-guard pattern from 3b.1).
- `split(',').map(trim).filter(Boolean)` -> forgiving (trailing comma / surrounding spaces tolerated); empties dropped; the store validates HEX64 + non-empty (DRY — one gate). An all-empty `--targets ,,,` -> `[]` -> the store's non-empty reject -> clean exit 1.
