# v3.9.x #80 — the 20-30 Python bootcamp corpus (PLANNING phase)

**Goal.** Curate a real, honesty-verified Python issue corpus and run it through the v3.9 bootcamp
pipeline → a populated recall graph + a calibration pass@k by tier, and (the #78 rider) a **control-screened
retrieval A/B** on failure-boundary targets — power CONTINGENT on a fail-cold sibling cluster existing
(feasibility back-of-envelope is a phase-1 deliverable, §9; the word "powered" is deliberately NOT used —
#78 proved only an existence-of-mechanism at n=1). This file is **phase 1 (PLANNING) only**: it locks the
selection criteria + an anti-hallucination sourcing methodology + a per-issue honesty-verify checklist. The
phase-1 planning section names zero commit SHAs — by design (§5); the later phase-2/3 status blocks (appended
as the plan executed) record the verified SHAs.

**USER directive (the reason this is staged, verbatim intent, 2026-06-14):** "Since this deals with
multiple issues, we have a higher risk of hallucinations. So, let's do a planning phase, then verify
EACH repo and issue with the honesty lens before starting the bootcamp."

So #80 is THREE gated phases, not a batch run (the cluster pool's ordering is REVERSED from the breadth
pool — see §7 / O2):
1. **PLAN** (this file) — criteria + methodology + shortlist + checklist. VERIFY-board it; fold; user-gate.
2. **MINE + HONESTY-VERIFY** — clone the shortlist, prove each repo gradeable, mine real candidate issues
   from `git log` (SHAs from git, never memory), honesty-verify EACH (the §8 checklist). For the cluster
   pool ONLY, a cheap control-screen runs FIRST (§7/O2) so the expensive human verify is spent only on
   confirmed fail-cold pairs.
3. **BOOTCAMP** — run the batch → recall graph + pass@k, then the control-screened A/B on the cluster(s).

## TWO DECOUPLED POOLS (the v1 plan wrongly treated Job B as a cheap subset of Job A — A1/A6)

Job A (calibration breadth) and Job B (the A/B cluster) pull on the SAME scarce axis (same-repo,
same-module, substantive, post-cutoff fixes), so they are budgeted as **separate pools**, not a
subset relationship:

| Pool | Purpose | Sourcing | Floor / measure | Novel? |
|---|---|---|---|---|
| **A — breadth** | calibration nodes + pass@k | broad: many repos, any difficulty | `per_tier` reports a real pass@k only at `withPk >= 20` (`N_CLEAN_LARGE_MIN`); else `INSUFFICIENT-N` — an ACCEPTED outcome (v3.9 EC1-4 already shipped it) | prefer novel, but do NOT chase it into gradeability risk (O1/A6) — high-confidence-gradeable-even-familiar beats low-confidence-novel |
| **B — cluster** | the retrieval A/B | REPO-FIRST: pick 1-2 repos with demonstrably MANY post-cutoff same-module fixes, mine the cluster from there | per-pair fail->pass delta (#78 ceiling: easy targets give ZERO signal) | NO — a `familiar` repo is fine; the pair measures a within-pair delta, NOT the N-floor; the SOURCE/TARGET pair does NOT count toward `clean_large_n` |
| **NC — negative-controls** | falsify the headline `negative_control_false_positive` metric (H2) | >=1-2 real post-cutoff refactor/docs/no-behavioral-fix commits, or constructed no-ops | the actor must FAIL them cold; `fail_to_pass` is `[]` or `[SENTINEL]` exactly | n/a |

The retriever HARD-gates on repo identity (`retrieve.js:76` — `repoMatch` false => score 0), so a cluster
MUST be same-repo. Decoupling dissolves O1's novel-vs-gradeable tension for Pool B entirely (it never
needed novel).

## §4 — Selection criteria

### HARD gates (a candidate that fails any one is OUT — verified mechanically in phase 2)

1. **Pure-Python**, no C/Cython extension (no compile/install step the sandbox can't do).
2. **Zero third-party deps**: the test suite imports nothing beyond stdlib + pytest (verify: fresh
   `pytest --collect-only` from the checkout with ONLY pytest installed collects with no `ModuleNotFoundError`).
3. **Importable without install**: `import <package>` resolves from the checkout cwd — NOT a `src/`-layout
   needing `pip install -e .`.
4. **pytest** with addressable nodeids.
5. **Post-cutoff**: the fix commit's `resolved_at` > `2026-01-01` => tier `clean-pending-probe` (NOT a proof
   of non-contamination — H6; the floor, not the ceiling, of the anti-leak argument).
6. **Source-fix + a test, separable**: the fix pairs a SOURCE change with a test that is RED on base / GREEN
   on fix. `accepted_diff` = SOURCE hunks only (survives the FULL `computeTamper` predicate, §3); `test_patch`
   = the test hunks. `pass_to_pass` nodeids curated from the same mine (so the pre-cert can guard regressions).
7. **Blind problem statement**: states the bug (expected vs actual), leaks NO fix detail; leak-checked vs
   `accepted_diff` (`rubricLeaks` >=12-char run) before commit.
8. **base_sha** = `git rev-parse <fix_sha>^` VERBATIM (full 40-hex lowercase parent of the fix; H5).
9. **id round-trips the retriever (A5)**: `id` MUST be `<repo-slug>__<slugifyTitle(real_upstream_commit_title)>`
   — the slug is the retriever's ONLY matchable surface (`retrieve.js:13-16`, the query title and every node's
   `issue_id` slug must come from the SAME `slugifyTitle`). Record the real title in the provenance/drop-log so
   the round-trip is auditable.

### Pool composition + SOFT preferences

- **Pool B (cluster)**: >=1-2 same-repo same-class clusters, each a SOURCE (solvable) + a failure-boundary
  TARGET (actor fails cold), where neither `accepted_diff` leaks the other AND the retrieved STRATEGY does not
  leak the target (H3, §8.C). Familiar repos OK.
- **Pool A (breadth) — EXPLICIT DIFFICULTY STRATIFICATION (USER 2026-06-14)**: do NOT build the corpus from
  all-simple issues (underfits the recall graph). Target a SPREAD across the three buckets — roughly a third
  each `lt1hr` / `1to4hr` / `gt4hr` (mine-time proxy per §5.4, confirmed/demoted at the control screen) — and
  REPORT which complexity bands actually POPULATE nodes vs. fail cold (a hard issue that fails won't mint a node,
  but IS a failure-boundary target where retrieval is tested). Prefer high-confidence-gradeable repos (even
  familiar) over low-confidence-novel (A6).
- **Negative-controls (NC)**: >=1-2 (H2), per the table above.
- **Model-authored labels are NOT git-sourced (H1)** — `repo_familiarity`, `difficulty_bucket`,
  `is_negative_control`, the `problem_statement`, and the cluster grouping are ALL model judgments and ALL
  gated in §8 (the §5 "fabrication impossible" guarantee covers the git FACTS, not these JUDGMENTS).

## §5 — Anti-hallucination sourcing methodology (the load-bearing part — git FACTS only)

**Invariant: every SHA, nodeid, diff, and date comes from a REAL clone via git — NEVER from model memory.**
The plan names zero commits; phase 2 produces them mechanically and re-verifies each. **Scope honesty (H1):
this guarantee covers the git-DERIVED facts; the model-AUTHORED labels (§4 list) are separately gated in §8.**

1. **Shortlist repos by gradeable properties** (§6) — repo names + dep profiles are stable-ish but EACH is an
   unproven claim until cloned (criteria 2/3 verified live).
2. **Per repo, prove gradeability FIRST** (§8.G) before mining issues. A repo that fails is dropped.
3. **Mine candidate issues from git, mechanically**: `git log --since=2026-01-01 --name-only`, filter to commits
   touching BOTH a source file AND a test file, inspect each diff to confirm BUG FIX (not feature/refactor/docs),
   extract `(fix_sha, base_sha=git rev-parse <fix_sha>^, source_hunks, test_hunks, fail_to_pass + pass_to_pass
   nodeids)` from the diff + the repo's test layout.
4. **Difficulty-proxy ranking at mine-time (A3)** — failure-boundary issues do not fall out of "source+test bug
   fix". After extraction, rank each candidate by cheap structural proxies (source-hunk count / LOC, files
   touched, new control-flow vs a one-line guard, fix-commit-message / linked-PR discussion length) and TAG a
   PROVISIONAL `difficulty_bucket`. Mine the cluster pool from the TOP of that ranking. `difficulty_bucket` is a
   provisional proxy judgment, CONFIRMED or demoted by the control screen; the A/B uses only TARGETs the screen
   confirms fail-cold.
5. **The model authors only**: each `problem_statement` (a blind bug restatement), the cluster grouping, and the
   provisional labels (§4) — all leak-checked / honesty-verified in §8. Everything else is git.

This makes FACT fabrication structurally impossible: an invented commit cannot survive `git log` + the §8
RED/GREEN re-grade on the real sandbox. The model JUDGMENTS are caught by §8.D + the control screen.

## §8 — Phase-2 per-issue HONESTY-VERIFY checklist (honesty-auditor lens, EACH issue)

For each candidate `(repo, fix_sha)`:

- **G. Repo gradeable** (once per repo, before its issues): clone HEAD; `install_requires`/`dependencies` empty;
  `import <pkg>` from checkout; `pytest --collect-only` clean with only pytest installed.
- **E. Exists + paired (H5)**: `git cat-file -e <fix_sha>` and `<base_sha>`; `base_sha == git rev-parse <fix_sha>^`
  VERBATIM (full 40-hex, the canonical parent — not merely "a 40-hex commit that exists").
- **R. RED on base (+ regression baseline, H4)**: clone@base + apply `test_patch` -> `fail_to_pass` FAIL AND
  `pass_to_pass` GREEN (the bug is present; the baseline is otherwise sound). If `fail_to_pass` already PASSES,
  the base/test pairing is wrong -> DROP.
- **GR. GREEN on fix (+ no regression, A4/H4)**: clone@base + `test_patch` + `accepted_diff` -> `fail_to_pass`
  PASS AND `pass_to_pass` GREEN. Run on the REAL sandbox (the §3 grading path), not a bare shell. NB this tier's
  live grade SKIPS the full suite — `pass_to_pass` is enforced HERE in curation precisely because the bootcamp
  grade will not (the curation gate must be STRICTER than the grade it pre-certifies).
- **S. Source-only**: `accepted_diff` touches NO test-infra path under the FULL `isTestInfraPath` predicate (§3,
  incl. `usercustomize.py`, `test/` singular, root-level `test_*.py`); `test_patch` is the ONLY test change.
- **B. Blind statement**: `rubricLeaks(problem_statement, accepted_diff)` is false; a human read confirms it
  states the bug, not the approach.
- **T. Tier**: `resolved_at > 2026-01-01`; `contamination_tier` ABSENT; `provenance: backtest`.
- **D. Model-label attestation (H1)**: `difficulty_bucket` cites the diff size / hunk count it was derived from
  (the §5.4 proxy); `repo_familiarity === 'novel'` cites the recorded familiar-set criterion; `is_negative_control`
  cites construction provenance. No subjective label ships un-attested.
- **ID. Slug round-trip (A5)**: `id` ends with `slugifyTitle(real_upstream_commit_title)`; the real title is in
  the provenance.
- **C. Cluster (if part of one) — leak surface is the STRATEGY, not the diff (H3)**: the SOURCE's `accepted_diff`
  shares NO >=12-char run with the TARGET's `accepted_diff` AND the SOURCE's `problem_statement` + the authored
  SOURCE strategy text share NO >=12-char run with the TARGET's `accepted_diff` (mirror recall-retrieval-test.js:127).
  Same class confirmed by a human read.
- **NC. Negative-control (for the NC pool, H2)**: `fail_to_pass` is `[]` or `[SENTINEL]` exactly (corpus.js:140);
  the actor FAILS it cold at the screen (a confident "pass" is the hallucination signal).

A candidate failing E/R/GR/S/B/T is DROPPED (not patched to pass). The drop log — how many inspected, how many
survived, and WHY each was dropped — is part of the deliverable (honest curation).
