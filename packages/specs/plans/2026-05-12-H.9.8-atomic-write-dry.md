## Option-axis design surface — cleanup-on-error semantic

### Option A — Enhance helper with built-in tmp cleanup (RECOMMENDED)

**Implementation**: add try-catch around `writeFileSync + renameSync` inside both helper functions; on error, attempt `fs.unlinkSync(tmp)` (best-effort, ignored if it also throws); re-throw original error.

```javascript
function writeAtomic(filePath, data) {
  fs.mkdirSync(path.dirname(filePath), { recursive: true });
  const tmp = filePath + _tmpSuffix();
  try {
    fs.writeFileSync(tmp, JSON.stringify(data, null, 2));
    fs.renameSync(tmp, filePath);
  } catch (err) {
    try { fs.unlinkSync(tmp); } catch { /* ignore — cleanup is best-effort */ }
    throw err;
  }
}
// Same shape for writeAtomicString
```

**Pros**:
- Uniform behavior across all 11 post-H.9.8 consumers (4 existing + 7 new files / 8 new sites)
- Strict improvement for existing 4 consumers (tmp-leak risk reduces; behavior strictly better; no breaking change since they don't rely on tmp persistence)
- Class B sites drop their try-catch-cleanup entirely after migration → maximal DRY
- Class C sites retain only their outer try-catch + log (cleanup absorbed into helper)
- Mirrors substrate convention precedent: hook fail-soft envelopes are at caller; cleanup-on-failure is helper-internal hygiene
- Helper LoC delta: +6 lines per function × 2 functions = +12 LoC; 84 → ~96 LoC

**Cons**:
- API contract change for 4 existing consumers (gain cleanup behavior they didn't request; harmless but observable). Patch bump 1.13.0 → 1.13.1 per HT.2.2 substrate-helper-API-additive precedent
- Helper internal complexity increases marginally (try-catch + nested try)
- If `fs.unlinkSync(tmp)` succeeds but `tmp` was already partially renamed (race), no harm — POSIX `renameSync` is atomic so partial state is impossible; only `tmp` exists OR target exists

### Option B — Preserve caller-side error handling asymmetry

**Implementation**: helper unchanged; Class A sites drop-in replace; Class B sites keep their try-catch + cleanup wrapper around the helper call; Class C sites keep their try-catch + log wrapper around the helper call.

```javascript
// Class B preserve (tree-tracker.js example):
function writeTreeAtomic(runId, tree) {
  const p = treePath(runId);
  try {
    writeAtomic(p, tree);
  } catch (err) {
    // tmp cleanup intentional even though helper doesn't do it
    // (rationale: defensive; rename failure is rare but cleanup is cheap)
    throw err;
  }
}
```

**Pros**:
- No helper API change (no manifest bump needed; pure-internal-refactor per HT.1.10/HT.1.12/HT.2.3 precedent)
- Explicit caller-side error handling visible at each site
- Existing 4 consumers' behavior unchanged

**Cons**:
- Less DRY (2 sites retain try-catch-cleanup boilerplate; redundant if helper later acquires cleanup behavior)
- Semantic asymmetry across 11 consumers (some get cleanup, some don't; depends on caller diligence)
- The "preserve cleanup" wrapper at Class B sites is awkward — it wraps the helper but adds nothing the helper couldn't do internally

### Option C — Parameterized cleanup via opts (REJECTED per architect LOW-1 absorption)

**Implementation**: helper accepts optional `{cleanup: boolean}` parameter (default true); consumers opt out if they need to inspect tmp post-failure.

**Rejection rationale** (single-sentence): adds API toggle for no consumer demand (zero current consumers want tmp-persistence-on-error); YAGNI applies symmetrically to telemetry deferral in §"Out of scope" — make universally-desirable behavior the helper default, not an opt.

### Recommendation: Option A

Rationale:
1. **Strict-improvement migration**: existing 4 consumers gain cleanup behavior with zero risk (they don't rely on tmp persistence; no code depends on `fs.existsSync(tmp)` after write failure)
2. **Maximal DRY**: Class B sites drop their wrappers entirely → ~6-10 LoC saved across 2 sites
3. **API stability**: patch bump (1.13.0 → 1.13.1) is appropriate signal magnitude — helper contract additive (post-condition strengthens: "no stale tmp on disk after error")
4. **Substrate convention alignment**: hook fail-soft envelope at caller (preserved); cleanup hygiene at helper (centralized)
5. **YAGNI counter**: cleanup behavior is 6 LoC; bounded surface; universally desirable; no evidence of "wait for need"

```javascript
// Test N: writeAtomic cleans up tmp on rename failure
const fs = require('fs');
const path = require('path');
const os = require('os');
const { writeAtomic } = require('./_lib/atomic-write');

const tmpdir = fs.mkdtempSync(path.join(os.tmpdir(), 'h98-cleanup-'));
const target = path.join(tmpdir, 'test.json');
const origRename = fs.renameSync;
fs.renameSync = () => { throw new Error('Mock rename failure'); };
try {
  let threw = false;
  try { writeAtomic(target, { test: true }); } catch (e) { threw = true; }
  assert.strictEqual(threw, true, 'writeAtomic should throw on rename failure');
  const tmpFiles = fs.readdirSync(tmpdir).filter((f) => f.includes('.tmp.'));
  assert.strictEqual(tmpFiles.length, 0, 'tmp file not cleaned up after rename failure');
} finally {
  fs.renameSync = origRename;
  fs.rmSync(tmpdir, { recursive: true, force: true });
}
```
