## Context

the integrator's `mintRejectEvent` wiring) as a SHADOW **producer**: the integrator
records every `quarantined` / `provenance-rejected` disposition as a content-addressed,

trust-DOWN / narrowing only; only a world-anchored merge HARDENS). The breaker stays
SHADOW (it halts nothing); the GATING consumer is v3.9 (first live beta). This is the
Producer-Consumer Phasing the v3.7 close named: the consumer is owed in the next phase.

## Threat-model declaration

- **Plant a foreign reject-event** (same-uid drops an internally-consistent reject-event for run X
  into run Y's dir) → OVER-count → OVER-narrow → **SAFE** (§0a.3.1 halt-only grants nothing). This
  is why the cross-run scan is over-count-safe and needs no run-binding/content-verify (D2).
- **The run-binding REVERSAL, named explicitly (honesty VERIFY M1)**: the v3.7 producer's read-side
  run-binding (`loadRejectEventFile(file, expectedRunId)`, reject-event-store.js:281 — added
  specifically to protect this breaker from a cross-run plant inflating a RUN's reject-rate) is
  **INTENTIONALLY DROPPED** by the cross-run scan. It was a per-run-count tamper isolation; the
  breaker's v3.8 count is GLOBAL + over-count-safe, so a cross-run plant can only over-narrow.
  This was NOT assumed by analogy — the hacker VERIFY probed it live (a planted
  shape-valid-but-content-garbage file only OVER-counts; a flipped `outcome` leaves the COUNT
  unchanged; no design-logic under-count path exists). A future PER-RUN reject-rate consumer must
  re-introduce run-binding (and must NOT trust the scan's `run_id` row field beyond its
  dir-name provenance).
- **UNDER-count via same-uid FS tamper — the full vector set (hacker VERIFY M1)**: **delete** the
  file / **rename** it off the `reject-event-<64hex>.json` prefix (the file-RE skips it) / **move**
  it out of the `reject-events/` subdir / **back-date its mtime into the past** (`utimes`). All four
  are the same same-uid FS-tamper class. UNLIKE the `manage-promote` TOMBSTONE (where suppression is
  self-defeating — deleting the record undoes the destruction), suppressing a reject-event does NOT
  un-quarantine the candidate, so under-count is a REAL vector here. **UNMITIGATED at the FS layer;
  closes at the Track-2 ContainerAdapter** (same class as the manage-promote C1 residual +
  `updateDisposition`).
- **Future-date mtime** → the breaker's `excluded_future` fires (a tamper / clock-skew signal); a
  v3.9 GATING consumer fails-CLOSED on `>0` (the promote.js precedent). v3.8-shadow correctly defers
  that fail-closed to v3.9 (hacker VERIFY confirmed: no live consumer reads the reject-event source
  today; the shadow boundary is genuine).
- **worktree ≠ sandbox**; all residuals are same-uid FS tamper that the ContainerAdapter closes.
