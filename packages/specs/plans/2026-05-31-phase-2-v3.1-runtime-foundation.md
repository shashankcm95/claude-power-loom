## Context

**Where we are.** Phase-1-alpha shipped the pure kernel transaction loop — 11 primitives
(K1,K2,K3,K3.b,K4,K7,K9,K10,K12,K13,K14) atop pre-existing K5. The MVP phase-gate review
(architect + honesty + QA → simulated PM) returned a **conditional GO**, then the hardening
pass proved the composed loop end-to-end on real git and the re-review board returned a
**unanimous GO**. Honest status today: *composition PROVEN end-to-end; NOT yet wired into a
live spawn hook; the runtime/persona layer does not exist yet.* main @ `492e20b`.

**What v3.1 delivers.** The first **runtime layer** atop the kernel, and the first **consumers
of the dormant primitives**. Two halves that are *coupled through the spawn-init / spawn-close
hooks* (which is why they belong in one phase):

1. **The persona/capability runtime** — R1 two-tier persona contracts, R2 migrate the 16
   personas, R3 capability traits, R4 trait composition, **K6** capability-subset check, **K8**
   capability-injection at spawn-init (K8 is the first production reader of the dormant **K3.b**
   context envelope).
2. **Make the transaction loop LIVE** — the four MVP carry-forwards: (#1) wire `resolve()` into
   a live spawn-close hook, (#2) validate `k14_ctx` at the resolver boundary, (#3) prove the
   non-genesis chained PROMOTE path, (#4) K13 spawn-id provenance under concurrency + the
   two-map reconciliation. Plus **R13** idempotency-key enforcer and **K2.c** per-tool-call
   observability.

**Intended outcome.** "Composition proven; not yet live" → "**a real runtime layer that *checks*
every spawn at spawn-init, plus a kernel loop that fires in production (shadow-first; real
promote/reject gated on the OQ-21 spike).**" K6/K8 turn persona contracts into spawn-time checks —
though only **Write** is runtime-*prevented* (worktree scope); Read/Network/Subprocess are
**audited post-hoc** at close per v3.3 §3.2 (L145-154), not blocked mid-spawn. The close hook makes
the kernel loop *observe* every spawn; it *constrains* only in the enforcing branch.

**Spec estimate (NOT a delivery contract):** ~24–36h, ~2,550–3,150 LoC (v6-substrate-synthesis
§6.6, L1447-1451). The 5-PR cadence is a plan-of-record; PR-3b's scope is spike-dependent
(enforcing > shadow), and PR-1/PR-4 carry pre-authorized splits.

## v3.1 Re-scope — capability injection is NOT viable (2026-05-31, ADR-0012)

**This supersedes the K8 / spawn-init-injection parts of PR-2 + PR-3 below.** Empirical probes
(headless `claude -p` with a throwaway `PreToolUse:Agent` hook + an observable sub-agent) proved a
PreToolUse hook's `updatedInput` is **INERT for Agent/Task spawns**: the Agent `tool_input` keys are
`{description, prompt, subagent_type}` (no `tools` field), and neither a `tools` nor a full `prompt`
rewrite reaches the sub-agent. Therefore:

- **K8 (capability-injection at spawn-init) is DROPPED** — its mechanism does not exist; building it
  would be enforcement-theater.
- **`pre-spawn-tool-mask` is UNREGISTERED** — it was inert since it shipped (rewrites a non-existent
  `tool_input.tools`); leaving it ran a no-op that gave false security.
- **Capability enforcement is STATIC**: the agent.md frontmatter `tools:` definition (which the
  harness honors) + the **agent.md↔contract reconciliation validator** (PR-2a, `contracts-validate.js`).
  **PR-2a (K6 + `buildEnvelope` + reconciliation) IS the v3.1 capability layer.**
- **K3.b `buildEnvelope`** ships as a dormant primitive; its per-spawn delivery channel does not exist
  (no `updatedInput` injection) → deferred.
- **`dormancy-assertion-k3b` is RETAINED** — NOT deleted as the PR-2 detail below + probe **P6**
  ("absent") state. `buildEnvelope` ships dormant → K3.b still has zero production readers → the gate
  correctly stays asserting dormancy. **P6 is superseded** (expect "present", not "absent").
- **`contextItem` shape + `consumes_context_envelope` opt-in were NOT defined** (the PR-2 detail lists
  them as deliverables). Both were consumers of K8's injected envelope; with K8 dropped there is no
  consumer, so both are **deferred (YAGNI)**, not omitted.

See `packages/specs/adrs/0012-capability-enforcement-is-static-not-runtime-injected.md` for the probe
evidence + the full decision. The PR-2/PR-3 detail below is retained as historical design; its
K8 / tool-mask / `updatedInput`-injection parts are obsolete.

## Architectural Decisions

1. **Live-wiring = spike-then-decide (USER).** PR-3a probes harness delta-observability; PR-3b ships
   enforcing if confirmed, else shadow (forward-compatible).
2. **Two-tier is ADDITIVE, not a migration.** Legacy contract keys stay; new `interface`/`defaults`
   added alongside; two validator-enforced equalities prevent drift. (KISS + no live-runtime breakage.)
3. **`trait-resolve.js` ships dormant in PR-1**, first runtime-consumed by K6 in PR-2 — mirrors the
   K3.b/K9 ship-dormant-then-activate discipline that the dormancy CI gates enforce.
4. **K6 *checks* capability subset deterministically; it does not *prevent* most axes at runtime.**

## Security review (the two CRITICAL surfaces drive this)

- **S1 — K8 `updatedInput` rewrite (CWE-20).** When a spawn carries a capability `request`, injection
  narrows to `request ∩ declared`. But real spawns **empirically carry no `tools[]`/capability request**
  (tool-mask's `audit-absent` path; v3.3 §3.1 L136 "no shipped hook currently exercises `updatedInput`
  for Agent input rewrite") — so in the common case K8 **writes** `declared_capabilities` onto a spawn
  that declared nothing. That still cannot widen past the persona's contract ceiling (no security hole),
  but the guarantee is "bounded by `declared`," NOT "narrows an existing request." **Single `updatedInput`
  owner** is *achieved* by the PR-2 tool-mask merge (Decision 11), not assumed. Whether the harness even
  *honors* injected capabilities is the **OQ-21/Wave-1** question. Confirm cumulative `updatedInput`
  semantics against the hook docs before build (general-purpose + WebFetch on code.claude.com, per H.7.23).
- **S2 — AGENTS.md / CLAUDE.md upward traversal (CWE-22).** Bound the walk at repo root via
  `path-canonicalize.checkWithinRoot(p, root).ok`; refuse symlinks escaping root; size-cap each file;
  capability data parsed from them can only **NARROW**, never widen. Never follow a path from file content.
- **S3 — k14_ctx boundary (CWE-20/22).** Whitelist keys + `path-canonicalize.checkWithinRoot(p,
  worktree_root).ok` for every path before `resolve()`'s blind spread (Decision 7; populator-side).
- **S4 — R13 key (forgery/missing).** Canonical-JSON key over harness `tool_use_id`; fail-CLOSED on a
  missing key. Replay-forgery in local-trust mode deferred to OQ-20 (HMAC envelope-signing, v3.1+ B2).
- **S5 — dormancy-flip blast radius.** The dormancy job **self-fails** if the first import lands without
  the same-commit gate deletion (the forcing tripwire); reviewer checkbox + P6/P7 absence-probes are
  backstops, not the primary control.
- **S6 — live-hook fail-posture.** K8 fail-SOFT; R13 fail-CLOSED on missing key but fail-SOFT on its own
  crash; the close hook wraps `resolve()` in try/catch so neither the hook nor a resolver throw reaches
  the harness.
