### Component A — `skills/next-js/SKILL.md`: Node-side ESM dep gotchas section

**Drift**: DRIFT-test3-017
**Symptom**: `Module not found: ESM packages (X) need to be imported. Use 'import' to reference the package instead.` — webpack 5 statically analyses `require.resolve('foo-esm/path')` and tries to bundle the ESM target, which fails.
**Triggered in test3 by**: `pdfjs-dist@4.5.136` from Next.js 14 Route Handler.
**Affects**: any persona spawning Next + ESM-only Node dep (pdfjs-dist, native deps with `.node` binaries, modern AWS SDK v3 sub-modules, etc.).

**Fix shape** — new `## Node-side ESM dep gotchas` subsection in `skills/next-js/SKILL.md`. Must NOT contradict the existing `## Native-Node packages in App Router` section (lines 182-213) — those handle different problems (externalize-then-import vs runtime worker-path).

1. **Anti-pattern catalog** (don't do):
   - `require('foo-esm/path/to/file.mjs')` from a Route Handler
   - `createRequire(import.meta.url).resolve('foo-esm/...')` — webpack STILL statically analyses
   - Setting `serverComponentsExternalPackages` alone — applies only to RSC, NOT Route Handlers

2. **Canonical pattern** (do — with explicit deployment caveat per code-reviewer HIGH):
   ```js
   import { join } from 'node:path';
   // Build the path at runtime from process.cwd() so webpack sees only an opaque string.
   const workerPath = join(process.cwd(), 'node_modules', 'foo-esm', 'path', 'to', 'file.mjs');
   ```

   > ⚠️ **`process.cwd()` is NOT reliable under `next build --standalone`**.
   > Under `next dev` / `next start` / `next build` (non-standalone), `process.cwd()` resolves to the project root and `node_modules/` is adjacent — pattern works. Under `--standalone` output (common for Docker/Railway/self-hosted Node deploys), the standalone server lives at `.next/standalone/server.js` with its own `node_modules/` at `.next/standalone/node_modules/`. If the operator launches from a different `cwd`, the path silently resolves to a non-existent location and `workerSrc` assignment fails at runtime.
   >
   > **For standalone deploys**: resolve relative to `__dirname` (CJS) / `import.meta.url` (ESM), OR pass the absolute worker path via an env var set at container build time.

3. **List of common offenders**:
   - `pdfjs-dist` (4.x is ESM-only; legacy build needs `GlobalWorkerOptions.workerSrc` set at module load)
   - native deps with `.node` binaries (better-sqlite3, sharp) — additionally need `serverComponentsExternalPackages` / `serverExternalPackages` (cross-ref existing § Native-Node packages)
   - `node-fetch` v3+ (ESM-only)

4. **When `serverComponentsExternalPackages` IS the right tool**: per existing § Native-Node packages (lines 182-213), use externalize-then-import for packages you want webpack to leave alone entirely. The runtime path-construction pattern is specifically for the **`workerSrc` / external-file-load** case where you need an absolute on-disk path that the package's own runtime will use.

### Component B — `swarm/personas/08-ml-engineer.md`: tiktoken model-alias convention

**Drift**: DRIFT-test3-018
**Symptom**: `Invalid model: gpt-4o-mini` thrown from `tiktoken.encoding_for_model('gpt-4o-mini')` in tiktoken@1.0.15 (and earlier within 1.0.x). The error names the BILLING model, misleading developers into thinking their API key lacks model access (curl probe to OpenAI `/v1/models/gpt-4o-mini` returns 200 fine).
**Triggered in test3 by**: per-chapter generation pipeline pinning `tiktoken: "^1.0.15"`.

**Surface choice (resolved per architect.theo H1)**: persona brief, NOT a new skill. The tiktoken alias-map is a *persona-specific runtime convention* (only ml-engineer encounters it); putting it in a SKILL would give it the wrong reach (every persona referencing `claude-api` inherits it) and the wrong reason-to-change axis per `kb:architecture/crosscut/single-responsibility`. Extend the v2.9.0 FIX-I8 scope-coherence locus (persona-brief + contract + agent-md as 3-surface coherence target) instead.

**Option-axis resolution (per architect.theo H2)**: **Option B (alias-only)**. Encoding-family-stable: gpt-4o-mini and gpt-4o share `o200k_base` — this is a property of the BPE tokenizer, not the package version. Option A (version bump) would hard-couple plugin-substrate releases to upstream tiktoken cadence. Option C (both) clones the FIX-I8 anti-pattern of pinning version policy in two places. Option B is zero-churn forever. Document A as deferred-with-criteria: "revisit if upstream eventually adds the alias and we want self-healing for any future model not yet in tiktoken."

**Concrete version fact (per code-reviewer HIGH)**: tiktoken **1.0.16** is the first release that recognises `gpt-4o-mini`. 1.0.15 throws. Document this in the persona brief so consumers know the minimum-for-self-healing if they want it; alias map is the safer default.

**Fix shape**:

Add `## Tiktoken model-name aliasing` subsection to `swarm/personas/08-ml-engineer.md`:

```markdown
### Tiktoken model-name aliasing

`tiktoken@1.0.15` and earlier throw `Invalid model: gpt-4o-mini` from
`encoding_for_model()` — the OpenAI billing model name precedes tiktoken's
internal recognition. Same applies to other model names landing in billing
before being added to the tiktoken model map.

**Canonical pattern**: ship an alias map in the cost-management module:

```ts
import type { TiktokenModel } from 'tiktoken';

const ENCODING_FOR_MODEL: Record<string, TiktokenModel> = {
  'gpt-4o-mini': 'gpt-4o',  // same o200k_base; mini name not in tiktoken@1.0.x
  'gpt-4o':      'gpt-4o',
  // Add new aliases as model names appear in billing before tiktoken adds them
};
```

`gpt-4o-mini` and `gpt-4o` share the `o200k_base` BPE encoding — token counts
for the same text are identical. Cost arithmetic is safe.

**Forward-looking (deferred Option A)**: tiktoken `>= 1.0.16` natively recognises
`gpt-4o-mini`. If you want self-healing for future model names, pin
`tiktoken: ">=1.0.16"` AND keep the alias map for any future
billing-precedes-tiktoken cases.

**Empirical origin**: DRIFT-test3-018; surfaced during test3 Phase-5 UAT.
```

### Component F — `skills/postgres-engineering/SKILL.md`: hand-written SQL migration journal pattern (NEW, promoted from MEDIUM)

**Drift**: DRIFT-test3-015 (promoted per code-reviewer MEDIUM — `pnpm db:migrate` ships broken for hand-written SQL scaffolds)
**Symptom**: `ENOENT: no such file or directory, open '.../meta/_journal.json'` from `drizzle-orm/better-sqlite3/migrator.migrate()` when the SQL was hand-written (not `drizzle-kit generate`'d).

**Fix shape** — add `## Hand-written SQL migrations + Drizzle migrator` subsection to `skills/postgres-engineering/SKILL.md`:

```markdown
### Hand-written SQL migrations + Drizzle migrator

When you ship hand-written `drizzle/migrations/NNNN_*.sql` files (no
`drizzle-kit generate` available at codegen time), the `drizzle-orm`
migrator throws `ENOENT: meta/_journal.json` — it expects the journal
file the codegen tool would produce.

**Two options**:

1. **Hand-write the journal** alongside the SQL:
   ```json
   {
     "version": "7",
     "dialect": "sqlite",
     "entries": [
       { "idx": 0, "version": "7", "when": <unix-ms>, "tag": "0000_initial", "breakpoints": true }
     ]
   }
   ```
   Then `pnpm db:migrate` works normally and tracks already-applied
   migrations via the internal `__drizzle_migrations` table.

2. **Bypass the migrator** with raw exec:
   ```ts
   import Database from 'better-sqlite3';
   import fs from 'node:fs';
   const db = new Database('./data/app.db');
   db.exec(fs.readFileSync('./drizzle/migrations/0000_initial.sql', 'utf8'));
   ```
   Trade-off: no idempotency, no migration tracking — re-running the
   script errors with "table exists". Acceptable for MVP / single-shot
   bootstrap.

**Recommended**: option 1 for production; option 2 only for prototype
bootstrap with manual cleanup discipline.

**Empirical origin**: DRIFT-test3-015; cost test3 Phase-5 UAT 1h before
the bypass was discovered.
```

**Tests**: grep-based smoke assertion (same pattern as Component A) — assert `## Hand-written SQL migrations + Drizzle migrator` heading present in skill + `meta/_journal.json` literal mentioned within 30 lines.
