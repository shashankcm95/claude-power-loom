# Plan — K11 Kernel Algorithm Library + A4-Binding Gate (v3.2 Wave 0, piece 3)

## Context

Wave 0's other two pieces shipped in #211 (12-security write-floor via `AGENT_NAME_ALIASES`; R10
recursion-depth on `budget-tracker.js`). **K11 is the remaining + design-heaviest Wave-0 foundation.**

Axiom **A4** (`v6-substrate-synthesis.md:387`): *"kernel scope SHALL include algorithmic logic.
Deterministic operations live in kernel code with unit tests — NOT prose discipline or embedded
pseudocode for LLM execution."* A4 was deliberately scoped to **v3.2+** (v6:389) — it *becomes binding
where the K11 algorithm library ships*. v3.0-alpha/v3.1 only needed `subset_check.js` (K6) +
`path_canonicalize.js` (K7) + `route-decide.js`. This plan ships the library structure + the gate that
makes A4 *trackable now* and *enforceable in Wave 3*.

**The design crux** (the thing this plan must nail, and the architect VERIFY must pressure-test):
`packages/kernel/algorithms/` is **one file today** (`route-decide.js`). The gate has almost nothing
real to bind on day one; its real subjects — R9 leaf-criteria + R11 routing — are Wave 2. So Wave-0 K11
is **scaffold + a WARN-first gate + a synthetic-violation fixture**, NOT "achieve full A4 compliance."
**The detection rule must be precise — false positives are the primary design risk.**

## Design — the gate's detection rule (the crux)

**The gate does NOT heuristically scan prose for "algorithm-shaped paragraphs."** A grep for
`if`/`for each`/`return`/`score =` across contracts + trampoline prompts to detect "embedded
pseudocode" is a **false-positive generator** (legitimate prose is full of conditionals). A4 cannot be
bound semantically at Wave 0. Instead the gate binds on an **explicit, author-maintained ledger** +
**structural integrity** — precise, deterministic, false-positive-free.

### The ledger — `packages/kernel/algorithms/manifest.json`

```jsonc
{
  "version": 1,
  "enforcement": "warn",          // Wave-0 = "warn"; Wave-3 flip = change to "error" (DATA, not code)
  "algorithms": [                  // REALIZED — each has a file + exports + a unit test
    {
      "id": "route-decide",
      "file": "route-decide.js",
      "exports": ["scoreTask"],
      "test": "tests/unit/kernel/algorithms/route-decide.test.js",
      "kind": "scorer",
      "summary": "Deterministic 7-dimension task-routing scorer (H.7.3)."
    }
  ],
  "planned": [                     // A4 WATCHLIST — known-future determinism not yet kernelized
    { "id": "leaf-criteria",      "owner": "R9",  "wave": 2,
      "note": "5 deterministic leaf-completion predicates — must land as a tested kernel algorithm, not contract prose." },
    { "id": "spawn-verify-route", "owner": "R11", "wave": 2,
      "note": "Discipline→verification routing — must land as a tested kernel algorithm, not dispatcher prose." }
  ]
}
```

Membership IS the status: `algorithms[]` = realized; `planned[]` = pending. The Wave-3 flip literally
*moves* an entry from `planned[]` to `algorithms[]` (and `enforcement: "error"` makes any residual
`planned[]` entry fail). No `status` enum needed (KISS).

**Schema rule — the two arrays have DIFFERENT required-field sets** (architect I-4 — the single most
likely build-stall spot; do NOT conflate them in one schema check):
- `algorithms[]` entry requires `{ id, file, exports[], test, kind, summary }`.
- `planned[]` entry requires `{ id, owner, wave, note }` — **no** `file`/`exports`/`test` (those modules
  don't exist yet). The structural-integrity checks apply to `algorithms[]` ONLY.

### Finding kinds

Every finding (error or warning) shares **one shape** — `{ kind, message, ...metadata, fix? }` (the same
shape contracts-validate violations already use). This makes the Wave-3 flip shape-preserving:
`errors.push(warnings.pop())` with **no** conversion (code-reviewer F5).

**Structural integrity — ERROR at Wave 0** (false-positive-free; the manifest is authored clean, so the
live gate is GREEN day one). Applies to `algorithms[]` entries only:

| kind | condition |
|---|---|
| `manifest-schema-invalid` | manifest missing `version`/`enforcement`/`algorithms`/`planned`; `enforcement ∉ {warn,error}`; an `algorithms[]` entry missing a field from its required set; a `planned[]` entry missing a field from ITS (different) required set. |
| `algorithm-file-missing` | `algorithms[].file` not on disk under `algorithms/`. |
| `algorithm-source-unreadable` | `readFileSync(file)` throws (file unreadable / vanished mid-scan). |
| `algorithm-export-missing` | a name in `algorithms[].exports` not found in the module's `module.exports = { … }` object-literal block (see "Export check = static analysis" below). |
| `algorithm-test-missing` | `algorithms[].test` path not on disk. |
| `algorithm-unregistered` | a file matching **`algorithms/*.js`** absent from `algorithms[]`. Catches "someone dropped a new algorithm without registering/testing it." |

**Export check = static analysis, NOT `require()`** (code-reviewer F1, HIGH — closes module-scope
side-effect contamination + require-cache pollution + a DoS surface): the gate **reads** each algorithm's
source and checks that every declared `exports[]` name appears inside the `module.exports = { … }`
object-literal block. It never `require()`s an algorithm module. This is purer (a deterministic
text check — itself A4-aligned), and it means a future R9/R11 module with module-scope I/O cannot poison
the `contracts-validate` process. The convention (README) **mandates** the object-literal export form +
module-scope purity + CLI-under-`require.main === module`, so the static check is sufficient and the style
is locked for every future algorithm. Injectable deps reduce to `{ existsSync, readFileSync, readdirSync }`
(no `requireModule`).

**The `algorithm-unregistered` filter is an extension ALLOWLIST (`f.endsWith('.js')`), never a blocklist**
(architect I-1 / code-reviewer F4): a blocklist (`f !== 'manifest.json' && f !== 'README.md'`) would
false-positive on `.DS_Store`, `.gitkeep`, or any future non-`.js` file. Scanning `algorithms/*.js` means
`manifest.json`/`README.md`/dotfiles are excluded by extension — no stringly-typed carve-out (KISS).

**A4-binding watchlist — WARN at Wave 0, ERROR at Wave 3** (governed by `manifest.enforcement`):

| kind | condition | Wave-0 | Wave-3 |
|---|---|---|---|
| `planned-not-realized` | each `planned[]` entry | one consolidated `⚠` stderr line ("A4 watchlist: N pending — leaf-criteria/R9, spawn-verify-route/R11") | `violations.push` per entry |

Precision argument: the integrity checks each resolve or don't (no judgment); the A4 teeth come from the
author maintaining `planned[]`. That is the **honest** Wave-0 mechanism — R9/R11 don't exist yet, so the
gate tracks them as declared intent and nags until they graduate. **Real enforcement arrives in Wave 2**
when R9/R11 land as kernel functions and move to `algorithms[]`; the gate then verifies they ARE tested
kernel modules (file+export+test integrity), which is exactly the flip precondition. **Wave-2
forcing-function (architect M-5): the PR that moves an entry `planned → algorithms[]` MUST land its test
file in the same PR** — the integrity check hard-ERRORs on a realized-without-test entry. That is the
designed teeth, not a surprise; documented in the README + the scope doc's Wave-2 section.

### Placement (ownership + wiring)

- **Ledger (data):** `packages/kernel/algorithms/manifest.json`.
- **Convention doc:** `packages/kernel/algorithms/README.md` — "what counts as a kernel algorithm":
  pure/deterministic; **module-scope MUST be side-effect-free** (no I/O, child procs, timers, or global
  handlers at top level); **CLI behavior ONLY under `if (require.main === module)`**; **exports via a
  `module.exports = { … }` object literal** (the form the static export-check parses); registered in the
  manifest; unit-tested under `tests/unit/kernel/algorithms/`. **Non-algorithm helpers (re-export shims,
  `_lib`-style utilities) live in `kernel/_lib/`, NOT `algorithms/`** (architect I-1 — so a future
  `foo-export.js` doesn't trip `algorithm-unregistered` with no guidance).
- **Audit fn (pure, injectable — itself the deterministic core):**
  `packages/kernel/_lib/kernel-algorithms-audit.js` exporting
  `auditAlgorithmLibrary({ rootDir, manifest?, deps? }) → { errors:[], warnings:[] }`. No `process.exit`,
  no module-scope I/O; `deps` (`existsSync` / `readFileSync` / `readdirSync`) injectable for fixture-free
  testing (Dependency Inversion). **No `require()` of algorithm modules** (static source analysis instead
  — F1). Lives in `kernel/_lib` (kernel owns its own algorithm-library logic).
- **CI adapter (thin, ~20 LoC):** `validators['kernel-algorithm-a4-binding']` in `contracts-validate.js`
  — `require('../../kernel/_lib/kernel-algorithms-audit')`, write `warnings` to stderr (`⚠`), return
  `errors` as the violations array. Inherits `Object.keys` enumeration + exit wiring (precedent:
  kb-architecture-* validators). Runtime→kernel dependency = legal.

### route-decide unit test (RECOMMENDED — flagged for architect scope-check)

The manifest's one real entry declares `test: tests/unit/kernel/algorithms/route-decide.test.js`, so the
hard integrity check requires that test to exist. **Decision (architect APPROVED in-scope): write it now**
(bounded ~100 LoC). Rationale: (1) dogfoods the gate with a genuinely-passing realized entry; (2) A4
literally requires the canonical kernel algorithm to have a unit test, and the probe confirmed it has
none; (3) bounded.

**Bounding constraint (architect M-2 — weight-version-resilient):** assert **behavioral BANDS**, not exact
score floats. route-decide.js:11–13 declares weights/thresholds LOAD-BEARING and re-derivable only by an
architect pass — so asserting `score_total === 0.375` would make the test a recalibration tax. Assert
instead: a clearly-route task → `recommendation: "route"`; a trivial task → `"root"`; a mid task →
`"borderline"`; `--force-route`/`--force-root` override; determinism (same input → same output); the
output-shape contract (`{recommendation, confidence, score_total, reasoning, …}`). Pin a sentinel float
ONLY alongside `WEIGHTS_VERSION` so a version bump is the signal to update the golden. **Import the
canonical `algorithms/route-decide.js`** (architect I-2), NOT the `_lib/route-decide-export.js` re-export —
that's the module A4 governs and the manifest registers.

## Principle Audit (SOLID / DRY / KISS / YAGNI)

- **SRP** — audit logic (pure fn, `kernel/_lib`) is separate from CI wiring (thin adapter,
  contracts-validate) is separate from data (manifest). Three concerns, three homes.
- **Open/Closed** — new algorithms register by *adding* a manifest entry; the gate slots into the
  registry by *addition*; the Wave-3 flip is a *data* change (`enforcement: "warn"→"error"`), not a code
  edit.
- **Dependency Inversion** — the audit fn depends on injected `deps`
  (`existsSync`/`readFileSync`/`readdirSync`), not raw `fs` — so the synthetic fixtures need no on-disk
  pollution. (It never `require()`s an algorithm module — static source analysis only, per F1.)
- **KISS** — no prose-scanning heuristic (the false-positive trap is explicitly rejected); membership in
  two arrays encodes status (no enum); the fixture is in-test data, not an on-disk fixture tree.
- **YAGNI vs. one forward-investment** — `enforcement` is the *only* forward-looking field; it is cheap
  (one field) and it de-risks the *known* Wave-3 flip by making it data + testing both modes now.
  Everything else (R9/R11 logic, prose-detection, multi-algorithm fan-out) is deferred to its wave.
- **Immutability** — the audit fn builds new `errors`/`warnings` arrays; mutates no input.
