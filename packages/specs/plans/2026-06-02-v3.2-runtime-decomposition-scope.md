# v3.2 — Runtime Decomposition — Phase Scope

> **Status: SCOPING.** This frames the phase, sequences the work, and surfaces the decisions that must
> be made before the first line of code. It is NOT a build plan — each wave gets its own `/build-plan`
> with HETS routing when we start it. **An architect pressure-test (2026-06-02) corrected several
> "current-state" premises in the first draft — see § Runtime Probes; those corrections are folded in.**

## Goal (one sentence)

Give HETS **real decomposition** — a persona can break a task into a bounded tree of verified leaves —
and make **Axiom A4 binding**: deterministic logic lives in kernel algorithms with unit tests, not as
prose discipline or LLM-executed pseudocode.

Today (post-v3.1) the substrate can *contain* a single spawn (worktree + capability contract +
transaction record). It cannot yet *decompose* a task, *bound* the recursion, or *verify* a leaf
against its declared discipline. v3.2 is that layer.

## Why now (the A4 trigger)

A4 ("kernel scope SHALL include algorithmic logic", v6:387) was deliberately **scoped to v3.2+** —
v3.0-alpha/v3.1 only needed `subset_check.js` (K6) + `path_canonicalize.js` (K7) + `route-decide.js`
(already in `packages/kernel/algorithms/`). v3.2 ships the **K11 algorithm library** and the
A4-binding gate, so "the deterministic bit is a tested kernel function, not a paragraph the LLM
re-derives each spawn" becomes enforced, not aspirational.

## Runtime Probes (current-state premises — corrected by the architect repo-probe)

The first draft asserted greenfield work that the repo already has. Verified facts that move the plan:

- **R10 is NOT greenfield.** `packages/runtime/orchestration/budget-tracker.js` already tracks tokens +
  wallclock + `contract.budget` + extension grant/deny with a `withBudgetLock` (15s) + writes
  `run-state/<run-id>/budgets.json`. R10 = **extend it** (add the recursion-depth dimension +
  budget-exhaust→`ABORTED` wiring), ~80–120 net LoC, NOT ~150 new. Wave -1 probe: it is **CLI-shaped
  (`process.exit`)** — confirm whether the trampoline consumes it as a subprocess or needs a new
  import-friendly library export.
- **K11 currently = ONE file** (`route-decide.js`). The A4 gate has almost nothing to bind on day one;
  its real subjects are the deterministic logic v3.2 *itself* introduces (R9 predicates, R11 routing).
- **R11 spawn-verify is a NEW dispatcher, not a 4th arm of `spawn-close-resolver.js`.** That resolver
  (at **`packages/kernel/hooks/post/spawn-close-resolver.js`** — not `runtime/orchestration/`)
  dispatches spawn-**CLOSE** disposition (git delta) by env-flag; R11 routes leaf-**VERIFY** by
  discipline. Different axis, trigger, and routing key. The known-debt resolver-dispatch-table cleanup
  is NOT an R11 prerequisite (see Known-debt #2).
- **The OQ-11 "full validators" pattern already exists** — `functional[]` declarative-check arrays in
  every `*.contract.json` + the `validators={}` + `Object.keys` registry in `contracts-validate.js` +
  a *shipped* `decomposition-discipline-valid` validator. The architecture has effectively chosen (see
  OQ-11 recommendation).
- **The R8 discipline vocabulary already exists** — `interface.decomposition_discipline =
  {primary, fallback_when_code_producing}` in all 18 contracts. R8 should **drive off that field**, not
  invent a parallel `tdd/spec-driven/exploratory` enum.

## THE decision — OQ-11 (must close at Wave -1, cannot defer) — ✅ RATIFIED FULL (2026-06-03)

> **USER ratified (2026-06-03 boundary session): FULL-structure + KISS bodies.** Five registered,
> individually-tested R9 validators reusing the Wave-1 `validators={}` registry; minimal bodies (2 of 5
> are one-liners — see the normative criteria table in Boundary Decisions below). **~450–600 LoC**, NOT
> the ~800–1,200 strawman: the architect showed the slim/full LoC dichotomy is partly false — A4-binding
> (via K11, shipped Wave 0) forces the registered+tested per-criterion STRUCTURE (the only seam K11 can
> enforce in Wave 3), and KISS keeps the BODIES minimal. "Slim-as-absorbed-into-R11-prose" is the literal
> A4 anti-pattern; "slim-as-tested-functions" *is* full-structure. Overturns the 2026-05-26 blueprint
> slim lock (whose evidence was about the PreToolUse-injector path, not R9's verify surface). The
> analysis below is retained as the rationale.

**Leaf criteria (R9): slim predicates vs full validators** — a 2–3× LoC swing on the largest area
(v6:1479).

- **Slim (~400 LoC)** — thin boolean checks *absorbed into* R11. Less code; but you can't unit-test "is
  this a valid leaf" without standing up the whole R11 routing path, and a deterministic check living
  inside the dispatcher prose is *itself* the LLM-pseudocode-as-prose shape A4 forbids.
- **Full (~800–1,200 LoC)** — standalone composable validator objects mirroring the contract
  `functional[]` shape; independently testable (each with a synthetic negative-path test, as every
  existing `contracts-validate.js` validator has); Open/Closed via the existing `validators={}` registry.

**Recommendation (architect, evidence-backed): FULL validators**, because the pattern *already exists
and is load-bearing* in the repo (the `functional[]` array + the `Object.keys`-enumerated registry +
the shipped `decomposition-discipline-valid` validator), and R9 is the component most directly serving
the phase's reason to exist (A4-binding testable determinism). Spend the LoC where the axiom lives.
**Escape hatch (KISS-inside-full):** ship the registry + 5 criteria as full validators, but allow a
criterion *body* to be a one-line predicate where that's honestly all it is — Open/Closed structure
without gold-plating each check. **The USER/Wave -1 session ratifies; the rest of the plan forks here.**

### R9 normative leaf-criteria (the 5 — promoted here as canonical)

**CRITICAL fix (architect RISK-1):** the v6/v3 RFCs assert "5 deterministic conditions" but never *define*
the set — the operational checks survived only in superseded drafts (`v3.3-substrate-synthesis-v1.md:176`,
falsifiability-corrected in `v2.md:211-217`). Promoted here as the **normative set** so the Wave-2 R9 builder
does not re-invent five criteria from a number. (Promote into the v6 RFC when R9 builds.)

| # | Criterion | Operational check (falsifiable) | Shape |
|---|---|---|---|
| 1 | `cost-justified` | `estimated_tokens ≥ ~500 OR wallclock ≥ ~30s` (cutoff is a *named tunable*, not hardcoded) | one-line body + boundary test |
| 2 | `semantically-cohesive` | one clear purpose; tag-overlap (Jaccard) proxy — **ADVISORY, not a hard gate** | configured validator (soft signal) |
| 3 | `interface-clean` | focused input + structured output (schema/shape check) | natural validator (reuses the idiom) |
| 4 | `validation-supported` | a deterministic gate exists for this leaf's discipline (**R12 adapter-registry lookup**) | natural validator + the `R9→R12` DAG edge |
| 5 | `resource-bounded` | `estimated_tokens ≤ N AND capability declaration forbids sub-spawning` (boolean field read) | one-line body |

So: **2 natural validators (#3, #4) + 1 configured-advisory (#2) + 2 one-liners (#1, #5)** — the empirical
case for FULL-*structure* + KISS-*bodies*. Honest rename: "5 *deterministic*" is **4 deterministic + 1
advisory** (#2).

### K6 — RETIRE (rationale)

`packages/kernel/enforcement/k6-subset-check.js` (capability-subset-check) was built in v3.1 for **K8 as its
first consumer — but K8 is CANCELLED (ADR-0012)**. R9/R11 design now reveals **no runtime subset-check
materializes**: criterion #5 is a boolean field read (not resolved-axis `⊆` math); R11 routing keys on the
`discipline` string (not capabilities); and the reconciliation validator's "inline set-math" (ROADMAP:86)
is actually biconditional `tool ⟺ trait` `.has()` checks (`contracts-validate.js:1228-1273`), not subset
math — so the speculated convergence target doesn't fit either. **Retire K6 + its inert importer
`pre-spawn-tool-mask.js` (itself ADR-0012 dead code) together, as a standalone Wave-2-opening PR** (confirm
no `checkSubset` importer remains incl. the K6 unit test; cross-reference ADR-0012 in the commit). git
history preserves the well-built code if a v3.5 ContainerAdapter ever needs real `⊆` math.

## Proposed wave sequence (dependency-ordered DAG; acyclic, architect-verified)

Edges: `R11 → {R9, R12, R8}`; `R6 → {R7, R10, R8}`; `R9 → R8` (leaf validates discipline). **R8's
discipline set is a freeze-point** — lock it before R6/R9 consume it.

- **Wave -1 — design session** (~6–9h; architect sniff): **close OQ-11** (recommend full); **produce a
  concrete `failure_signature` field sketch** (ADR-0011); **freeze the R8 discipline set** (off the
  existing `decomposition_discipline` field); flag K6 (decide later); confirm no concurrent
  budget/depth writers; probe budget-tracker CLI-vs-import. *Nothing builds until OQ-11 + the schema
  sketch + the R8 set are closed.*
- **Wave 0 — foundations + known-debt**: `12-security` write-floor; **extend `budget-tracker.js`**
  (R10: recursion-depth + abort-wiring); **K11 scaffold** (library structure + A4 gate in WARN + a
  synthetic-violation fixture). (Opportunistic: resolver dispatch-table.)
- **Wave 1 — decomposition primitives**: **R7** TodoWrite-checkpoint + **R8** disciplines (frozen) +
  **R6** Pattern-A trampoline. Exit demo: a 3-leaf task tracked through TodoWrite + folder hierarchy
  within an R10 budget.
- **Wave 2 — verification tier**: **R12** adapters (incl. node-runner) → **R9** leaf-criteria (full,
  per OQ-11) → **R11** spawn-verify dispatcher (routes by R8, consumes R9 + R12) + **ADR-0011** freeze.
  **Decide K6 converge-or-retire here** (R9/R11 design now reveals the answer). Exit demo: 5 failing
  fixtures rejected by spawn-verify.
- **Wave 3 — close**: flip the A4 gate to enforcing — **precondition: R9 + R11 deterministic cores are
  tested kernel/runtime functions, not contract-prose or trampoline-prompt text** (not merely "K11
  complete"). Validate exit criteria; phase-close review (architect + code-reviewer + honesty-auditor,
  per v6:1544).

## Exit criteria (v6:1685)

1. A **Pattern-A trampoline completes a 3-leaf task within budget** (R6 + R7 + R10).
2. **5 failing fixtures are rejected by spawn-verify** (R9 + R11 + R12, incl. a node-runner fixture).
3. **A4 is binding** — the algorithm library shipped and the A4 gate enforces it over R9/R11's cores.

## Rollback / escape hatches

- Spawn-verify dispatcher: `LOOM_DISCIPLINE_OVERRIDE=spec-driven` → schema-only verification (v6:1492).
- A4 gate ships WARN-first, flips only once the current tree (incl. R9/R11) is clean — the repo's
  established warn→fix→flip pattern (e.g. `kb-architecture-related-bidirectional`).
- R10 extension behind the existing budget escape-hatch family.

## Biggest risk

Not sequencing (the DAG is sound). It's **sizing/sequencing R9 + R11 + the A4 gate against an imagined
clean slate** when the repo already has the budget primitive, the validator-registry, the discipline
field, and an empty K11 — that inflates the "new build" surface and hides the real work (extension +
integration), exactly where effort estimates rot. The Runtime-Probes section above is the antidote;
re-probe at Wave -1 before committing effort numbers.
