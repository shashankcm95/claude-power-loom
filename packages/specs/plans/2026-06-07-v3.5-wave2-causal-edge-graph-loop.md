2. **v6 §10b/OQ-24:1815 + §4.2:584 EXPLICITLY PERMIT this:** "a DERIVED index (regeneratable from the JSONL WAL) is a legitimate v3.5+ ... It is NOT the §4.2 dual-write trap — that trap is about a CANONICAL second-source that can disagree with the WAL; an advisory store that never feeds canonical state cannot.

**=> D1 is v6-conformant (§10b/OQ-24 + the E1/E4 advisory-store precedent). We do NOT touch the kernel schema or `validateTransactionRecord`.** Secondary reason (the recon trap that started this): the kernel JSON schema is **documentary, not runtime-enforced** — there is no `ajv` in the repo (0 hits); `validateTransactionRecord:358` self-documents "without pulling in a full JSON-schema library"; `:397` "the JSON-schema pattern is never enforced." ## Wave 2 sub-builds

### W2.1 — Semantic-edge store (`packages/lab/causal-edge/store.js` + `cli.js` + test)

- **Record shape** (a Lab record): `{ node_type:'causal-edge', edge_id, relation, source_block, target_block, conflict_type?, faithfulness_status, source_origin, recorded_at }`. NOTE: `source_block`/`target_block` are Lab-edge endpoints — we do NOT carry a kernel `affected_records` field; these are semantically unrelated to v6 §4.2's SUPERSEDE/TOMBSTONE `affected_records` targets (architect carry — prevents conflation).
- **R4 (closed enums + canonicalization) — self-owned NEW Lab validation:**
  - `relation` closed enum (9 values: `caused_by|depends_on|validated_by|contradicts|supersedes|regressed_after|fixed_by|reviewed_by|blocked_by`); `conflict_type` (`temporal|factual|contextual|conditional|null`, required iff `relation==='contradicts'`); `faithfulness_status` (`unvalidated|surface_overlap_only|advisory_llm_checked|human_confirmed`), **R1 fail-closed default `unvalidated`**.
  - **NFC/homoglyph defense (code-reviewer OK):** `String.prototype.normalize('NFC')` then reject any codepoint > U+007F in an enum-candidate field BEFORE the membership check (catches Cyrillic lookalikes, combining sequences, zero-width/BOM). Genuinely NEW (0 NFC logic in `packages/kernel`).
- **Identity + status are SEPARATE (code-reviewer FAIL-fix — the load-bearing store contract):** `edge_id = sha256(canonicalJsonSerialize([relation, source_block, target_block, conflict_type ?? null]))` — the WHOLE basis through the kernel canonical-json leaf (sorted keys -> cross-node reproducible; `MAX_FIELD_LEN=512` makes it total, no sentinel). `faithfulness_status` and `source_origin` are NOT in the basis. **The store DEDUPS on `edge_id`: one live row per edge identity** (NOT E1-accumulate — E1 accumulates distinct EVENTS; an edge is a stable identity with a mutable property). A separate **`updateEdgeStatus(edge_id, newStatus)`** does a findIndex-replace under `withLabLock` (the W1 `enrichRecord` analog) — this is how a faithfulness upgrade supersedes. `createEdge` on an existing `edge_id` is idempotent (returns the live row), never a second row.
- **Field caps:** `MAX_FIELD_LEN=512` on ALL stored string fields incl. `source_origin` (bounds ledger bloat + keeps the id basis total — code-reviewer carry).
- **Store mechanics (E1/W1 template, verbatim):** `withLabLock` advisory soft-lock (never `process.exit`); read-modify-atomic-rename-whole-ledger (`writeAtomicString`, never raw append); `readJsonlBounded` reads; `MAX_LEDGER_RECORDS`/`MAX_LEDGER_BYTES` caps; `LOOM_LAB_STATE_DIR` ENV-BEFORE-REQUIRE; ledger `<base>/causal-edges/ledger.jsonl`.
- **K12-clean:** import ONLY `kernel/_lib/{canonical-json, jsonl-read, atomic-write, lock}`. Ship the containment test (scan `require()` -> 0 `record-store|transaction-record|spawn-state|runtime/`), mirroring `negative-attestation/store.test.js:155-163`.

### W2.2 — The OQ-27 read-side walker (`packages/lab/causal-edge/walker.js` + test)

- PURE over a passed-in edge array (the `provenance-walk.js` purity contract; the STORE does the bounded `readJsonlBounded` read and feeds records in — bounds read/parse at the store boundary).
- **NET-NEW traversal (architect carry — NOT "generalize the leaf"):** `cluster`/`related` are UNDIRECTED multi-edge walks — a different shape than `provenance-walk`'s backward single-parent chain or its evidence-closure BFS. `collectEvidenceClosure` is the closest template (bounded-BFS + seen-set + maxNodes incl. its seed-phase cap — `provenance-walk.js:153` VALIDATE M-fix); the undirected walk needs the SYMMETRIC cap on neighbor expansion.
