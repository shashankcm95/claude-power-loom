# v3.11 W3 — the trap seam (the payoff, DATA-GATED)

W1 built the organ (derive -> signature -> leak-guard -> store) and minted the first real lesson
(N=1 dogfood). W2 built the confirmation gate + the first `confirmed-by` edge (a lesson can only
enter the predictor lane on a same-requirement verified delta). W3 is the **payoff wave**: capture
the **failed** attempt so `contrast(wrong-diff, accepted-fix)` is computable, wire the **confirmed**
recurrence weight (now that the W2 edge exists), and build the **signature-match** retriever.

**The honest frame (RFC Sec 7/8, OQ-NS-6, the BETA-internal-verification mandate):** W3 is
DATA-GATED. The machinery is the easy half; the *discrimination claim* ("signature-retrieval
measurably beats the lexical floor") needs N beyond the seed corpus with **real same-signature
collisions present** — which N=1 lesson cannot provide. So W3 delivers: (a) the three machinery
pieces, TDD-proven on fixtures; (b) a small real dogfood of the failed-attempt seam (the trap-seam
re-run, RFC Sec 10); (c) a measurement harness that reports **INSUFFICIENT-N** below the corpus
floor and never asserts a premature beat. This mirrors W1 (machinery + N=1 dogfood) and the v3.9
bootcamp (EC1-4 = MACHINERY-met-but-INSUFFICIENT-N). The N-floor re-run itself is a carry-forward.

## Design (three pieces; KISS, Open/Closed, additive)

### Piece 1 — the failed-attempt capture seam (`failed_attempt_ref`)

The RFC's core W3 idea: a lesson's gotcha is most legible from `contrast(wrong-diff, accepted-fix)`.
W1 contrasted the *passing* candidate vs the accepted diff; W3 adds the *failed* attempt's wrong-diff
as the primary trap signal.

- **`lesson-derive.js`** — extend `contrastInput` with an optional `failed_patch`. The leg now
  contrasts `{problem, failed_patch (the trap), candidate_patch (the passing approach), accepted_diff
  (the answer key)}`. Leak-guard UNCHANGED (only `accepted_diff` is sealed; `failed_patch` is a wrong
  attempt, never the answer key). Backward-compatible: absent `failed_patch` -> W1 behavior.
- **`candidate-sidecar.js`** — REUSED (module verbatim; a NEW guarded call-site is added — honesty
  LOW-1). `failed_attempt_ref = writeCandidate(failed_patch).sha` (content-addressed; `readCandidate`
  verifies on read). **Sidecar-write-failure disposition (hacker M2):** compute the ref ONLY from the
  confirmed-OK write (never `sidecarSha` before the write succeeds); on `!cw.ok` set
  `failed_attempt_ref = null` + count `n_failed_sidecar_failed` and **still mint the node** (degrade to
  a trap-less-but-valid lesson — the passing node anchors the lesson and is valuable on its own; this
  differs from the W1 candidate sidecar, whose failure skips the WHOLE node because that patch IS the
  node identity). TDD: no node ever carries a `failed_attempt_ref` for which `readCandidate` returns null.
- **`recall-graph.js`** — `attachLesson` / `buildWorkedExampleNode` accept an additive
  `failed_attempt_ref` (top-level, **NOT** in `LESSON_HASH_FIELDS` -> no orphaning; node_id stays
  patch-stable). `classifyLessonLayer` is UNCHANGED (the field is outside the hash, like `fail_to_pass`).
- **`lesson-capture.js`** — items accept an optional `failed_patch`; length-guard it (hacker M1 — the
  NEW untrusted input surface; mirror `LESSON_BODY_MAX`: an oversize `failed_patch` is treated as absent
  -> no trap, not a derive); sidecar it (per the M2 disposition); thread `failed_patch` into
  `deriveLesson` + `failed_attempt_ref` into `buildWorkedExampleNode`.

**Security note — the enforceable evidence-only boundary (hacker H2 + architect 2):**
`failed_attempt_ref` is **evidence + contrast-fuel, never a gate/authorization input.** Verified
firsthand: the W2 gate (`confirmsLesson`, `lesson-confirm.js:67-83`) reads `node_id`, `fail_to_pass`,
`candidate_patch_sha`, `accepted_diff_ref`, `worked_example_ref.issue_id` — never `failed_attempt_ref`;
the consolidation weight does not read it; the derive leg consumes the patch BYTES, not the ref. Its
integrity is the sidecar content-address (verify-on-read), not the node hash. But because it sits
*inside* a node whose other fields ARE tamper-evident, a future dev could mistake the whole node for
integrity-covered. So the boundary is made **enforceable, not just documented:** (a) a test asserts
`failed_attempt_ref` is NOT in `LESSON_HASH_FIELDS` AND that `confirmsLesson`/`consolidateLessons` do
not read it (a grep-able guard); (b) a code comment at each site forbids it as a trust input. The day
it becomes a trust input it requires a **versioned** `lesson_content_hash` (freeze-doc:140), never an
in-place add — a hard gate in R-W3-2, not a soft "if a future phase…".

**M1 leak note (hacker M1):** `failed_patch` is a wrong attempt, not the sealed answer key, so the
output leak-guard correctly stays `lessonLeaks(lesson_body, accepted_diff)` only. But `failed_patch` is
a new input the leg *sees*, and a wrong fix often shares context lines with the right one — so the
existing body-vs-`accepted_diff` guard (which scans the output regardless of which input inspired it)
remains the catch, and the new surface is length-bounded. Residual R-W3-5 (the contrast is now
two-channel; sub-threshold/homoglyph residuals from W1 `lessonLeaks` carry).

### Piece 2 — the confirmed-recurrence consolidation trust-weight (DEF-3) [C1 — the headline fold]

**The board CRITICAL (hacker C1, architect 3 — live-probed):** W1's `consolidateLessons` member
filter (`lesson-consolidate.js:35-39`) re-validates only the three enums + re-derives `lesson_signature`
— it NEVER re-derives `lesson_content_hash`. In W1 this fed only a `confirmed:false` *raw diagnostic*
(cosmetic). W3 turns the SAME function into a TRUST weight, so a p-writescope attacker who writes N
forged-but-self-consistent lesson nodes (each with its own correctly-computed hash — the #310 lesson:
a content-hash authenticates a *caller-chosen* field) + N self-consistent `confirmed-by` edges mints a
fully-forged `recurrence_count_confirmed` with **zero W2 gate runs**. This is the #273 family a THIRD
time in v3.11 (W1 H1, W2 H-A, now W3 C1). The fix is belt-and-suspenders — defend at BOTH the function
and the I/O boundary:

- **(a) Harden the pure core to speak the minter's FULL language.** The member filter (for BOTH the raw
  AND confirmed paths) requires `classifyLessonLayer(n) === 'valid'` (full `lesson_content_hash`
  re-derivation), not just the enum + signature re-check. Legit nodes (minted by `attachLesson`, which
  computes the hash) pass; a hash-lying forge is rejected. `consolidateLessons` is EXPORTED, so it must
  self-defend regardless of caller (completing the W1 M2 defense-in-depth note, which stopped short of
  the hash).
- **(b) Gate-enforce at the I/O boundary (the W2 H-A pattern).** Add an impure, dir-injectable
  **`runConsolidationPass({ nodeDir, edgeDir, reportFile, now })`** that sources nodes from
  `listNodes(nodeDir)` (verify-on-read passed) and the confirmed set from
  `confirmedNodeIds(listEdges(edgeDir))` (verify-on-read passed) — NEVER raw JSON / raw edge arrays
  (a raw forged edge passes `confirmedNodeIds`, which only checks `edge_type` + `isHex64`; only
  `listEdges` content-verifies). It calls the pure `consolidateLessons(verifiedNodes,
  { confirmedNodeIds: verifiedSet })`. The verification lives INSIDE the wired path, not a caller
  contract.

- **`consolidateLessons(nodes, opts = {})`** — when `opts.confirmedNodeIds` supplied:
  - each `per_signature` entry gains `recurrence_count_confirmed` = filtered-`list` members whose
    `node_id` is in the Set (the TRUST-bearing weight — DEF-3; ranges over the post-`classifyLessonLayer`
    filtered list, never the raw `nodes` arg — architect 3);
  - the report flips `confirmed: true`;
  - a `confirmed_under_separation_signatures` field (signatures with >1 distinct **confirmed** issue);
  - the RAW diagnostic (`recurrence_count_raw` / `distinct_issues` / `under_separation_signatures`) is
    KEPT (DEF-3 = confirmed weight + raw diagnostic).
  - When `opts.confirmedNodeIds` absent -> W1 behavior verbatim (`confirmed:false`, no confirmed field).

**Discipline:** the confirmed weight is computed over **verified nodes ∩ verified edges**. A node not in
the verified-edge set contributes to the raw diagnostic but NOT the trust weight (false >> missing: an
unconfirmed lesson is hazard-lane). TDD red test (hacker C1): a hash-lying node + a raw forged edge must
NOT be counted in `recurrence_count_confirmed`.

### Piece 3 — the signature-match trigger retriever + INSUFFICIENT-N harness

- **`_spike/retrieve-signature.js`** (NEW; lab spike, out of the live K4 CLI path) — a PURE retriever:
  - query = `{repo, trigger_class}` (the situation recognizer axis; the retriever needs *what
    situation you are in*, RFC Sec 6). repo stays a soft signal; `trigger_class` is the primary key.
  - **input contract (hacker H1 + architect 1):** ranks over store-loaded nodes (sourced from
    `listNodes()` — verify-on-read passed) AND additionally filters to `classifyLessonLayer(node) ===
    'valid'` (defense-in-depth, mirroring `consolidateLessons`'s self-defense). A forged / off-floor /
    hash-lying node is EXCLUDED from the ranked vector, never merely ranked low. Any internal keyed
    index uses `Object.create(null)` or reuses `groupByKey` (never a `{}` literal keyed on a
    node-derived string — a `trigger_class: "__proto__"` would crash a `{}` map; `groupByKey` is
    null-proto and safe).
  - rank lesson nodes by signature match: `trigger_class` exact match (primary), tie-broken by
    `recurrence_count_confirmed` (the trust weight — inherits C1's integrity: only trustworthy once
    Piece 2 is fixed), then `gotcha`/`corrective` overlap.
- **the measurement harness** — compare signature-retrieval vs the lexical floor (`_spike/retrieve.js`)
  on a held-out lesson set. **Collision-gated INSUFFICIENT-N (architect 5 + honesty HIGH-4):** report a
  `discrimination_margin` only when N >= the corpus floor AND **real same-signature collisions are
  present** (>=2 valid lesson nodes sharing a `lesson_signature` across DISTINCT issues — exactly the
  consolidation report's `under_separation_signatures`, `lesson-consolidate.js:64`). Collision-presence,
  NOT mere attempt-count, is the binding gate (the RFC Sec 7 "real same-signature collisions present"
  made precise — 20 lessons in 20 distinct cells cannot be discriminated at all). Otherwise report
  `'INSUFFICIENT-N'` (the `computeJudgeAgreement` precedent: `recall-graph.js:316`). NEVER a small-sample
  number dressed as a result — TDD: a below-gate fixture whose raw margin *would* favor signature-match
  STILL returns `'INSUFFICIENT-N'` (the leak-the-beat guard — honesty MEDIUM-2).

### The dogfood (real seam, out of CI)

`_spike/lesson-capture-rerun.js` is WIRED for the trap seam: when an item carries a `failed_patch`, it
rides into the real-leg prompt as the primary contrast (`contrast(wrong-diff, accepted-fix)`).

**What was ACTUALLY run this wave (honesty VALIDATE MED-2, corrected):** a **`--dry` deterministic
smoke** (a stub leg) that confirms the seam THREADS end-to-end — `failed_patch` -> sidecar ->
`failed_attempt_ref` (content-recoverable) -> minted node. **No real `claude -p` run was performed**
(unlike W1, which minted a real lesson via the real leg — `DOGFOOD-SAMPLE.md`). This is an honest scope
choice, not a gap: the derive leg is **behaviorally unchanged** from W1 (the prompt grew a `failed_patch`
contrast field — pure string construction, no new parse/verdict path), and W1 already dogfooded that the
real leg yields on-floor leak-free output. Every W3-new behavior (the threading, the confirmed weight,
the retriever, the gates) is **deterministic and unit-proven**. The genuinely-new real-data question
(does `contrast(wrong, accepted)` yield a *better* lesson?) is the DATA-GATED discrimination question,
deferred (R-W3-1).

**Verify-the-failure is a DOCUMENTED PRECONDITION, not yet code-enforced (honesty VALIDATE MED-3):** the
deferred real driver must run the failed candidate through the REAL test-runner (`pytest-runner.js`) and
assert `behavioral_verdict !== BEHAVIORAL_PASS` BEFORE sidecaring it as the trap (a "wrong" diff that
actually passes is a second valid approach, not a trap). This discipline is a comment in the spike
(`lesson-capture-rerun.js`), NOT a coded gate — the real N-floor driver that would enforce it does not
exist this wave (R-W3-1). **From-zero (honesty MEDIUM-1):** the live `~/.claude/lab-state/` lesson
population is currently **0** (only the 11 lesson-less `recall-graph-backtest/` nodes); the W3 `--dry`
smoke wrote to a temp dir, so "verifies through the store" is against an injected dir, not the live store.

## Residuals / carries (named, not surprises)

- **R-W3-1 (DATA-GATE, the headline — honesty HIGH-4 fold):** the discrimination exit
  ("signature-retrieval measurably beats the lexical floor") has **two** gates, not one: (a) N >= the
  corpus floor (`N_CLEAN_LARGE_MIN = 20`; today's live lesson pop = 0), AND (b) the corpus must actually
  **contain same-signature collisions** — non-trivial against the 24-cell frozen floor (4x3x2), where
  under-separation (DEF-3 raw diagnostic firing) is the *expected* near-term signal, not free
  collisions. The N-floor re-run must therefore be *engineered to produce collisions*, not just "collect
  20+ nodes" -> a v3.9-bootcamp-scale carry-forward. W3 proves the machinery on fixtures + smoke-tests
  the seam threading via `--dry`; the real `claude -p` trap-seam re-run (verify-the-failure coded) AND
  the real-corpus discrimination beat are BOTH deferred here (no real claude run this wave — MED-2/3).

  versioning rule, freeze-doc:140 — adding it to the hash would orphan every W1 node). Integrity = the
  content-addressed sidecar (verify-on-read). It is NOT a gate input (asserted by test + comment).
  Consistent with the standing #273 model (W2's `to_delta_ref`). **HARD GATE:** the day it becomes a
  trust input it requires a **versioned** `lesson_content_hash` (`lesson_content_hash_v2` over the
  extended field-set), NEVER an in-place add to `LESSON_HASH_FIELDS`.
- **R-W3-3:** the signature retriever stays in `_spike/` — the live K4 recall-CLI is MAJOR-protected
  and untouched (RFC Sec 9). No live-retriever wiring this wave (OQ-7 firewall unprobed for live).
- **R-W3-4 (G-AGNOSTIC):** a model-portability check for lesson *content* remains a future test.
- **R-W3-5 (hacker M1):** the contrast is now **two-channel** (`failed_patch` + `candidate_patch` both
  fed to a leg that also sees `accepted_diff`). The output leak-guard (`lessonLeaks` body-vs-accepted)
