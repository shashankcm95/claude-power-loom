## Phases

#### Phase 1 — `gh-emit.js`: the sanitized `gh api --input -` subprocess primitive (Risk: HIGH)
1. **`runGh(args, { env, input, timeoutMs, maxBytes })`** — `execFileSync('gh', args, { env, input, timeout, maxBuffer, killSignal:'SIGKILL', stdio:['pipe','pipe','pipe'], encoding:'utf8' })`. NO shell; array args; the sanitized `env` from `buildEmitEnv`; the JSON request body passed as `input` (stdin, consumed by `gh api --input -`). **Fail-CLOSED: throw on any non-zero exit / timeout / oversize (`maxBuffer`) / unparseable JSON.** Injectable via `deps.runGh` for tests.
   - **Contract note (board LOW-1):** `runGh` is fail-CLOSED, in DELIBERATE CONTRAST to `safe-exec.js`'s fail-OPEN (ADR-0001 returns `null`). Documented so a future "alignment" can't silently turn a network failure into a swallowed error on the highest-stakes path. No code-sharing — different binary, different failure contract.
2. **Transport — `gh api <endpoint> --method <M> --input -` with the FULL JSON body on stdin; NEVER `-f`/`-F` for ANY field (board CRITICAL-1, re-probed firsthand on gh 2.90.0):** `-F/--field` magic-reads a value starting with `@` as a HOST FILE (`@-`=stdin). The blob `content` is actor-influenced, multi-line, multi-MB — a `@path`-leading line would exfiltrate a host file into the first live PR. `--input -` sends the body as opaque JSON; no field-flag magic touches actor bytes. Rationale for the subprocess (vs node-https) unchanged: the shipped EC1b.1 killswitch=env-sanitization proof only has meaning for a credential-resolving subprocess.

#### Phase 2 — positional post-image reconstruction from the scrubbed diff (Risk: HIGH)
1. **`reconstructPostImages(scrubbedDiff)`** → `[{ path, mode:'100644', content }]`. **Scope (MVP, fail-closed): NEW-FILE adds ONLY.** A new-file stanza (`new file mode` + a single `@@ -0,0 +1,N @@` hunk) → the post-image is the N added lines, taken **positionally** from the hunk body (strip exactly one leading `+` per content line), so a content line that itself begins with `+` (a markdown doc about diffs) is preserved — **never** a `startsWith('+++')` string test (board HIGH-4). The line count MUST equal the hunk header's `+N` — refuse otherwise.
   - **Whole-file DELETE is DROPPED from the MVP (board LOW-3):** the `sha:null` delete path is doc-sourced AND would not be exercised by the new-file-add dogfood — shipping an un-exercised mutating path is the wrong cut. New-file-add is the only post-image case this wave emits; everything else fails-closed `cannot-reconstruct-postimage`.
   - **Canonical path per stanza (board HIGH-4):** `parseDiffPaths` ingests BOTH `diff --git b/<p>` and `+++ b/<p>` (probed: a crafted stanza yields two paths). Reconstruction takes the canonical path from the stanza and **refuses if the two forms diverge**, then **re-runs `isEgressDeniedPath` on each emitted path** (defense-in-depth: the emitted path is exactly a validated, non-denied path or the emit refuses).
   - **Forward-Contract source:** content is derived from `draft.diff` (SCRUBBED, secret-free) — NEVER a worktree (raw secret).
   - Probe: new-file fixture (incl. a `+`-leading body line) → content === the post-image; a modify-hunk fixture → throws `cannot-reconstruct-postimage`; a divergent-path stanza → throws.

#### Phase 3 — the gh-REST sequence + the REAL round-trip self-check (Risk: HIGH)
1. **`ghEmit({ draft, token, ghConfigDir, approvalHash }, deps)`** — what `armedEmit` delegates to:
   - **The REAL self-check FIRST (board CRITICAL-2, 3-lens convergence):** (a) `computeEmissionHash(draft) === approvalHash` — a genuine cross-check (the gate computed `approvalHash` from the draft at emit-pr.js:382; `ghEmit` independently recomputes; mismatch ⇒ refuse, zero network). (b) every reconstructed path ∈ the validated path-set of `draft.diff`, and `isEgressDeniedPath` passes per emitted path. (c) the reconstruction consumed exactly the new-file `+` content (positional count == `@@ +1,N`). Any failure ⇒ refuse before ANY `runGh` call. (NOT a re-hash of the draft against itself — that was the tautology.)
   - **Envelope = KERNEL CONSTANTS, zero actor bytes (board HIGH-3):** the commit `message` and PR `body` are fixed kernel strings (they may interpolate the already-validated integer `issueRef` and the `approvalHash`, never free actor text); `title` is the existing kernel template. The signed approval binds only `{repo, issueRef, diff}` — the envelope is kernel-templated + deterministic + NOT part of the content-address (stated honestly as a residual, not claimed as "approved").
   - **Sequence (all via `runGh` + the sanitized env; base ref is the API-resolved DEFAULT branch, never actor-supplied):**
     1. `GET repos/{repo}` → `default_branch`. **Validate it** (`^[A-Za-z0-9._/-]+$`, non-empty, no `..`, not a `owner:branch` cross-fork form) before it rides into argv / the ref path (board MEDIUM-3).
     2. `GET repos/{repo}/git/ref/heads/{default_branch}` → base commit SHA; `GET repos/{repo}/git/commits/{baseCommitSha}` → base tree SHA. (**base_tree-accepts-a-commit-ish is doc-sourced; probe at dogfood time — if `base_tree`=commit-sha is accepted, drop this commit→tree GET; board MEDIUM-1.**)
     3. `POST repos/{repo}/git/trees` `--input -` body `{ base_tree, tree:[{path,mode:'100644',type:'blob',content}] }` → new tree SHA.
     4. `POST repos/{repo}/git/commits` `--input -` `{ message:<kernel const>, tree:newTreeSha, parents:[baseCommitSha] }` → commit SHA.
     5. `POST repos/{repo}/git/refs` `--input -` `{ ref:'refs/heads/loom/issue-{issueRef}-{approvalHash.slice(0,12)}', sha:commitSha }` → branch. **RESERVE point.** The branch name is kernel-built from the validated integer + the hash — deterministic, no actor string.
     6. `POST repos/{repo}/pulls` `--input -` `{ title, head:<branch>, base:default_branch, body:<kernel const>, draft:true }` → DRAFT PR. Return `{ pr_url, number, branch }`.
   - **`draft:true` is a HARD CONSTANT** (security.md non-bypassable-guard) — no opts/data path can set it false; ready-PR is a future arming step.
   - **422 dedup-on-key (board HIGH-1 + MEDIUM-5, the single highest-leverage fix):** the deterministic branch name IS the idempotency key. On step 5 returning 422 "Reference already exists", `GET repos/{repo}/pulls?head={owner}:{branch}` → return the existing `{ pr_url, number, branch, deduped:true }` rather than throwing. This makes the emit **idempotent**: a retry after a network-success-but-record-failure (the EC1b.5e partial-commit case) reconciles to the existing PR — **no duplicate PR**, which was the real harm. This closes the duplicate-emit risk EC1b.5e flagged "deferred to 3.2.5".
   - **Reserve→rollback:** if step 6 fails after step 5, `DELETE repos/{repo}/git/refs/heads/loom/...` (best-effort; logged; never throws over the original error). **Residual stated:** a rollback DELETE that itself times out leaves an orphan ref — recoverable because the branch name is deterministic (a retry 422-dedups to it). No circuit-breaker (YAGNI; the per-window cap + human gate bound volume).
   - Probe: a mock `runGh` records the call sequence → assert exact order + every write uses `--input -` and zero `-f`/`-F`; a self-check mismatch → refuse, zero `runGh`; a 422 on step 5 → the dedup GET fires, `deduped:true`; a step-6 failure → the rollback DELETE fired.

## What this DOESN'T claim to fix

- It does NOT harden trust (OQ-NS-6: only a real maintainer-MERGE does). A self-owned DRAFT PR is not an external acceptance signal.
- It does NOT close integrity→provenance by itself — the cross-uid broker (5b, **deployed**) makes the signature provenance; un-deployed/same-uid the dogfood proves only the MECHANISM (and the gate is not an authorization boundary same-uid).
- **The human's signature covers the FILE CONTENT (the scrubbed diff), NOT the PR envelope.** The commit message, PR body, base branch, and branch name are kernel-templated + deterministic but ride out UNSIGNED. "Byte-identical approved draft" is true for the secret-bearing file content, NOT the envelope (board HIGH/honesty).
- **The human reviews the DIFF text; the emitted post-image is a RECONSTRUCTION of that diff** (hash-bound, not separately rendered at the approve-CLI). The reconstruction's faithfulness rests on the positional parse + the count-assert + the STRICT-single-hunk fail-closed (a multi-hunk / extra-line stanza refuses rather than truncating) — adequate for SHADOW, but a reconstruction defect would emit content the human approved only transitively (VALIDATE-honesty MEDIUM).
- **Dedup closes the duplicate-OPEN-PR; it does NOT auto-recover a closed-PR-then-retry.** A 422 with a matching open loom PR reconciles; a 422 with NO open PR FAILS CLOSED (`ref-exists-no-open-pr`, operator-verified) rather than auto-creating on a pre-existing branch (the laundering guard). A retry also leaves an orphan tree+commit created before the ref step — bounded, not a duplicate PR (VALIDATE-hacker MEDIUM + honesty MEDIUM).
- **Scrub over-redaction becomes literal file content** — the post-image reconstructs the SCRUBBED `+` lines verbatim, including any `[REDACTED-ENTROPY]`/`[REDACTED-BASE64]` markers. The human at approve-cli sees these markers in the rendered scrubbed diff, so over-redaction is human-catchable at the sign-what-you-see gate; an automated path would emit corrupted content (board honesty MEDIUM).
- The post-image reconstruction is new-file-add only; a real loom solve's modify-diff is not yet emittable.

## Why this is the right shape

- **`gh api --input -` subprocess:** the `buildEmitEnv`/killswitch=env-sanitization apparatus + the shipped EC1b.1 proof only have meaning for a credential-resolving subprocess; `--input -` neutralizes the `-F` `@file`-read exfil channel while keeping that design.
- **Post-image from the scrubbed diff, positionally:** only the scrubbed diff is the approved, secret-free artifact; positional reconstruction (not string tests) is the only faithful derivation; the round-trip self-check + per-path re-validation make "emit exactly what was approved & validated" enforceable, not aspirational.
- **Kernel-constant envelope:** rather than expand the merged+frozen approval axiom, the unsigned envelope is bytes-free of actor influence — the smallest cut that closes the un-approved-field channel.
- **Dedup-on-deterministic-branch-name:** one mechanism makes the emit idempotent — closing both the EC1b.5e duplicate-emit hole and the human-double-approve case.
- **A separate `gh-emit.js`:** emit-pr.js is already 421 lines; the network mechanism is its own SRP module; `armedEmit` is a thin delegator.
