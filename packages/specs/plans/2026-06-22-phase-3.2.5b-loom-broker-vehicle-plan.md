# ③.2.5b — the dedicated cross-uid `loom-broker` vehicle (provenance-ABLE, not provenance-closed)

## Goal

Build the cross-uid broker **vehicle** that, *once deployed under a separate uid and attested out-of-band*, closes
the integrity -> provenance gap left open by ③.2.5a. **This PR closes NOTHING on its own** — un-deployed, the
wrapper runs same-uid and the host can still read the key, so it proves no uid separation (PACT NS-7). 5a built the
SIGNED-approval *verify* half (`verifyApproval` checks a broker ed25519 sig over the freshness-bound basis); the key
is still same-uid / in-process, so a host process can co-forge a byte-valid approval — 5a is **integrity +
mechanism, NOT provenance**. 5b ships the vehicle to move the signing key **out of the host heap into a dedicated
cross-uid broker** so that **WHEN DEPLOYED cross-uid**, the actor process is structurally unable to mint a valid
approval. Provenance is conferred by the *deployment* + the operator's out-of-band attestation (NS-7) — never by
this merged code. (Mirrors PACT broker-sign §0: "a custody MECHANISM, world-anchorABLE; NOT custody-real here.")

> **The honest one-liner (carry into ROADMAP/MEMORY/PR-title verbatim):** *5b builds the cross-uid signing vehicle;
> it does not deploy it and closes no provenance gap on its own — only the operator's out-of-band uid attestation
> hardens the vehicle into provenance (OQ-NS-6: merged code NARROWS; the out-of-band attestation HARDENS).*

**This PR is the broker VEHICLE + recompute-bind + caller-auth + custody-verify + runbook.** It is purely
ADDITIVE: no existing egress module changes, `armedEmit` STILL throws, EC6 holds, SHADOW. The production caller
(the **approve-CLI**, the human "sign-what-you-see" gate that drives `recordApproval` with the broker signFn) is
**split out to ③.2.5b.2** for reviewability — this PR ships the security-critical signer mechanism; a test/operator
injects the signFn.

## The port (PACT -> Loom) — what carries verbatim, what adapts

PACT and Power Loom **share the same `edge-attestation` ed25519 leaf + the `(id, body)->sig|null` signer shape**, so
the mechanism ports near-verbatim. The DEDICATED trust domain (USER decision — separate uid/key/registry from
`pact-broker`, blast-radius isolation) means every `PACT_*` token becomes `LOOM_BROKER_*` and the registry/key are
the loom-broker's own.

**Carries verbatim** (re-derive, do not import across repos):
- **env-allowlist client** (`brokerSigner`): child env built FROM SCRATCH, `process.env` NEVER spread (the
  NODE_OPTIONS/--require defense, HIGH-1); `RESERVED_ENV` refuses `NODE_OPTIONS|LOOM_BROKER_KEY_FILE|LD_|DYLD_`;
  fixed argv with the strict-hex basis as the FINAL arg (no shell -> no argv injection); positive-int `timeout` +
  `maxBuffer` DoS bounds (0 = footgun -> default); canonical-base64 + 64-byte output re-gate (defense-in-depth).
- **O_NOFOLLOW key wrapper** (`broker-sign`): `O_RDONLY|O_NOFOLLOW|O_NONBLOCK` open (symlink refused atomically;
  FIFO opens-then-rejected, no hang) -> `fstat` the RESOLVED fd (swap-immune TOCTOU defense) -> `isFile()` +
  no-group/world-writable (`& 0o022`) -> read THAT fd; close-before-fail (process.exit skips finally); stderr-only
  fixed messages (NEVER key bytes / NEVER err.stack); empty stdout + non-zero exit on any fail.
- **validated sudo launcher** (`broker-launch`): `crossUidLoomBrokerSigner({brokerUser,wrapperPath,sudoPath})` ->
  `sudo -n -u <user> <wrapper>`; `USERNAME_RE=/^[a-z_][a-z0-9_-]{0,31}$/`; absolute + no-dotdot + no-control-char
  path guards (sudo flag-injection); command PINNED to sudo; `-n` (never block on a password prompt).
- **custody-verify** (`custody-verify`): pure `assessCustody(facts)` + impure `gatherCustodyFacts`; C0-root /
  C1-keypresent(lstat) / C2-denied(host-can't-read AND owner-differs) / C3-liveness(real sign+verify) /
  C2.5-wrapper; `hostObservableChecksPassed` (NOT `custodyVerified` — NS-9); the out-of-band uid attestation is the
  SOLE custody-real determiner; `--attested-cross-uid` changes only the exit code, never the proof.

**Adapts (the Loom-specific deltas — VERIFY-board-folded):**
1. **Recompute-bind = ctx<->basis CONSISTENCY, NOT human-approval (the honest framing — VERIFY hacker/arch C1).**
   The broker receives `basis` (argv, 64-hex) + `ctx={emission,approvedAt,nonce,key_id}` (stdin) and **re-derives
   the hash from the emission BODY** — it IGNORES any presented `hash` field (#273: verify the body, not the
   asserted field) — via the SAME kernel module (`require('./approval')`, never a re-impl):
   `h' = computeEmissionHash(ctx.emission)`; `basis' = approvalSigBasis({ hash: h', approvedAt: ctx.approvedAt,
   nonce: ctx.nonce, key_id: ctx.key_id })`; refuse unless `basis' === basis`. It then **signs `basis'` (the
   RECOMPUTED value), never the argv basis** — argv is consumed ONLY for the `===` gate then DROPPED (PACT
   request-auth.js:81). **This binds the signed basis to a self-consistent emission context — it does NOT prove a
   human approved that emission** (the actor controls both argv and ctx, so a forged-emission ctx passes the `===`
   gate). The provenance gate is NOT here — see the trust model: it is (a) cross-uid key custody + (b) caller-auth
   restricting WHO can reach the broker (via sudo) + (c) the human-at-the-approve-CLI presenting the ctx (5b.2).
   Recompute-bind is the WHAT-consistency defense-in-depth layer; it also inherits 5a's freshness (a TTL-bump /
   nonce-swap flips `basis'`).
2. **Exact-shape ctx type-gate BEFORE recompute, fail-closed (VERIFY arch C2 + hacker H1).** The ctx is untyped
   stdin JSON; a number-vs-string `approvedAt` produces a DIFFERENT basis (probed live by the hacker), and
   `computeEmissionHash([...])` / a scalar returns a valid hex (PACT request-auth.js:119). So `loom-broker-bind.js`
   validates the EXACT shape — `approvedAt` a finite number, `nonce` a non-empty string, `key_id` a non-empty
   string, `emission` a plain non-array object, **reject extra keys** — and DENIES (never coerces) on any miss,
   mirroring `verifyApproval`'s field gates (approval.js:103-106). Recompute-bind is ALWAYS-ON (no PACT
   "require-frame" toggle — 5b is greenfield, no blind-oracle back-compat to preserve).
3. **Bounded AND deadlined stdin (VERIFY arch L1 + hacker H2).** Port PACT's `readStdinBounded` (broker-sign.js:36):
   a byte cap (`MAX_CTX_BYTES`, sized for a scrubbed `emission.diff` — larger than a PACT persona frame) bounds
   VOLUME; a wall-clock `READ_DEADLINE_MS` bounds TIME (a never-EOF slow-loris would otherwise hang forever —
   `fs.readFileSync(0)` is FORBIDDEN). Both positive-int-guarded (0 = footgun). Drain the ctx FIRST — before any
   gate that can `process.exit` — so the host's `execFileSync input:` write completes (else an early refuse EPIPEs
   the host).
4. **Caller-auth unset => DENY (fail-closed), not PACT's "disabled-LOUD" — but it is contingent, NOT independent
   (VERIFY hacker C1).** For an egress signer the WHO gate is load-bearing: an unset `LOOM_BROKER_ALLOWED_UIDS`
   denies all + emits a LOUD stderr (the `fail-CLOSED security decision must be OBSERVABLE` rule, graduated this
   session). BUT `SUDO_UID` is host-forgeable on a DIRECT (non-sudo) invoke — so **caller-auth provides NO security
   on a direct invoke; its soundness is ENTIRELY contingent on the cross-uid deployment (sudoers `env_reset,!setenv`
   so sudo injects an unforgeable `SUDO_UID`, + the wrapper/key dir unreadable to the actor uid so a direct invoke
   gains nothing). The real, independent control is the cross-uid KEY CUSTODY** (the actor can't read the key). The
   operator sets the allowlist to the human/approve-CLI uid, NEVER the actor uid, in the root-owned wrapper.
5. **Bind + caller-auth are broker-CLI-side ONLY (VERIFY arch M3).** The CLI is directly invokable, so a client-side
   gate is not a security boundary (PACT broker-sign.js:110). The client's only gates are the env-allowlist + DoS
   bounds + output re-gate (defense-in-depth). All security lives in the key-holding wrapper.
6. **Naming + domain:** `LOOM_BROKER_KEY_FILE`, `LOOM_BROKER_ALLOWED_UIDS`; modules under `packages/kernel/egress/`;
   the registry/verify-key is the loom-broker's own custody file (the 5a `custodyVerifyKeyPath` the egress already
   reads) — NEVER `LOOM_EDGE_VERIFY_KEY` (the edge/lesson domain; H2 dead-pin lesson).

## Trust model (what the DEPLOYED vehicle provides — every guarantee below is contingent on the cross-uid deployment; un-deployed/same-uid, ALL are vacuous)

- **NON-EXFILTRATION** (the HSM-shaped core — the ONE independent control): WHEN deployed cross-uid, the signing key
  lives behind a separate-uid process the actor cannot read (`cat <key>` -> Permission denied) nor ptrace. The
  actor cannot self-sign. **Un-deployed (same-uid), this bullet is vacuous** — the host reads the key (PACT
  broker-sign §0). Deployment + the out-of-band attestation is what makes it real (NS-7).
- **WHO** (caller-auth) — a CONTINGENT, defense-in-depth narrowing, NOT an independent control: only an allowlisted
  `SUDO_UID` can request a signature. This holds ONLY when sudo injected `SUDO_UID` (sudoers `env_reset,!setenv`);
  on a DIRECT (non-sudo) invoke the actor forges `SUDO_UID` freely (hacker-probed). So caller-auth's security rests
  on the NON-EXFILTRATION control (a direct invoke gains nothing because the key is unreadable + the wrapper/key dir
  is actor-unreadable). The actor uid is NEVER on the allowlist.
- **WHAT** (recompute-bind) — ctx<->basis CONSISTENCY, NOT human-approval: the broker signs only a basis that
  re-derives from the presented emission body. The actor controls the ctx, so this does NOT prove a human approved
  the emission — it prevents signing a basis decoupled from ANY emission, and inherits 5a's freshness. Defense in
  depth, not the provenance gate.
- **The provenance chain (what actually makes a forged sign-off impossible), WHEN deployed:** (a) NON-EXFILTRATION —
  the actor can't sign; (b) caller-auth via sudo — the actor uid can't reach the broker; (c) only the human at the
  approve-CLI (5b.2), running as the allowlisted uid, presents the ctx after sign-what-you-see. Remove ANY of the
  three (esp. un-deployed (a)) and provenance collapses.
- **Provenance is a DEPLOYMENT property** (NS-7): the host cannot observe uid separation — only the operator's
  out-of-band `id`/`ls -l`/`cat` attestation HARDENS the vehicle into provenance. custody-verify checks the
  host-observable NECESSARY conditions, reports `hostObservableChecksPassed` (NEVER `custodyVerified`), never claims
  sufficiency.
- **Carried LOUD residual (do NOT report closed):** the entitled operator (the human at the approve-CLI) can sign
  ANY well-formed emission — the single-operator-payload-authority ceiling. That is BY DESIGN (PATH-1 = human is the
  sole gate); the deployed vehicle makes the human's approval UN-forgeable by the actor, it does not adjudicate the
  human's judgment. "Did a human SEE this exact draft" is the approve-CLI's sign-what-you-see job (5b.2).
