## Design

### G1 — dedup-by-subject (the `verdict-fail` source only)

The dedup is a **source-local transform** inside `SOURCES['verdict-fail'].list` — the store keeps
every reviewer record (probe 4: accumulation is the store's contract; E4 reputation NEEDS the
distribution), and the window/threshold math stays source-agnostic (probe 7).

- **Dedup ONLY among the countable class** (VERIFY code-reviewer F2, HIGH): partition each
  group first — records with a parseable `recorded_at <= now` are the countable class; the
  group's deduped event = the LATEST of those. Future-dated and undated records PASS THROUGH
  un-deduped, so the projection loop still lands them in `excluded_future`/`excluded_undated`.
  Without this partition, a hand-written future-dated line sharing a real build's `agent_id`
  would become the group representative and the window loop would silence the WHOLE group —
  a forged record HIDING a real in-window denial (the CR-1 under-count class re-entering via
  the dedup layer). With it, the count is preserved AND the tamper diagnostics stay intact.
- **Timestamp of the deduped event = the LATEST countable `recorded_at` in the group.**
  Rationale: the narrowing-safe direction. The breaker's bad failure direction is UNDER-count
  (fails to trip — the W4 H1 lesson class); latest-sighting keeps the denial in-window longest
  and reads honestly as "when fail evidence for this build was last attested". (VERIFY
  architect F3 confirmed latest over earliest; the latch-interaction residual is named in
  Out-of-Scope.)
- **Fail-soft for a hand-written line missing `agent_id`**: a per-record POSITIONAL sentinel —
  agent_id-less rows NEVER collapse (VERIFY architect F5 + code-reviewer F1 HIGH closed the
  `undefined`-key collapse; CodeRabbit #305 Major then removed the intermediate attestation_id
  rung — an id-keyed collapse of hand-written rows is itself an under-count lever, since a
  forged row can REUSE another row's attestation_id to suppress it. Positional keys make the
  forged-input class strictly over-count-safe; a hand-duplicated line counting twice is the
  SAFE direction). Store-written records always carry agent_id (probe 2), so this path is
  forged-input-only.
- Persona of the deduped event: from the latest record in the group (H-1 makes mixed-persona
  groups store-impossible; a hand-written mislabel mis-buckets only its own event).
- The other three sources are per-EVENT streams (a decompose-reject / a committed mint / an
  integrator reject) — one record IS one denial; no dedup applies. G1 is verdict-fail-specific
  and documented as such at the source registry.
- `denials_in_window` semantics for verdict-fail becomes "distinct failed subject spawns in
  window" — update the D6 comment + cli.js USAGE + the ACTIVATION-LEDGER E11 row.

### G2 — source-validation in the API layer (not just the CLI)

Move the starved-source knowledge INTO `project.js` so every consumer gets it — the CLI warn
stays, but derives from the same registry (single source of truth):

- Each `SOURCES[id]` entry gains `starved: boolean` — **a STATIC registry fact, not a dynamic
  probe**: `negative-attestation: true` (E1's producer, the decompose tier, is probe-dead);
  `verdict-fail` / `manage-promote` / `reject-event`: `false` (live producers — Rule-4 routine,
  promote-path mints, integrator rejects). Rationale for static-over-dynamic: "0 records" from a
  LIVE source is a legitimate all-clear (the E11-M1 lesson: a clear/empty result is not a safety
  signal — in BOTH directions), while a dynamic store-life probe cannot distinguish "healthy,
  no denials" from "dead producer" and introduces a bootstrap deadlock (the first-ever promote
  would see an empty scan dir and refuse). When E1 un-starves (decompose tier goes live), the
  flag flips in the same PR that wires the producer — exactly like `NON_STARVED_SOURCES` today.
- `projectBreaker` + `evaluate` outputs gain `source_starved: boolean` (additive; no existing
  key changes — non-breaking for promote.js and the A3b consumer).
- `evaluate` gains **`opts.requireLive`**: when `true` and the resolved source is starved, THROW
  a clear Error (the REFUSE arm — "a kernel gate must never silently read a probe-dead tier and
  report all-clear"). The v3.9 gating consumer passes `requireLive: true`; advisory consumers
  omit it and read the flag. Default stays non-throwing (§0a.3.1: the advisory tier never
  hard-fails a caller). Three VERIFY folds on the throw arm:
  - **The throw composes with the shipped gating-consumer shape** (architect F1, HIGH —
    premise-probed firsthand): `promote.js:312-316` already wraps `evaluateBreaker` in
    try/catch and converts ANY throw into `refuse('breaker-source-unavailable')` — so a
    requireLive throw IS a clean fail-closed refusal for a promote-shaped consumer. The
    project.js header documents the contract: requireLive callers MUST wrap in try/catch — a
    starved source under requireLive is an exception BY DESIGN (fail-closed-loud). The two
    refusal idioms deliberately diverge and the doc names it: tamper signal
    (`excluded_future>0`) → a refuse() VALUE; starved source under requireLive → a THROW. Do
    NOT soften starved to a value — a probe-dead tier must never read as a benign all-clear
    field a careless consumer ignores (the M-1 trap re-opening).
  - **Bypass wins over requireLive** (code-reviewer F3, HIGH): `evaluate` checks `bypassed`
    FIRST (the existing CR-4 ordering) and returns the bypassed decision BEFORE the
    requireLive starved-source guard. Bypass is the operator's explicit, exact-`'1'` override
    of ALL trip logic, including source-health refusal. Test-locked.
  - CLI passthrough (code-reviewer F5): parseArgs stores a valueless flag under its HYPHEN
    key — the mapping is `const requireLive = args['require-live'] === true` →
    `evaluate({..., requireLive})` (bracket access, NOT `args.requireLive` which is silently
    undefined). Test: `check --source negative-attestation --require-live` → exit 1 via the
    existing catch path, with the starved-source message on stderr.
- cli.js: `NON_STARVED_SOURCES` is DELETED; `warnIfNonDefaultSource` re-derives from the
  project.js registry (warn when resolved source is starved — which today reproduces the exact
  current behavior: only `negative-attestation` warns).

### The hysteresis LATCH — stateless look-back (no state file; the deterministic-theorem property holds)

(Named the HYSTERESIS latch, not "half-open" — VERIFY architect F4: a true half-open breaker
admits one trial request and re-opens on its failure without human action; this latch has no
single-trial admission. For the HUMAN-GATED v3.9 consumer that distinction is immaterial — the
human IS the trial-admission decision — so **this discharges A-2 for v3.9** ("is a half-open
probe needed?" → no active probe; hysteresis + human-as-prober suffices). **A-2 RE-OPENS for
autonomous v4.x**, where a true probe semantic is owed — named in Out-of-Scope.)

The A-2 risk: a live gating consumer flaps — denials age out, trip clears optimistically,
destructive work resumes, fails again, re-trips. A stateful open/half-open/closed breaker would
break the v6:509 pure-theorem contract (W4's deliberate choice). The latch keeps statelessness:

- **`latched(now)` = the window-count was at-or-over threshold at SOME moment `t` in
  `(now - LATCH_MS, now]`** — `t` over CONTINUOUS time, not just denial timestamps. (The
  draft's denial-timestamps-only formalization was WRONG — VERIFY code-reviewer F8's fixture
  math exposed it: with all denials at `t0` and `LATCH = W`, crossing-at-denial-times makes
  the latch expire at `t0 + LATCH` — INSIDE the window-trip period, a no-op. The count stays
  >= threshold until the burst ages out at ~`d_max + W`, so the latch must run from THERE.)
  **Discrete check (O(n) two-pointer over the per-plane, exclusion-filtered, sorted denial
  timestamps `d_1..d_n`, threshold `k`):** latched ⟺ ∃ j:
  `d_{j+k-1} - d_j < W` (the k denials fit one window) AND `d_{j+k-1} <= now` AND
  `d_j > now - LATCH_MS - W` (some window containing them ends inside the look-back).
  For the all-at-`t0` fixture this yields trip-ends-at `t0 + W + LATCH_MS` — the latch
  EXTENDS the window trip, as intended ("a trip lives ~2 windows" at the default).
- The look-back applies the SAME exclusions as the window loop (future-dated and undated
  denials never enter the latch math — the CR-1 class must not re-enter via the look-back).
- **Scan-source fetch horizon (SELF-CAUGHT post-board, plan-fold):** the manage-promote and
  reject-event sources pre-filter their scans to `sinceMs = now - windowMs()` — the look-back
  would see NO aged-out mints and the latch would be structurally INERT for exactly the
  sources the v3.9 gating consumer uses. The source fetch horizon becomes
  `now - (windowMs() + latchMs())`; the window loop still counts only `(now - W, now]` (the
  extra records fall to the existing `ts <= windowStart` branch, feeding ONLY the latch math).
  Test-locked over a real scan-source fixture (test 25: 10 TOMBSTONE mints at now-15min →
  window count 0, global latched true).
- **View-vs-decision split (probed against the existing locks):** the VIEW (`projectBreaker`)
  reports per-AXIS facts — row/global `tripped` stays WINDOW-only (`denials_in_window` keeps
  its in-window meaning; project.test.js tests 2/3/4 stay green as window-axis locks); rows
  and `global` gain `latched`; a persona row now appears when `denials_in_window >= 1 OR
  latched` (a latched-but-aged-out persona must be visible — no existing test seeds that
  state, so no lock breaks; test-11/verdict-7's `persona absent` cases are future-only
  denials, which never latch). The DECISION (`evaluate`) combines:
  `tripped = window_tripped OR latched`.
- `tripped = window_tripped OR latched` (at the decision) — the latch only ADDS trips (narrows-only; §0a.3.1-safe
  by the same argument as every other E11 widening-free change).
- After the latch expires with no fresh denials, the next `evaluate` is clear — the consumer's
  next attempt IS the probe; a fresh fail re-trips and re-latches.
- Computed per-persona AND global (both trip planes latch independently, each with its own
  threshold).
- **Decision-shape: the latch is a SEPARATE AXIS — `scope` stays the PLANE enum, unchanged**
  (the two lenses disagreed here; synthesis): the architect (F2) argued plane and time are
  orthogonal axes — `scope:'latched'` for a latched-global trip LOSES the plane information
  and would rank a recently-crossed global cap below a live persona trip; the code-reviewer
  (F4) flagged that `scope:'global'` with `global_tripped:false` is the W4-CR-M1 misleading
  triple. **Both are satisfied by the explanatory pair**: add `latched: boolean` +
  `latched_global` + `latched_persona` (mirroring the existing `global_tripped` /
  `persona_tripped` split); `scope` reports the plane of the EFFECTIVE trip (window OR
  latch; global still supersedes persona); no `'latched'` scope value. A consumer reading
  `scope:'global', global_tripped:false, latched_global:true, latched:true` sees both facts
  with no contradiction — the triple is self-explaining. promote.js + A3b backward-compat is
  preserved for free (the scope value set never changes; all new fields additive).
- `LOOM_BREAKER_LATCH_MS` env-tunable with the W4 clamp discipline: floor = `WINDOW_MS_FLOOR`
  (60s), ceiling = 24h, default = `windowMs()`. A latch shorter than the window adds little
  by construction (the window itself still trips) — safe.
- **Env-stability note (code-reviewer F7)**: the latch is a pure function of
  `(ledger, now, LOOM_BREAKER_WINDOW_MS, LOOM_BREAKER_LATCH_MS)` — the look-back re-evaluates
  history with the CURRENT window env. Narrowing the window between calls can false-CLEAR a
  historical crossing (the under-count direction); widening can false-latch (the safe
  direction). Documented in the project.js header: a gating consumer keeps both envs stable
  across a wave; narrow only between waves.
- `bypassedView` stays shape-stable: it gains `source_starved`, `latched:false`,
  `latched_global:false`, `latched_persona:false`, `latch_ms` (same keys as the live view —
  the CR-4 no-NPE discipline).
