## 2. Why this is more than a one-line swap (the pre-build catch)

The naive read — "import `canonicalJsonSerialize` from `transaction-record.js`, swap it in at `store.js:147`" — is **falsified by the test contract**. `store.test.js` Test 8 (containment, F7) explicitly **forbids** importing `transaction-record`:

```
const forbidden = requires.filter((r) => /agent-identit|spawn-state|record-store|transaction-record|identity\//.test(r));
assert.deepStrictEqual(forbidden, [], 'store.js imports no kernel/identity STATE store (advisory-only)');
```

`transaction-record.js` is blacklisted because it is the kernel's **record-STATE** module (idempotency, INV-22 false-merge defense) — the Lab must touch NO kernel state module (advisory boundary, RFC §2 Layer 3). `canonicalJsonSerialize` happens to *live* there but is a **pure, stateless serializer** — the same KIND of leaf the lab already legally imports (`kernel/_lib/atomic-write`, `kernel/_lib/lock`).

**Resolution: extract the pure serializer to its own leaf** `packages/kernel/_lib/canonical-json.js`, re-export from `transaction-record.js` for back-compat (byte-identical → INV-22 / M1 preserved), and import the leaf into the lab store. Rejected alternatives: (a) relax Test 8's ban — erodes the containment invariant; (b) duplicate canonical-serialize into the lab — DRY violation + **drift risk on a hashing function** (a drifted copy hashes differently → defeats the very cross-node-determinism goal).

## 5. The design

### 5a. Extract — `packages/kernel/_lib/canonical-json.js` (NEW)
- Move `canonicalJsonSerialize` + `MAX_CANONICAL_DEPTH` (100) + `MAX_CANONICAL_NODES` (10000) **verbatim** (carry the hardening rationale comment `transaction-record.js:43-54`).
- Export `{ canonicalJsonSerialize, MAX_CANONICAL_DEPTH, MAX_CANONICAL_NODES }`.
- Pure, stateless, zero deps beyond none (it uses only `JSON.stringify` on scalars + recursion).

### 5b. Re-export — `packages/kernel/_lib/transaction-record.js`
- `const { canonicalJsonSerialize } = require('./canonical-json');` (replace the inline fn + consts).
- Keep `canonicalJsonSerialize` in `module.exports` (back-compat — `record-store.js` is unaffected).
- **The re-export is the SAME function object** (not a second copy that could drift) — `canonical-json.test.js` asserts `txn.canonicalJsonSerialize === <leaf>.canonicalJsonSerialize`, and `transaction-record.test.js` byte-pins the output. (Honesty-auditor note: prefer "same function object → no copy to drift" over the looser word "byte-identical".)

### 5c. Wire — `packages/lab/negative-attestation/store.js`
- `const { canonicalJsonSerialize } = require('../../kernel/_lib/canonical-json');`
- A local helper: `canonicalSigBasis(sig)` → `try { return canonicalJsonSerialize(sig); } catch { return JSON.stringify(sig); }` — **canonical-with-fallback**: canonical (key-stable) on the normal path; falls back to `JSON.stringify` if the (untrusted-ingest) signature trips the depth/node bound, so P6's batch-abort surface stays CLOSED. The fallback is deterministic-per-input (a pathological sig dedups with itself).
- `store.js:147`: `const sigHash = sha256(canonicalSigBasis(o.failureSignature));`
- **Scope discipline:** lines :148-150 (the `[runId, leafRef, sigHash]` array wrap) stay `JSON.stringify` — string arrays in fixed order are already canonical; switching buys nothing + adds throw-surface. Update the `:140-146` comment to note the basis is now canonical.

### 5d. Migration note (surface for VERIFY/honesty-auditor)
The `attestation_id` **basis changes** (canonical sorts keys; `JSON.stringify` preserved insertion order). A pre-change ledger record and its post-change replay compute **different ids** → won't dedup. **Precise blast bound (the falsifiable claim):** ≤ 1 duplicate per *currently-live* pre-change event, self-healing within ≤30 days (expiry), and **no consumer reads `attestation_id`** — `listAttestations`/`pruneExpired` filter by wall-clock only; the id is used solely for the internal dedup-on-write scan (`store.js:171`). **Acceptable:** advisory + expiring; v3.3 records are dogfood artifacts; "a dup is far safer than dropping a real failure" (store.js:24-25). No migration script (a migration touches a live ledger = more risk than the bounded dup it prevents). **Forward-caveat (honesty-auditor):** the bound holds only while `attestation_id` stays read-only-internal — a future v3.4 E4 that KEYS on the id must re-derive it the SAME canonical way (the M1 forward-coupling trap), or it inherits a silent join break.

### 5e. Architect VERIFY — corrections folded (APPROVE)
1. **Throw-policy is principled, not ad hoc — cite the charter inversion.** The kernel fails-CLOSED on a hash failure because dropping/mis-keying a record is a correctness/security event (INV-22 false-merge defense). The Lab is **advisory** and its bias is the inverse — *never drop a witness* (`store.js:24-25`). Fail-closed here would contradict the layer charter; **canonical-with-fallback matches it.** The fallback path is **dead code on the producer path** (the signature is a flat 8-scalar-field object, depth 1 / ~9 nodes — `failure-signature.js:56-65` — it cannot approach the depth/node bounds); it exists to defend the **C1 untrusted-outbox** ingest path (a hand-crafted `decompose-result.json`), which `record-from-decompose.js:70-86` does NOT try/catch.
2. **Assert the fallback is total.** `JSON.stringify` itself throws only on `BigInt` or a circular ref; the signature arrives via `JSON.parse` (ingest) or the frozen producer → neither is possible. Add a one-line code comment asserting the catch can't itself throw (the property that makes the fallback safe).
3. **Carry + breadcrumb.** Move the hardening-rationale comment (`transaction-record.js:43-54`) to the new leaf verbatim; leave a one-line pointer at the old site so the "why the bounds exist" memory doesn't rot.

## 8.5 VALIDATE — 3-lens findings folded

| Lens | Verdict | Outcome |
|---|---|---|
| `code-reviewer` (correctness) | **Approve**, 0 findings | runtime-verified the re-export is the same function object; all 8 `transaction-record` importers unaffected; scope + tests sound. |
| `honesty-auditor` (claim-vs-evidence) | **Grade A / NO-OVERCLAIM** | all 6 claims TRUE; 2 wording notes folded (§5b "same function object"; §5d E4 forward-caveat). |
| `hacker` (adversarial-security) | **HIGH (fixed) + 2 MEDIUM** | the false-assurance regression I introduced — fixed below. |

**The HIGH (fixed):** the original `canonicalSigBasis` fallback `JSON.stringify(sig)` is **itself non-total** — it throws `RangeError` on deep nesting (it recurses, like canonical), *outside* the guard. A hand-crafted deep `failure_signature` via the untrusted outbox → canonical throws (caught) → `JSON.stringify` throws (uncaught) → the un-try/caught ingest loop fails the WHOLE run's ingest, denying every good leaf its witness. **And my original Test 10 (depth 130) certified the DoS as fixed while testing inside the safe zone** (130 < JSON.stringify's ~6100 overflow) — false confidence. *The single-`code-reviewer` + the honesty-auditor both rated the "fallback is total" comment as fine by REASONING; only the hacker's empirical probe found it — the canonical case for why kernel/data-mutation diffs get the full 3-lens tier.*

**Fix (two-part, both folded):**
1. **Ingest shape-guard (the boundary fix)** — `record-from-decompose.js` `isFlatScalarSignature`: reject any `failure_signature` carrying a nested object/array value (a real ADR-0015 block is 8 flat scalars) → `skipped++`, BEFORE the recursive serializers see it. Rejects by value-TYPE → **depth-agnostic + deterministic** (no reliance on actually overflowing; forward-tolerant — a new scalar field still passes).
2. **`canonicalSigBasis` total** — on the canonical bound firing, emit a unique non-content sentinel (total by construction; `randomBytes`+concat can't throw) instead of re-serializing the attacker blob via `JSON.stringify`. Honest comment; the misleading Test 10 replaced by a **deterministic helper-level totality test** (canonical's bounds are call-local counters → stack-independent).

**Deferred (pre-existing — NOT introduced by this diff; tracked for the v3.4 live-wiring + circuit-breaker wave, design-input a):** the verbatim-store byte-bloat (a wide all-scalar signature stored verbatim — needs a signature-size policy that touches the E1 verbatim invariant) + a hardened outbox-parse bound. These are untrusted-LIVE-input hardening — they belong with the wave that makes the input live + untrusted, where the circuit-breaker lands (per the symmetric-guard-activation directive). The tier is INERT today (0 `hooks.json` refs; CLI-only), which correctly caps the HIGH at HIGH-not-CRITICAL.
