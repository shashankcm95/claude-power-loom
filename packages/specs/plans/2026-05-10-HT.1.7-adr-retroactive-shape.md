# HT.1.7 — ADR-0001 retroactive shape + forward-looking ADR-0003 sub-plan

## Context

The substrate's first ADR — `swarm/adrs/0001-substrate-fail-open-hook-discipline.md` — was codified retroactively in H.8.2. The discipline (every hook has a top-level try/catch + observability via `logger('error', ...)` + never block on hook errors) existed across 14 hook scripts before the ADR primitive shipped. ADR-0001 documents the convention rather than introducing it.

The chaos theo F4 finding (post-H.8.x cleanup arc) called this out: ADR-0001 has a structural mismatch with the canonical ADR lifecycle. The canonical lifecycle is `proposed → accepted → superseded | deprecated` — an ADR is drafted, deliberated, accepted, then maintained or replaced. ADR-0001 entered the system at `accepted` because the discipline was already in place; there was no `proposed` phase.

The master plan v3.1 line 344 captures the refactor candidate:

> ADR-0001 retroactive shape (chaos theo F4 — should be reshaped as `status: seed` and a forward-looking ADR-0002 created — refactor candidate; document state).

The HT.0.6 audit (E.6 finding lines 193-205 + 239) re-documented this and offered three options:

- **(a)** leave as-is — author-field text ("codified retrospectively in H.8.2") + status-note text already disclose retroactive shape; schema unchanged
- **(b)** add `seed` enum value to lifecycle and re-tag ADR-0001 as `seed`
- **(c)** author forward-looking ADR-0002 with proper `proposed → accepted` lifecycle (preserving ADR-0001 as-is)

The HT.0.9 synthesis selected **(b) + (c) combined** as the canonical resolution — the schema-level reshape both preserves ADR-0001 as historical institutional ledger AND establishes a forward-looking institutional commitment in a fresh ADR. HT.0.9-verify architect FLAG-1 committed-path resolution removed the original "OR ADR-0007 supersedes ADR-0001" branch — ADR-0001 retags in place via the `seed` enum (no formal supersession; supersession would imply replacement, which mis-frames the relationship).

Note on numbering: the master plan + HT.1 backlog say "forward-looking ADR-0002" — but ADR-0002 was claimed by HT.1.3 for the bridge-script entrypoint criterion. The forward-looking fail-open discipline ADR will therefore be **ADR-0003**.

**Why (b) + (c) and not (a)** (leave as-is): the author-field text is human-readable disclosure; the schema is what tooling consumes. `adr.js` `isActive()` returns true for `status === 'accepted'`; `validate-adr-drift.js` filters by `isActive()`. The retroactive shape is invisible to the substrate's machine-readable surface. Adding `seed` to the schema makes the retroactive disclosure machine-readable + institutional.

**Why this is the right shape now** (HT.1.7 timing): the ADR system is small (1 active ADR pre-HT.1.3; 2 active post-HT.1.3 with ADR-0002; ADR-0003 will be the third). The schema extension is cheap and backward-compatible; the cost grows with each ADR added. Doing the schema reshape early — before the ADR system has a dozen entries — is sequencing-correct.

## Design choice — `seed` as active-for-drift (Design B)

The semantically load-bearing design choice in HT.1.7: when `isActive()` is widened to admit `seed` ADRs, does the drift detection hook emit `[ADR-DRIFT-CHECK]` for `seed` ADRs?

**Decision: YES — `seed` ADRs participate in drift detection** (`isActive()` returns true for both `accepted` and `seed`, gated by empty `superseded_by`).

Two designs were considered:

- **Design A**: `seed` = NOT active-for-drift. `isActive()` returns true only for `accepted`. ADR-0001 becomes a museum piece — readable via `adr.js read 0001` but invisible to drift detection. ADR-0003 takes over the live rule for the same 14 files. Cleaner drift output (one ADR per file edit), but ADR-0001's discipline is no longer surfaced when its `files_affected` are touched.

- **Design B (chosen)**: `seed` = active-for-drift. `isActive()` returns true for `accepted` OR `seed` (with empty `superseded_by`). Both ADR-0001 (seed; retroactive context) and ADR-0003 (accepted; forward-looking commitment) surface in drift detection when their shared `files_affected` are touched. Two-ADR output for the affected files, but every ADR continues to carry its institutional weight at the file boundary.

**Why Design B**: the "preserve ADR-0001 as historical institutional ledger" framing in the HT.1 backlog (line 169) implies ledger continues to be reachable in normal substrate operation — including drift detection, which is the substrate's primary surface for ADR awareness. If ADR-0001 silently drops from drift output post-retag, future readers editing a hook script won't see the historical context that explains why fail-open is the rule. Design B keeps both the historical evidence (ADR-0001) and the forward-looking commitment (ADR-0003) live at the edit boundary.

**Cost of Design B**: drift detection emits two ADRs per affected-file edit (ADR-0001 + ADR-0003) instead of one. The two ADRs' invariants overlap; reader has to understand they encode the same discipline. Mitigated by: (a) ADR-0001 status-note explicitly links to ADR-0003 ("forward-looking institutional commitment captured in ADR-0003"); (b) ADR-0003 `related_adrs: [0001]` cross-references the historical ledger; (c) the drift-check forcing instruction surfaces ADR titles, which differ ("hooks fail open with observability" vs "fail-open hook discipline (forward-looking)") signaling complementary roles to the reader.

**Alternative considered (Design C)**: `seed` is its own non-overlapping category — like documentation-tier ADR. Rejected because the substrate already has pattern docs (`kb/architecture/`) for documentation-tier knowledge. ADR-0001 is decision-tier, not documentation-tier — adding a third tier would further fragment the ADR system.

## ADR-0003 shape — governance-tier institutional commitment (FLAG-3 absorption: load-bearing reframe)

- **ADR-0001 (seed; technical-tier)**: codifies the 4 mechanical invariants of the fail-open discipline. WHAT the discipline is.
- **ADR-0003 (accepted; governance-tier)**: codifies the institutional commitment + code-review gate that enforces the discipline on all future hooks. HOW the discipline is maintained as the substrate grows.

This relationship — technical ADR + governance ADR for the same domain — is genuinely ADR-worthy and avoids the "two ADRs saying the same thing" critique that motivated the FLAG. ADR-0003's `invariants_introduced` contains ONLY governance-novel content; ADR-0001's mechanical invariants are referenced by ID in prose.

**Sections** (architect FLAG-3 reframe applied):

- **Context** (lead with the technical-vs-governance distinction): ADR-0001 codifies the technical discipline (4 mechanical invariants for fail-open hook authoring) — codified retroactively in H.8.2 from existing convention across 14 hook scripts. ADR-0003 codifies the **governance** layer above the technical layer: the institutional commitment to enforce ADR-0001's discipline on all future hooks via code review. The two ADRs address distinct concerns: ADR-0001 = WHAT the discipline is; ADR-0003 = HOW the discipline is maintained as the substrate grows. Without the governance layer, ADR-0001 is a snapshot — it documents what existed at H.8.2 but does not commit the substrate to maintaining the discipline as new hooks are added. Subsequent phases (H.8.3 through HT.1.6) added 6+ new hooks all of which followed the discipline because authors knew about ADR-0001 and its invariants. That implicit institutional commitment is what ADR-0003 makes explicit + load-bearing.
- **Decision**: Substrate hooks shipped after 2026-05-10 (HT.1.7 onward) MUST satisfy ADR-0001's four mechanical fail-open invariants. Compliance verification is institutional: (1) hook author includes verification of each invariant in the PR body or commit message; (2) code reviewer applies the four invariants as a load-bearing review gate; (3) PRs that don't verify all four invariants are NEEDS-REVISION regardless of other merit. The substrate's governance commitment is to enforce the technical discipline — not to weaken it via case-by-case exception.
- **Consequences**: positive (substrate stability primitive remains load-bearing as the hook count grows; new hook authors have a canonical reference for the four mechanical invariants — namely ADR-0001; code reviewers have a load-bearing gate that survives reviewer turnover; the substrate's institutional discipline is machine-readable via this ADR's invariants); negative (institutional commitment creates onboarding cost for new contributors; the discipline is one more rule among many — Convention G's cap-rule headroom must be considered when adding more institutional gates; PR review tax is non-zero for hook authors).
- **Alternatives Considered**:
  - **A) Leave ADR-0001 as the sole reference**: rejected — ADR-0001 documents the discipline's technical shape but does not commit the substrate to enforcing it on future hooks; the implicit commitment was institutionally fragile as the hook count grew.
  - **B) Merge into ADR-0001 with a status-note commitment**: rejected — would mix retroactive technical evidence with forward-looking governance commitment in one document; the distinct scopes (WHAT vs HOW) deserve distinct ADRs for clarity at drift-detection time + during PR review.
  - **C) Author as pattern doc instead of ADR**: rejected — pattern docs describe canonical concepts (e.g., "fail-fast"); ADR-0003 captures a substrate-specific governance commitment, not a canonical concept.
  - **D) Do nothing**: rejected — the gap between "retroactively codified" (ADR-0001) and "forward-looking governance commitment" (ADR-0003) is real; closing it strengthens institutional discipline as the substrate grows past the original 14 hooks.

## Why per-phase pre-approval is INVOKED for HT.1.7

Three triggers (any one is sufficient; all three apply):

1. **Substrate-fundament-touching schema change**: `isActive()` widening changes the semantic of "active ADR" — propagates to `validate-adr-drift.js` (drift hook) + `cmdActive()` + `cmdTouchedBy()`. Schema-level changes deserve design-time review.
