# v3.8a W3 — K4 live-recall: the recall-layer retrieval-suppression view

## Context

The v3.5 manage layer RECORDS destructive intent (quarantine/cull/dedup/merge proposals); v3.6
ENFORCES approved intent (committed kernel `TOMBSTONE`/`SUPERSEDE` mints). But NOTHING on the
read side honors any of it: record-store reads do not filter by `operation_class` (probed —
zero hits in record-store.js), and the only read consumer is the per-txid `lifecycle` CLI
verdict. The loop is dark at its read edge. The v3.5 phase-close named this wave verbatim:
"**un-darkening the loop into live K4 recall**" (ROADMAP:242), and promote.js:52 parked the
fourth op for it: "**`quarantine` is a recall-layer retrieval-suppression (v3.8a), NOT a
kernel op**". The charter (combined-roadmap:162-167) inherits "K4 live-recall" as v3.8a spine,
characterized as **narrowing-safe-mechanical** (explicitly NOT the causal-edge walker — edge
traversal in production reads stays PROHIBITED until an R3-honoring walker lands, RFC v3.5:396).

The thesis: a **set-level recall-suppression VIEW** — given a candidate set of kernel
transaction_ids (a recall-class result set), partition into `surfaced` / `suppressed` /
`flagged` per the LIVE manage state. Narrowing-only (it can only remove/annotate, never add),
advisory (§0a.3.1 — the consumer decides; the view never gates a kernel read).

## Design questions for the architect (D1-D5)

- **D1 — scope**: confirm the thesis (set-level suppression view over the KERNEL-RECORD
  recall class; loom-recall.js untouched per the K12 + target-schema probes) vs alternatives
  (extend loom-recall's corpus to records [not mechanical]; suppress inside record-store reads
  [REJECT: an advisory Lab store gating a kernel read violates §0a.3.1 + K12]).
- **D2 — suppression semantics (the tier question)**: which states land where?
  Proposed: `suppressed` = kernel-COMMITTED facts only (tombstoned/superseded — the v3.6
  mints); `flagged` = ADVISORY intent (approved-but-unpromoted destructive ops + quarantine
  proposals — any disposition? approved-only?); `surfaced` = the rest (active/stale/archived/
  unknown annotated). The view stays a partition + per-txid reasons; the CONSUMER decides
  (advisory, never a gate).
- **D3 — placement + name**: a `manage-proposal` sibling (`recall-suppression.js`) vs a new
  `packages/lab/recall/` package. Lab imports kernel projections (legal) + sibling
  projections. CLI surface: a new `recall-filter --txids a,b,c` subcommand on the manage cli
  vs its own cli.
- **D4 — SP-1a (persona-scoped recall)**: in or out of this wave? Probed reality: nothing
  persona-keys recall today; the suppression view is persona-agnostic. Expectation: OUT
  (defer; keep the wave lean per the kickoff's candidate discipline).
- **D5 — the lossy-recall caution** (the v3.5 carry item (c): "live K4 recall is
  high-precision/lossy-recall until the OQ-21 calibration"): does the view need a
  `--show-suppressed` escape hatch so a consumer can always see the full partition
  (anti-silent-memory-loss)? Expectation: yes — suppression that cannot be inspected is the
  failure mode OQ-21 calibrates against.
