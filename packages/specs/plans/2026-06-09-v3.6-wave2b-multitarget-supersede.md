# v3.6 Wave 2b.1 — multi-target single-run SUPERSEDE/cull (generalize the W2a mint)

## Context

W2a (#270) shipped the manage-promote MINT in its narrowest real form: a human promotes ONE approved `cull`
proposal with ONE target -> a COMMITTED `TOMBSTONE` -> the W1 reader reports the target `tombstoned`. W2a hard-
REFUSED everything wider with explicit `*-deferred-w2b` codes. W2b.1 lifts those refusals into real handling:

- **multi-target** (`target_records.length > 1`) -> ONE op naming all targets in `affected_records`;
- the **`SUPERSEDE` ops** (`content-dedup` / `merge` op_types) on top of `cull` -> `TOMBSTONE`;
- the post-condition generalized from a single-element check to **exact-SET-equality** (the W2a NEW-1 CRITICAL,
  promoted to a verify discipline);
- **per-target** IDOR eligibility (every target must pass, not just one).

It stays opt-in (`LOOM_MANAGE_ENFORCE=1`) + shadow-default + human-gated, under the same cooperative / trusted-
local-fs / human-in-loop threat model as W2a. **Cross-run** multi-target and the **E11 promote-path breaker**
are deferred (see Out of Scope) — both need distinct design, not a mechanical lift.

## Scope

### In scope (W2b.1)
1. **`OP_MAP` extension** — add `'content-dedup': 'SUPERSEDE'`, `'merge': 'SUPERSEDE'` (alongside `cull: 'TOMBSTONE'`).
2. **Canonicalize-at-the-boundary (hacker HIGH)** — the orchestrator computes `wantTargets = canonicalizeTargets(proposal.target_records)` ONCE (dedup+sort, the store's exported helper) and uses that single array for the mint input AND the post-condition `want`. Closes the self-DoS where a hand-planted *authentic* but non-canonical row (`[t2,t1,t1]`) makes a correct approval un-promotable. Empty canonical set -> `refuse('no-targets')`.
3. **Blast-radius re-cap (hacker MEDIUM)** — `if (wantTargets.length > MAX_TARGETS) refuse('too-many-targets', {count})` at the promote boundary (the store's `MAX_TARGETS` is create-time only; a planted authentic row bypasses it). This is what actually bounds the per-op blast radius.
4. **Multi-target single-run** — replace the `length !== 1` refusal with `resolveSingleRun(wantTargets, opts)`: call `findRecordRun` on EVERY target, collect distinct runIds, refuse on any phantom/ambiguous, refuse `cross-run-deferred-w2c` if >1 distinct run, else the unanimous `runId`. Build ONE op naming all targets in `affected_records`.
5. **Exact-SET-equality post-condition** — the stored op's `affected_records` must equal `new Set(wantTargets)` EXACTLY (same cardinality, no dupes, bidirectional containment). Generalizes the W2a single-element check; defends the same INV-22 poison-key (superset/subset/dup-pad/foreign decoys all fail).
6. **Per-target eligibility (AFTER run-resolution)** — `eligibilityRefusal(readById(t, {runId}))` runs on EVERY target (the resolved single run); ANY kernel-owned / manage-op target refuses the WHOLE op (no partial mint).

### Out of scope (deferred — honest discipline)
- **The E11 promote-path breaker -> W2b.2.** A separable rate-bound (new promote-event source over the E11 pattern). W2b.1's per-invocation explicit `--proposal-id` + INV-22 dedup + `MAX_TARGETS=256` is the same rate-bound W2a shipped.
- **True cross-run (multi-run) mints -> W2c (or fold later).** Needs per-`(proposal,run)` idempotency + a reader-side cross-run load convention + cross-run atomicity (partial-failure). Distinct design + threat surface.
- **The `merge` survivor-CREATE** (the synthesized record the merged-away targets fold into) — the v3.5 carry-(b) structured-summary slot; rides `justification` today. W2b.1 mints only the destructive SUPERSEDE half (marking the merged-away targets superseded), which is meaningful standalone (for `content-dedup` the kept canonical is simply not in `target_records`).

## Key design decisions

- **D1 — single-run is a probed constraint, not a shortcut.** The reader is structurally fed one run's records (probe rows 6-7). Cross-run isn't "harder code" — it's a different identity model (per-`(proposal,run)` keys) + a reader change. Fail-closed refuse is the cumulative-coherent boundary.
- **D2 — exact-SET-equality over the CANONICALIZED APPROVED set, NOT subset `.includes` and NOT the raw stored array.** Promoted from the W2a NEW-1 CRITICAL. The `want` source-of-truth is the **approved intent**, canonicalized at the boundary: `wantTargets = canonicalizeTargets(proposal.target_records)` (NOT the per-target `findRecordRun`-resolved set — that's a lookup artifact that could legitimately differ in a future cross-run world; the post-condition's job is "did we mint EXACTLY what was approved"). The SAME `wantTargets` array is passed to `buildManageOpRecord` so build-input and `want` cannot drift. FAIL unless `got.length === wantSet.size && new Set(got).size === wantSet.size && got.every(t => wantSet.has(t))` — rejects superset/subset/dup-pad/foreign; accepts reorder (a SET is order-free). **Canonicalizing `want` (not trusting `proposal.target_records` as-stored) is load-bearing**: a planted authentic non-canonical row would otherwise self-DoS a correct approval (hacker HIGH).
- **D3 — `content-dedup` and `merge` are the SAME kernel op (`SUPERSEDE`).** The semantic difference is at the proposal/justification layer; the kernel records a SUPERSEDE naming the superseded targets. No new operation_class (the v6 enum is LOCKED). **Acknowledged sacrifice:** a SUPERSEDE-without-a-survivor-CREATE is an intentional PARTIAL mint of the `merge` semantics (the synthesized survivor record is the deferred carry-(b)); the merged-away targets reading `superseded` is meaningful + coherent standalone (for `content-dedup` the kept canonical is simply not in `target_records`).
- **D4 — per-target eligibility is all-or-nothing, and runs AFTER run-resolution.** A multi-target op is atomic; one ineligible target poisons the whole set (you cannot half-tombstone). `eligibilityRefusal` needs the resolved record (`readById(t,{runId})`), so it is downstream of D5's unanimous-`runId`. Refuse on the first ineligible target, before the single `appendRecord`. (No partial-mint window: it is ONE atomic `appendRecord` naming all N, not a per-target loop of appends.)
- **D5 — the run resolution is per-target then unanimity (`resolveSingleRun(wantTargets, opts)`).** For EVERY target: `findRecordRun(t)` -> refuse `target-not-found` on `null`, refuse on `ambiguous`; collect the distinct `runId`s; refuse `cross-run-deferred-w2c` if >1 distinct; else return the unanimous `runId`. **Must resolve each target's OWN run** (architect CRITICAL): doing `readById(t,{runId_0})` for a target in another run returns `null` -> the WRONG refusal code (`target-not-found` not `cross-run-deferred-w2c`) -> EC3 fails.
- **D6 — the blast-radius cap is re-asserted at the promote boundary.** `MAX_TARGETS=256` is enforced only at `createProposal` time; a hand-planted authentic row bypasses it and `buildManageOpRecord` has no cap (hacker MEDIUM). The orchestrator re-checks `wantTargets.length > MAX_TARGETS` post-canonicalization. The cap stays a single Lab-policy constant (NOT pushed into the kernel builder, which is policy-free — KISS/layer-discipline).

## Threat model / security

- **Same threat surface as W2a** (cooperative / trusted-local-fs / human-in-loop). Multi-target does NOT widen the *rate* surface: still one explicit human-approved proposal per invocation; the canonical `proposal_id` binds the sorted-unique target SET (a swap of any target changes the id -> named lookup fails); INV-22 dedup makes re-promote a no-op.
- **Blast-radius honesty (hacker MEDIUM, corrected from the W2a assumption):** multi-target DOES widen the per-op blast radius vs W2a's structural single-target. The bound is NOT the create-time `MAX_TARGETS` (a planted authentic row bypasses it) — it is the **D6 boundary re-cap** + the D2 boundary canonicalize. With those, the per-op radius is bounded at `MAX_TARGETS=256` distinct records.
- **Human-scope honesty (hacker LOW):** a single human `approve` of one `proposal_id` now authorizes a SET of up-to-`MAX_TARGETS` destructive marks (vs one in W2a). The human-review scope-gate does more work; the per-target eligibility gate is NOT bypassable (refuses the whole op on any ineligible target) but the documented W2a residual — an ordinary non-kernel record is targetable, the human is the scope gate — now applies set-wide. Accepted under the cooperative model; closes at the sandbox.
- **The poison-key fail-open** (W2a CRITICAL) is the load-bearing defense and is GENERALIZED here: exact-SET-equality over the CANONICAL approved set is strictly stronger than the W2a single-element check, and the W2a NEW-1 superset-decoy regression test is subsumed.
- **Residuals carried (unchanged, documented):** (a) a hostile same-uid persona automating approve+promote (the OQ-E NO-GO; closes only at the Track-2 sandbox); (b) the approve->execute TOCTOU swap window (mitigated by the canonical-`proposal_id` content-address; fully closed at the sandbox). Neither is widened by multi-target.

## Exit criteria

- **EC1** — a multi-target `merge`/`content-dedup` proposal (2+ targets, same run) promotes to ONE COMMITTED SUPERSEDE; the W1 reader reports each target `superseded`. A multi-target `cull` -> each target `tombstoned`.
- **EC2** — the exact-SET post-condition FAILS a superset decoy, a subset decoy, and a dup-pad decoy (regression tests green); a planted non-canonical authentic row promotes correctly (canonicalize self-DoS closed); a planted over-cap row refuses `too-many-targets`.
- **EC3** — a partially-ineligible target set (one kernel-owned, SAME run) refuses with no mint; a cross-run set refuses `cross-run-deferred-w2c` (not `target-not-found`).
- **EC4** — shadow-default unchanged; INV-22 re-promote dedups; the full gate green (install.sh --hooks --test 118/0, kernel suite, signpost --check, contracts-validate).
- **EC5** — 3-lens VALIDATE clean (hacker re-probe of the BUILT code per Rule 2a); CodeRabbit folded; PR opened for the USER merge gate.
