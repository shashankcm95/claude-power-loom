## Goal

Lift the `cross-run-deferred-w2c` refusal (`promote.js:111`): allow a human-approved manage-proposal whose
`target_records` span **more than one run** to promote — minting the COMMITTED destructive op (SUPERSEDE/TOMBSTONE)
into **each target's own run**, so the W1 reader (`manageLifecycleStatus`) reports every target's lifecycle correctly
regardless of which run it lives in.

This is a real (not speculative) capability gap: `runId = sha256(session_id)[:16]` **rotates at compaction**
(harness-runid-session-rotation), so records a human curates across sessions WILL span runs. A multi-session cull
hits the current refusal today.

**NOT in this wave** (the "then close" half — separate, after this merges): EC1 explicit re-verify adjudication,
EC3 residual-doc + R-abort verification, EC4 `/phase-close`. This wave is the cross-run BUILD only.

## Design

### D1 — mint topology: PER-RUN PARTITION (one op per run, each naming that run's subset)

Partition `wantTargets` by their resolved run; for each run mint ONE COMMITTED op whose `affected_records` is
**exactly that run's subset**, appended INTO that run. Rationale (preserves the reader's locality invariant): a
tombstone of run-X's target must be co-located in run X, because the reader matches a target by scanning a record
set and finding an op whose `affected_records` names it (`provenance-projections.js:54-66`). The rejected
alternative — ONE op in a single "home" run naming foreign-run targets — breaks that locality (the reader scanning
run Y for target t never finds an op that lives in run X) and would force a cross-run MERGE into every read. Per-run
partition keeps **reads simple** (a single target's op lives in the target's own run) and pushes all cross-run
complexity to MINT time.

### D2 — identity: per-(proposal, run) idempotency via runId-seeded writer_spawn_id (UNIFORM)

Fold the resolved `runId` into the spawn id **uniformly** (single-run is just the one-run special case):

```
writer_spawn_id = `manage-promote:${proposalId}:${runId}`   // was `manage-promote:${proposalId}`
```

**M3 (hacker VERIFY) — assert `proposalId` is 64-hex in `buildManageOpRecord`.** The colon-join is
unambiguous ONLY because `proposalId` is a fixed-width 64-hex sha256 (no colon) — so the tail after
`manage-promote:<64hex>:` is unambiguously the `runId` (even though `runId` may itself contain a `:`, which
`isSafePathSegment` permits). A future variable-width/colon-bearing `proposalId` would reopen a
`(p1,"aa:bb")` vs `(p1:aa,"bb")` collision. Make the invariant ENFORCED: add `if (!isHex64(proposalId)) throw`
to the builder (matches the existing `approvalAxiomHash` 64-hex assertion at `manage-op-record.js:69`). KISS:
an enforced invariant, not a hash-of-the-join.

Each run's mint -> distinct `writer_spawn_id` -> distinct `content_hash` -> distinct `idempotency_key` -> NO cross-run
INV-22 collision (every run's record lands). Re-promoting the same proposal -> same `(proposalId, runId)` per run ->
same key -> INV-22 idempotent no-op per run (the recovery property; see D4). `runId` stays **internally derived** (from
the resolved run), not a free-form caller param — preserving the architect MED-1 "no drift from affected_records"
property. `buildManageOpRecord` gains a required `runId` param.

**Identity churn note (architect to confirm):** this changes the single-run mint's `writer_spawn_id`/`content_hash`/
`idempotency_key`/`transaction_id` vs W2a/W2b.1. Acceptable because manage-records are advisory SHADOW (opt-in
enforcement; no production data to migrate) and existing tests assert *behavior* (dedup / post-condition / refusal
codes), not specific hash literals — re-verified RED-first in TDD. The uniform form is DRY (one identity model, not a
single-vs-multi branch).

### D3 — flow: `resolveTargetRuns` (replaces `resolveSingleRun`) + per-run mint loop

`promote.js` steps revised:

1-3. (unchanged) flag gate; load + content-authenticate proposal (`listProposals` re-derives `proposal_id`);
   op_type->operation_class (hasOwnProperty-guarded); canonicalize + MAX_TARGETS re-cap.
4. **`resolveTargetRuns(wantTargets, {stateDir})`** — for each target `findRecordRun(t)`: `null` -> refuse
   `target-not-found` (fail-closed); `{ambiguous,runs}` -> refuse `target-in-multiple-runs-w2b` (a SINGLE target id
   duplicated across runs is under-determined; KEEP refusing — orthogonal to cross-run-DIFFERENT-targets; D7); else
   add `runId -> push(target)` into an ordered map. Returns `Map<runId, target[]>` (>1 entry is now LEGAL — the
   lifted refusal). Deterministic: iterate runIds **sorted**.
5. **Per-target IDOR eligibility** — for every target, `readById(t, {runId: itsResolvedRun, stateDir})` (each target's
   OWN run, from the step-4 map — the W2a CRITICAL: never read against a guessed run) -> `eligibilityRefusal`. Any
   kernel-owned / manage-op target refuses the WHOLE op (all-or-nothing, BEFORE any mint).
5.5 **Breaker** — evaluate ONCE up front (D5).
6. **Mint loop (sorted runIds):** for each `(runId, subset)`: build the manage-op (affected_records = subset;
   writer_spawn_id seeds runId) -> `appendRecord(record,{runId,stateDir})` -> re-read + `postConditionOk(stored,
   opClass, new Set(subset))`. Accumulate `{runId, transaction_id, targets: subset, deduped}`. On the FIRST per-run
   failure: STOP and return the honest partial result (D4).
7. Success: `{ ok:true, operation_class, targets, mints:[{runId, transaction_id, targets, deduped}] }` — the
   top-level operation_class/targets are PROMOTION-level; the per-run facts are in mints[] (per-mint has no
   operation_class — code-reviewer LOW).

### D4 — cross-run atomicity: pre-flight-maximal, mint-per-run, report partials, idempotent retry (NO rollback)

A cross-run promotion is **not atomic** across runs (no distributed transaction over per-run files). Posture:

- **Pre-flight maximally**: ALL eligibility (step 5, every target across every run) + the breaker (step 5.5) happen
  BEFORE any mint, so the common failures (ineligible target, breaker open, tamper signal) abort with ZERO mints.
- **Mint per-run in sorted order**; on a mid-sequence failure (append error / post-condition mismatch), STOP and
  return `{ ok:false, failed:'partial-cross-run', minted:[{runId,transaction_id,targets}], unminted:[runId...],
  cause:<the failing refusal/failed code>, ... }` — the EXACT partial state, never hidden.
- **No rollback.** Un-committing a destructive op is itself a destructive op (§0a.3.1: a halt narrows, it does not
  amplify; we never mint a tombstone-of-a-tombstone). Irreversibility is by-design. Recovery = **re-invoke**: the
  per-(proposal,run) key (D2) makes already-minted runs INV-22 no-ops and retries only the unminted runs. The human
  (the trust anchor) sees the partial result and re-runs.
- This partial-failure window is a NEW EC3 residual (documented in the close wave): cross-run is best-effort +
  idempotent-recoverable, fully atomic only at the sandbox/Track-2 boundary.

### D5 — breaker semantics: evaluate ONCE up front (one promotion = one destructive intent) [ARCHITECT-ADJUDICATE]

A cross-run promotion mints K records (K = #runs). Two options:

- **(a) once up front** (proposed): one human-approved promotion is ONE destructive intent that happens to touch K
  runs; evaluate the breaker once (as today, `promote.js:209`). A single promotion can therefore push the window
  count up to K past the threshold — but K <= #targets <= MAX_TARGETS (256), so the overshoot is BOUNDED, and the
  NEXT promotion's scan sees all K, so the long-run RATE stays bounded. Simplest; matches "blast-radius is already
  MAX_TARGETS-capped; the breaker bounds the rate of distinct destructive ACTIONS."
- **(b) per-run re-evaluate**: strict per-record rate bound, but a mid-sequence trip GUARANTEES a partial promotion
  (D4) on a path the human didn't cause — turning a rate cap into a partial-failure generator.

**RESOLVED (both lenses): option (a) — but PREDICTIVE.** Keep ONE up-front evaluation (per-run re-eval
manufactures partials — strictly worse), BUT make it account for the K records this promotion is about to
mint. After `evaluate`, in addition to the existing `breaker.tripped -> breaker-open`, refuse
**`breaker-would-exceed`** with **ZERO mints** if `breaker.denials_in_window + K > breaker.threshold`
(K = #runs = the partition-map size, known at step 4; pin the exact `>=`/`>` boundary RED-first to match
`projectBreaker`'s own trip condition). The hacker DEMONSTRATED (probe-w2c-2.js) the non-predictive check
mints 20 records past a cap of 10 in one promotion — the predictive check closes the within-promotion blast
window while staying a single source-agnostic evaluation (promote.js does the arithmetic from the breaker's
already-surfaced `denials_in_window` + `threshold` — NO breaker-API change, NO identity coupling; architect
SRP guard honored). A legitimately large cross-run cull (K > threshold) now refuses cleanly with a legible
`{threshold, K, denials_in_window}` message → the human batches, raises `LOOM_CIRCUIT_BREAKER_*`, or sets
`LOOM_DISABLE_CIRCUIT_BREAKER` — a conscious override, not a silent partial.

### D6 — reader-side cross-run load convention: `loadRecordsForTarget(txid, opts)` helper + CLI fix

With D1 (per-run partition), a single target's op lives in the target's OWN run, so the reader needs only that run's
records — but the CONSUMER doesn't know which run a txid is in (the CLI passes none today -> `unknown`). Add a small
read-only helper (lab-side; reuses `findRecordRun` + `listByRun`):

```
loadRecordsForTarget(txid, {stateDir}):
  if (!HEX64.test(txid)) return [];                 // F8: hex-gate at the helper's OWN boundary (defense-in-depth)
  loc = findRecordRun(txid, {stateDir})
  if (!loc) return [];                              // absent -> [] (reader -> 'unknown', the safe narrowing default)
  if (loc.ambiguous) return loc.runs.flatMap(r => listByRun({runId:r, stateDir}));  // M2: UNION across the dup runs
  return listByRun({runId: loc.runId, stateDir});   // the normal locate-then-load (content-verified)
```

Wire it into the CLI's **`lifecycle`** command (NOT `status` — there is no `status` command; architect F7) at
`cli.js:125`, so `manageLifecycleStatus(txid, {records: loadRecordsForTarget(txid, {}), proposals})` reports the
real `kernel_state` instead of `unknown`. This is the "reader-side cross-run load convention" — a
**locate-then-load** per-target assembly. **M2 (hacker):** on a cross-run-duplicated (`ambiguous`) txid the helper
UNIONS `listByRun` over all the dup runs (more honest than collapsing to `unknown` for a target that IS tombstoned
in one of them) — safe because the reader is advisory / narrowing-safe (it reports more facts, never fewer gates).
Routes through `loadRecordFile`'s #273 content-verify (via `listByRun`). Read-only.

## Threat surface (hacker VERIFY pre-build + VALIDATE re-probe of BUILT code)

- **Run-resolution TOCTOU**: a target's run changes between `resolveTargetRuns` (step 4) and the eligibility read
  (step 5) or the mint (step 6). Cooperative model accepts; the map must be resolved ONCE and reused (no drift —
  W2a CRITICAL). Re-resolving per step would risk a target landing in a different run than it was eligibility-checked
  in.
- **Cross-run leak**: a run's op must name ONLY that run's subset; naming a foreign-run target puts the op where the
  reader won't find it (silent under-report) — closed by the per-run exact-SET post-condition (step 6) on that run's
  subset.
- **Partition poisoning**: a planted/duplicated target steering the partition (e.g. forcing a target into an
  attacker-controlled run). Eligibility + content-addressed `findRecordRun` (readById content-verifies, #273) gate it.
- **Breaker overshoot** (D5): the bounded K-record overshoot under option (a).
- **Idempotent-retry correctness**: a re-invoke after a partial must not double-mint or mis-report (per-(proposal,run)
  key dedups minted runs; the post-condition re-verifies each).
- **`loadRecordsForTarget`**: must reuse the enumeration gate triple (no path-traversal via a hostile txid -> HEX64
  guard; `findRecordRun` already gates run basenames).

### Documented EC3 residuals (NOT code — folded into the close wave's residual doc)

- **R-W2c-1 (F3, architect)** — a partial cross-run promotion's already-minted records COUNT toward the breaker
  window on a retry within the window; if the partial pushed the count to threshold, the idempotent retry is
  `breaker-open`-refused until the window (default 10 min) elapses, then completes. The human sees `breaker-open`
  and waits / overrides. Accepted under the cooperative + human-in-loop model. NOT fixed by coupling the breaker to
  manage-op identity (architect SRP guard: the breaker stays source-agnostic).
- **R-W2c-2 (M1, hacker)** — re-invoke recovery re-runs `resolveTargetRuns` from scratch; under concurrent same-uid
  store mutation a target can resolve to a DIFFERENT run on retry -> a second (benign, narrowing) tombstone instead
  of an INV-22 no-op. Accepted (cooperative model assumes no concurrent mutation between a partial and its retry); a
  resume-from-recorded-map API is YAGNI here, closes fully at the sandbox/Track-2.
- **R-W2c-3 (H1 bound honesty)** — even with the predictive check, a single promotion can mint up to
  `min(#runs, threshold - denials_in_window)` irreversible ops at once; MAX_TARGETS(256) is the hard per-promotion
  blast-radius cap; the human approval + the predictive breaker are the rate bound. Stated honestly.
