## Context

**What R9 is** (scope:70, OQ-11 RATIFIED FULL): the **leaf-criteria validators** — the deterministic
checks that decide whether a decomposition leaf is well-formed enough to be spawned/verified. Per OQ-11
(USER-ratified), **FULL-structure + KISS-bodies**: 5 registered, individually-tested validators reusing
the Wave-1 `validators={}` registry idiom, minimal bodies (2 are one-liners). Budget **~450–600 LoC**.

**The normative 5** (scope:126-136 — promoted to canonical there; "5 deterministic" is honestly
**4 deterministic + 1 advisory**):

| # | Criterion | Operational check | Shape |
|---|---|---|---|
| 1 | `cost-justified` | `estimated_tokens ≥ COST_MIN_TOKENS OR wallclock ≥ COST_MIN_WALLCLOCK_S` (named tunables) | one-liner |
| 2 | `semantically-cohesive` | one clear purpose; a cohesion proxy — **ADVISORY, never a hard gate** | configured (soft) |
| 3 | `interface-clean` | focused input + structured output (shape check) | natural validator |
| 4 | `validation-supported` | a deterministic gate exists for the leaf's discipline (**R12 registry lookup**) | natural + `R9→R12` |
| 5 | `resource-bounded` | `estimated_tokens ≤ RESOURCE_MAX_TOKENS AND capability forbids sub-spawning` | one-liner |

**Why it's architect-led**: R9 must (a) **define a NEW extended leaf schema** (the current leaf is only
`{id, content, status, discipline?}` — it carries none of the fields these criteria read), (b) settle the
**advisory-vs-hard** mechanism, (c) own the **`R9→R12` + `R9→R8`** cross-component contracts. Route-decide
returned `borderline` (0.313, matched `schema`) — it escalated on its own, agreeing with the architect path.

## Design (PROPOSED — the architect VERIFY settles the open questions)

**Module**: `packages/runtime/orchestration/leaf-criteria.js` (alongside trampoline/todo-checkpoint). Tests:
`tests/unit/runtime/contracts/leaf-criteria.test.js` (matches the Wave-1 test home; CI-covered).

**Extended leaf schema** (what the criteria read — a superset of the R7 leaf):

```
{ id, content, status, discipline,          // R7/R8 (discipline now REQUIRED for an R9-validated leaf)
  estimated_tokens?: number,                 // #1, #5
  estimated_wallclock_s?: number,            // #1 (alt)
  tags?: string[],                           // #2 (cohesion proxy)
  inputs?: string[], output_schema?: object|string,  // #3 (focused in + structured out)
  verification?: { runner: string, testFile?: string },  // #4 (tdd → R12 ctx)
  allows_subspawn?: boolean }                // #5 (capability: forbids sub-spawn)
```

**The 5 validators** (`LEAF_CRITERIA = {}`, each `(leaf, ctx) → violations[]`, violation `{criterion, kind, severity, message, ...}`):

- **#1 `cost-justified`** (one-liner): pass iff `estimated_tokens ≥ COST_MIN_TOKENS` OR `estimated_wallclock_s ≥ COST_MIN_WALLCLOCK_S`. Else `{severity:'error', kind:'cost-unjustified'}`. Cutoffs = named tunables.
- **#2 `semantically-cohesive`** (ADVISORY): a cohesion proxy (see **Q2** — the scope's "Jaccard" is underspecified for a single leaf). Emits ONLY `severity:'advisory'` — never fails the leaf. Default-soft (Goodhart-guard, scope:153).
- **#3 `interface-clean`** (natural): pass iff a structured output is declared (`output_schema` present + non-empty) AND inputs are focused (`inputs` count ≤ `INTERFACE_MAX_INPUTS`). Else `{severity:'error', kind:'interface-unclean'}`.
- **#4 `validation-supported`** (natural + `R9→R12` + `R9→R8`): first validate `isValidDiscipline(leaf.discipline)` (R8) → else `{kind:'unknown-discipline'}`. Then the **discipline branch**: `tdd` → build `ctx={testFile:leaf.verification?.testFile, runner:leaf.verification?.runner}` and require `registry.isVerificationSupported(ctx)` true (a registered runner can verify it) → else `{kind:'validation-unsupported'}`; `spec-driven` → **PASS** (R9's own #1/#3/#5 ARE that leaf's gate — there is no test-run, and that is correct; architect R12-VERIFY Q1).
- **#5 `resource-bounded`** (one-liner): pass iff `estimated_tokens ≤ RESOURCE_MAX_TOKENS` AND `leaf.allows_subspawn !== true`. Else `{severity:'error', kind:'resource-unbounded'}`.

**Aggregate entry point** (what R11 consumes): `validateLeaf(leaf, ctx) → { ok, errors, advisories }` where it enumerates `Object.keys(LEAF_CRITERIA)`, partitions violations by `severity`, and `ok = errors.length === 0` (advisories never fail). Plus `listCriteria()` (Object.keys) for parity with the registry idiom.

**Named tunables** (frozen, documented defaults): `COST_MIN_TOKENS=500`, `COST_MIN_WALLCLOCK_S=30`, `RESOURCE_MAX_TOKENS=` **(Q4 — scope leaves N open)**, `INTERFACE_MAX_INPUTS=` (Q3 default).

## Open Questions (for the architect VERIFY — these reshape the build)

- **Q1 (leaf schema — required vs optional + absent-field policy):** which extended fields are REQUIRED for an R9-validated leaf vs optional? Esp. `estimated_tokens` — a leaf omitting it: hard-fail #1/#5, or skip-as-advisory? Does requiring the heavy fields conflict with R6's minimal `{id,content,discipline}` leaves (backward-compat)? Recommended: extended fields are REQUIRED **only at the R11 verify boundary** (a leaf submitted for verification must be fully specified), NOT at R6 decompose time. Confirm.
- **Q2 (criterion #2 is underspecified — the murkiest):** the scope says "tag-overlap (Jaccard) proxy" but Jaccard compares TWO sets — what two, for a SINGLE leaf? Options: (a) leaf.tags vs content-derived tokens (cohesion = do the tags reflect the content?); (b) drop Jaccard, use a simpler honest proxy (e.g., has ≥1 tag AND content length within a band AND a single declared output_schema → "one clear purpose"); (c) cross-leaf overlap (a leaf's tags vs siblings — too much overlap = not distinct) — but that needs sibling context R9 may not have. Pick the HONEST, non-Goodhart proxy. It STAYS advisory regardless.
- **Q3 (#4 contract + #4-vs-discipline-validity placement):** is building the R12 ctx from `leaf.verification` right, and is the `isValidDiscipline` check part of #4 or a separate precondition criterion? Does a `tdd` leaf at definition-time reliably know its `verification.runner`/`testFile` (it may not have authored the test yet)? If not, what does #4 check for a not-yet-authored tdd leaf — "declares a runner kind R12 supports" (lighter than a testFile)?
- **Q4 (#5 — is it degenerate? + the N value):** Pattern-A (R6) is serial/persona-internal — it NEVER sub-spawns. So is `allows_subspawn !== true` always-true-by-construction (a degenerate check), or a real guard against a leaf DECLARING sub-spawn (forward-defense for Pattern-B, deferred v3.5)? And what is `RESOURCE_MAX_TOKENS` (N)? The scope leaves it open — propose a default + rationale.
- **Q5 (who populates the fields + R9's caller):** confirm R9 is a VERIFY-time gate (called by R11), not a decompose-time gate (R6). If so, the persona/R11 populates the extended fields when submitting a leaf for verification. Is there an R11 contract for this the R9 entry-point shape must anticipate (so R9 isn't re-opened at R11)?

## Principle Audit

- **SRP** — each criterion owns one check; `validateLeaf` owns aggregation; R9 owns "is this leaf well-formed", NOT "run it" (R11) or "decompose" (R6).
- **OCP** — 5 criteria register into `LEAF_CRITERIA` without editing each other (the `validators={}` idiom); a 6th criterion is additive. R11 depends on `validateLeaf`'s shape, not the criteria (DIP).
- **DRY** — reuse the `validators={}` registry idiom, `isValidDiscipline`/`disciplineBlockViolations` (R8), `isVerificationSupported` (R12); do NOT re-implement discipline or runner logic.
- **KISS/YAGNI** — KISS bodies (2 one-liners); do NOT gold-plate #2 into a hard semantic gate (Goodhart); do NOT build Pattern-B sub-spawn handling (#5 forward-defense only if non-degenerate per Q4).
- **A4-binding** — R9's deterministic cores are tested kernel/runtime FUNCTIONS (not contract-prose/trampoline-prompt text) — the Wave-3 precondition for flipping the K11 A4 gate to enforcing (scope:201-203).
- **Immutability** — `validateLeaf` reads the leaf, returns fresh `{ok,errors,advisories}`; never mutates the leaf; frozen tunables.

## Verification Probes

- **Per-criterion negative path:** each of the 5 emits a violation on a malformed leaf (mirrors every contracts-validate validator's synthetic-negative test); a well-formed leaf → zero errors.
- **#1 boundary:** `estimated_tokens` exactly at `COST_MIN_TOKENS` passes; one below (and wallclock below) fails.
- **#2 advisory:** a low-cohesion leaf yields an `advisories[]` entry but `ok:true` (never fails the leaf).
- **#4 discipline branch:** a `tdd` leaf whose `verification` R12 supports → pass; a `tdd` leaf with no supported runner → error; a `spec-driven` leaf → pass (no test-run required); an unknown discipline → `unknown-discipline` error.
- **#5:** over-budget tokens → error; `allows_subspawn:true` → error.
- **Aggregate:** `validateLeaf` partitions errors vs advisories; `ok === (errors.length===0)`; `listCriteria()` returns the 5 names.
- **Full gate:** kernel suite green; widened runtime suite green (+ the new leaf-criteria suite); `bash install.sh --hooks --test` 119/0.

## Pre-Approval Verification — architect `ac3455` → NEEDS-REVISION → all folded (USER ratified 6 criteria)

**The design below SUPERSEDES the proposals above where they differ.** Architect verdicts: Q1 CONFIRM+tighten, Q2 OVERRIDE, Q3 OVERRIDE+split, Q4 FLAG, Q5 CONFIRM+tighten.

1. **SIX criteria, not five (USER-ratified 2026-06-03; resolves Q3b + F1).** `LEAF_CRITERIA` = `cost-justified`, `semantically-cohesive` (advisory), `interface-clean`, `validation-supported`, `resource-bounded`, **`discipline-gate`** — set-equal to the frozen ADR-0015 `failed_criterion_id` enum (ADR-0015:59), so R11 maps a failure straight to `failed_criterion_id` (INV-FS-CriterionEnumMirrorsR9). `discipline-gate` = a one-liner reusing `isValidDiscipline` (R8); ordered BEFORE #4 (precondition). A fitness test asserts `listCriteria()` set-equals the enum verbatim (catches future drift).
2. **Schema + absent-field policy (Q1).** Required-set at the R11 verify boundary: `{id, content, discipline, (estimated_tokens OR estimated_wallclock_s), output_schema}` + `verification` **required-iff-`tdd`**. **Discriminating rule (header-documented):** a REQUIRED field absent = the owning criterion's `error`; an OPTIONAL field absent = a defined default evaluated normally; **NO criterion passes vacuously on a missing field** (the A4-binding anti-rubber-stamp). `inputs` absent→`[]`; `tags` absent→#2 advisory; `allows_subspawn` absent→`false`. A bare-R7 `{id,content,discipline}` leaf passed to `validateLeaf` is an **expected fail-closed reject** (R11 populates the fields first), NOT a bug — documented so R11 isn't "fixed" by loosening R9.
3. **#4 `validation-supported` (Q3a).** A `tdd` leaf checks the LIGHT **`registry.getAdapter(leaf.verification?.runner) !== null`** (a registered runner exists) — NOT `isVerificationSupported({testFile})`, because the live adapter requires an ABSOLUTE on-disk testFile that does not exist at leaf-definition time (node-runner.js:126,173). `spec-driven` → PASS (R9's own #1/#3/#5 are that leaf's gate). The actual run is R11's job at verify time.
4. **`validateLeaf(leaf)` return shape (Q5) — keyed by criterion-id (the load-bearing R11 contract):** `{ ok, criteria: { <criterion-id>: { ok, severity, violations:[...] } }, errors:[...], advisories:[...] }`. `ok === (no error-severity violation)`. The keyed `criteria` map lets R11 do `Object.keys(criteria).find(c => !criteria[c].ok && criteria[c].severity==='error')` → `failed_criterion_id` with ZERO re-mapping. Flat `errors`/`advisories` kept as a convenience view. `listCriteria()` → the 6 ids.
5. **Drop the speculative `ctx` (F5).** `validateLeaf(leaf)` and each `criterion(leaf) → violations[]` — no criterion reads out-of-leaf state (tunables are module constants; R12/R8 are imports). Adding `ctx` now is YAGNI; a future override seam is additive (OCP).
6. **#2 `semantically-cohesive` (Q2) — structural proxy, NOT Jaccard.** Jaccard is incoherent for a single leaf (no second set). Pass iff `tags.length≥1` AND exactly one `output_schema` AND `content.length ∈ [COHESION_MIN_CHARS, COHESION_MAX_CHARS]`. **Advisory-only** (`severity:'advisory'`, never fails `ok`). Header label: "a deliberately-weak STRUCTURAL proxy — a deterministic check cannot honestly judge semantic 'one clear purpose'; advisory, feeds v3.3 E2 reputation, NEVER harden into a gate (Goodhart; scope:153)."
7. **#5 `resource-bounded` (Q4).** `estimated_tokens ≤ RESOURCE_MAX_TOKENS` (a LIVE bound) AND `allows_subspawn !== true` (a **degenerate-now FORWARD-GUARD** — Pattern-A/R6 never sub-spawns, so it's true-by-construction today; its job is fail-closed forward-defense for Pattern-B/v3.5). Header labels the sub-spawn clause as a forward-guard, NOT a live check. **`RESOURCE_MAX_TOKENS = 100000`** — a SINGLE-leaf bound (not the run-budget, which R10/budget-tracker owns; not a model context limit); 200× the 500 floor so #5 fires only on genuinely under-decomposed leaves; provisional (no calibration data yet).
