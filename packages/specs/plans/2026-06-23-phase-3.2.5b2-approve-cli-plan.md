# ③.2.5b.2 — the approve-CLI (human sign-what-you-see gate)

## Goal

The **production caller** for the ③.2.5b broker: a CLI a HUMAN runs to review the exact emission a draft would
produce and, on confirmation, mint a **signed** approval (via the cross-uid broker `signFn`) into the custody
approvals dir. emitPR (5a) later finds + verifies that approval keyed to `computeEmissionHash(draft)`. **SHADOW** —
`armedEmit` still throws, so a minted approval gates nothing live until 5c arms emission. This is the human-gate UI;
it does NOT change the kernel, emitPR, or the broker — it WIRES them.

## Design

**The flow** (the CLI runs as the operator/human uid — the caller-auth-allowlisted one, DISTINCT from the actor uid):
1. Parse args (`--draft <draft.json>` = the actor-produced `{repo, issueRef, diff}`; `--approvals-dir`;
   `--broker-user` + `--wrapper` [+ `--sudo`] = the broker wiring; `--verify-key <pem>` = the custody verify key;
   `--key-id` optional (default `v0`, advisory); `--ttl-ms` optional).
2. Read + **validate the draft by REUSING emitPR's OWN exported egress validators, on the RAW draft, in emitPR's
   order** (VERIFY arch/hacker/honesty HIGH — all probed EXPORTED): `assertDataIsPolicyFree(data)` ->
   `assertSafeRepoRef(repo)` -> `assertSafeIssueRef(issueRef)` -> `assertEgressSafeDiff(RAW diff)`. The CLI is the
   FIRST place these run on the human-facing path -> LOAD-BEARING, not redundant: the human may then only ever
   review+mint a draft emitPR would accept AND normalize identically, and the `.github`/`.git*`/CI-path denylist +
   traversal/charset/size gates run on the SAME RAW bytes both sides see (probed: a `.github` diff is REJECTED).
   Fail-closed on any throw, BEFORE any render or prompt.
3. **Build ONE frozen scrubbed-draft object** `const scrubbed = Object.freeze({ ...draft, diff: scrubEmitDiff(draft.diff) })`
   (`scrubEmitDiff` idempotent, probed) and flow the SAME object into render + hash + recordApproval — no re-scrub,
   no second draft variable (VERIFY hacker LOW: a render/hash byte-divergence would silently break sign-what-you-see).
4. `hash = computeEmissionHash(scrubbed)` (the content-address). recordApproval re-derives the SAME hash.
5. **`assertCustodyApprovalsDir(dir, selfUid)`** as a FAIL-FAST UX check (don't prompt the human if the mint target
   is poisoned). NOTE: this early check is NOT the atomic guard — `recordApproval`'s OWN re-assert + the `wx`
   exclusive-create is (VERIFY arch LOW; already true in the store).
6. **Render sign-what-you-see** to `/dev/tty`: the normalized repo, issueRef, the SCRUBBED diff, the full `hash`.
7. **Read the confirmation from `/dev/tty`, and assert `tty.isatty(fd)` on the opened fd** — never fd 0 (the actor
   can pipe `y` to stdin). Require the human to type the **hash-prefix handshake token** (first 8 hex of `hash`).
   **read-once** (one prompt; wrong/blank -> refuse, no retry loop). **No tty (ENXIO, probed) / not-a-tty -> fail-closed.**
8. On the correct token: fresh `nonce` (`crypto.randomBytes`), `signFn = crossUidLoomBrokerSigner(...)`, read the
   `verifyKeyPem` SAFELY (below), call `recordApproval(dir, scrubbed, { now: Date.now(), nonce, signFn, keyId,
   verifyKeyPem, selfUid })`. Report the minted `hash`.

**The `--verify-key` read is fail-closed** (VERIFY hacker MEDIUM): read it with `O_NOFOLLOW` + `fstat`-the-fd +
uid-ownership (reuse the `.approved` read discipline), NOT a bare symlink-following `readFileSync`, and the path
comes from the operator (NEVER derivable from the actor-written draft). It is the mint-boundary trust anchor.

**Security model (stated honestly — the CRITICAL reframe):**
- **The LOAD-BEARING boundary is the cross-uid deployment, NOT `/dev/tty`.** The CLI runs as the OPERATOR uid (on
  the broker's caller-auth allowlist), DISTINCT from the ACTOR uid. An actor that spawns the CLI runs it as the
  ACTOR uid -> the broker's caller-auth DENIES it (probed: `SUDO_UID=actor` -> `caller-not-in-allowlist`) -> no
  mint. So the actor cannot drive a mint even by PTY-wrapping a CLI it spawns.
- **`/dev/tty` + `isatty` is DEFENSE-IN-DEPTH, not the boundary.** It defeats a NAIVE same-uid pipe (`echo token |
  cli`), but a uid-co-resident actor can allocate a controlling PTY (`pty.fork`/`script`/`expect`) so `/dev/tty`
  opens and returns an actor-written token — the actor KNOWS the token (`hash.slice(0,8)` over its OWN draft via
  the exported `computeEmissionHash`). `/dev/tty` is a separate-*fd* test, NOT a separate-*principal* test.
- **The confirm token is a SIGN-WHAT-YOU-SEE HANDSHAKE, not a secret** (the actor can derive it). Its only job:
  force the confirmer to have READ the rendered content-address (a blind `<enter>` fails); a swapped draft changes
  the hash -> the old token no longer confirms. It adds NO cryptographic entropy; the `/dev/tty` requirement must
  never be relaxed into a stdin-readable challenge.

**Honest scope (carried LOUD — do NOT report closed):**
- **Single-operator PAYLOAD authority** — the human can approve ANY draft shown (the CLI does not adjudicate judgment).
- **Operator AUTHENTICITY ceiling** — the CLI cannot verify the entity at the tty is a human, the INTENDED human,
  or uncoerced/undeceived. A pty-driving process, a duped/coerced operator, or a shoulder-surfer is OUT of this
  gate's threat model — carried by the cross-uid deployment + operator trust, not the CLI.
- **Same-uid is VACUOUS** (consistent with 5b) — if actor-uid == operator-uid, the actor defeats the CLI entirely
  (PTY-wrap, `ptrace`, …) AND the broker would authorize it. Custody is real only under the cross-uid deployment.
- It is the production minting path; **SHADOW until 5c** (a minted approval gates nothing live).
- `keyId` is ADVISORY (bound into the basis, NEVER selects the key — the pin selects); the CLI mints under
  `--key-id` (default `v0`); a future rotation must keep mint-vs-consumer `keyId` aligned (documented).
- `recordApproval` independently re-validates issueRef/repo/dir + recompute-binds the hash before the `wx` write —
  the CLI's validation is the FIRST floor (render + sign-what-you-see correctness), the store is the second.
