## Design

### Rejected mechanisms (named, with reasons — the board should re-derive these)

- **HMAC** — the key is same-uid-readable; a forger reads the key and mints a valid MAC.
  Security theater in the trusted-local-fs model. REJECTED.
- **dir perms** — OS-dependent, does not bind CONTENT to a producer, silently breaks the
  cooperative multi-tool story. REJECTED.
- **Hot-path re-derivation** — re-projecting E4 from the verdict ledger and comparing breaks
  the O(1)/no-ledger-read constraint (probe 2); off-path re-derivation is FUZZY (the ledger
  prunes on write, so a later re-projection legitimately differs). REJECTED as the gate
  mechanism (an off-path audit CLI may still cite it as a soft cross-check — out of scope).
- **A kernel TRANSACTION record per materialize** — would need a new `operation_class`
  (a v6 enum amendment; the W3a lesson: do not touch it for Lab advisory artifacts). REJECTED
  this wave; the witness ledger below leaves room to upgrade later if a kernel gate demands
  chain-anchoring.

### The mechanism: a materialize-WITNESS ledger (two-artifact coherence + audit trail)

**The contract lives in the SAME kernel/_lib leaf** (`evolution-snapshot-read.js`) — one leaf
= one cross-layer contract; a second leaf would re-open the CR-HIGH-2 writer/reader
split-brain class for the witness path/format:

- `resolveWitnessLedgerPath()` — `${LOOM_LAB_STATE_DIR || ~/.claude/lab-state}/
  snapshot-provenance.jsonl` (env-overridable via `LOOM_SNAPSHOT_WITNESS_PATH`, mirroring the
  snapshot path formula). Root placement accepted (nothing readdirs lab-state root for
  snapshots; the reader opens fixed names — VERIFY A-LOW-4 disposition).
- `appendSnapshotWitness({content_hash, generated_at, record_count, now?})` — appends ONE
  witness line `{witness_id, schema_version, content_hash, generated_at, recorded_at,
  record_count}` where:
  - **`witness_id = sha256(canonical(body minus witness_id))` — the WHOLE body, not a field
    pair** (VERIFY synthesis: the draft's `[content_hash, generated_at]` basis was DOUBLY
    wrong — architect A-HIGH-1: `generated_at` is INSIDE the snapshot's hashed body, so
    `content_hash` already determines it — zero independent entropy, and it made a
    caller-choosable timestamp load-bearing, the exact reject-event-store anti-pattern;
    hacker H1, live-probed: every field OUTSIDE a pair-basis is attacker-mutable while the
    witness still vouches — a `record_count: 999999` mutation re-derived fine. The #273
    whole-body discipline closes both: ANY mutated field breaks the id).
  - **`recorded_at` is LEDGER-APPLIED at append time** (hacker M1): the body's
    `generated_at` is attacker-chosen (it is inside the snapshot hash); the append-time stamp
    plus FILE POSITION are the order signals. Honest framing: `recorded_at` is authenticated
    WITHIN the witness (in the id basis) but is still same-uid-forgeable at authoring time —
    the trail is an ORDER-of-materialize record, not a timestamp-trustworthy one.
  - Bounded + idempotent: locked RMW (the lab-lock pattern) capped to the newest
    `WITNESS_LEDGER_MAX_RECORDS` (1024); an identical `witness_id` re-append dedups.
  - Writer-side deps (`jsonl-read`/`lock`/`atomic-write`) are **LAZY-required inside the
    functions** (architect A-MED-2): the leaf is loaded by the <50ms spawn-close hook, whose
    no-flag path must not pay the writer's module-load tax.
- `verifySnapshotProvenance({content_hash}, [opts])` → `{witnessed: boolean, reason:
  'witnessed' | 'unwitnessed' | 'no-ledger' | 'invalid-snapshot'}`: bounded tail read (newest
  1024 — read cap == prune cap, so a surviving witness is always readable; architect
  A-MED-3), then per row: re-derive the id over the row's body-minus-id (**per-row try/catch
  — hacker L1: a syntactically-valid but 200-deep row must SKIP, not throw**; the leaf's
  never-throw contract), require `row.witness_id === derived` AND `row.content_hash ===
  snapshot.content_hash`. A self-inconsistent id is SKIPPED (forged-id rejection — #273).
  **The tail-window invariant is explicit**: provenance is verifiable only for witnesses
  inside the bounded tail; aged/flooded-out reads `unwitnessed` (fail-closed direction).
- `readEvolutionSnapshot` gains OPTIONAL `{verifyProvenance: true}` → the result carries
  `provenance: 'witnessed' | 'unwitnessed'` (computed via the fn above; one extra bounded
  file read). **The spawn-record hot path is UNTOUCHED** — it does not pass the flag, its
  result shape is unchanged (additive-only for opted-in callers). The GATING consumer (v3.9)
  passes the flag and FAIL-CLOSES on `provenance !== 'witnessed'` (the `excluded_future`
  value-check shape — probe 7; provenance is a property of the DATA, not of the caller's
  authority, so a value beats a throw here).
- `materializeSnapshot` appends the witness AFTER the atomic snapshot rename
  (**write-then-witness**: a crash between the two leaves an UNWITNESSED snapshot → the gate
  fail-closes → re-materialize heals; witness-first would attest a snapshot that may not
  exist). The return gains `witnessed: true` + the `witness_id`.
- CLI: `reputation verify-snapshot` — reads + verifies, prints the JSON, exit 0 on
  witnessed / 1 otherwise (the operator surface; also the dogfood probe).

### Honest scope (the load-bearing honesty — say it everywhere)

A same-uid actor who can write the snapshot can ALSO append a witness — the ledger validates
FORMAT and COHERENCE, not authorship (identical concession to every store: p-writescope; the
store is not a sandbox; closes at the ContainerAdapter). **Same-uid is BOTH a forge AND a
denial axis** (hacker H2, live-probed): flooding the ledger past the bounded tail makes a
GENUINELY-witnessed snapshot read `unwitnessed` — narrowing-safe (over-halt), and
**re-materialize HEALS it** (the fresh witness lands back inside the tail; test-locked).
What the witness BUYS: (1) the gate-PASS semantic becomes "produced by a materialize event
that left a coherent two-artifact trail" — naive hand-edits, partial tampering, and
accidental corruption now FAIL a check they previously passed; (2) an append-only
**ORDER-of-materialize trail** (positional + append-stamped; body timestamps are
honest-but-same-uid-forgeable — hacker M1's down-scope: NOT "audit-grade when"); (3) the
machine-checkable surface M1 demanded ("the write-path needs provenance") so the v3.9 gating
consumer has something to fail-close ON. What it does NOT buy: same-uid tamper-PROOFNESS.
The ledger row and the reader docs must carry `witnessed != authentic-beyond-same-uid`.
