# HT.2.1 — Measurement-methodology codification doc

## Goal

Author a single substrate-internal convention doc at `swarm/measurement-methodology.md` (~300-400 LoC) capturing **observed dogfooded best practice** on measurement-via-grep/wc methodology for substrate audits. Document the seven-instance audit-count-drift + option-axis-conflation pattern as 6 active case studies + 3 in-scope-resolution case studies (drift-note 76 spans both cohorts), with 5 canonical measurement patterns derived from observed practice. Sibling shape with `swarm/path-reference-conventions.md` (HT.1.10; 203 LoC).

**What the doc IS**: substrate-internal convention doc capturing existing dogfooded practice; lightweight institutional decision record per HT.1.6 BACKLOG.md `decision-record-pattern: lightweight` precedent (3rd convention-doc-shape institutional artifact across HT.1 + HT.2: HT.1.10 path-conventions + HT.2.1 measurement-methodology + HT.2.5 soak-gate-readiness readout).

**What the doc is NOT**: prescriptive discipline charter; new institutional invariant; ADR-shape commitment. Discriminator vs ADR (architect MEDIUM-2 absorption at HT.2.0): captures EXISTING dogfooded pattern, not new institutional invariant. Substrate ADR ledger stays at 4 post-HT.2.

## Context — why HT.2.1 now

Six convergent audit-count-drift case studies + one option-axis-conflation case study (architect LOW-2 re-cohorting at HT.2.0) have accumulated across HT.1.4–1.15. Each demonstrates the same underlying pattern: an audit's raw-count claim drifted from empirical reality at the per-site / per-line / per-caller / per-option-axis level. Substrate dogfooded the corrective discipline — empirical pre-validation at sub-plan time — across the closing 8-phase cohort (HT.1.8 → HT.1.15) with 100% phase-success rate.

The pattern has sufficient empirical mass to justify single-doc codification. Per HT.0.9-verify FLAG-5 right-sizing precedent, lightweight convention doc is the right institutional shape (not ADR — convention doc captures existing practice; ADR codifies new invariant).

## Canonical measurement patterns (observed practice)
[5 patterns derived from case studies; bullet-list form]

### Pattern 1 — Inventory-via-grep + per-site classification
Never cite a raw grep count without per-site classification. Classify per-site:
- in-scope vs out-of-scope for the action under consideration
- static-value vs hot-path vs lukewarm
- live-call vs dead-code
- module-load-time vs per-call
Example: HT.1.11 9-site grep → 4-tier classification → 6 sites with migration value (case study 4).

### Pattern 2 — Audit-method-and-currency awareness (REFRAMED per architect HIGH-1 + code-reviewer HIGH-1 convergent absorption)
Name the measurement method when citing — visual estimate, analytical claim, and fence-line empirical can yield different counts. AND: audit findings have audit-date currency; substrate evolves between audit-T and implementation-T+N; re-validate before treating as ground truth.
- **Visual estimate**: rough order-of-magnitude scan; useful for prioritization, not for thresholds
- **Analytical claim**: counted across a finding's named scope (e.g., "function `X`"); accurate only if scope is well-bounded
- **Fence-line empirical**: `wc -l` or `grep -c` against the actual filesystem state at sub-plan time; the ground truth
- **Audit-date currency**: any count cited in an audit was true at audit-T; substrate may have evolved since (workflow.md grew via H.7.x additions post-2026-05-09 audit — drift-note 74)
Examples:
- HT.1.4 (case study 1; drift-note 63): HT.0.9-verify FLAG-2 cited 537 LoC for `run_smoke_tests`; empirical reality was 1188 LoC pre-HT.1.4 — visual-estimate-vs-fence-line gap (+121% understatement)
- HT.1.5 (case study 2; drift-note 64): 3 different counts produced by HT.0.3 visual estimate + HT.0.9-verify FLAG-2 + HT.1.5 fence-line empirical — same measurement, three methods, three numbers
- HT.1.13 (case study 6; drift-note 74): HT.0.8 cited 228 LoC for always-on rules; empirical reality at HT.1.13 was 322 LoC (+41% understatement) — likely caused by H.7.x additions to workflow.md post-audit-date; audit-date currency drift

### Pattern 3 — Reference count grounding
Count broken references against the actual filesystem at sub-plan time, not against the audit's expected target list. Audits authored at time T may have stale-by-time-T+N counts if the substrate tree shape evolved.
Example: HT.1.12 cutover claim 11/7/8 vs empirical 7/5/5 (case study 5).

### Pattern 4 — Caller-count empirical re-validation
Before "delete + migrate" decisions, grep ALL invocation forms:
- `require('./X')` (CommonJS)
- `import` (ESM — none in substrate currently but future-proofing)
- `spawnSync('node', [..., 'X', ...])` (subprocess)
- `execFile('X', ...)` (subprocess)
- `bash invocation referencing X`
- documentation / comment references (not invocations but document the consumer relationship)
Example: HT.1.15 cited 1 caller → empirical 2 callers / 3 call sites (case study 9).

### Pattern 5 — Option-axis disambiguation
When a decision-block enumerates N options, identify the underlying axes. Collapsed axes lead to silent decision gaps. A 3-option enumeration may actually be a 2×2 matrix with 1 axis silently undecided.
Example: HT.1.6 decision-block "3 options" → 2 independent axes (contract field shape × roster membership) → one axis silently undecided until runtime exposure (case study 3).
