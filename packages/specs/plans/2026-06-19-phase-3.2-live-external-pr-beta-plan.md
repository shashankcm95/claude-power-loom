# Phase 3.2 — the LIVE external-PR beta (the trust unlock)

## Context

Phases ③.0 (foundation-hardening) and ③.1 (the dry-run loop) are **phase-closed** (2026-06-17 / 2026-06-19): the full
recall -> solve -> grade -> learn loop runs end-to-end on a **real `claude -p` actor** + **real Docker-graded sealed
tests**, DRAFT-only and SHADOW (trust ZERO — OQ-NS-6). ③.2 is the apex: the system pulls real `good-first-issue`s,
solves them, and the **user pushes each PR** for a real maintainer to merge — the **only lever that HARDENS trust**
(everything to here only NARROWS). This plan turns the 2026-06-16 charter's high-level "③.2 — REAL-PR submission"
section into reviewable sub-waves, instantiating the **named ③.2 carry-list** the ③.1 phase-close handed forward
([ROADMAP.md:616](../../../docs/ROADMAP.md)). ③.2 is **apply-then-gate**: no real PR ships until the egress safety
kernel exists fail-closed and either nothing from the lab tier gates the decision (stay shadow) or the authenticated
minter is fully landed.

## The load-bearing design decision (resolve FIRST — it scopes everything below)

The ③.1 phase-close finding #3 established that the SHADOW-tolerable security residuals (the co-forgeable
`confirmed-by` edge, the unauthenticated `built_by.role`, the unbounded forged-metric trust, the conftest-poison /
last-wins grade-forgery) are each tolerable *only because nothing gates* — and **all trip the moment a slice / weight /
grade GATES a real-PR decision**. ③.2 ingests real external PRs (adversarial-capable). So ③.2 must pick a path:

- **PATH 1 — human is the SOLE gate; every lab weight/grade stays SHADOW (RECOMMENDED for the first real PR).**
  Structurally guarantee that **nothing from the lab tier gates the real-PR decision** — the user reviews the diff and
  pushes; a maintainer merges; that merge is the world-anchored signal (OQ-NS-6). On this path the **trust-weight
  provenance SET is NOT required** for the first real PR (minter P1/P2 vehicle, signed-edge lane, two-axis reconcile
  stay deferred to the named trigger "a lab weight first GATES an action"). This is the smallest honest path and the
  charter's stated safe default ("keep every weight shadow").
- **PATH 2 — a lab weight/grade GATES (admit/deny/auto-anything).** Then the **full** integrity!=provenance SET must
  land first (minter P1 + the recompute-inside vehicle + P2 consumer-flip + signed-edge + two-axis). Heavier; not
  needed for a human-sole-gate first PR.

**This plan is written for PATH 1.** Two residuals do NOT depend on the path and remain hard ③.2 gates even under
Path 1, because the human reviews a **grade computed over a stranger's repo code**: (a) the **grade-integrity** subset
(RFC-R3 conftest-poison / last-wins — a poisoned `conftest.py` can make a broken patch grade "passing", so the human
pushes a bad PR), and (b) **egress safety** (the PR-egress kernel + clone-hardening). Those are in-scope; the
trust-weight provenance SET is explicitly deferred (see Out of Scope).

### ③.2.0 — Readiness (internal hardening; no external surface; unblocked NOW)

Decomposed into three sub-slices by the pre-build VERIFY board (below): **A** (actor-clone security hardening — the

#### ③.2.0-A — Actor-clone security hardening (SECURITY; the coupled unit)

The VERIFY hacker **live-reproduced a host-RCE that the originally-planned fix does NOT close** — so this slice is a
real security build, not a one-line route-through. Five coupled folds across the two actor paths +
`_clone-lifecycle.js`:

- **A1 — `.git/`-poisoning RCE close (VERIFY C1, CRITICAL — pulled FORWARD from ③.2.1a).** The actor runs `claude -p`
  with `cwd=actorDir` and the `Write` tool, so it can write `.git/config` + `.gitattributes`; the harness's subsequent
  `git add -A` (`real-solve.js:171` / `earned-grounding-run.js:366`) then executes an attacker `[filter "x"] clean=…`
  driver as host shell — **proven live, UNDER the full `GIT_HARDEN` set** (`core.hooksPath`/`protocol.ext.allow`/
  `core.fsmonitor` do not touch filter drivers). **Fix: snapshot `.git/config` immediately after clone+checkout
  (before the actor spawns) and RESTORE it to that pristine state before any post-actor `git` op** (a pristine config
  with no `[filter]`/`core.*`-exec directive makes a poisoned `.gitattributes filter=x` a no-op — git can't resolve the
  driver), and run the capture ops with `-c core.attributesFile=/dev/null` as belt-and-suspenders. EC re-runs the
  hacker's live probe against the BUILT fix.
- **A2 — converge the actor CLONE onto the hardened path (VERIFY H1 — the probe premise was wrong).** `real-solve.js`
  does NOT use `prepareClone` (`:160` is a **bare** `git clone` — no `--`, no `--no-hardlinks`, no `GIT_ENV`); the
  residual surface is the clone too, not only post-clone ops. Route the clone + checkout + add + diff through
  `_clone-lifecycle.js`'s hardened `git`/`prepareClone` (both actor files; they are duplicated — keep in sync or
  extract).
- **A3 — `GIT_ALLOW_PROTOCOL` https-only on the actor path (VERIFY H3 — pulled FORWARD from ③.2.1a).** Today it is
  `file:https:http` (`_clone-lifecycle.js:64`); combined with A1's attacker-controlled `.git/config`, the `file:`
  allowance is a local-file-read primitive. Parameterize the protocol set: **`https` only** for the actor + grader
  paths; the `file:` allowance survives ONLY behind the existing `allowLocalRepo` test flag.
- **A4 — clone-DoS bound (VERIFY H2 — and NOT `--depth 1`).** The architect proved `--depth 1` BREAKS the
  `checkout base_sha` reproducibility pin (a shallow clone of the branch tip cannot check out an arbitrary pinned
  historical sha). Bound instead with `--filter=blob:none` (partial clone — lazy blobs) + a **post-clone `du`/byte
  ceiling -> `safeDiscard` on breach** + a tighter-than-120s clone-step wall-clock. (`maxBuffer` bounds stdout only;
  `git clone` writes to disk.)
- **A5 — `maxBuffer`/cap reconcile WITHOUT a regression (VERIFY F6 + Item 2c).** The hardened `git` has NO `maxBuffer`
  override -> Node's 1MiB default; the actor closures deliberately raise it (`real-solve.js:147` ~2MiB so a
  `[1MiB,2MiB)` diff reaches the size-cap check instead of `ENOBUFS`; `earned-grounding-run.js:357` 4MiB). **Add an
  optional `opts` param to the hardened `git(args, cwd, {maxBuffer, timeout})`** and standardize BOTH actor paths on
  `MAX_PATCH_BYTES`+slack (the 4MiB drift is unjustified) — do NOT silently inherit 1MiB (it would regress a fixable
  candidate to UNAVAILABLE).

**Exit criteria (A — each proven INTERNALLY against the BUILT code; SECURITY tier).**
- EC0a.1 — the VERIFY hacker's live `.git/config` filter-driver probe (and a `.gitattributes` variant) **cannot execute
  host shell** against the BUILT fix; a `core.fsmonitor` config-exec is also neutralized.
- EC0a.2 — both actor paths clone + checkout + add + diff through the hardened `git`; `GIT_ALLOW_PROTOCOL=https` blocks a
  `file::`/`ext::` reference on the actor path; a crafted leading-`-`/non-https repo is refused (`assertSafeRepo`).
- EC0a.3 — the clone is `--filter=blob:none`-bounded and aborts + `safeDiscard`s on an oversized / long-running hostile
  remote; the `base_sha` checkout still resolves (no `--depth 1` regression).
- EC0a.4 — a `>1MiB`-but-legal diff still grades (no `ENOBUFS`->UNAVAILABLE regression); both actor caps reconciled to
  one documented ceiling.
- EC0a.5 — full **3-lens VALIDATE** (correctness + adversarial-security re-probe of the BUILT code per Rule 2a +
  honesty on every "closed" claim) clean.

#### ③.2.0-B — ARCH-PC-4 close-path concurrency probe (re-targeted by VERIFY)

The original target was **wrong**: `library-catalog.js`'s `withLockSoft` is the *catalog* lock, NOT the spawn-close
path; the close path's only lock is **K13's `acquireLock`, which SHADOW stubs to a no-op** (`spawn-close-resolver.js:687-688`)
— so there is no close-path drop-rate to measure in the default mode. Re-scoped:

- Measure **two seams, labeled honestly** (do NOT conflate): (B-i) the catalog `withLockSoft` drop-rate via N
  concurrent catalog writes — a real, tractable contention measurement; (B-ii) the K13 `acquireLock` contention via
  N concurrent direct `serialAdmission` calls (or the `LOOM_RESOLVER_ENFORCE=1` path), which is the actual close-path
  lock. Plus the close-resolver **wall-time** via a synthetic `tool_response` close payload over a real on-disk git
  worktree fixture (`status_git_ms`/`producer_git_ms` are already journaled).
- **Must fork N OS processes**, not single-process async — `acquireLock`'s `wx`-flag contention is invisible to the
  event loop (a single-process probe reports a false 0% drop-rate; the `T108` precedent forked children). Drive
  against an ephemeral `LOOM_SPAWN_STATE_DIR`/`LOOM_LAB_STATE_DIR` tmpdir, never the real state dirs.

**Exit criteria (B).** EC0b.1 — the probe forks N processes and records wall-time + a real (non-zero-reachable)
drop-rate for each labeled seam; the 10s hook budget holds (or the regression is quantified). EC0b.2 — the journal
labels which lock each number measures (no "the close hook's lock" over the catalog lock).

#### ③.2.0-C — trace-store `nextSeq` (sub-linear + collision-safe)

`trace-store.js:9-20` documents BOTH defects: `nextSeq()` re-reads the whole file (O(n^2) over a run) AND two emitters
to the same `run_id` race read-max->append and can COLLIDE. **Fix: a per-`run_id` `withLockSoft` (soft-fail is correct
— SHADOW/best-effort, a dropped trace degrades observability, never corrupts) wrapping read-max+append, + an O(1)
per-run counter sidecar.** Reuse `kernel/_lib/lock` (the lab->kernel-lock precedent already exists at
`causal-edge/store.js:45`). Contract-preservation: `readTimeline` keeps reading `seq` from the JSONL lines (the counter
is a WRITE-side optimization only); the store keeps owning seq (overrides caller seq); increment the counter only after
a successful append, inside the lock; a `withLockSoft` drop must NOT bump it.

**Exit criteria (C).** EC0c.1 — `nextSeq` is sub-linear (a >=10^4-record volume test in budget) AND collision-safe
under concurrent same-`run_id` emitters (a multi-process test asserts no duplicate seq). EC0c.2 — `appendTrace`/
`readTimeline` contracts byte-preserved; old (counter-less) timelines still read correctly.

**Build order (VERIFY): `[C ∥ B] then A`** — C and B are independent of each other and of A; A is the coupled
security unit (one builder owns `_clone-lifecycle.js` + both actor files). A is the priority (it carries the CRITICAL).

### ③.2.1b — The PR-egress kernel (emission safety; the single `emitPR` chokepoint)

The fail-closed barrier between "draft artifact" and "anything leaves the box". **Needed only before anything LEAVES
(③.2.3), so it can be built and hacker-re-probed without blocking the ③.2.2 DRAFT loop.** Kernel/security tier ->

- **Probe the submit mechanism FIRST (VERIFY F2 + Runtime Probes).** Before any `hooks.json` commitment, resolve: is

  a `PreToolUse:Bash` hook is advisory/evadable for a direct subprocess `gh`/`git push` (the same lesson
  `network-egress-audit.js` already encodes as "advise, never block"). **Do NOT wire the egress kernel as a
  `PreToolUse:Bash` hook.**
- **A single in-process `emitPR(...)` is the SOLE emission path (the architectural gate, not a hook).** It is the
  **only holder of the GitHub token** (the token is never in the actor / grader / clone process env), it performs

  token. This makes "fail-closed by construction" true: with the killswitch set there is **no resolvable token**, so
  the emission capability is *removed*, not *intercepted*.
- **`LOOM_BETA_KILLSWITCH`, re-checked atomically (board-missed M1).** Re-read the killswitch **immediately before the
  irreversible network call**, not only at `emitPR` entry; concurrent `emitPR` invocations **serialize on a lock** (the
  Fibonacci ramp is exactly the concurrent multi-agent surface the MEMORY "M1 TOCTOU" watch-item anticipated).
- **Secret-scrub = defense-in-depth, NOT the primary control (VERIFY F5).** The primary token control is custody
  (above). The scrub (base64/encoded-secret decode + entropy normalization — ARCH-PC-5) is a second layer over the
  **bounded sandboxed candidate diff only** — the PR body/diff are built ONLY from that diff, never from actor stdout /
  logs / trace. The clone uses anonymous https + `GIT_TERMINAL_PROMPT=0` (no credential helper).
- **Caps + etiquette.** A hard per-window cap + 429/abuse backpressure; an etiquette gate (one PR/issue; respect the
  reservation + `CONTRIBUTING`; no re-open / unsolicited churn).
- **Disposition ownership (board-missed M2).** The DRAFT-vs-real and dry-run-vs-live dispositions are set **only in the
  egress-kernel process**, sourced from a path no actor / grader / clone process can write — proven un-influenceable by

**Exit criteria (each proven INTERNALLY before "done").**
