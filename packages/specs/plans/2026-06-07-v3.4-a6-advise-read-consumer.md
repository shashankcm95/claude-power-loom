# A6 advise read-consumer — close the last E4/A6 dark edge

## Goal

Wire the **orchestrator persona-selection step** to consume the **A6 reputation snapshot** as an advisory
**preference** input — the "advise" half that pairs with the E11 "halt" half shipped in #250. This closes
the last E4/A6 dark edge (the ledger E4 row: "the only remaining dark edge is a ROUTER consuming the A6
advisory read") and completes the loop: **produce (verdict) -> advise (A6) + halt (E11) -> consume
(persona-selection)**.

The deliverable is a **consumer convention** (the honest altitude for an A3b / LLM-orchestrator consumer,
exactly like the E11 halt convention) + a thin **read-ergonomic** so the orchestrator can pull a candidate
SET in one call. **No new producer code; no scalar score** (see the load-bearing reframe below).

## The load-bearing reframe (firsthand-probed) — "NOT A SCORE"

`reputation/project.js:13` + the runtime label: *"advisory-verdict distribution over kernel-attested
spawns — NOT a quality score."* The substrate **deliberately refuses a scalar reputation grade**. So
"prefer high-rep personas" CANNOT mean a substrate-computed rank — it means the **orchestrator reads the
DISTRIBUTION and judges**, among >=2 viable candidates for the same role/lens, which has the stronger
track record (pass-rate, `distinct_spawns`, recency, `pending_enrichment` caveat). The recommendation is
the orchestrator's (v6 §0a.3.1:173 "MAY recommend"), grounded in the displayed distribution. This wave
builds NO ranking algorithm.

## Routing Decision

route-decide -> `root` (score 0; the known substrate-meta vocab false-negative, as every prior wave).
**Judgment-override -> ROUTE / per-wave rigor:** this is a **§0a.3.1 anti-amplification surface** (a
derived view informing spawn selection) — the highest-stakes coherence class. `/verify-plan` + 3-lens
VALIDATE with the **honesty-auditor weighted** (does any surface imply a score/rank?) + architect
(coherence) mandatory.

## Design

### D1 — consumer = orchestrator JUDGMENT over the distribution, not a substrate score

The convention: before a delegated builder spawn, when choosing among >=2 candidate personas **for the
same role/lens**, the orchestrator reads the A6 snapshot and MAY prefer the stronger distribution. The
substrate emits the distribution; the orchestrator emits the preference. No scalar, no sort-by-metric.

### D2 — deliver the candidate-set read as a FILTER, not a `compare`/rank verb (USER scope-fork, reframed)

USER leaned toward a `compare` helper; the reservation (a `compare` verb erodes NOT-A-SCORE + invites a
"return the winner" slippery slope) is addressed by delivering the SAME one-call candidate-set ergonomic
as a **filter on the existing read path**: `reputation snapshot --personas a,b,c` (and `--persona P`
single, parity with `show --persona`). It returns each requested persona's distribution **in caller
order** (NEVER sorted by a metric), with **no derived comparison scalar**, keeping the loud NOT-A-SCORE
label. Flippable to a literal `compare` verb if USER prefers post-verify-plan. (Same value as scope-fork
option 3; framing of option 2.)

### D2.1 — filter implementation spec (verify-plan code-reviewer Check 1 FAIL + Check 2/3/5 FLAGs, folded)

- **Parse defensively** (Check 1 FAIL — `parseArgs` returns the boolean `true` for a bare `--personas`,
  and raw comma-strings carry empty/whitespace/duplicate tokens): collect names from `--personas` (ONLY
  if `typeof args.personas === 'string'`: `split(',')`, `.trim()` each, drop empties) + `--persona` (if
  `typeof === 'string'`: the single trimmed name); **dedup**, preserving first-occurrence order. If the
  resulting list is EMPTY (bare flag / all-empty/whitespace) -> fall back to the **whole-snapshot**
  (no-filter) behavior — non-crashing + backward-compat-consistent (an advisory read returning MORE is
  not dangerous; it never masks a subset as the whole).
- **Caller-order via Map-lookup** (Check 2 — `snap.value` is alpha-sorted by `project.js:113`, so
  `.filter()` would yield SNAPSHOT order, not caller order): build
  `const byPersona = new Map(snap.value.map((p) => [p.persona, p]));` then
  `requested.map((name) => byPersona.get(name) || { persona: name, status: 'no-data' })`. The Map is also
  **prototype-safe** (Check 5 — a `__proto__`/`toString` persona name cannot collide; NEVER `obj[name]`).
- **Absent snapshot** (Check 3 — `present:false` AND a filter requested): return the snapshot envelope
  UNCHANGED (`present:false`, no synthesized markers) -> reputation-blind. The filter applies ONLY when
  `present:true` and `value` is an array. (Conflating "no snapshot at all" with "these personas
  unmeasured" would mislead the consumer.)
- **`no-data` vs `pending_enrichment`** (architect pt 4): the `no-data` marker is ONLY for a persona
  ENTIRELY absent from `value`. A persona WITH a row but `total:0` / `pending_enrichment>0` is returned
  as-is — the orchestrator reads `pending_enrichment` as "unverified, NOT a negative." The convention doc
  states both `no-data` and `pending_enrichment>0` are "not a negative signal" (the E11-M1 generalization).

### D3 — no-data != low-rep (the E11-M1 lesson, generalized)

A requested persona ABSENT from the snapshot returns an explicit `{persona, status: "no-data"}` marker —
NEVER silently dropped (the orchestrator must not read absence as a bad track record). This is the
direct generalization of E11-M1 ("a clear/empty result is not a safety signal"): here, "no rows for P"
is not "P is bad", it is "P is unmeasured".

### D4 — reputation is a TIE-BREAKER after lens-fit; NEVER feeds verification-tier (no widening)

Two §0a.3.1 guardrails baked into the convention:
- **Tie-breaker after lens-fit, not a primary selector.** Lens-fit (the CLAUDE.md route-by-LENS rule)
  picks the candidate set; reputation only breaks ties among same-lens candidates. This BOUNDS the
  rich-get-richer feedback loop (prefer high-rep -> more spawns -> more verdicts -> entrench) — without
  it, the advisory could ossify one persona.
- **Preference NARROWS the selection; it must NEVER WIDEN capability.** Reputation must NOT feed the
  verification-tier / `skipChecks` decision (`agent-identity.js recommend-verification`) — good rep ->
  less oversight is exactly the amplification §0a.3.1 forbids ("never gates/widens"). Selection-preference
  is safe; oversight-reduction is not.

### D5 — read the A6-MEDIATED snapshot, not the live projection; absent -> reputation-blind

The decision path reads the **precomputed A6 snapshot** (`reputation snapshot`, content-addressed,
kernel-recorded — v6 §3.6 mediation for any reputation->decision flow), NOT the live `show` projection.
An absent snapshot is NOT an error -> **reputation-blind is the safe default** (the orchestrator selects
on lens-fit alone). `show` stays the human/live-inspection surface.

### D6 — reconcile the loose "RANK" wording from #250

The E11 wave's doc (`agent-identity-reputation.md:469`) loosely says "E4 / A6 = advisory RANK ('prefer
the higher-reputation persona')". Soften to "advisory PREFERENCE over the distribution (NOT a score)" so
the doc is consistent with the NOT-A-SCORE invariant.

## Non-goals (explicitly deferred / forbidden)

- **A scalar reputation score / rank** — forbidden by the NOT-A-SCORE invariant (`project.js:13`).
- **A `compare`/sort verb** — the filter-framing (D2) is deliberate; flippable only by explicit USER call.
- **Reputation -> verification-tier / `skipChecks`** — widening; §0a.3.1-forbidden (D4).
- **route-decide integration** — that is COST-routing; persona-reputation there is a category error.
- **An in-code fix for the rich-get-richer loop** — bounded by advisory + tie-breaker-after-lens (D4); a
  harder structural bound (e.g., exploration epsilon) is a future concern, noted not built.
- **The live `show` projection as the DECISION input** — the decision path uses the A6-mediated snapshot
  (D5, v6 §3.6); `show` stays live-inspection only.

## §0a.3.1 / coherence

The consumer reads the **A6-mediated** snapshot (v6 §3.6 — the required mediation for any reputation->
decision flow; built W3), interprets the **distribution** advisorily (line 173 "MAY recommend"), and only
**narrows the selection** among same-lens candidates — it NEVER widens capability, never feeds K9, never
becomes a scalar. The NOT-A-SCORE invariant is preserved structurally (the filter emits distributions in
caller order, no derived metric). The rich-get-richer feedback is bounded (tie-breaker-after-lens +
advisory override). INV-W1 (enriched-only) is inherited from the projection/snapshot. No kernel hook
consults this -> no NEW A6 mediation needed (the snapshot read IS the mediation).
