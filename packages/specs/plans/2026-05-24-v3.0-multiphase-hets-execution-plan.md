# v3.0 Multi-Phase HETS Execution Plan

Implementation plan for the locked v3.2 architecture (Power-Loom-Trace). Decomposes the work across HETS personas with explicit route-decide gates, per-component spawn dispatch, bounded-output conventions, and per-phase acceptance criteria.

## Plan provenance + locked architecture context

**Architectural contract**: `swarm/thoughts/shared/design/causal-recall-graph-rfc.md` (v3.2-LOCKED; 659 lines; absorbed 25+ flags across 4 review rounds).

**Substrate state**:
- v2.9.1 SHIPPED on `main` @ commit ~`224cc85`
- RFC v3.2 + GC component spec on disk; uncommitted due to git wedge (deferred; non-blocking for planning)
- No active branches off main
- All 16 persona contracts at current versions; ready for `state_interface` field addition

**Total v3.0 effort**: ~1050 LoC across 4 phases; ~90-130h wallclock including pair-reviews + dream prompt iteration + fixture testing.

## Plan-level conventions

### Route-decide policy

Per `~/.claude/rules/toolkit/core/workflow.md` §"Route-Decision for Non-Trivial Tasks":

| Work classification | Route policy |
|---|---|
| Verification probes (P1-P5) | **ROOT-direct** — mechanical bash/curl/file ops; convergence_value < 0.10 |
| Schema design + persona-contract authoring | **HETS-routed** — substrate-fundament; convergence_value ≥ 0.30 |
| Implementation of new substrate primitives (~JS files in `scripts/`) | **HETS-routed** — pair architect + node-backend; convergence_value ≥ 0.20 |
| Mechanical content (CHANGELOG, README, version bumps) | **ROOT-direct** — convention is settled |
| Dream-prompt design | **HETS-routed MANDATORY** — option-axis (per-cycle prompts) + institutional encoding |
| Fixture tests | **ROOT-direct** — deterministic specs from RFC §Validation criteria |

**Rule of thumb**: if the architect would produce a meaningful disagreement, HETS-route. If it's mechanical execution of a spec, ROOT-direct.

### Pair-review policy

Per HT.1.6 trigger rubric (5 of 5 fires for the overall v3.0; per-component varies):

| Component | Pair-review? | Cadence |
|---|---|---|
| Spawn-state-namespace infra | YES (substrate fundament) | architect + code-reviewer in parallel before implementation |
| Delta capture mechanism | YES (security surface: snapshot scope) | architect + security-auditor |
| Dream-lite (per cycle) | YES (load-bearing prompt design) | architect + honesty-auditor per cycle |
| GC machinery | YES (operational hygiene; failure modes) | architect + code-reviewer |
| Interface contract per persona | NO (mechanical extension after schema locked) | code-reviewer post-implementation |
| Recall CLI verbs | NO (deterministic theorems; trivial to verify) | code-reviewer post-implementation |
| Theorem memo cache | YES (hash-spec discipline is load-bearing) | architect + code-reviewer |
| ADR-0008 (MAJOR-bump criterion) | YES (governance) | architect + planner |

### Spawn convention (canonical per `kb:hets/spawn-conventions`)

Every persona spawn writes its bounded output to:
```
swarm/run-state/v3.0-phase{N}-{component-id}/node-actor-{persona}-{identity-name}.md
```

Bounded output format:
- YAML frontmatter (persona, identity, run_id, phase, component, status)
- Findings section (severity-laddered: CRITICAL / HIGH / MEDIUM / LOW)
- Skills used + KB references resolved (per H.9.20 KB-citation discipline)
- 800-2000 words

### Acceptance gate convention

Each phase has a binary GATE: all acceptance criteria green → phase closes → next phase opens. Phases serialize (no parallelism between phases; intra-phase components can parallelize per dependency graph).

### PR cadence (critical lesson from DRIFT-test3-020 — 3rd occurrence in test3 series)

**NO stacked PRs.** Each phase opens ONE branch; squash-merges to main when phase closes. Sub-components within a phase are commits on that branch, not separate PRs.
