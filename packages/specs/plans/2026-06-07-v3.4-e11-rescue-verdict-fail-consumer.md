# E11-rescue — verdict-fail producer + persona-selection consumer

## Goal

Un-strand the W4 denial-rate circuit-breaker (`packages/lab/circuit-breaker/`). W4 shipped it
projecting over the **E1 negative-attestation** store (decompose-rejects) — which is **starved** (the
decompose tier is probe-dead for shipped personas: depth-1 / general-persona, per the W1 spike). This
wave gives E11 a **live producer** + a **real consumer** instead of retiring it:

1. **Producer** — generalize E11's input to a pluggable **denial-source**, default it to the **W6
   verdict-`fail`** stream (a LIVE, evidence-linked producer), retain E1 as an opt-in source.
2. **Consumer** — wire the **orchestrator's persona-selection step** to consult the breaker before a
   delegated builder spawn: `check --persona P` -> `tripped` -> reroute / halt / surface. Advisory,
   narrows-only, never a kernel gate.

Net arc: `W6 verdict producer -> E4 (advise) + E11 (halt) -> persona-selection consumer`. Still
SHADOW/advisory (it halts nothing in the kernel). The honest win is "**E11 gets a real producer + a real
consumer, end-to-end wired + fixture-tested**" — NOT "E11 defends production" (see Honest Scope below).

   orchestrator CLI + convention (0 kernel-hook refs; stays that way).

## Design

### D1 — pluggable denial-source (generalize the input; re-aim, NOT union)

`project.js` replaces the single hard `negStore` coupling with a small **denial-source registry**. A
source = `{ id, list(nowMs) -> [{persona, recorded_at}] }`; each normalizes its own store's records into
denial-events. The sliding-window / per-persona / global math is UNCHANGED and source-agnostic.

- `verdict-fail` (DEFAULT): `verdictStore.listVerdicts({now, filter: r => r.verdict === 'fail'})` ->
  `{persona: r.subject.persona, recorded_at: r.recorded_at}`.
- `negative-attestation` (opt-in, the W4 behavior preserved): `negStore.listAttestations({now})` ->
  `{persona: r.identity.subagent_type, recorded_at}`.
- Selection: `LOOM_BREAKER_SOURCE` env -> `'verdict-fail'` (default) | `'negative-attestation'`. An
  unknown / empty value **fails safe to the default** (the live producer), mirroring the W4 clamp
  discipline. `'both'` (union) is a **non-goal** (see below — semantic-conflation risk + YAGNI).

**Implementation invariants (verify-plan code-reviewer CR-1/CR-2 + architect A-F2 — folded):**
- **Unconditional requires (CR-2, the ENV-BEFORE-REQUIRE trap).** BOTH stores are `require`d
  UNCONDITIONALLY at module-top of `project.js`. The registry selects between the **already-loaded**
  modules at call-time (env read at call-time, like `windowMs()`). Do NOT lazy/conditionally `require`
  on the env — each store resolves its `LAB_STATE_BASE` at module-load (`verdict store.js:49`,
  `neg store.js:45`), so a gated require would break ENV-BEFORE-REQUIRE for the non-default source.
- **Defensive persona fallback (CR-1).** `personaOfVerdict` + `personaOfNeg` BOTH apply the existing
  `|| 'unknown'` guard (the current `personaOf`, `project.js:73-75`) so a hand-corrupted ledger line
  (parsed JSON, malformed shape) cannot poison the per-persona `Map` accumulator.
- **Tamper-counters carry transparently (A-F2).** `excluded_future` / `excluded_undated` (the
  security-relevant counters at `project.js:119-120`) live in `projectBreaker`'s loop, NOT in the
  source. A source returns only `{persona, recorded_at}` pairs, so future-/undated-record exclusion
  stays in the projection — still load-bearing for the verdict source (the verdict store has no
  future-date guard of its own).
- **`projectBreaker` stays < 50 lines.** The source-`list(nowMs)` call REPLACES the inline
  `negStore.listAttestations` (net ~0 growth); the registry + accessors are module-level helpers.

### D2 — only `verdict === 'fail'` is a denial

`pass` / `partial` do NOT count. A circuit-breaker halts on FAILURE; `partial` is a qualified pass.
Counting it would over-trip (halt personas that mostly succeed). (A future `partial`-at-0.5-weight knob
is conceivable — YAGNI now; mirrors the trust formula's partial-credit discussion.)

### D3 — consumer = CLI + documented convention (A3b advisory), not a code gate / kernel hook / route-decide

The orchestrator consults `check --persona P` at spawn-selection and narrows its own choice. Documented
in the agent-team patterns (the discoverable, on-demand surface) + the activation ledger. NOT promoted to
the always-on persona-selection rule this wave (ADR-0005 context-tax discipline — deferred; see non-goals).

### D4 — stateless-windowed stays (half-open still deferred)

The breaker auto-resets optimistically as fails age out of the window (W4 "stateless windowed", not a
3-state breaker). For an ADVISORY orchestrator consumer (re-queried per spawn; no long-running process to
flap), this is correct. The W4 note "revisit half-open before E11 gates LIVE/kernel work" STILL STANDS —
this wave does not make it gate kernel work, so half-open stays deferred.

### D5 — safety rests on narrowing, not on the evidence-link

The verdict-fail source is evidence-linked anyway, but (per W4) the breaker's §0a.3.1-safety comes from
"a halt only narrows", independent of the source. So no INV-W1 trace gate is added (it would be vacuous).
The evidence-link is enforced at the store WRITE boundary (recordVerdict requires agentId); the breaker
does NOT re-verify a hand-written ledger line at read (VALIDATE hacker M2) — acceptable precisely because
a halt only narrows (an injected `fail` over-halts a persona, never grants). A read-path evidence filter
is a noted deferred defense-in-depth (it would only stop accidental/malformed records — a tamperer with
ledger-write access can forge an agentId — so it adds consistency, not a security boundary).

### D6 — count fail-VERDICT records, not distinct failed-builds (build-emergent; KISS)

A `fail` verdict = ONE reviewer's verdict; a 3-lens review of one bad build emits up to 3 `fail` records
(one per reviewer). The breaker counts fail-VERDICT records (source-agnostic, matching E1's record-count
+ W4's "count records" math UNCHANGED), so a multi-reviewer fail of a single build inflates the count.
This is a **disclosed characteristic**, not a bug — `denials_in_window` honestly means "fail-verdict
records in window" (a caution signal), the breaker is advisory + thresholds tunable, and **dedup-by-
subject-spawn** (`evidence_refs.agent_id`) is a backlog refinement (YAGNI now; would thicken the thin
normalizer). A code comment in the verdict-fail normalizer states this.

## Non-goals (explicitly deferred, not dropped)

- **`both` / multi-source union** — E1 (decompose-rejects) and verdict-fail (build-review-fails) are
  DIFFERENT denial semantics; unioning them under one threshold conflates two halt signals. The
  abstraction MAKES it trivial later (one `.concat`), but YAGNI + conflation-risk -> deferred.
- **Promoting the convention to the always-on persona-selection rule** (`rules/core/workflow.md`) —
  ADR-0005 context-tax; keep it in the on-demand skill-doc + ledger this wave. Promote only if the
  pattern recurs / the user wants it always-on.
- **Threshold retune for the verdict stream** — the W4 defaults (5 persona / 10 global per 10-min window)
  are reasonable + the clamp ceilings are load-bearing safety; retune once real volume exists. Backlog.
- **Half-open / hysteresis** (D4) — deferred until E11 ever gates LIVE/kernel work (not this wave).
- **A kernel hook consuming E11** — out of scope; that would require A6-snapshot mediation (a separate
  high-risk phase, like `LOOM_RESOLVER_ENFORCE`).
- **High-volume capture from arbitrary user repos** — the producer is dogfood/self-measurement volume
  (Axiom 5); user-repo capture needs a shipped-surface convention + a capture hook (later follow-on).

## §0a.3.1 / coherence

The breaker is a DERIVED VIEW over an evidence-linked producer that only NARROWS (halts) -> monotonically
safe (v6:173), exactly as W4 established. No widening, no trust-grant, no kernel state-transition. The
verdict-fail source's evidence-link (`agentId`) is enforced UPSTREAM by the store; the breaker inherits a
clean input. E4 stays advisory/display-only; A6 unaffected (the consumer is orchestrator-level, not a
kernel read). Coherent with the v6 §3.6 / §0a.3.1 boundary.
