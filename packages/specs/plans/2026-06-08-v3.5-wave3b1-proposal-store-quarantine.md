## What 3b.1 is — the manage-write-layer's safest closed loop

The Wave 3b scope chose a **dedicated advisory Lab proposal store** (D1, NOT a kernel PENDING record — falsified on 3 probed axes). 3b.1 builds that store + the **safest op (`quarantine`)** end-to-end: producer (`quarantineRecord`) -> store -> consumer (`quarantinedRecords` projection) + the human-disposition primitive + a CLI loop. A COMPLETE closed loop in shadow, mirroring 3a (which shipped flag-conflict + conflicted + the loop + a CLI in one wave).

`quarantine` is the safest first op because the RFC places it **OUTSIDE R2** — "retrieval-suppression, not deletion; the record stays WAL-retained; deliberately Lab-advisory, NOT a kernel write-path op." It is the least destructive (no SUPERSEDE/TOMBSTONE, no `operation_class`), the closest analog to 3a's non-destructive `flag-conflict` CREATE.

## Disambiguation (load-bearing — firsthand-probed) — two unrelated "quarantine"s

The kernel ALREADY has `packages/kernel/_lib/quarantine-promote.js` — the **PR-3c spawn-delta materialization** library (materialize a spawn's `delta_sha`, build genesis/spawn records, cherry-pick into a K9 STAGING worktree). That "quarantine" is the **spawn-state delta-staging** concern (K-layer integration). **Wave 3b.1's `quarantine` is a different, unrelated concept** — a Memory **Manage-Layer** op that marks a memory RECORD as quarantined (retrieval-suppressed). They collide only on the English word; the path (`lab/manage-proposal/` vs `kernel/_lib/quarantine-promote.js`) + the `node_type:'manage-proposal'` namespace them. Every 3b.1 file header MUST state this disambiguation so a future reader does not conflate them. No functional overlap; no shared code.

### Consolidated fold (the corrected build contract — the build follows THIS)

**FAIL-1 (empty array):** `validateCreateProposalInput` rejects a non-array OR `target_records.length===0` with a clean error BEFORE the element loop (a vacuous `[].every(HEX64.test)` returns `true` — the trap). **FAIL-2 (non-string coercion):** per-element check is `typeof el==='string' && HEX64.test(el)` — NOT a bare `HEX64.test(el)` (which coerces: `HEX64.test({toString:()=>'a'.repeat(64)})` -> `true`). The `typeof` guard precedes the regex (the `validateFreeString` discipline).

**FAIL-3 (`quarantineRecord` target presence):** the wrapper owns a THIRD guard — presence-check `target` (`== null` -> `quarantineRecord: target is required (a 64-hex kernel transaction id)`) BEFORE `[target]` wrapping; the HEX64 FORMAT validation delegates to the store.
