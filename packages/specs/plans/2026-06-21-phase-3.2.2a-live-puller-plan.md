# ③.2.2a — The read-only `good-first-issue` live puller + repo-eligibility filter

> Phase ③.2 (live external-PR beta) · sub-wave ③.2.2a · trust-ZERO · emits nothing · read-only.
> Plan-of-record for the FIRST half of ③.2.2. The DRAFT-on-real-repos solve/grade loop is ③.2.2b (separate wave).

## Why this wave (and the premise correction)

The pre-compact memory framed ③.2.2 as "wire Docker as the grading tier first." **Firsthand recon corrected that**:
the Docker grade tier is already SATISFIED (charter map: `Docker/gVisor ContainerAdapter — SATISFIED (③.0 #346;
docker-backend.js)`; `earned-grounding-run.js` already creates+attests a Docker backend and grades through it — mechanism wired; grading
efficacy is a separate prior residual, not this wave's claim). The
real net-new work of ③.2.2 is **the live puller** — the bridge from the sealed 18-issue corpus to live, uncurated,
real-world inputs. Per the ③.2 plan's own ③.2.2 section: *"a read-only pull of `good-first-issue?l=python`, filtered
to repos that accept unsolicited PRs + are PR-capable + license-compatible … replaces the sealed corpus as the input."*

This wave builds ONLY the read-only puller + filter. It does **not** clone, solve, grade, or emit. That keeps it a
small, reviewable, side-effect-light PR and isolates the stranger-code-execution risk to ③.2.2b.

## Scope (EC2.1 only)

Map to the ③.2 plan exit criterion **EC2.1**: *"the puller returns a live, filtered issue set (unsolicited-PR-OK +
license-compatible); a test asserts the filter rejects assignment-required + non-PR-capable repos."*

**IN scope**
- A new lab module `packages/lab/issue-corpus/live-puller.js` that, read-only, runs a **per-item validate-and-DROP**
  loop (VERIFY H3 — never batch-validate; one crafted item drops only itself):
  1. SEARCHES GitHub for `label:"good first issue" language:python state:open type:issue` (injectable gh runner).
  2. PARSES + GUARDS the `owner/repo` slug from each item's `repository_url` with a puller-owned slug guard
     `assertSafeOwnerRepo` (VERIFY H1 CRITICAL — `assertSafeRepo` is a full-URL guard that does NOT cover a bare
     slug, and the enrichment call below runs on this slug): `^[A-Za-z0-9_.-]{1,39}\/[A-Za-z0-9_.-]{1,100}$`, reject a
     leading `-` in either segment and any `.`/`..`-only segment. A failing slug DROPS the item.
  3. ENRICHES via `gh api repos/{owner}/{repo}` — the endpoint is a SINGLE positional (BUILD-probed: `gh api`
     rejects `--` before the endpoint, so "separate positionals after `--`" is not how `gh api` works), re-asserted
     to a fixed `repos/<owner>/<repo>` shape and passed as ONE argv element via `execFileSync` (no shell/word-split);
     the slug guard already excludes `?`/`..`/`-`-lead/whitespace/control, so the interpolated endpoint cannot
     traverse or arg-inject → metadata (`license.spdx_id`, `archived`, `disabled`, `default_branch`, `allow_forking`,
     `is_template`, `assignees`).
  4. RESOLVES `base_sha` = the default-branch HEAD `.sha`, run through `assertSafeSha`; a missing/non-string/non-40-hex
     `.sha` DROPS the item (VERIFY M2 — the gh `.sha` field is untrusted data, never coerced).
  5. FILTERS to eligible repos: license-compatible allowlist + PR-capable + **unassigned** (VERIFY F4 — reject an
     already-assigned issue: the unsolicited-PR signal) + accepts-unsolicited-PRs heuristic.
  6. CONSTRUCTS the canonical `repo` = `https://github.com/{owner}/{repo}` from the guarded slug and round-trips it
     through `assertSafeRepo` (VERIFY F2 — pin the full-URL form the 3.2.2b consumer requires; never trust the raw
     `repository_url`/`html_url`).
  7. BOUNDS the `problem_statement` (title + body): truncate to a hard 64 KiB cap and strip C0 control chars except
     `\t`/`\n` (VERIFY M1 — no reused module bounds it; it rides into a `claude -p` prompt in 3.2.2b).
  8. MAPS each survivor to the public record `{id, repo, base_sha, problem_statement}` and validates it with a NEW
     `validatePublicRecord` (added to `corpus.js`, open/closed) — NOT `validateIssueCorpus` (VERIFY F1/H2/HIGH-1:
     that is a FULL raw-record validator and hard-throws on a public-only record; there is no public-only validator
     today). `validatePublicRecord` checks the four `PUBLIC_FIELDS` are non-empty strings + `base_sha` is 40-hex,
     reusing `PUBLIC_FIELDS` + the existing sha regex.
- A pure/IO split (the slug guard, filter predicates, record-mapping, problem_statement bound are pure & unit-testable
  WITHOUT network; the `gh` calls are an injectable seam) — mirrors `docker-backend.js`'s pure-vs-IO split.
- A small open/closed addition to `corpus.js`: export `validatePublicRecord` alongside the existing validators
  (does NOT touch `validateOne`/`validateIssueCorpus` — the SEALED boundary is untouched).
- Unit tests `tests/unit/lab/issue-corpus/live-puller.test.js` (+ a `validatePublicRecord` case in the corpus suite).

**Forward contract to ③.2.2b (VERIFY F3 — stated, not implied):** the public record drives the **clone+actor** half
(`repo`/`base_sha`/`problem_statement`) but carries **NO sealed oracle** (`fail_to_pass`/`pass_to_pass`/`test_patch`/

**OUT of scope (deferred to ③.2.2b or later)**

- Any scheduled/cron `routine` invocation (the puller is a callable module this wave; scheduling is ③.2.2b+).

## Security & correctness model (this is an untrusted-input boundary)

The GitHub search/metadata response is **untrusted DATA** — anyone can open an issue with a crafted title/body and
name a repo arbitrarily, so the WHOLE input set is attacker-influenceable. Discipline:
- **Per-item validate-and-DROP** (VERIFY H3, load-bearing): each candidate's full validation chain (slug guard →
  enrich → sha guard → filter → URL round-trip → public-shape check) is wrapped per-item; a throw DROPS that item and
  the loop continues. NEVER batch-validate (the batch `validateIssueCorpus` throws on the first bad record, so one
  crafted item would zero the entire pull — a DoS).
- **The owner/repo slug is the CRITICAL net-new surface** (VERIFY H1): it flows into the puller's OWN
  `gh api repos/{owner}/{repo}` enrichment call, and `assertSafeRepo` does NOT cover a bare slug. `assertSafeOwnerRepo`
  (puller-owned) guards it (`^[A-Za-z0-9_.-]{1,39}\/[A-Za-z0-9_.-]{1,100}$`, no leading `-`, no `.`/`..` segment); the
  endpoint is then a single `repos/<owner>/<repo>` string, re-asserted against `ENDPOINT_RE` and passed as ONE argv
  element via `execFileSync` (no shell/word-split). (BUILD-probed: `gh api` rejects `--` before the endpoint, so the
  original "separate positionals after `--`" wording was falsified — the guarded single positional is the control.)
  Read-only does NOT mean injection-free.
- `base_sha` → `assertSafeSha` (reuse); a missing/non-string/non-40-hex `.sha` DROPS the item, never coerced/fallback.
- `repo` is CONSTRUCTED as `https://github.com/{owner}/{repo}` from the guarded slug and round-tripped through
  `assertSafeRepo` (reuse) — never trust the raw `repository_url`/`html_url`.
- `problem_statement` (title + body) is untrusted text: truncated to a hard **64 KiB** cap + C0 control chars stripped
  (except `\t`/`\n`), stored as DATA, never eval'd/executed. The 3.2.2b fence-as-DATA is the SECOND layer, not a
  license to store unbounded text here.
- License gate = an SPDX **allowlist** (default permissive: `MIT, Apache-2.0, BSD-2-Clause, BSD-3-Clause, ISC, 0BSD,
  Unlicense`), exact-set membership — NOT a substring/`.includes` test (which would mis-accept `Apache-2.0 AND GPL-3.0`
  or `MIT-but-actually-GPL`). A `null`/`NOASSERTION`/unknown license → INELIGIBLE (fail-closed: absence of a
  known-permissive license is a reject, never a default-allow).
- PR-capable = `archived === false && disabled === false && allow_forking === true && is_template === false`; and the
  issue is UNASSIGNED (`assignees` empty — VERIFY F4, the unsolicited-PR signal).
- Accepts-unsolicited-PRs = a conservative, DOCUMENTED heuristic this wave (not a guarantee): the repo is
  PR-capable + permissively licensed + the issue is unassigned. The human reviews the pulled set before ③.2.2b ever
  clones — the puller does not assert a repo WANTS the PR, only that nothing structurally forbids one.
- No mutation: pure functions return fresh objects; no in-place edits of the gh response.
- No secrets in source (ADR-0006: zero eslint-disable; ASCII-only).

**Explicitly NOT claimed**: this wave does not establish that any pulled issue is solvable, that a repo welcomes the
PR, or any efficacy. It returns an *eligible candidate set*; trust stays ZERO.

- **EC2.1c** — the gh layer is an injectable seam: the full unit suite runs with NO network (a mock runner), proven
  by a test that supplies canned responses and never shells out.
- **EC2.1d** — the module contains no egress call-site and statically imports no egress module; **a unit test

  NOT a drop case — `boundProblemStatement` TRUNCATES it and the item survives; covered by EC2.1h, not here.)
- **EC2.1g** — **slug-injection rejected** (VERIFY H1): `assertSafeOwnerRepo` rejects a slug containing `/` beyond the

  `repos/<owner>/<repo>` positional, re-asserted to a fixed shape, and passed as one argv element (no shell) — a

- **EC2.1h** — **regression fences** (VERIFY N1, held surfaces): the license gate is exact-set and rejects

## Post-PR folds (CI + CodeRabbit on #390)

Two issues surfaced AFTER the VALIDATE board — exactly the cross-PR / integration class the per-wave board
structurally cannot see (it reviewed `live-puller.js` in isolation; `emit-pr.test.js` was not in its file set):

- **CI FAIL — the egress sole-chokepoint lint (③.2.1b EC1b.2a) flagged the puller** (`emit-pr.test.js`): the lint
  asserted "no production module outside `emit-pr.js` spawns `gh`", and the read-only puller spawns `gh api`. This is
  a genuine integration seam: ③.2.2a introduces the FIRST legitimate non-egress `gh` consumer. **Resolved by refining
  the invariant** from "sole `gh`-spawner" to "sole **WRITE-egress** `gh`-spawner + sole token-reader": a named
  read-only consumer (`live-puller.js`) is exempt from the `gh`-SPAWN cap, **conditional on it proving GET-only via a
  POSITIVE runtime gate**, and still barred from `git push` + token reads. **This is a security-gate refinement —
  flagged to the USER for ratification.**

  A focused **hacker re-probe** of the refinement (Rule 2a, since it touches the egress chokepoint) returned
  **NO-HOLE today** (exact-path exemption; current puller clean; token-custody control EC1b.2b intact) but a **HIGH
  latent gap**: my first fix ENUMERATED gh write-verbs (`-X|--method POST...`), which is the
  `syntactic-gate-extension` anti-pattern — gh's write surface is effectively unbounded (it AUTO-POSTs on `-f`/`-F`

  misses forms, and the puller runs on **write-capable ambient auth**. **Redesigned to the bounded positive
