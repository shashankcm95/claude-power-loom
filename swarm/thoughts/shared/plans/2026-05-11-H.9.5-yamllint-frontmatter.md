# H.9.5 — yaml-lint on substrate frontmatter

## Context

Per user reframe of v2.0 readiness: clean-phase counter tracks "no new institutional commitments to soak", NOT "substantive coverage gaps closed". H.9.5 attacks the latter — closes a real coverage gap in the substrate's testing framework: substrate frontmatter is **not strict YAML 1.2 valid**, yet substrate's permissive `parseFrontmatter` silently tolerates the divergence.

### The coverage gap

H.9.0 + H.9.1 + H.9.2 added local-verification gates for markdown content + bash scripts + JSON syntax. Frontmatter (YAML embedded in markdown) was not gated. Substrate's runtime parser is `parseFrontmatter` — a permissive YAML subset that handles substrate's specific usage (scalars, inline arrays, block lists, inline `#` comments per HT.2.2). It doesn't validate strict YAML 1.2 conformance.

Practical implication: any future tooling expecting standard YAML (yamllint, python yaml, js-yaml, pyyaml, jq -y, GitHub Actions YAML linting, etc.) would fail on substrate frontmatter. The substrate is effectively coupled to its own internal parser — a "yaml-ish" convention rather than strict YAML.

### What yaml-lint baseline surfaced

`npx --yes yaml-lint` against extracted frontmatter blocks (130 substrate `.md` files with `---` markers) surfaced **12 failing files** at H.9.5 start:

- `swarm/thoughts/shared/HT-state.md`
- 11 `swarm/thoughts/shared/plans/*.md` plan files (HT.1.10, HT.1.12, HT.1.13, HT.2.3, HT.3.2, HT.3.3, H.9.0, H.9.1, H.9.2, H.9.3, H.9.4)

Two distinct YAML-invalidity patterns:

1. **Narrative scalar values with embedded colons**: substrate's ledger writing convention puts long prose-with-colons inline as YAML scalar values (e.g., `last_session_phase: H.9.4 SHIPPED on ... phase: (a) ...`). YAML parsers see `:` as key-value separator; cascading parse failures.

2. **Duplicate mapping keys** in HT-state.md: substrate's stack convention (each phase ship demotes prior `last_session_phase` to a new `last_session_phase_prior:` entry) produces 8 duplicate keys. YAML disallows duplicate keys at any single mapping level.

## Implementation strategy

### Phase 1 — `parseFrontmatter` quote-handling validated (no parser change needed)

Verified at impl line 142 (`val.replace(/^["']|["']$/g, '')`): substrate's `parseFrontmatter` already strips outer double-quotes from scalar values. Wrapping narrative values in `"..."` produces no runtime behavior regression. Block list shape (`key:\n  - item1\n  - item2`) is already handled per HT.2.2 impl lines 104-114.

No parser enhancement needed.

### Phase 2 — Migration: narrative scalar value wrapping

Migration script applied (`scripts/h95-fix-frontmatter.js`, deleted post-ship):

- Walk 12 failing files
- Per line, identify `key: value` or block-list item `  - value` patterns
- Skip already-YAML-valid forms (empty, inline array, already-quoted, numeric, boolean, null, timestamp, git hash)
- For narrative scalars + narrative block-list items:
  - Replace internal `"..."` with `` `...` `` (semantic-preserving for substrate inline-code convention)
  - Escape literal backslashes (rare)
  - Wrap entire value in double quotes

Migration applied: 223 entries wrapped (199 scalar values + 24 block-list items) across 12 files.

### Phase 3 — Migration: HT-state.md duplicate-key consolidation

Migration script applied (`scripts/h95-fix-htstate-dupkeys.js`, deleted post-ship):

- 8 duplicate `last_session_phase_prior:` keys in HT-state.md
- Consolidate into a single block-list under `last_session_phase_priors:` (plural; YAML-valid single-key form)
- Position immediately after `last_session_phase:` (current state)
- Preserves substrate's stack convention semantically (each prior entry preserved as separate list item)

**No programmatic consumers** read `last_session_phase_prior` (verified via grep on `scripts/` and `hooks/`). Renaming + reshape is safe.

### Phase 4 — Add Test 83 to `tests/smoke-ht.sh`

Insert between Test 82 (H.9.2 jq empty) and Test 65 (H.8.7 trailer). Test 83 invocation:

```bash
T83_TMPDIR=$(mktemp -d)
T83_COUNT=0
while IFS= read -r T83_FILE; do
  if head -1 "$T83_FILE" 2>/dev/null | grep -q "^---$"; then
    T83_COUNT=$((T83_COUNT + 1))
    awk 'BEGIN{state=0} /^---$/ { if(state==0){state=1;next}; if(state==1){state=2;exit} } state==1' "$T83_FILE" > "$T83_TMPDIR/fm-$T83_COUNT.yaml"
  fi
done < <(find "$SCRIPT_DIR" -name "*.md" -not -path "*/node_modules/*" -not -path "*/.git/*" -not -path "*/swarm/run-state/*")
T83_OUT=$(npx --yes yaml-lint "$T83_TMPDIR"/fm-*.yaml 2>&1)
T83_EXIT=$?
rm -rf "$T83_TMPDIR"
```

`find` enumerates all `.md` (excluding `node_modules`, `.git`, `swarm/run-state` chaos artifacts). `awk` extracts frontmatter block between first `---` markers. yaml-lint validates each extracted block. Exit code 0 → substrate frontmatter is strict YAML 1.2 valid.

## Out of scope (deferred)

- **CI parity for yaml-lint**: H.9.5 establishes local-layer gate only. Adding `yaml-lint` job to `.github/workflows/ci.yml` is sibling-track work; drift-note candidate
- **Convention codification**: the "wrap narrative scalar values in double quotes; use backticks for internal inline-code" convention is established by H.9.5 in practice but not yet codified as a substrate authoring convention doc (could be a future H.9.x or v2.x phase if drift recurs)
- **Stylistic yaml-lint rules**: yaml-lint only does syntax validation (no per-rule config); upgrading to `yamllint` (Python) would add stylistic rules (indentation, line length) but those would surface noise without safety value
- **Frontmatter schema validation**: yaml-lint validates YAML syntax; schema validation (e.g., persona contract schema) is `contracts-validate.js` scope
- **`parseFrontmatter` quote-escape handling**: H.9.5 doesn't need enhanced escape handling because the migration uses semantic-preserving substitution (internal `"` → `` ` ``). Future drift requiring inner double quotes would need parser enhancement; deferred
