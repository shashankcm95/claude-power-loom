### Why JSON syntax check (not full JSON lint)

For JSON, the lint dimensions are:
1. **Syntax** — parseable by `JSON.parse()` / `jq empty` (mechanical correctness)
2. **Style** — indentation, trailing commas, line length (cosmetic)
3. **Schema** — conformance to a declared schema (semantic correctness)

H.9.2 gates dimension 1 only. Rationale:

- **Style**: not load-bearing for JSON (which has rigid syntax). Adding `prettier --check` would surface mostly indentation noise; not a clean preventive gate.
- **Schema**: already covered by `contracts-validate.js` for persona contracts + plugin manifest schemas. Adding generic schema validation at this layer is redundant.
- **Syntax**: the minimum-viable preventive gate. Catches future JSON drift (typo, malformed bracket, trailing comma without parser tolerance) at local-verification layer before CI.

This matches the H.9.0 + H.9.1 pattern: gate the most-fundamental layer (markdownlint at error-severity per `.markdownlint.json` lenient defaults; shellcheck at `--severity=error`; JSON at parse-correctness). Stricter dimensions are separate substantive phases.

### Why jq (not Node-inline / npx jsonlint)

Tool choice rationale:

| Tool | Pros | Cons | Pick? |
|------|------|------|-------|
| **`jq empty FILE`** | Pre-installed on macOS + Ubuntu CI; 1-line pipeline shape; fast compiled binary; standard Unix util | System dependency (though widely available) | ✓ chosen |
| `node -e '...'` inline walker | No system deps (Node already required for substrate) | Multi-line inline script via `node -e` is bash-quoting fragile; substrate-internal but verbose | rejected |
| `npx jsonlint FILE` | Pattern-consistent with H.9.0 (`npx markdownlint-cli2`) + H.9.1 (`npx shellcheck`) | Slower per-invocation (npm fetch); overkill for syntax-only check | rejected |
| Standalone `scripts/json-syntax-check.js` | Maintainable; reusable; substrate-controlled | Adds new substrate file; larger surface than H.9.0/H.9.1's tool-call shape | rejected |

`jq` is the most pragmatic choice given the substrate already has it on every relevant platform. The CI parity note: GitHub Actions ubuntu-latest images include jq pre-installed.

## Implementation strategy

### Edit 1 — Add Test 82 to `tests/smoke-ht.sh`

Insert between Test 81 (H.9.1 shellcheck) and Test 65 (H.8.7 trailer). Test 82 invocation:

```bash
echo -n "  Test 82 (H.9.2 JSON syntax in local smoke harness; jq empty against substrate .json files): "
T82_OUT=$(cd "$SCRIPT_DIR" && find . -name "*.json" -not -path "./node_modules/*" -not -path "./.git/*" -not -path "./swarm/run-state/*" -print0 | xargs -0 -n1 jq empty 2>&1)
T82_EXIT=$?
T82_COUNT=$(cd "$SCRIPT_DIR" && find . -name "*.json" -not -path "./node_modules/*" -not -path "./.git/*" -not -path "./swarm/run-state/*" | wc -l | tr -d ' ')
if [ $T82_EXIT -eq 0 ]; then
  echo "OK (substrate JSON syntax clean; $T82_COUNT files checked, 0 errors)"
  passed=$((passed + 1))
else
  echo "FAIL: jq reported JSON syntax errors"
  echo "$T82_OUT" | tail -5
  failed=$((failed + 1))
fi
```

`find` enumerates `.json` files with 3 exclusions (node_modules, .git, swarm/run-state). `xargs -0 -n1 jq empty` parses each file individually; `jq empty` returns exit 0 if file is valid JSON, non-zero otherwise. xargs continues across all files; final exit code is non-zero if ANY file fails. `T82_COUNT` is computed via a parallel `find | wc -l` to report file count in success message.
