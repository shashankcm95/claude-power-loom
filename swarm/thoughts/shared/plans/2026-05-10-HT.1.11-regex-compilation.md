## Empirical pre-validation — per-site classification

Verified live (read 2026-05-10):

### Tier 1 — STATIC pattern (1 site; clear win)

| Site | Current | Migration |
|------|---------|-----------|
| `scripts/agent-team/verify-plan-spawn.js:84` | `const re = new RegExp(\`\\n## Pre-Approval Verification[\\s\\S]*?(?=\\n## \|$)\`, 'g');` per `appendSection` call | Promote to module-top regex literal: `const PRE_APPROVAL_RE = /\n## Pre-Approval Verification[\s\S]*?(?=\n## \|$)/g;` |

**Why migrate**: pattern is fully static (no template variables); compile-once at module load is strictly better than compile-per-call. Standard regex literal syntax is more idiomatic than `new RegExp` for static patterns.

### Tier 2 — HOT site with bounded dynamic keyspace (1 site; biggest win)

| Site | Current | Migration |
|------|---------|-----------|
| `scripts/agent-team/route-decide.js:292-295 buildKeywordRegex` | Recompiles per keyword per call; called ~90× per `scoreTask` invocation (9 dims × ~10 keywords) | Memoize by keyword key in module-scope `Map<string, RegExp>`; cache miss compiles + stores; cache hit returns cached regex |

**Why migrate**: highest call frequency (~90× per scoreTask). Keyspace is bounded (~100 unique keywords across all dims + SUBSTRATE_META_TOKENS). Memoization eliminates per-call compile after first invocation.

**Migration shape**:
```js
const _keywordRegexCache = new Map();
function buildKeywordRegex(keyword) {
  if (!_keywordRegexCache.has(keyword)) {
    const escaped = keyword.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
    _keywordRegexCache.set(keyword, new RegExp(`(?:^|[^a-zA-Z0-9_])${escaped}(?=$|[^a-zA-Z0-9_])`, 'i'));
  }
  return _keywordRegexCache.get(keyword);
}
```

### Tier 3 — Per-section dynamic; small keyspace (4 sites; consistency win)

| Site | Current | Migration |
|------|---------|-----------|
| `hooks/scripts/validators/validate-plan-schema.js:108 findSection` | `const re = new RegExp(\`^## \${escaped}(?:\\\\s*\\\\([^)]*\\\\))?\\\\s*$\`, 'm');` per call | Memoize by section name |
| `hooks/scripts/validators/validate-kb-doc.js:60` | `const sectionRe = new RegExp(\`^## \${escapeRegExp(sectionName)}\\\\s*$\`);` per call | Memoize by section name |
| `hooks/scripts/validators/verify-plan-gate.js:76 findSection` | Same shape as validate-plan-schema | Memoize by section name |
| `scripts/agent-team/kb-resolver.js:189 + :192 extractSections` | `const startRe = new RegExp(\`^## \${escapeRegExp(startName)}\\\\s*$\`);` + `endRe = new RegExp(\`^## \${escapeRegExp(endName)}\\\\s*$\`)` per call | Memoize by (startName, endName) keys via shared section-regex cache |

**Why migrate**: each call recompiles; section name is dynamic but bounded (small set of section names like "Summary", "Quick Reference", "Pre-Approval Verification", "HETS Spawn Plan", etc.). Memoization adds consistency across the validator family.

**Migration shape (shared)**:
```js
const _sectionRegexCache = new Map();
function getSectionRegex(name, suffixPattern, flags) {
  const key = `${flags}|${suffixPattern}|${name}`;
  if (!_sectionRegexCache.has(key)) {
    const escaped = escapeRegExp(name);
    _sectionRegexCache.set(key, new RegExp(`^## ${escaped}${suffixPattern}`, flags));
  }
  return _sectionRegexCache.get(key);
}
```

Each site keeps its own cache (no shared module across hooks/_lib boundary unless extraction is justified — per HT.1.8 "extract at 3+ callers" rule, 4 callers IS the threshold, but sub-plan defers extraction to HT.2 to keep HT.1.11 scope mechanical).
