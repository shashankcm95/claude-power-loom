## Slopfiles primitive design (proposed)

### Primitive shape

**`<important if "task involves X">...</important>`** block markers:

```markdown
<important if "task involves git commits, PRs, or branch operations">

## Git Conventions

- Conventional commits: `feat:`, `fix:`, `refactor:`, `docs:`, `test:`, `chore:`
- ...

</important>
```

**Why block markers vs alternatives**:

- **Block markers** (chosen): visible in source; explicit predicate co-located with content; one-shot LLM read of full file with conditional interpretation
- **Frontmatter conditionals** (rejected): `applies_when:` per-file granularity insufficient (workflow.md has 6+ orthogonal conditional sections); per-section conditionality requires section-level metadata syntax that doesn't exist
- **File splitting** (rejected as pure shape): would require Claude Code-side conditional file-loading (no such mechanism today); adopt as augmenting strategy for purely-informational content (move OUT of rules/ entirely)

### Conditionality predicate vocabulary

Predicates use natural-language descriptions (LLM interprets task fit). Recommended vocabulary:

| Predicate | Triggering task patterns |
|-----------|--------------------------|
| "task involves git commits, PRs, or branch operations" | `git commit`, `gh pr create`, `git push`, branch naming |
| "task involves running or writing tests" | `npm test`, test authoring, coverage discussion |
| "task involves code review" | `/code-review`, reviewing diffs, review feedback |
| "task involves deploying or release" | deploy commands, release notes, version bumps |
| "task involves multi-file changes (≥2 distinct files)" | refactors, multi-file features, sweeping changes |
| "task involves invoking /build-team or sub-agent orchestration" | HETS routing, multi-actor spawn |
| "task involves Hardening Track or HETS-routed phase" | per-phase pre-approval gate, plan-vs-research forbidden-phrase grep |
| "task involves CI workflow or install.sh changes" | new GitHub Actions, install scripts, cross-environment validation |
| "task involves markdown authoring" | .md edits, documentation drafts |
| "task involves API/library/spec research" | research questions, citation discipline |
| "task involves user-prompt-vagueness handling" | (rare; mostly automatic via hook layer) |
| "task involves Memory→Rule promotion or skill forge" | self-improve queue actions, /forge invocation |

Predicates are advisory — LLM uses judgment per README "What this toolkit is NOT" (best-effort instruction-following, not deterministic enforcement).

## ADR-0005 shape

**Title**: Substrate slopfiles authoring discipline — `<important if>` conditionality for always-on context surfaces.

**Status**: accepted (per HT.1.7 precedent — pre-approval gate IS the acceptance ceremony; transient `proposed` introduces test-timing-window risk).

**Context**: Always-on context surfaces (e.g., `~/.claude/rules/toolkit/core/*.md` auto-loaded into every session via Claude Code rules layer) accumulate substrate-discipline content over time. HT.0.8 audit + HT.1.13 empirical pre-validation found 322 LoC × ~12 tokens/LoC ≈ 3870 tokens of always-on rules content; ~90% of sessions don't use the conditional sections (per HT.0.8 framing). Without an authoring discipline, every new piece of substrate guidance lands as "one more always-on rule" rather than being scoped to its actual relevance window.

**Decision**: adopt the **slopfiles `<important if>` block-marker convention** as substrate authoring discipline. Always-on context surfaces are partitioned into:

1. **Core-always content** — universal discipline that applies to every session (security guardrails, factual-claim discipline, fundamental coding principles). NOT wrapped.
2. **Conditional content** — task-scoped discipline. Wrapped in `<important if "task involves X">...</important>` block markers with natural-language predicates.
3. **Substrate-meta documentation** (informational; not active discipline) — moved OUT of always-on context entirely, into substrate KB docs.

**Invariants (governance-tier)**:

1. **Safety-critical content invariant** — security guardrails, factual-claim discipline, and any content whose absence creates session-level safety regression MUST remain core-always (NOT wrapped in `<important if>`).
2. **Authoring discipline invariant** — new substrate guidance is presumptively conditional. Adding NEW content to core-always (post-HT.1.13) requires explicit safety-criticality justification + code-reviewer gate.
3. **Predicate-vocabulary invariant** — predicates use natural-language task descriptions. The vocabulary is curated (HT.1.13 establishes 12 starter predicates); extensions go through ADR-update or code-review.

**Consequences**:

- HT.1.13 partitions current `rules/core/*.md` content into core-always + conditional + out-of-rules-migrated.
- Future always-on accumulation surfaces (e.g., per-project rules at `rules/<topic>/`) inherit the discipline.
- LLM-side compliance is best-effort per README "What this toolkit is NOT" (instruction-following, not deterministic). Author-side discipline is fully under our control.
- Migration path: if Claude Code adds native `<important if>` parsing, the substrate is already prepared.

**Files affected**: `rules/core/fundamentals.md`, `rules/core/prompt-enrichment.md`, `rules/core/research-mode.md`, `rules/core/security.md`, `rules/core/self-improvement.md`, `rules/core/workflow.md`, `swarm/adrs/0005-slopfiles-authoring-discipline.md` (NEW), `skills/agent-team/SKILL.md` (ledger entry), `CHANGELOG.md` (HT.1.13 entry), `swarm/thoughts/shared/HT-state.md` (cutover).

**Out of scope (deferred)**:

- Per-project rules (`rules/typescript/`, `rules/web/`) — already project-conditional via Claude Code's rules layer; not auto-loaded into unrelated sessions
- `~/.claude/projects/<project>/memory/MEMORY.md` user-memory layer — different surface; out of scope
- ADR-0001/ADR-0003 fail-open hook discipline alignment with `<important if>` (different surface entirely; hooks are deterministic enforcement, slopfiles are LLM-instruction-following)
- Automated lint enforcement of `<important if>` block-marker syntax — HT.2 sweep candidate
